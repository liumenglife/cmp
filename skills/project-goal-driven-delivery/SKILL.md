# project-goal-driven-delivery

## 何时使用

当任务属于以下任一类型时必须使用：

1. 新功能实现。
2. Bug 修复。
3. 重构。
4. 任何需要明确验证闭环的交付任务。

## 执行步骤

1. 先写出成功标准，而不是直接开始改文件。
2. 将任务拆成最小步骤，每一步都配一个验证动作，并说明该步骤在支撑哪条成功标准。
3. 按任务选择合适的验证维度，先完成最关键的正确性验证，再做补充验证。
4. 如果验证失败，不宣称完成，先回到失败点修复。
5. 最终验证必须逐项覆盖已定义的成功标准，不能只验证部分步骤或只展示局部结果。
6. 交付时同时汇报“改了什么”和“如何证明它达标”。

## 验证模板

以下是可按任务选用的示例维度，不是固定顺序清单：

1. 行为结果验证。
2. 测试或回归验证。
3. 接口或输出验证。
4. 文件或结构验证。
5. 导航或引用验证。

## 完成标准

1. 成功标准已提前定义。
2. 每个主要步骤都有验证记录。
3. 最终验证已逐项覆盖全部成功标准。
4. 最终交付不依赖口头保证，而依赖已执行的检查结果。
