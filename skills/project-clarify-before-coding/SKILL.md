# project-clarify-before-coding

## 何时使用

当出现以下任一情况时，应先澄清，再决定是否实现：

1. 用户需求存在两种及以上合理解释。
2. 继续执行前，必须先显式暴露并确认关键假设。
3. 不同实现路径在范围、复杂度、风险或交付速度上存在明显权衡。
4. 已识别到更小、更简单或更低风险的方案，但是否采用仍需对齐。

## 不适用

以下场景通常不需要启用该 skill：

1. 需求、边界和成功标准已经明确，且不存在关键权衡。
2. 任务是纯信息查询、事实检索或文档阅读，不涉及实现决策。
3. 任务是一次性微小改动，影响范围清晰，且基本不会引入返工或方向性偏差。

## 执行步骤

1. 先把当前任务中可能影响实现结果的关键假设显式列出来。
2. 标出其中哪些假设一旦判断错误，会导致返工、偏题或引入不必要复杂度。
3. 明确列出主要分歧点，以及各自会带来的影响。
4. 给出推荐方向，并说明为什么它更合适、更小或更稳妥。
5. 只提出最少但必要的澄清问题，默认一次只聚焦当前最关键的一个问题，避免一次抛出多个问题。
6. 在边界与成功标准明确前，不进入实现。
7. 澄清完成后，用简短几行总结最终理解，再继续后续工作。

## 不该做的事

1. 不要因为“看起来大概是这个意思”就静默开工。
2. 不要一次性抛出大量低价值问题，把真正关键分歧淹没掉。
3. 不要把个人偏好的实现方案伪装成唯一结论强推给用户。
4. 不要在关键边界尚未明确时，一边实现一边猜。
5. 不要把澄清阶段写成冗长方案文档，而忽略真正需要确认的决策点。

## 完成标准

1. 关键假设已被显式写出并完成确认。
2. 高风险歧义、主要分歧点或关键权衡已完成澄清。
3. 推荐方向及其理由已清楚说明。
4. 后续实现边界和成功标准已足够明确，执行时无需继续依赖隐含假设。
