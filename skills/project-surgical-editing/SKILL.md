# project-surgical-editing

## 何时使用

当任务需要修改已有代码或文档时，应优先使用，尤其包括以下情况：

1. 局部修复或小范围增强。
2. 对现有命名、文案、规则或结构做定点调整。
3. 在既有实现上补充必要逻辑，而不是从零新建一套实现。
4. 更新已有文档、说明或规范条目。

## 执行步骤

1. 先明确当前目标直接要求解决什么，不把相邻问题自动并入本次任务。
2. 确认本次允许修改的文件、段落或代码路径，并识别哪些相邻区域虽然相关但不在本次范围内。
3. 编辑时只做能直接服务当前目标的改动，不顺手调整无关命名、格式、注释或结构。
4. 如果本次改动直接产生了未使用导入、死引用、失效文案或孤儿段落，应一并清理。
5. 如果发现已有问题但与当前目标无直接关系，记录风险或说明边界，不顺手扩展修改范围。

## 检查问题

在结束前逐项自问：

1. 每一处修改都能否直接映射到当前任务目标？
2. 是否夹带了本次并不需要的重构、美化或风格统一？
3. 是否误删、误改了原本存在且与当前目标无关的内容？
4. 清理动作是否只覆盖了由本次修改直接产生的冗余？

## 完成标准

1. 改动范围小且边界清晰，能够解释每一处修改的必要性。
2. 没有与当前目标无关的顺手修改。
3. 只清理了本次变更直接引入的冗余或失效内容。
4. 最终结果解决了当前目标，同时保持既有文件的其他部分不被无关扰动。
