# gotchas

## 2026-04-02

- 需求判断、范围界定、字段命名、页面命名、文档命名，必须以甲方正式招标文件为唯一来源；禁止沿用旧立项报告、讨论稿或任何猜测性基线。
- 项目正式名称统一使用 `湖南星邦智能装备股份有限公司合同管理平台`；平台简称统一使用 `合同管理平台`；英文缩写统一使用 `CMP`。
- 主 agent 只负责维护 Todo、派发 subagent、收集反馈、分析与汇总；禁止在主会话中直接修改代码或文档。
- 当用户要求“每一条需求基线都标注来源的位置级引用”时，需求基线文档中的每一条需求项、边界项、验收项、实施项都必须逐条附来源位置引用；禁止只在章节标题或章节说明处做章节级引用。
- 只有正式需求基线文档需要逐条来源位置引用；实施计划、API 草稿、里程碑等派生草稿文档默认不需要逐条来源引用，除非用户再次明确要求。
- 招标文件存在冲突时，如用户已给出收口原则，必须按该收口原则直接落文并消除歧义；禁止继续保留并列选项、含糊表述或未收口的冲突描述。
- 支付与性能范围按已确认口径执行：微信支付已排除，不得再写入范围；企业微信对接纳入一期，必须按一期能力落文；性能指标如存在冲突，统一取更严格口径。
- 当招标文件要求平台具备完整能力时，即使业务主路径优先走外部系统，也必须保留平台内完整能力定义与落地范围；禁止弱化成“仅保留 API 壳子”“可选后备能力”或等效表述。
- 当文档写“审批流可视化配置”时，禁止只写流程节点类型；必须明确每个审批节点都可关联组织架构中的部门、人员或组织架构规则，因为审批流程基于组织架构运行。
- 当招标文件同时出现一般兼容条款与更具体的技术实现条款时，当前项目正式收口必须以更具体条款为准；禁止继续保留被后文明确实现选择排除的并列候选项，例如后文已明确“选用 MySQL 数据库”时，不得再将 PostgreSQL 作为最终交付候选项。

## 2026-04-03

- 后端技术栈沿用当前既定口径；后续讨论、方案描述与文档落文默认按当前后端栈展开，除非用户再次明确调整。
- 当前前端技术栈固定为 `React (SPA) + Tailwind CSS + shadcn/ui`；新增页面、组件、方案描述与实现讨论必须默认按该组合落文，禁止擅自切换到其他前端栈。
- 即使项目最终数据库已按 `MySQL` 收口，架构设计也必须保留数据库抽象层；禁止在架构层直接绑定 `MySQL` 语义、能力或实现细节。
- AI 能力设计必须保留模型层与 Provider 层抽象；禁止把提示编排、能力接口、业务流程或配置结构直接绑死到单一模型或单一 Provider。
- 部署环境默认按 `Docker Compose / 企业内网` 约束讨论、设计与落文；未获用户明确变更前，禁止默认按公有云托管、Kubernetes 或互联网直连口径展开。
- 主 agent 负责协调、派发、收集与汇总；凡可独立处理且互不冲突的子任务，默认优先并行派发给 subagent，禁止无故在主会话中串行堆积。
- 外部测试账号只为企业微信对接准备；加密软件、电子签章属于平台内自研子模块，禁止再按外部测试账号依赖项落文。
- 加密软件的正确平台语义固定为：文件进入文档中心自动加密，默认仅可在平台内按权限自动解密使用；管理端授权模块必须支持按部门、人员授予“解密下载”权限，获授权对象导出的明文文件可脱离 `CMP` 使用，但解密下载行为必须纳入权限控制与审计留痕。
- 后续任何文档都不得把加密软件主语义写成“外部对接对象”或“外部依赖系统”；如需描述其关系，应明确其为平台内能力或平台内自研子模块。
- 后续任何文档都不得再使用“平台外绝对不可解密/不可使用”这类绝对化表述；必须写成“默认密文文档不能脱离平台使用，管理端授权解密下载是受控例外”。

## 2026-04-04

- 向用户汇报“已更新的文件”时，只能列本轮真实参与当前正式交付链路的文件；未迁正的旧影子文档、历史草稿、自动维护的 `docs/planning/*` 真相文件，除非本轮交付明确以其为正式产物，否则不得混入交付清单。
- 新文档体系一旦确定，旧 `docs/drafts/planning/`、旧 planning 草稿或其他影子文档必须立即二选一处理：迁入正式链路，或清理/归档；禁止继续留在主链路旁边与正式文档并存误导执行。
- `docs/planning/current.md`、`docs/planning/history.md`、`docs/planning/decisions.md` 仅表示项目推进真相与过程决策，不等同于 Requirement、Architecture、API、Detailed、Implementation 这条正式文档链路；汇报、引用和交付归类时必须分开表述。
- 只要向用户声称“某个文档已修改/已修好/已迁移”，必须同时附该文档的主要修改列表，便于快速评审；禁止只报文件名而不说明改动点，迫使用户通读全文找差异。
- 以后汇报文档更新时，除列出“改了哪些点”外，必须再附该文档 1-2 条关键原文摘录，便于用户快速打开文档核实；禁止只给文件名，或只给抽象总结而不附文内原文。
- 写中文文档时，行长尽量控制在 80 个中文字符以内，英文尽量控制在 160 个英文字符以内；这是“尽量”而不是“必须”，不要为了追求短行把本来一行能说清楚的话硬拆成 2-3 行；只有排比句、并列项明显较多时，才优先改成 Markdown 有序/无序列表；不再沿用此前“40 个中文字左右就强拆”的过严规则。

## 2026-04-18

- 当用户连续推进多条主线时，汇报必须严格对应当前正在处理的那一条主线；禁止把上一轮或其他模块的修改结果串线汇报到当前问题上。若当前问题是解释、评审或回答某一文档段落，就先直接回答该段落，禁止先跳去汇报无关文档变更。

## 2026-04-25

- 主 Agent 不得在主会话中直接改生产文档；生产文档修改必须派独立执行 SubAgent 完成。
- 审查与修改必须由不同 SubAgent 分离执行；同一 SubAgent 不得同时承担修改和审查。
- 专项设计文档的 `## 1. 文档说明` 顶部只保留直接父文档 `Detailed Design`；禁止在顶部列跨模块设计基线、同级专项参考、`API Design` 或 `Implementation Plan`。

## 2026-04-26

- 后续派发子代理时，返回格式和状态词必须使用简体中文，例如“状态：已完成 / 阻塞”；不要在面向用户或主 Agent 的汇报要求中写 `DONE / BLOCKED` 这类英文状态词。
- `docs/superpowers/specs/` 每新增一份规格文件，规格编号递增一次；`docs/superpowers/plans/` 不单独递增规格主编号，计划编号前缀的第一部分必须与对应规格编号一致。一份规格只需一份计划时，计划直接复用规格编号；一份规格需要多份计划时，计划使用 `规格编号-计划子编号`，计划子编号从 `01` 开始递增，例如 `101-01-*`、`101-02-*`，以及 `102-01-*`、`102-02-*`。

## 2026-04-27

- 功能分支命名必须覆盖实际交付范围；当最小骨架同时包含后端、前端、本地编排和验证入口时，禁止使用只指向后端的分支名，应使用 `minimal-skeleton` 这类范围完整的名称。
- 进入编码实现阶段后，`docs/planning/current.md` 只保留当前编码批次的目标、Todo、阻塞和下一步；文档阶段长列表必须迁入 `docs/planning/history.md`，禁止继续在 `current.md` 中混放历史批次。
- 编码阶段每个功能点必须执行 `编码 -> QA -> 修复 -> QA`，直到明确全绿；全绿后进入下一个功能点前按提交规范自动提交，不需要再向用户确认是否提交。
