## 3. 平台共享模块拆分

### 3.1 拆分原则

平台共享底座按“共享真相源、横切能力、运行时治理”三类拆分，
避免把子模块私有逻辑塞进平台底座，也避免各业务模块重复建设基础能力。

- 共享真相源模块：持有跨模块复用且必须统一治理的数据真相。
- 横切能力模块：为多个业务模块提供认证、权限、审计、通知、集成等能力。
- 运行时治理模块：为平台提供缓存、任务、配置、恢复、观测等公共机制。

### 3.2 共享底座模块清单

#### 3.2.1 组织与权限底座

- 持有组织、部门、人员、岗位、角色、权限、数据授权等共享主数据。
- 为审批节点绑定组织架构、数据权限过滤、通知路由、审计归属提供统一基线。
- 不负责审批流程定义本身，也不负责业务表字段的具体含义。

#### 3.2.2 合同主档底座

- 持有合同主档、合同编号、来源标识、分类主链、生命周期总状态、主关联键。
- 作为起草、审批、签章、履约、变更、终止、归档、检索的统一业务主键中心。
- 不替代模板、履约、归档等子模块的领域规则实现。

#### 3.2.3 文档主档底座

- 持有文件对象主记录、文档与业务对象绑定关系、主版本引用、存储定位、
  加密状态摘要。
- 统一提供“哪个文件对象存在、属于谁、当前主版本是谁”的平台真相。
- 具体版本链、转换产物、预览页、批注结构下沉到文档中心详细设计。

#### 3.2.4 统一任务中心

- 平台全部异步任务统一进入任务中心建模。
- 提供任务主键、任务类型、状态机、调度信息、重试信息、人工介入标记。
- 不在平台层把各任务的私有执行参数和子步骤无限展开。

#### 3.2.5 审计与日志中心

- 统一记录安全审计、关键业务动作、接口交换、配置变更、解密下载、
  AI 人工确认等留痕。
- 负责平台级追踪字段、关联对象键、操作人、来源端、结果摘要。
- 不替代各模块的业务详情表。

#### 3.2.6 配置中心

- 统一管理系统级、组织级、模块级共享配置。
- 负责配置版本、生效状态、环境范围、灰度开关、敏感配置引用方式。
- 不把复杂流程规则、模板正文、提示词全文直接塞入系统配置表。

#### 3.2.7 通知与集成交换中心

- 统一承接站内信、邮件、企业微信、外部回调、入站交换、出站重试等平台能力。
- 管理消息通道、交换对象、重试状态、回执摘要、签名校验结果。
- 外部系统字段映射与业务语义适配，仍由具体集成模块下沉实现。

#### 3.2.8 DB abstraction layer

- 对业务服务暴露稳定仓储接口、事务边界、查询封装。
- 把 `MySQL` 实现细节隔离在数据访问层内部。
- 平台共享表和子模块表都必须经该层访问，不允许业务模块直接散写方言 SQL。

### 3.3 共享模块职责边界

| 模块 | 持有内容 | 不持有内容 | 主要服务对象 |
| --- | --- | --- | --- |
| 组织与权限底座 | 组织主数据、角色权限、数据授权 | 审批图定义、业务状态机 | 审批、通知、管理、检索 |
| 合同主档底座 | 合同主键、分类主链、生命周期总状态 | 模板正文、审批节点细节、履约明细 | 起草、审批、签章、履约、归档 |
| 文档主档底座 | 文件主记录、业务绑定、主版本引用 | 完整版本链与预览内部实现 | 起草、审批、签章、归档、加密 |
| 统一任务中心 | 任务主表、状态、调度、重试 | 各任务私有算法与执行步骤 | OCR、AI、导出、回调、索引 |
| 审计与日志中心 | 平台统一留痕、追踪、交换日志 | 子模块私有业务明细表 | 全模块 |
| 配置中心 | 平台共享参数、开关、组织级配置 | 子模块复杂规则正文 | 全模块 |
| 通知与集成交换中心 | 消息分发、回调交换、通道治理 | 外部系统本体逻辑 | 审批、签章、集成、管理 |

### 3.4 模块间内部调用边界

平台共享模块之间只允许通过稳定内部服务接口或仓储接口调用，
不允许跨模块直接访问对方私有表。

关键边界如下：

- 审批、签章、履约、归档模块读取合同基础信息，只能经合同主档服务，
  不能直接拼接其他模块私有表得到“第二合同真相”。
- 各模块需要文件对象信息时，只能经文档主档服务读取文件主记录与主版本引用。
- 各模块发起异步任务时，只能经统一任务中心创建任务，再由任务执行器领取。
- 各模块发通知或外部交换时，只能经通知与集成交换中心出站。
- 各模块写平台级审计时，只能经审计与日志中心统一落库。
- 各模块获取组织、角色、数据权限时，只能经组织与权限底座。
- 所有持久化读写统一经 `DB abstraction layer`，不允许业务服务层绕过抽象层。

### 3.5 平台层不直接展开的子模块边界

总平台层只保留以下共享接口面：

- 流程引擎：仅保留“平台审批流是一期正式能力、任务中心和审计中心如何与其协作”。
- 文档中心：仅保留“文件对象真相与主版本引用由谁持有”。
- 加密模块：仅保留“默认不可脱离平台使用、授权解密下载受控留痕”的平台约束。
- `Agent OS`：仅保留“AI 统一收口、任务与审计如何接入”的平台约束。

### 4.3 共享表之间的主关系

- `contract_master` 是合同生命周期的统一根表。
- `document_asset` 通过 `owner_type + owner_id` 绑定 `contract_master`
  或其他允许的业务对象，但文件对象真相仍由文档主档持有。
- `platform_job` 可绑定 `contract_id`、`document_id`、`exchange_id`、
  `message_id` 等平台对象，作为统一异步处理入口。
- `audit_event` 可追踪任一共享对象，但不替代对象自身状态表。
- `integration_exchange` 负责交换过程真相，不反向改写业务对象归属。
- `notification_message` 负责消息投递真相，不替代业务动作本身。

### 4.4 哪些数据必须下沉到子模块表

以下内容不进入总平台共享主表范围，应下沉到模块级文档：

- 流程定义节点、连线、条件表达式、会签策略、催办策略等流程引擎细表。
- 文档版本链、预览转换产物、批注锚点、修订关系等文档中心细表。
- 加密策略明细、密钥轮换、授权单、加解密流水等加密模块细表。
- AI 会话上下文、prompt 版本、tool 调用明细、memory 单元等 `Agent OS` 细表。

## 5. 缓存与并发控制公共机制

### 5.1 Redis 使用边界

`Redis` 在平台层是增强层，不是真相源。

允许放入 `Redis` 的内容：

- 幂等键。
- 短时分布式锁。
- 热点只读缓存。
- 登录会话短态。
- 任务短态与消费者协调信息。
- 限流计数、短期验证码、回调防重放票据。

必须落 `MySQL` 的内容：

- 合同主档与生命周期状态。
- 审批实例正式状态与关键动作结果。
- 文档主档、文件绑定关系、主版本引用。
- 解密下载授权结果与下载审计。
- 异步任务正式状态、重试次数、人工介入标记。
- 系统配置正式版本。
- 审计日志、集成交换记录、通知投递结果。

### 5.2 幂等键机制

平台所有写接口和异步任务创建接口统一支持 `Idempotency-Key`。

- 幂等键首选落 `Redis`，用于高频快速判重。
- 首次成功受理后，关键写操作的最终关联结果必须同步落库，
  避免仅凭缓存记录历史受理结果。
- 幂等键冲突时返回首次请求对应的受理结果，
  不允许再次执行副作用动作。
- 对外回调和入站交换同时以 `source_system + external_request_id`
  在 `integration_exchange` 落正式唯一约束，`Redis` 只做前置减压。

### 5.3 锁机制

平台统一使用短 TTL 分布式锁控制高冲突修改。

适用场景：

- 合同编号生成。
- 合同状态关键跃迁。
- 同一文档对象的受控解密下载。
- 同一任务的执行器领取。
- 同一回调请求的重复处理。

规则如下：

- 锁只用于互斥，不用于持久化状态。
- 锁粒度优先按业务对象，例如 `contract:{id}`、`document:{id}`、`job:{id}`。
- 锁失效后必须依靠数据库版本号、唯一约束或状态条件二次兜底。
- 不允许仅依赖 `Redis` 锁判断业务是否已经成功提交。

### 5.4 热点缓存机制

热点缓存用于提升高频读取性能，但缓存失效不影响真相恢复。

适合缓存的对象：

- 组织树、部门树、人员基础信息摘要。
- 权限字典、菜单字典、分类字典。
- 系统共享配置只读快照。
- 合同列表检索中的稳定统计摘要。

规则如下：

- 缓存键必须带版本或失效标签，避免配置更新后旧缓存长期污染。
- 共享配置和权限变更后优先事件驱动失效，辅以 TTL。
- 合同详情、审批详情、文档详情这类状态频繁变化对象，
  不采用长 TTL 粗暴缓存，优先按摘要字段或只读投影缓存。

### 5.5 会话短态与任务短态

会话短态允许仅存在 `Redis`，但必须满足可过期、可重建、可失效。

- 登录会话票据、验证码、一次性跳转态允许仅存 `Redis`。
- 用户长期身份、角色、组织绑定必须回到 `MySQL` 真相。

任务短态是执行期间的运行辅助信息，不是正式任务状态。

- 例如执行器心跳、进度百分比快照、分片游标、临时结果句柄。
- 最终成功、失败、超时、取消、人工介入等正式状态，必须回写 `platform_job`。

### 5.6 并发控制基线

平台共享对象统一采用“数据库条件更新 + 版本号 / 状态校验 +
必要时短锁”的三段式控制。

- 合同、文档、任务、交换对象都应带 `version_no` 或等效乐观并发字段。
- 关键状态迁移必须使用条件更新，避免后写覆盖先写。
- 对需要绝对串行的动作，再叠加短锁保护。
- 审批、签章、解密下载、回调处理都不得依赖前端按钮防重代替后端并发控制。

## 6. 异步任务与调度公共机制

### 6.1 平台统一任务模型

平台异步任务统一抽象为 `platform_job`。

每个任务至少具备以下平台级属性：

- `job_id`
- `job_type`
- `owner_type` / `owner_id`
- `idempotency_key`
- `job_status`
- `priority`
- `payload_ref`
- `retry_count`
- `max_retry_count`
- `scheduled_at`
- `next_run_at`
- `timeout_at`
- `manual_intervention_required`
- `result_code` / `result_message`

任务载荷采用“主表摘要 + 载荷引用”的方式，
避免把大对象直接堆进任务主表。

### 6.2 统一任务类型范围

一期共享任务类型至少包括：

- `OCR`
- `AI_TASK`
- `EXPORT`
- `DECRYPT_DOWNLOAD`
- `INDEX_REFRESH`
- `NOTIFICATION_DELIVERY`
- `CALLBACK_RETRY`
- `INTEGRATION_SYNC`

子模块可新增类型，但必须复用平台统一状态机、重试与审计规则。

### 6.3 任务状态机

平台统一任务状态建议收口为：

```text
PENDING -> READY -> RUNNING -> SUCCEEDED
                   -> FAILED_RETRYABLE -> READY
                   -> FAILED_TERMINAL
                   -> TIMED_OUT
                   -> CANCELLED
                   -> WAITING_MANUAL
```

状态说明：

- `PENDING`：已创建但未到调度条件。
- `READY`：可被执行器领取。
- `RUNNING`：已被某执行器持有。
- `SUCCEEDED`：执行成功并完成结果回写。
- `FAILED_RETRYABLE`：本次失败，但允许重试。
- `FAILED_TERMINAL`：确定不可自动恢复。
- `TIMED_OUT`：超过超时边界，需补偿或人工处理。
- `CANCELLED`：被上游业务显式取消。
- `WAITING_MANUAL`：进入人工介入池。

### 6.4 调度与领取机制

- 调度器按 `job_status + next_run_at + priority` 选取可执行任务。
- 执行器领取任务前，先以数据库条件更新锁定，再用 `Redis` 作为辅助互斥。
- 同一 `job_id` 任何时刻只允许一个有效执行上下文。
- 调度器与执行器异常退出后，依据 `heartbeat_at` 和 `timeout_at`
  回收悬挂任务。

### 6.5 重试、超时、补偿

- 瞬时性失败采用指数退避重试。
- 可重试失败仅限网络抖动、外部系统超时、临时资源不足、
  可重入的下游依赖失败。
- 数据不合法、权限不足、业务前置条件不满足这类错误直接进入
  `FAILED_TERMINAL`。
- 超时任务先进入 `TIMED_OUT`，再依据任务类型判断是否自动补偿。

补偿边界如下：

- 通知失败：允许自动重试或切换通道。
- 回调失败：允许按策略重试并保留交换留痕。
- 索引刷新失败：允许重建，不影响主业务真相。
- 导出失败：允许重新生成，不回滚主业务对象。
- OCR / AI 失败：允许重新投递，不覆盖原业务真相。
- 解密下载失败：不得静默补偿为“已成功”，必须重新授权校验并重新执行。

### 6.6 死信与人工介入边界

平台一期不以独立 `MQ` 死信队列为前提，
而是在 `platform_job` 内显式建模人工介入边界。

- 达到最大重试次数后，任务进入 `WAITING_MANUAL` 或 `FAILED_TERMINAL`。
- 需要人工确认的数据风险、权限风险、法律风险任务，不做无限自动重试。
- 人工介入池至少覆盖：解密下载异常、外部回调长期失败、
  AI 高风险结果待确认、外部系统状态不一致。

### 6.7 与子模块的边界

平台层只统一任务模型、状态机、调度、重试、恢复与审计。

以下内容下沉到各模块详细设计：

- OCR 识别管线步骤。
- AI prompt 编排与工具调用细节。
- 文档解密执行流水。
- 搜索索引分词、倒排与重建策略。
- 流程引擎内部节点级任务编排。

## 7. 跨模块数据归属

### 7.1 平台真相源原则

跨模块对象必须先判断“谁是平台真相源”，再决定谁可以投影、索引或派生。

平台真相源对象如下：

- 组织、部门、人员、角色、权限。
- 合同主档与生命周期总状态。
- 文档主档与文件对象绑定关系。
- 平台审批定义与平台审批实例。
- 解密下载授权结果与下载审计。
- 统一任务正式状态。
- 审计事件、集成交换结果、通知投递结果。
- 系统共享配置。

### 7.2 只能作为投影 / 索引 / 扩展状态的对象

以下对象允许存在，但不是真相源：

- 搜索索引中的全文倒排、关键词摘要、高亮片段。
- 统计报表宽表、看板聚合结果。
- Redis 热点缓存、任务短态、会话短态。
- 外部系统映射副本、同步快照、回调请求体归档副本。
- AI 摘要、AI 比对结果、风险提示、问答结果。
- 文档预览产物、OCR 中间文本、导出临时文件。

### 7.3 关键对象归属边界

#### 7.3.1 合同

- `contract_master` 是合同真相源。
- 审批、签章、履约、归档模块只能推动合同状态变化，
  不能各自维护第二份合同主状态。
- 搜索与报表中的合同数据仅为读模型。

#### 7.3.2 文档

- 文档中心持有文件对象真相与版本链。
- 总平台层只保留文档主档、业务绑定与主版本引用。
- 其他模块只允许引用 `document_id`，不自建附件真相源。

#### 7.3.3 审批

- `OA` 仍可作为主审批执行路径，但平台内的审批承接结果、
  审批模式选择、审批摘要归属仍受平台治理。
- 平台审批引擎是平台正式能力，其定义与实例是平台真相源。
- `OA` 回传结果是外部事实输入，不等于平台把审批对象所有权交给 `OA`。

#### 7.3.4 签章

- 签章模块持有签章专业状态和执行结果。
- 对合同是否已签、签章是否完成的业务结论，必须回写合同主档或合同时间线。

#### 7.3.5 加密

- 加密模块负责文档读写链路上的加密控制和受控解密。
- 文档是否受加密控制、是否允许解密下载、谁何时执行了解密下载，
  必须沉淀到平台共享主档和审计链路。
- 密钥、算法、设备协同等内部细节不在总平台层展开。

#### 7.3.6 AI 结果

- AI 结果统一收口到 `Agent OS` 输出，但 AI 结果不是合同、审批、
  签章、归档的正式真相源。
- AI 可形成建议、摘要、比对、风险提示、待确认结论。
- 最终业务状态必须由人工动作、规则判断或正式业务流程确认后落库。

### 7.4 写入边界原则

- 真相源对象只能由其归属模块写入主状态。
- 其他模块只能发起动作请求、写事件、写投影，不能越权改主状态。
- 跨模块写入必须通过应用服务而不是跨表直写。
- 外部系统输入只能作为来源事件或同步输入，不能绕过平台校验直接篡改真相源。

## 8. 平台级审计、日志、观测与恢复

### 8.1 必须记录的内容

平台层必须统一记录以下审计事件：

- 登录、登出、登录失败、身份切换。
- 角色、权限、组织架构、系统配置变更。
- 合同创建、关键字段变更、状态跃迁。
- 审批发起、审批承接、审批模式切换、审批结果回写。
- 文档上传、主版本切换、解密下载授权、解密下载执行。
- AI 任务创建、AI 结果确认、人工驳回。
- 外部系统入站、出站、回调验签失败、重试、终止。
- 异步任务失败、超时、人工介入、补偿执行。

平台日志建议按三层拆分：

- 应用日志：排查代码和运行异常。
- 审计日志：用于合规、安全、责任追踪。
- 交换日志：用于外部系统对账、回放和重试分析。

### 8.2 指标与告警建议

平台级核心指标：

- API 成功率、P95 / P99 响应时间。
- 异步任务积压量、失败率、超时率、人工介入量。
- 回调验签失败次数、外部系统重试次数、交换失败持续时长。
- Redis 命中率、锁竞争失败率、缓存失效率。
- 合同状态更新失败数、解密下载失败数、审计写入失败数。
- 文档上传失败率、索引刷新延迟、通知送达率。

平台级核心告警：

- 任务队列持续积压。
- 审计日志写入失败或明显下降。
- 回调失败率异常升高。
- Redis 大面积不可用导致锁和幂等异常增多。
- `MySQL` 主库不可写或复制延迟过高。
- 解密下载异常激增。

### 8.3 备份与恢复边界

必须备份并可恢复的数据：

- 组织与权限主数据。
- 合同主档。
- 文档主档及文件对象映射。
- 平台审批定义与平台审批实例。
- 解密下载授权与下载审计。
- 统一任务主表和任务尝试明细。
- 审计日志、集成交换、通知结果。
- 系统配置。

允许重建的数据：

- Redis 缓存、幂等键、短锁、会话短态。
- 搜索索引与检索投影。
- 统计报表宽表。
- OCR 中间文本、AI 临时中间结果、导出临时文件。

### 8.4 恢复策略原则

- 恢复顺序优先保证 `MySQL` 主数据和文件对象映射恢复。
- 恢复后先重建配置与权限，再恢复任务调度，再重建索引和缓存。
- 对外交换和异步任务必须支持基于主表记录补跑、重试或人工对账。
- 恢复后不得仅凭缓存或索引认定业务完成状态。
