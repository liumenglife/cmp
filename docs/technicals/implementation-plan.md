## 2. 计划边界与原则

本文档只回答“先做什么、后做什么、依赖什么、何时进入联调和验收准备”。
以下内容不在本文展开：

- 不复述一期需求范围与业务清单
- 不复述技术选型理由、系统拓扑和平台分层
- 不复述物理表设计、索引策略、任务状态机和接口字段细节
- 不写成按人天拆分的项目管理表，也不写负责人分配

实施顺序遵循以下原则：

- 先平台底座，再业务闭环，再增强能力，再集成验收
- 先建设共享真相源，再建设依赖共享真相源的业务模块
- `OA` 主审批优先，但平台审批引擎是一期正式能力，必须在一期完成可用闭环
- 文档中心是平台文件真相源，凡正文、附件、签章稿、归档稿相关链路，均以文档中心先成立为前提
- 数据库交付以 `MySQL` 为准，但实施中必须始终保留 `DB abstraction layer`
- 前端实施口径固定为 `React (SPA) + Tailwind CSS + shadcn/ui`
- 部署基线固定为 `Docker Compose / 企业内网`
- 加密软件、电子签章为平台内自研子模块，不作为外部测试账号依赖处理

## 3. 前置条件

### 3.1 企业微信测试账号

- 企业微信测试租户、测试应用和可用的测试通讯录需在外部联调前准备完成
- 需至少具备登录验证、部门同步、人员同步、消息触达所需测试条件
- 测试账号应覆盖普通员工、部门负责人、管理端操作人员等最小验证角色

### 3.2 环境准备

- 开发、测试、联调环境需按 `Docker Compose / 企业内网` 基线可重复部署
- `MySQL`、文件存储、缓存、日志与基础任务运行环境需提前就绪
- 各环境应具备基础网络连通、域名或访问地址、时间同步、日志留存与备份策略
- 前后端构建链路、配置注入方式、镜像构建方式需在基础阶段前收口

### 3.3 文档与配置准备

- 正式需求、总平台技术文档和当前 3 个 `docs/technicals/foundations/` 模块 + 7 个 `docs/technicals/modules/` 模块文档需作为当前实施唯一正式依据
- 组织架构、角色权限、字典、流程编码、环境变量清单等实施必需配置项需提前整理
- 外部系统联调时使用的接口地址、签名方式、回调地址、字段映射责任边界需先明确
- 测试与验收所需样例合同、模板样例、条款样例、附件样例需在业务闭环阶段前准备

### 3.4 其他正式约束

- 需求与正式文档链路以 `specifications/` / `technicals/` 为准
- planning 文档是项目真相文件，但不是正式交付链路的一环
- 审批路径采用 `OA` 主审批优先，但平台审批引擎为一期正式能力
- 文档中心是文件真相源
- 默认不可脱离平台使用，但管理端可按部门、人员授权解密下载
- 企业微信需要测试账号；加密软件、电子签章不依赖外部测试账号

## 4. 总体阶段划分

总平台第一版实施建议按五个阶段推进，阶段之间以前一阶段完成标志作为下一阶段
进入条件，而不是并行无序展开。

### 4.1 阶段一：基础底座就绪

#### 目标

- 建立总平台可持续开发和可重复部署的基础底座
- 把组织、权限、合同主档、任务中心、审计、集成入口、文档主档入口等共享能力先收口
- 为后续流程引擎、文档中心和核心业务模块提供稳定挂载点

#### 核心产出

- 可部署的基础运行环境与基础配置体系
- 总平台共享底座主链路的第一版可用实现
- 合同主档、组织权限、任务中心、审计日志、集成交换、配置中心的基础能力
- 文档中心最小写入入口与文件主档绑定能力
- 统一前端应用壳、基础菜单、认证与权限框架

#### 前置依赖

- 正式文档链路已完成当前轮收口
- 开发与测试环境已具备基础部署条件
- 组织架构与权限基础口径已确认

#### 完成标志

- 平台可以在目标环境稳定启动
- 共享底座能力可支撑后续模块不重复建设基础设施
- 合同主档、文档主档、统一任务、统一审计、统一集成入口均已有可落地实现
- 后续模块可以基于已落地共享能力接入，而不需要回头重写基础能力边界

### 4.2 阶段二：核心业务闭环成立

#### 目标

- 先打通一期最核心的合同主链路，形成“起草/录入 -> 文档入库 -> 审批发起 -> 状态回写 -> 查询”的平台闭环
- 以首批重点流程为目标建立可验证业务路径，而不是先铺开全部外围与增强功能

#### 核心产出

- 合同模板、条款库、合同起草与合同主档管理能力
- 合同正文与附件进入文档中心的标准写入路径
- `OA` 主审批发起与审批摘要回写的第一版闭环
- 合同台账、详情、基础检索、基础通知的业务可用版本
- 首批重点流程的端到端演示链路

#### 前置依赖

- 阶段一共享底座已可用
- 文档中心至少具备文件主档写入、版本登记和基础读取能力
- 集成入口和审计能力已可复用

#### 完成标志

- 平台内可完成一条合同从创建到审批结果回写的最小业务闭环
- 合同正文、附件、审批摘要均已回到平台统一治理
- 核心页面和核心接口已能支撑内部联测
- 后续流程引擎增强、文档协作、签章、归档不再缺少基础业务挂载点

### 4.3 阶段三：流程引擎与文档中心能力完善

#### 目标

- 把一期正式能力中的平台审批引擎与文档中心能力从“最小可挂载”提升到“正式可承接”
- 建立后续签章、归档、协作、受控解密下载等能力所依赖的平台基础

#### 核心产出

- 平台审批引擎可视化配置、定义发布、实例运行、任务推进与回写能力
- 节点绑定组织架构的完整能力
- 文档中心版本链、预览、协作挂载、受控读取与授权解密下载能力
- 文档协作、修订、批注、版本切换等围绕文件真相源的正式能力
- 面向签章、归档、搜索、`OCR` 的稳定挂接入口

#### 前置依赖

- 阶段二已建立合同主链路和文档中心最小闭环
- 组织架构、权限、任务中心、审计、通知已具备复用条件
- `OA` 主路径与平台路径的切换规则已在正式文档中收口

#### 完成标志

- 平台审批引擎可独立承接需要由平台执行的审批场景
- 审批节点已可稳定绑定部门、人员或组织规则
- 文档中心已成为正文、附件、签章稿、归档稿的正式真相源
- 管理端受控解密下载链路已具备实现基础并纳入权限与审计
- 后续签章、归档、搜索、智能能力可直接挂接，不再需要补基础设施

### 4.4 阶段四：智能能力与业务增强

#### 目标

- 在核心闭环和平台底座稳定后，补齐一期要求中的智能与增强能力
- 让 AI、`OCR`、搜索、履约、统计、通知等能力建立在既有真相源和统一任务体系上

#### 核心产出

- `OCR`、全文检索、AI 辅助、智能摘要与风险提示的接入能力
- 履约跟踪、变更/终止、归档准备、统计分析等业务增强能力
- 通知中心、报表导出、索引刷新、批量处理等增强型任务链路
- 面向多语言、电视端适配的前端补齐工作

#### 前置依赖

- 阶段三的流程引擎和文档中心已形成稳定挂接入口
- 统一任务中心、审计中心和配置中心已可承接增强任务
- 文档版本、审批摘要、合同主档状态均已可作为智能能力输入源

#### 完成标志

- 智能能力只作为增强能力挂接，不破坏业务真相源边界
- 搜索、`OCR`、AI、履约、报表等能力已建立可联测链路
- 多语言与多终端适配达到一期可验收状态
- 平台功能从核心闭环扩展到一期正式能力范围内的完整可验收形态

### 4.5 阶段五：集成联调、验收与上线准备

#### 目标

- 把平台从“模块可用”推进到“环境可交付、联调可验收、上线可切换”
- 完成外部系统联调、验收脚本准备、上线包准备和运维交接准备

#### 核心产出

- `OA`、企业微信及其他一期范围内外围接口的联调结果
- 覆盖核心业务场景、异常场景、权限场景的测试与验收用例
- 部署脚本、初始化配置、运维手册、回退预案、培训材料
- 上线窗口所需的数据初始化、账号初始化、权限初始化方案

#### 前置依赖

- 前四阶段功能和接口已基本冻结
- 联调环境、测试数据和外部测试账号均已就绪
- 核心缺陷已收敛到可控范围

#### 完成标志

- 外部联调通过，关键业务场景验收通过
- 部署、备份、恢复、日志排查与基础运维动作可执行
- 上线所需文档、脚本、配置和培训材料齐备
- 平台具备一期上线或试运行条件

## 5. 阶段依赖关系与推进顺序

当前技术目录已收口为 3 个 `docs/technicals/foundations/` 模块和 7 个 `docs/technicals/modules/` 模块，但推进时仍应按依赖关系分批进入实现。

如需查看正式的“五批分组清单”、批内并行关系、批间关键依赖、批次启动门禁与验收门禁等统一口径，应以 [`Implementation Batch Plan`](./implementation-batch-plan.md) 为准；本文只保留总平台层的阶段顺序与总体依赖判断。

整体依赖关系建议如下：

1. 先落地 `docs/technicals/foundations/` 下的模块：`identity-access`、`agent-os`、`integration-hub`
2. 再建立核心业务闭环：`contract-core`、`document-center`、`workflow-engine`
3. 然后推进依赖主链路真相源的模块：`e-signature`、`encrypted-document`、`contract-lifecycle`
4. 在主链路稳定后接入 `intelligent-applications` 等智能与增强能力，避免倒逼前面模块反复返工
5. 最后统一进入外部联调、综合测试、验收与上线准备

这一顺序的关键原因是：

- `identity-access`、`agent-os`、`integration-hub` 先成立，才能为后续模块提供统一身份、任务 / 审计 / Agent 运行底座与外部承接边界
- `contract-core`、`document-center`、`workflow-engine` 共同构成合同主链路、文件真相源与审批承接主线，应在第二批形成可消费的接口、状态、事件与挂载点
- `e-signature`、`encrypted-document`、`contract-lifecycle` 均具备正式设计输入，应作为第三批在启动时检查上游合同状态、审批状态、文档版本与事件是否可消费
- `intelligent-applications` 具备正式设计输入，并依赖稳定输入源和异步任务体系；第四批启动时检查上游接口、状态、事件与挂载点是否可消费
- 如第三 / 四批启动门禁发现依赖未满足，应冻结问题清单和责任主线，再按 `Implementation Batch Plan` 调整批次推进条件，不表达为缺少设计或需要补文档

## 6. 模块推进顺序

### 6.1 当前模块现状

当前总平台已形成 3 个 `docs/technicals/foundations/` 模块：`identity-access`、`agent-os`、
`integration-hub`；以及 7 个 `docs/technicals/modules/` 模块：`workflow-engine`、
`document-center`、`contract-core`、`e-signature`、`encrypted-document`、
`contract-lifecycle`、`intelligent-applications`。

其中 10 条主线均具备正式设计输入，后续统一按 `Implementation Batch Plan` 的五批依赖顺序推进：批次启动时检查上游接口、状态、事件与挂载点是否可消费；批次验收时确认本批输出可被下游继续消费。本文不重复展开详细矩阵。

### 6.2 建议的模块实施先后

建议按以下顺序推进：

1. `docs/technicals/foundations/` 下的模块先行：`identity-access`、`agent-os`、`integration-hub`
2. 主链路成型：`contract-core`、`document-center`、`workflow-engine`
3. 依赖文档真相源与审批真相源的模块：`e-signature`、`encrypted-document`、`contract-lifecycle`
4. 智能与增强能力：`intelligent-applications`
5. 综合联调、验收和上线准备

### 6.3 推进顺序的原因

- `identity-access` 先落地，后续组织、权限、审批节点绑定和多端访问才有统一主数据入口
- `agent-os` 与 `integration-hub` 先落地，后续任务协同、审计追踪、外部系统承接和异步编排才不会在各模块内重复建设
- `contract-core`、`document-center`、`workflow-engine` 是合同主链路的三条主线，应在第二批优先形成可被第三批消费的稳定闭环
- `e-signature`、`encrypted-document`、`contract-lifecycle` 都直接依赖合同状态、审批状态和文档版本真相，应在第三批启动门禁中检查这些上游输出是否可消费
- `intelligent-applications` 依赖前述模块沉淀出的稳定输入、事件和文档资产，应在第四批启动门禁中检查上游接口、状态、事件与挂载点是否可消费
- 若启动门禁未满足，应冻结问题清单和责任主线，由对应上游主线补齐可消费输出，不表达为缺少设计或需要补文档

## 7. 外部联调准备

本项目一期中，真正需要作为外部测试账号前置准备的重点对象是企业微信。

### 7.1 企业微信联调前置项

- 测试企业、测试应用、回调配置、密钥信息与测试通讯录可用
- 测试部门、测试人员与平台组织架构映射规则已确认
- 登录入口、身份同步、消息触达、回调地址在联调环境中可验证
- 联调双方对字段映射、失败回执、重试策略和责任边界已确认

### 7.2 `OA` 与其他外围系统联调准备

- 需要的是接口连通、报文样例、字段映射、回调路径和责任边界准备
- 不应把外围系统本体开发工作写成平台实施前置完成项
- 平台侧应优先保证接口承接、预留、字段调整和异常回执能力 ready

### 7.3 不作为外部测试账号前置项的内容

- 自研加密软件不依赖外部测试账号
- 自研电子签章不依赖外部测试账号
- 这两类能力应作为平台内子模块，按内部能力建设、内部联测、内部验收方式推进

## 8. 测试、验收与上线准备

### 8.1 测试准备

- 按阶段建立测试入口，不等到全部开发完成后再一次性补测
- 阶段二开始建立核心业务闭环测试集
- 阶段三补齐平台审批引擎、文档中心、受控解密下载等平台级能力测试集
- 阶段四补齐智能能力、搜索、履约、归档、统计、多语言与多终端适配测试集
- 阶段五完成回归测试、集成测试、权限测试、异常恢复测试与部署验证测试

### 8.2 验收准备

- 验收以正式文档链路中的需求、接口和设计文档为依据组织场景
- 验收用例应覆盖：
  - 合同核心闭环
  - `OA` 主审批路径
  - 平台审批引擎承接路径
  - 文档中心真相源与版本链
  - 管理端授权解密下载
  - 企业微信登录与组织同步
  - 归档、检索、统计与智能增强的关键场景
- 验收材料需包含环境说明、测试账号、操作脚本、样例数据和结果留痕方式

### 8.3 上线准备

- 固化部署脚本、初始化脚本、配置项清单和环境差异说明
- 准备数据库初始化与回退方案，但不改变 `MySQL` 正式交付口径
- 准备日志排查、任务补偿、接口重试、备份恢复和运行检查清单
- 准备上线培训与操作手册，覆盖业务用户、管理端用户和运维支持角色

## 9. 当前不展开项与后续细化入口

本文只给出总平台第一版正式实施顺序，不下沉到子模块实施细节。

后续细化应按以下入口承接：

- 当前 3 个 `docs/technicals/foundations/` 模块与 7 个 `docs/technicals/modules/` 模块均已有各自 `Implementation Plan`；后续细化应优先回到 `docs/technicals/foundations/<module-name>/implementation-plan.md` 或 `docs/technicals/modules/<module-name>/implementation-plan.md` 承接
- 各主线均具备正式设计输入，后续按 `Implementation Batch Plan` 的五批依赖顺序进入模块内细化与实现承接
- 第三 / 四批启动时检查上游接口、状态、事件与挂载点是否可消费；如未满足，冻结问题清单和责任主线，不在本文改写为模块缺少设计或需要补文档
- 总平台层只继续维护跨模块依赖、阶段顺序、联调窗口与验收准备，不再重复下沉模块内批次

总平台层与子模块层的关系应保持为：

- 总平台 `Implementation Plan` 只负责阶段、依赖、顺序、联调与验收准备
- 子模块 `Implementation Plan` 负责把各自阶段进一步细化为模块内实施批次与联调批次

## 10. 本文结论

总平台第一版正式实施顺序应按“`docs/technicals/foundations/` 模块 -> 合同核心主链路 -> 依赖主链路的 `docs/technicals/modules/` 模块 ->
智能增强 -> 集成联调与验收上线准备”推进。

当前真实现状是 3 个 `docs/technicals/foundations/` 模块与 7 个 `docs/technicals/modules/` 模块都已成型并具备正式文档输入；但从实施顺序上，仍应优先让 `identity-access`、`agent-os`、
`integration-hub`、`contract-core`、`document-center`、`workflow-engine` 先形成稳定闭环，
再推进 `e-signature`、`encrypted-document`、`contract-lifecycle`、`intelligent-applications`，
以减少跨模块返工。
