# 外围系统集成主线 Implementation Plan

本文档是 `CMP` 外围系统集成主线的第一份正式
`Implementation Plan`。

本文用于在既有正式文档边界内，明确外围系统集成主线的实施阶段、
任务依赖、推进顺序、联调准备、测试验收与上线准备。

## 2. 计划边界与原则

- 本计划只覆盖外围系统集成主线的平台侧实施安排，不承担外部系统本体改造职责。
- 业务模块不得私接外部入站、出站或回调入口，所有外围系统接入统一经由集成主线推进。
- 企业微信是正式移动承载端之一，实施顺序遵循原生优先、小程序兜底，不回桌面 `Web`。
- `OA` 是默认主审批路径外部系统，但平台仍需建设并保有正式审批承接能力。
- `CRM` / `SF` / `SRM` / `SAP` 提供外部主数据或业务事实，不直接成为平台真相源。
- 合同主档是业务真相源，文档中心是文件真相源，流程引擎是运行时真相源。
- 本计划按“先统一接入，再承接入站与绑定，再闭环出站与回调，再补齐治理与补偿，最后联调验收上线”推进，避免先接单点接口、后补公共底座造成返工。

## 3. 前置条件

### 3.1 企业微信测试账号

- 已取得可用于开发、测试、联调的企业微信测试账号与应用配置。
- 已明确登录、组织同步、消息触达、小程序兜底所需的回调地址、可信域名与环境白名单。
- 已确认移动端联调设备、测试人员身份与基础账号映射规则。

### 3.2 `OA` 联调前置条件

- 已明确 `OA` 作为默认主审批路径的联调范围，只覆盖平台侧桥接、状态回写与结果承接。
- 已取得 `OA` 测试环境访问方式、接口凭证、回调开通条件与审批实例查询能力。
- 已确认 `OA` 无法满足时由平台流程引擎正式承接的切换判定口径，避免联调后仍存在审批责任不清。

### 3.3 `CRM` / `SF` / `SRM` / `SAP` 数据交换前置条件

- 已明确各系统在一期内提供的是主数据或业务事实交换，不将其写成平台真相源。
- 已取得最小可联调的数据交换样本、业务主键规则、幂等键规则、失败回执方式与测试环境连通条件。
- 已确认平台侧仅负责接口承接、字段映射、状态治理、审计与补偿，不承担外围系统本体改造。

### 3.4 平台真相源承接前置条件

- 合同主档已具备承接外部合同事实或业务绑定的入口条件，且以合同主档作为统一业务落点。
- 文档中心已具备承接外部附件、扫描件、签章结果稿或归档产物的正式入口条件。
- 流程引擎已具备承接 `OA` 审批摘要、外部审批结果转译与平台内审批承接的基础入口条件。
- 上述承接条件未就绪前，不进入外部系统正式联调，只进行样例报文与连通性验证。

## 4. 总体阶段划分

本主线按五个阶段推进。

1. 统一接入与适配基础能力阶段
2. 入站承接与绑定建立阶段
3. 出站派发与回调闭环阶段
4. 运行治理与补偿完善阶段
5. 联调、验收与上线准备阶段

阶段之间按严格依赖推进；后一阶段可以预研，但不得跳过前一阶段的完成标志直接进入正式交付。

## 5. 分阶段实施计划

### 5.1 第一阶段：统一接入与适配基础能力

#### 目标

- 先建立外围系统统一接入面、统一适配层与统一治理入口，防止各业务模块各自对接外部系统。
- 为后续企业微信、`OA`、`CRM` / `SF` / `SRM` / `SAP` 的入站、出站、回调提供共同底座。

#### 核心产出

- 统一入站、出站、回调的接入边界与路由骨架。
- 外围系统适配器注册、端点配置、鉴权验签、幂等控制、审计追踪的基础能力。
- 面向合同主档、文档中心、流程引擎的统一投影与承接入口约束。
- 联调所需的环境连通、回调域名、凭证引用与最小样例报文准备清单。

#### 前置依赖

- 上游正式文档边界已固定。
- 企业微信测试账号与各外围系统测试环境信息已至少完成最小收集。
- 合同主档、文档中心、流程引擎已明确各自的承接边界。

#### 完成标志

- 新增外围系统接入时无需绕开集成主线单独开入口。
- 入站、出站、回调都已有统一平台侧入口与治理约束。
- 联调前不再存在“先接哪个系统就先写一套专用直连逻辑”的实施风险。

### 5.2 第二阶段：入站承接与绑定建立

#### 目标

- 先打通外部事实进入平台后的正式承接路径。
- 建立平台对象与外部对象、实例、组织、人员之间的稳定绑定关系。

#### 核心产出

- `CRM` / `SF` / `SRM` / `SAP` 的通用入站承接流程与合同类、主数据类最小落地路径。
- 企业微信组织 / 人员身份同步的承接路径与平台身份映射。
- `OA` 审批桥接前置绑定能力，包括平台对象与外部审批实例关系管理。
- 入站受理结果、待承接状态、待人工确认状态与绑定状态的统一处理口径。

#### 前置依赖

- 第一阶段统一接入与适配基础能力完成。
- 合同主档已能承接外部合同事实或建立业务绑定。
- 文档中心已能承接外部文件对象。
- 流程引擎已能承接外部审批摘要或实例绑定。

#### 完成标志

- 外部主数据或业务事实进入平台后，可以落到正式承接路径，而不是停留在报文层。
- 平台对象与外部对象已建立可追踪绑定，后续出站与回调可以基于同一绑定关系闭环。
- 企业微信身份同步、`OA` 实例绑定、业务系统业务主键绑定口径一致。

### 5.3 第三阶段：出站派发与回调闭环

#### 目标

- 在已有入站与绑定基础上，补齐平台向外发送业务事实、等待外部处理、承接回调并回写平台真相源的闭环。
- 形成 `OA`、企业微信、业务系统的可联调主链路。

#### 核心产出

- 面向 `OA` 的审批桥接出站、审批摘要承接与回调处理闭环。
- 面向企业微信的登录换取、消息触达、轻量动作回执闭环。
- 面向 `CRM` / `SF` / `SRM` / `SAP` 的合同事实、文档事实或主数据反馈出站能力。
- 统一回调受理、去重、顺序校验、结果回写与异常挂起机制。

#### 前置依赖

- 第二阶段入站承接与绑定建立完成。
- 平台对象对外投影口径已可稳定生成。
- 各外部系统最小联调样本与回调开通条件已就绪。

#### 完成标志

- `OA` 默认主审批路径可完成“发起 - 回调 - 平台承接”闭环。
- 企业微信可完成“登录 / 同步 / 触达 / 回执”闭环。
- 业务系统可完成至少一条稳定的“平台出站 - 外部确认 - 平台留痕”闭环。
- 回调不再由业务模块分散接收，全部经由集成主线治理。

### 5.4 第四阶段：运行治理与补偿完善

#### 目标

- 在主链路已经跑通后，补齐失败重试、补偿恢复、对账核查、监控告警、人工介入等运行治理能力。
- 把“能联通”提升为“能稳定运行、能恢复、能审计”。

#### 核心产出

- 集成任务视图、交换审计视图、失败工单或待处理清单。
- 入站失败、出站失败、回调冲突、顺序异常、绑定缺失的补偿与人工处理入口。
- 面向 `OA`、企业微信、业务系统的对账与重放策略。
- 上线后的监控、告警、日志追踪与恢复演练基线。

#### 前置依赖

- 第三阶段闭环链路完成并可稳定复现。
- 关键失败场景与外部系统超时、重复回调、错误回执样本已收集。

#### 完成标志

- 关键链路失败后可定位、可重试、可补偿、可人工接管。
- 平台具备对外部系统交互过程的统一审计与问题追踪能力。
- 上线后不依赖人工翻日志猜测问题归因。

### 5.5 第五阶段：联调、验收与上线准备

#### 目标

- 将前四阶段形成的能力收敛为可交付状态。
- 完成跨系统联调、场景验收、切换方案、回退方案与上线前检查。

#### 核心产出

- 企业微信、`OA`、`CRM` / `SF` / `SRM` / `SAP` 的联调清单与通过记录。
- 覆盖主链路、异常链路、补偿链路的测试结果与验收结论。
- 生产环境配置清单、回调白名单、凭证装配、监控告警、发布窗口与回退预案。
- 上线后首轮观察项、人工值守项与问题升级路径。

#### 前置依赖

- 前四阶段完成标志全部达成。
- 外部系统联调窗口、测试账号、网络开通、回调配置均已落实。
- 平台真相源承接链路在测试环境已稳定通过。

#### 完成标志

- 外围系统集成主线具备正式上线条件。
- 联调结果、验收口径、上线操作与回退策略完整可执行。
- 已形成第一版可用于生产运行的主线级实施基线。

## 6. 模块推进顺序

本主线推进顺序固定为：统一接入与适配 -> 入站与绑定 -> 出站与回调 -> 运行治理。

原因如下：

- 先统一接入与适配，是为了先把平台侧唯一正式入口立住，避免企业微信、`OA`、业务系统各接一套，后面再返工统一。
- 再做入站与绑定，是因为平台必须先具备承接外部事实、建立跨系统绑定的能力，出站和回调才能知道“要发给谁、收到后回写到哪”。
- 然后做出站与回调，是因为没有入站与绑定，出站只会形成一次性发送，无法保证回调能回到合同主档、文档中心或流程引擎。
- 最后补齐运行治理，是因为补偿、对账、重放、告警都必须建立在真实链路已经跑通、真实异常模式已经出现的前提上，过早设计会失真，过晚补齐会直接影响上线稳定性。

对应模块实施顺序建议如下：

1. 统一适配层、统一交换入口、统一审计追踪骨架
2. 企业微信身份同步与移动入口承接
3. `CRM` / `SF` / `SRM` / `SAP` 入站承接与对象绑定
4. `OA` 审批桥接绑定与审批摘要承接
5. `OA` / 企业微信 / 业务系统的出站与回调闭环
6. 任务治理、补偿恢复、对账与上线前运行保障

## 7. 外部联调准备

### 7.1 企业微信最小联调准备

- 测试账号、应用标识、回调配置、可信域名与测试终端准备完成。
- 明确原生能力承接范围与小程序兜底范围。
- 准备登录、身份同步、消息触达、轻量动作回执的最小联调样例。

### 7.2 `OA` 最小联调准备

- 测试环境、接口凭证、审批发起入口、审批结果回调与实例查询能力可用。
- 明确默认主审批路径的适用范围，以及不能满足时切回平台流程引擎的判定口径。
- 准备最小审批样例，包括发起、处理中、完成、驳回、异常回调几个基础场景。

### 7.3 `CRM` / `SF` / `SRM` / `SAP` 最小联调准备

- 为每个系统准备至少一组可重复使用的业务样例与主数据样例。
- 明确业务主键、幂等键、失败回执与重试约定。
- 明确平台侧承接对象、绑定对象与允许的最小字段集。

### 7.4 联调准备边界

- 联调准备只覆盖平台侧接入所需条件、样例与环境，不展开外围系统本体改造计划。
- 如外围系统侧存在改造需求，应作为外部依赖记录，不并入本主线实施责任。

## 8. 测试、验收与上线准备

### 8.1 测试重点

- 统一入口测试：验证业务模块未绕过集成主线私接外部入口。
- 入站测试：验证鉴权、验签、幂等、绑定、待承接、待人工确认等处理结果。
- 出站测试：验证投影生成、目标路由、同步结果、异步等待与重试行为。
- 回调测试：验证去重、顺序冲突、状态回写、异常挂起与审计留痕。
- 治理测试：验证失败补偿、人工介入、对账、重放、告警与恢复路径。

### 8.2 验收重点

- 企业微信可作为正式移动承载端之一稳定运行，且复杂业务不回桌面 `Web` 兜底。
- `OA` 作为默认主审批路径可稳定桥接，且平台保留正式承接能力。
- `CRM` / `SF` / `SRM` / `SAP` 的外部事实可被平台承接，但不改写平台真相源边界。
- 合同主档、文档中心、流程引擎三类真相源承接关系清晰，未被集成链路冲散。
- 关键异常场景具备可追踪、可补偿、可恢复能力。

### 8.3 上线准备

- 确认生产环境端点、白名单、凭证引用、回调地址、日志与监控已按正式环境配置。
- 确认首批上线系统范围、灰度顺序、切换窗口、问题升级路径与回退方案。
- 确认上线后首轮对账、异常值守、回调监测、失败补偿与人工介入安排。
