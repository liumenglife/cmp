# 外围系统集成主线专项设计：签名与票据安全治理

## 1. 文档说明

本文档用于继续下沉 `integration-hub` 主线中“签名与票据安全治理”能力，重点回答以下问题：

- `OA`、企业微信及统一适配层中的签名算法应如何统一建模与分层治理
- 系统级票据换取、续期、失效、回调验签与重放防护应如何落地
- 密钥托管、证书轮换、灰度切换、审计追溯、失败补偿与应急降级边界应如何收口

本文是以下文档的下游专项设计：

- [`integration-hub Detailed Design`](../detailed-design.md)

`workflow-engine`、`identity-access`、`contract-core`、`document-center` 是本专项设计的强依赖或关键消费方：前者消费 `OA` 审批桥接与回调可信结论，后者分别消费主体绑定结果、合同桥接结果与附件交换安全上下文。但这些关系属于跨模块依赖关系，不属于本文档的父文档关系，也不构成本文档的上游来源。

本文不展开以下内容：

- 不重写外围系统对外 `API` 路径、请求体、响应体、错误码与字段样例
- 不写联调脚本、实施排期、责任分工、上线步骤或运维操作手册
- 不写具体类名、SDK 调用方式、数据库索引、配置键名或代码实现细节
- 不把外部系统本体改造方案、外部证书平台改造或外部账号运营流程写成 `CMP` 内部责任

## 2. 设计目标

- 为 `OA`、企业微信与 `integration-hub` 统一适配层建立一套可复用的签名与票据安全框架，避免每个适配器各自维护口令、签名串和换票逻辑
- 保证所有入站、出站、回调链路都能回答“谁发起、凭什么可信、何时失效、是否被重放、如何追溯”
- 将密钥、票据、证书、签名材料与业务投影解耦，避免业务模块直接感知安全细节
- 在企业内网 `Docker Compose` 部署前提下，提供可执行的托管、轮换、灰度和应急降级边界
- 把失败补偿限制在 `integration-hub` 自身边界内，不把安全补偿误写成跨系统事务回滚或外部系统本体修复

## 3. 适用范围与重点对象

本专项设计覆盖 `integration-hub` 中所有需要系统级签名、验签、票据换取、票据刷新、证书轮换与回调可信性判断的链路，包括入站、出站、回调、对账与人工重放前的安全校验。

治理重点如下：

- `OA`：重点治理审批桥接请求签名、系统级会话票据、回调验签、审批回放前的可信性校验
- 企业微信：重点治理网页登录/小程序会话换取、服务端 `access token` 类票据、回调消息签名校验、消息发送签名与票据续期
- 统一适配层：重点治理系统注册、签名模型抽象、密钥引用、证书版本切换、审计与恢复边界

统一原则如下：

- `OA`、企业微信是重点系统，但签名与票据安全框架归属始终在 `integration-hub`
- 其他外围系统后续可复用同一安全框架，但不作为本文档同等重点展开
- `identity-access`、`workflow-engine`、外部系统都不是本文档父文档，也不改变 `integration-hub` 的归属边界

## 4. 统一安全模型

### 4.1 分层模型

签名与票据治理统一拆为四层：

1. `SecurityProfile`：定义某个外部系统、某类端点、某个方向的安全模式、算法族、票据模式与生效窗口
2. `SigningContext`：描述一次请求或回调在签名时使用的规范化串、密钥引用、证书版本、时间戳、`nonce` 与摘要结果
3. `TicketContext`：描述一次票据换取或刷新得到的票据类型、作用域、失效时间、缓存状态与来源证据
4. `VerificationResult`：描述验签、验票、重放检查、时钟偏差判断与最终可信结论

四层职责必须分离：

- `SecurityProfile` 负责定义“某系统应该怎么安全接入”
- `SigningContext` 负责定义“这次签名实际用了什么材料”
- `TicketContext` 负责定义“这次换到的票据是什么、何时失效”
- `VerificationResult` 负责定义“平台最终是否接受这次请求或回调”

### 4.2 核心对象

统一治理中至少存在以下概念对象：

- `system_name`
- `endpoint_code`
- `direction`：`INBOUND`、`OUTBOUND`、`CALLBACK`
- `signature_mode`：如摘要签名、共享密钥签名、证书签名、平台包裹签名
- `ticket_mode`：如静态凭证换取、短期访问票据、会话交换票据、免票据模式
- `key_ref`
- `certificate_version`
- `nonce`
- `issued_at` / `expires_at`
- `ttl_seconds`
- `clock_skew_tolerance`

统一约束如下：

- 所有签名与验签必须绑定 `system_name + endpoint_code + direction + security_profile_version`
- 所有票据必须绑定明确作用域，不允许一个票据默认覆盖多个不相关能力面
- 签名材料、票据缓存与业务投影结果不得混存于同一逻辑对象
- 历史交换记录必须保留命中的安全版本、证书版本与验签结论，便于审计解释
- 父文档主表与审计对象层至少在 `ih_inbound_message`、`ih_outbound_dispatch`、`ih_callback_receipt`、`ih_integration_audit_event` 正式承接 `security_profile_version`、`certificate_version`、`verification_result`；其中 `verification_result` 在入站、回调与审计事件中表示正式验签结论，在出站记录中表示发送前安全校验与对端同步安全接受结论的统一结果字段

## 5. 签名模型

### 5.1 规范化签名串

统一签名模型不直接依赖单一供应商协议，而是先生成平台侧规范化签名串，再由各适配器映射到目标协议。规范化签名串至少包含：

- 请求方法或事件类型
- 逻辑资源标识或 `endpoint_code`
- 规范化查询参数与主体摘要
- `timestamp`
- `nonce`
- `ticket` 或等效短期凭证摘要（如协议要求）
- `body_hash`
- `key_id` 或证书版本标识

统一要求如下：

- 同一系统同一方向必须固定字段顺序与编码规范，避免验签口径漂移
- 原始敏感值不进入业务日志，只记录摘要、版本和引用
- 若外部协议要求专有签名串，平台仍需保存“平台规范化串摘要 + 外部协议串摘要”两份证据

### 5.2 算法治理原则

- 平台按“算法族 + 参数画像”治理，不在设计层把自己绑死到某一个具体供应商写法
- `OA`、企业微信可各自声明不同算法画像，但必须落到统一 `SecurityProfile`
- 允许同一系统并存老版本与新版本算法画像，用于轮换和灰度切换
- 禁止适配器私自拼接签名串、私存共享密钥或绕过统一验签组件

### 5.3 回调验签

回调验签除校验签名本身外，还必须同时校验：

- 回调来源系统与 `endpoint_code` 是否匹配
- `timestamp` 是否落在可接受时钟偏差窗口内
- `nonce` 是否已被使用
- 如协议包含票据或回调密钥版本，是否命中当前可接受版本集合
- 关联 `dispatch_id`、外部回执号或绑定关系是否可被定位

验签结果统一分为：

- `VERIFIED`
- `REJECTED_SIGNATURE`
- `REJECTED_EXPIRED`
- `REJECTED_REPLAY`
- `REJECTED_CONTEXT_MISMATCH`
- `WAIT_MANUAL`

## 6. 票据交换模型

### 6.1 票据类型分层

统一把票据分为三层：

1. `BootstrapCredential`：静态托管的初始凭证，只用于换取短期票据或建立首次信任
2. `AccessTicket`：系统级短期访问票据，用于出站调用或消息发送
3. `SessionTicket`：与用户动作、网页登录或小程序会话相关的短期票据，用于换取平台受控会话

治理要求如下：

- 上层票据不得反向降级为长期凭证存储
- `SessionTicket` 失效不影响 `AccessTicket` 生命周期，两者分别治理
- 所有票据必须有明确 `ttl`、作用域、刷新条件与失效原因

### 6.2 票据换取流程

统一票据换取流程分为五步：

1. 读取 `BootstrapCredential` 或前序有效票据引用
2. 生成本次换票的 `SigningContext`
3. 调用外部换票端点并记录 `ticket_exchange_trace`
4. 形成 `TicketContext`，写入安全缓存与审计摘要
5. 按剩余 `ttl`、失败类型和系统策略决定是否预刷新、按需刷新或禁止继续使用

系统侧重点如下：

- 企业微信：重点区分系统级访问票据与前端临时登录票据，禁止把前端换到的短票据直接当作平台长期会话
- `OA`：重点区分系统桥接票据与人工联调临时票据，生产链路只承认受控系统票据

### 6.3 缓存与并发控制

- 票据缓存必须以 `system_name + ticket_scope + tenant_context + credential_version` 作为最小键
- 同一缓存键只允许一个活跃刷新任务，避免雪崩式并发换票
- 票据剩余寿命进入预刷新窗口时可后台刷新，但旧票据在兼容窗口内可继续服务短时间
- 票据换取失败时，不允许所有业务请求同步击穿外部换票端点

## 7. 密钥分层与托管边界

### 7.1 密钥分层

统一把密钥材料分为四层：

1. `RootTrustAnchor`：平台信任根或外部证书根引用，只用于建立信任，不参与业务请求直签
2. `SystemCredential`：系统级长期凭证或换票初始凭证
3. `SigningKeyMaterial`：用于请求签名或回调验签的当前有效密钥或证书引用
4. `EphemeralSecret`：短期票据、一次性 `nonce`、临时会话密钥与重放检查材料

分层原则如下：

- 长期凭证、签名密钥、短期票据必须分层托管，禁止混用
- 业务模块只能拿到“已验签后的可信结论”，不能直接读取上述任一层原文密钥材料
- `integration-hub` 统一维护引用与生命周期，适配器只通过受控解析器取用

### 7.2 托管边界

- 长期密钥、共享密钥、证书私钥、换票初始凭证只允许托管在平台受控密钥存储中
- 业务表、审计表、任务表、对象映射表只保存引用、版本、摘要和最后验证时间，不保存明文
- 日志、告警、人工工单只展示脱敏摘要，不展示完整票据与完整签名串
- `identity-access` 可以消费主体可信结论，但不接管外部系统密钥托管责任
- 外部系统证书更新、应用 `secret` 变更若需要人工介入，属于跨系统协作事项，不等于其托管边界转移到外部系统

## 8. 证书轮换与灰度切换

### 8.1 轮换模型

证书或签名密钥轮换统一采用“双版本并行 + 明确退出窗口”模型。

必须同时维护：

- `current_version`：当前主用版本
- `next_version`：待切换版本
- `accept_versions`：回调验签允许接受的版本集合
- `retire_after`：旧版本退出时间

### 8.2 轮换原则

- 出站签名可优先切到 `next_version`，回调验签在灰度期内同时接受 `current_version` 与 `next_version`
- 所有轮换都必须先验证新版本换票、签名、验签链路，再扩大流量
- 轮换失败时，只允许回退版本选择，不允许篡改历史审计记录
- 历史请求、历史回调、历史审计必须保留命中的证书版本，便于事后追溯

### 8.3 灰度切换

灰度维度至少包括：

- 按 `system_name`
- 按 `endpoint_code`
- 按方向 `OUTBOUND / CALLBACK`
- 按租户、组织或特定集成配置组
- 按时间窗口

灰度发布要求如下：

- 新安全版本先做只验不切或影子签名比对
- 企业微信与 `OA` 先在低风险接口放量，再扩展到审批、消息等关键链路
- 灰度期间必须保留快速回退到上一稳定版本的能力

## 9. 时钟偏差、重放防护与 `nonce` / `ttl`

### 9.1 时钟偏差治理

- 所有签名请求、回调事件、换票请求都必须携带可校验的时间戳
- 平台统一维护 `clock_skew_tolerance`，默认按系统配置，不允许由单个适配器私自放宽
- 超出偏差窗口但存在高价值业务上下文的请求，可进入 `WAIT_MANUAL`，不得直接当作成功受理

### 9.2 重放防护

重放防护至少同时依赖以下维度：

- `system_name`
- `endpoint_code`
- `nonce`
- `timestamp_bucket`
- 请求体摘要或回调体摘要
- 外部请求号或回执号

统一要求如下：

- `nonce` 在 `ttl` 窗口内必须唯一
- 相同 `nonce` 对应不同主体摘要时直接判定为重放或篡改冲突
- 已过期请求即便签名正确也不得进入正常承接链路

### 9.3 `nonce` 与 `ttl` 约束

- `nonce` 必须具备足够随机性和一次性语义，不允许使用递增数字或可预测业务主键直出
- `ttl` 必须显式配置到 `SecurityProfile` 或 `TicketContext`，不得隐式写死在适配器中
- 短期用户票据、系统级访问票据、回调验签窗口的 `ttl` 必须分开治理，不共用一个全局值

## 10. 审计追溯、失败补偿与应急降级边界

### 10.1 审计追溯

每次签名、换票、验签、验票、轮换切换都应留下最小可追溯证据，至少包括：

- `trace_id`
- `system_name`
- `endpoint_code`
- `direction`
- `security_profile_version`
- `key_ref` 或证书版本
- `ticket_type` 与 `ttl` 摘要
- `nonce_hash`
- `verification_result`
- `failure_code` 与处理结论

### 10.2 失败补偿

签名与票据安全失败后的补偿只覆盖 `integration-hub` 自身边界：

- 重新换取短期票据
- 切换到可接受的证书或密钥版本重新发起签名
- 在安全窗口内重放此前未送达的出站请求
- 重新执行回调关联、验签或安全上下文修复后重放承接
- 将无法自动修复的记录转入人工恢复工单

明确不在补偿边界内的事项：

- 不承诺自动修复外部系统错误证书配置或外部账号权限问题
- 不承诺绕过验签失败继续推进合同、文档、流程状态
- 不承诺把安全补偿升级为跨系统事务回滚

### 10.3 应急降级边界

允许的应急降级只包括：

- 对非关键出站链路临时停止发送，避免持续消耗失效票据或错误签名
- 对关键回调链路改为“先入审计隔离区、后人工确认”
- 对已知短时换票故障启用受控缓存兜底，但不得突破已过期 `ttl`

明确禁止的降级方式：

- 全局关闭验签
- 接受无时间戳或无 `nonce` 的生产回调
- 使用测试票据替代生产受控票据
- 直接把验签失败请求伪装成业务成功

## 11. 治理落地原则

- 安全配置、签名规则、票据策略、证书版本入口由 `integration-hub` 统一维护，业务模块不私接
- 新增系统优先复用既有 `SecurityProfile`、换票框架和轮换框架，确有必要时才扩展新画像
- 灰度切换、轮换切换、补偿重放都必须先留下审计证据，再决定是否推进业务承接
- `OA` 与企业微信的安全实现可各有差异，但差异必须停留在适配器画像层，不得扩散为业务模块条件分支
