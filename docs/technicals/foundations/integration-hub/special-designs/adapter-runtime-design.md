# 外围系统集成主线专项设计：适配器运行时

## 1. 文档说明

本文档用于继续下沉 `integration-hub` 主线中“适配器运行时”能力，重点回答以下问题：

- `OA`、企业微信、`CRM`、`SF`、`SRM`、`SAP` 各类适配器如何在统一运行时抽象下成立
- 统一生命周期、调用上下文、方向模型、错误归一、幂等与限流隔离应如何收口
- 观察性、恢复边界与人工介入边界应如何限定在 `integration-hub` 内部

本文是以下文档的下游专项设计：

- [`integration-hub Detailed Design`](../detailed-design.md)

`workflow-engine`、`identity-access`、`contract-core`、`document-center` 是本专项设计的强依赖或关键消费方：前者消费 `OA` 审批桥接运行时结论，后者分别消费主体上下文、合同事实投影与文件交换结果。但这些关系属于跨模块依赖关系，不属于本文档的父文档关系，也不构成本文档的上游来源。

本文不展开以下内容：

- 不重写外围系统对外 `API` 路径、请求体、响应体、错误码与字段样例
- 不写实施排期、联调顺序、责任分工、上线步骤与运维操作手册
- 不写具体类名、代码实现、数据库索引、脚本细节或过细类图与时序图
- 不把外部系统本体改造方案、组织运营流程或外部账号准备写成 `CMP` 内部设计责任

## 2. 设计目标

- 为 `OA`、企业微信、`CRM`、`SF`、`SRM`、`SAP` 建立统一适配器运行时壳层，避免每个适配器各自定义生命周期与错误语义
- 保证入站、出站、回调三类方向都能使用同一套上下文、执行阶段、审计口径与恢复入口
- 让业务模块只消费平台统一交换语义，不直接承接外部协议差异、供应商错误码与调用限流策略
- 将系统差异收敛在“系统画像 + 方向画像 + 运行时策略”内，而不是散落在业务链路分支中
- 把恢复边界限制在交换链路、绑定关系、重放与人工修复，不越界承诺外部系统本体修复或跨系统分布式事务

## 3. 适用范围与系统分组

本专项设计覆盖 `integration-hub` 内所有需要通过统一适配器运行时承接的链路，包括：

- 外部事实入站
- 平台事实出站
- 外部结果回调
- 安全前置校验后的重放、补偿、对账与人工恢复动作

各系统在运行时中的角色重点如下：

- `OA`：重点承接审批桥接请求、审批进展回调、实例绑定与审批摘要回写
- 企业微信：重点承接身份换取、组织同步、消息触达、轻量动作回执与移动端渠道状态
- `CRM` / `SF`：重点承接客户、商机、销售订单、合同来源与销售侧业务事实交换
- `SRM`：重点承接供应商、采购协同、采购单据与供应侧合同协作交换
- `SAP`：重点承接财务主数据、编码、凭证摘要、经营单据与状态回写交换

统一范围原则如下：

- 各系统都只是适配器运行时的接入对象，不构成本文档父子文档关系
- 运行时设计只定义统一抽象与治理边界，不写成各系统独立实施手册
- 适配器运行时的归属始终在 `integration-hub`，其他模块只消费归一后的运行结果

## 4. 统一运行时抽象

### 4.1 分层模型

适配器运行时统一拆为五层：

1. `AdapterProfile`：定义某个外部系统在平台中的能力画像、方向画像、认证模式、节流策略与恢复策略
2. `AdapterRuntimeContext`：描述一次交换实际携带的系统信息、方向信息、身份信息、绑定信息、追踪信息与策略快照
3. `AdapterExecutionPlan`：描述本次执行要走的阶段、同步 / 异步模式、回调预期、幂等语义与失败分支
4. `AdapterExecutionResult`：描述一次执行的统一结果，包括接受结论、外部交互结果、平台承接结果与可恢复性判断
5. `AdapterRecoveryHandle`：描述一次失败链路在平台侧可被再次执行、人工修复、对账修正或终止受理的边界

分层职责必须分离：

- `AdapterProfile` 负责定义“这个系统通常应该怎么跑”
- `AdapterRuntimeContext` 负责定义“这次实际带着什么上下文在跑”
- `AdapterExecutionPlan` 负责定义“本次准备按什么阶段执行”
- `AdapterExecutionResult` 负责定义“这次跑完后平台如何理解结果”
- `AdapterRecoveryHandle` 负责定义“失败后平台还能做什么、不能做什么”

### 4.2 核心对象

统一运行时中至少存在以下概念对象：

- `system_name`
- `direction`：`INBOUND`、`OUTBOUND`、`CALLBACK`
- `capability_type`：如 `APPROVAL`、`MESSAGE`、`MASTER_DATA`、`DOCUMENT_REF`、`ORG_SYNC`
- `runtime_mode`：`SYNC`、`ASYNC`、`SYNC_WITH_CALLBACK`
- `profile_version`
- `trace_id`
- `idempotency_key`
- `binding_ref`
- `security_verdict`
- `rate_limit_bucket`
- `recovery_policy`

统一约束如下：

- 所有适配器执行都必须绑定 `system_name + direction + capability_type + profile_version`
- 运行时上下文必须显式带出真相源引用，不能让适配器自行猜测合同、文档、流程归属
- 同一适配器可以支持多个方向与能力，但必须通过同一运行时对象表达，不拆成互不相干的私有流程
- 历史交换记录、任务记录、审计记录必须保存命中的 `profile_version` 与运行时结论，便于追溯与回放
- 父文档主表至少在 `ih_inbound_message`、`ih_outbound_dispatch`、`ih_callback_receipt`、`ih_integration_job`、`ih_integration_audit_event` 正式承接 `profile_version`，避免运行时画像只存在专项设计描述中

## 5. 统一生命周期

### 5.1 生命周期阶段

统一适配器生命周期分为七个阶段：

1. `ACCEPT`：接收请求或恢复触发，建立 `AdapterRuntimeContext`
2. `PRECHECK`：完成来源确认、安全结论注入、方向判定、幂等预判与限流判定
3. `NORMALIZE`：把外部输入或平台投影归一到运行时可消费的规范对象
4. `EXECUTE`：执行真正的外部调用、平台承接、回调转译或结果写入
5. `COMMIT`：写交换主表、绑定关系、任务状态、审计事件与运行摘要
6. `OBSERVE`：产出指标、日志、告警候选与人工介入信号
7. `RECOVER`：在需要时生成重试、补偿、重放、对账或人工恢复入口

统一要求如下：

- 任何系统、任何方向都不能跳过 `PRECHECK` 和 `COMMIT`
- `NORMALIZE` 负责协议差异归一，不能顺带承接业务真相写入决策
- `EXECUTE` 失败后也必须进入 `OBSERVE` 和 `RECOVER`，不能只停留在日志中
- 生命周期阶段是运行时协议，不要求在代码实现上强制一一对应为独立类或独立进程

### 5.2 生命周期与方向的关系

- `INBOUND`：重点在 `ACCEPT`、`PRECHECK`、`NORMALIZE` 与下游承接前的归一判断
- `OUTBOUND`：重点在 `NORMALIZE`、`EXECUTE`、`COMMIT` 与回调预期建立
- `CALLBACK`：重点在 `PRECHECK`、顺序校验、绑定定位、结果转译与摘要回写
- 恢复链路重跑时，允许从某一中间阶段重新进入，但必须保留上一次运行证据，不覆盖历史结论

## 6. 调用上下文模型

### 6.1 上下文组成

每次适配器执行的 `AdapterRuntimeContext` 至少应包含：

- `trace_context`：`trace_id`、父交换记录、来源链路、触发原因
- `system_context`：`system_name`、`endpoint_profile`、`profile_version`、方向能力画像
- `identity_context`：来源身份、操作主体、租户 / 组织范围、可信结论引用
- `binding_context`：合同、文档、流程、用户、部门等已知绑定关系与候选绑定提示
- `delivery_context`：同步 / 异步模式、超时预算、回调预期、限流桶、重试预算
- `payload_context`：原始报文引用、归一报文、投影版本、映射版本摘要
- `recovery_context`：是否首跑、是否重放、关联工单、人工决议、对账补丁标记

### 6.2 上下文使用原则

- 上下文由运行时统一构造并向下传递，业务模块不得自行拼装外部系统上下文
- 安全结论、绑定引用、映射版本、限流策略都应作为上下文显式字段，而不是隐藏在适配器内部状态中
- 同一 `trace_id` 下可以存在多次外部尝试，但每次尝试都需要独立记录尝试号与结果快照
- 人工恢复进入重放时，必须继承原始上下文证据，同时追加新的人工操作上下文，而不是覆盖原上下文

## 7. 方向模型与系统画像

### 7.1 统一方向模型

适配器运行时只承认三种基础方向：

- `INBOUND`：外部系统把业务事实、主数据、状态或轻量动作结果送入平台
- `OUTBOUND`：平台把受治理后的业务事实、消息、审批桥接请求或摘要送往外部系统
- `CALLBACK`：外部系统对平台已发起动作回传处理结果、进展或送达状态

这三种方向共享同一运行时协议，但在以下方面可配置差异：

- 幂等键组成方式
- 顺序控制要求
- 超时与重试语义
- 结果是否需要等待回调闭环
- 是否允许人工修正后继续重放

### 7.2 系统画像差异

- `OA`：以 `OUTBOUND + CALLBACK` 为主，强调审批实例绑定、回调顺序、摘要回写与外部拒绝语义归一
- 企业微信：同时覆盖 `OUTBOUND`、`CALLBACK`、部分 `INBOUND`，强调消息触达、身份换取、组织同步与移动动作回执
- `CRM` / `SF`：以 `INBOUND` 和受控 `OUTBOUND` 为主，强调来源业务事实受理、合同来源绑定与外部业务拒绝归一
- `SRM`：以 `INBOUND`、`OUTBOUND` 为主，强调供应商和采购协同事实交换的幂等与对账
- `SAP`：覆盖 `INBOUND`、`OUTBOUND`、少量 `CALLBACK`，强调高价值主数据与经营单据交换时的顺序、限流与错误分级

统一要求：

- 允许系统画像不同，不允许运行时语义漂移到每个系统各自定义一套状态语言
- 新增系统时，只能扩展画像与策略，不应重新发明生命周期与上下文模型

## 8. 错误归一与结果语义

### 8.1 统一错误分层

运行时统一把错误分为五层：

1. `SECURITY_REJECT`：验签失败、票据失效、来源不可信、重放命中
2. `CONTRACT_REJECT`：外部或平台契约不满足，如缺字段、类型不符、版本不兼容
3. `BUSINESS_REJECT`：业务规则拒绝，如审批拒绝、状态不允许、对象不存在或不可承接
4. `RUNTIME_FAILURE`：网络超时、外部不可用、平台执行异常、资源耗尽
5. `RECOVERY_REQUIRED`：顺序冲突、绑定不完整、需人工补全、需对账修正

统一原则如下：

- 运行时必须先判断错误层级，再决定是否重试、是否告警、是否进入人工恢复
- 外部供应商原始错误码必须保留，但业务侧消费的是平台归一后的错误语义
- `BUSINESS_REJECT` 不默认重试，`RUNTIME_FAILURE` 可按策略重试，`RECOVERY_REQUIRED` 必须产生明确恢复入口

### 8.2 统一结果状态

`AdapterExecutionResult` 统一收口为以下结果状态：

- `ACCEPTED`：平台已接受并进入后续承接或异步链路
- `SUCCEEDED`：本次方向所要求的关键动作已完成
- `WAIT_CALLBACK`：外部已受理，但结果必须等待回调闭环
- `REJECTED`：被安全、契约或业务规则明确拒绝
- `FAILED_RETRYABLE`：运行失败，可自动重试
- `FAILED_MANUAL`：需要人工介入、对账或补丁处理

同一状态可附带不同错误层级与恢复建议，但不允许返回只对单个系统有意义的私有结果术语。

## 9. 幂等、顺序与限流隔离

### 9.1 幂等模型

运行时幂等统一按“方向 + 系统 + 能力 + 业务锚点”建模。

- `INBOUND`：优先使用外部请求号、事件号、主对象标识与事件类型组成幂等键
- `OUTBOUND`：优先使用平台对象标识、派发类型、投影版本与目标系统组成幂等键
- `CALLBACK`：优先使用外部回执号、关联派发、事件序号与回调类型组成幂等键

统一要求如下：

- 幂等判断结论必须写入交换记录，不允许只在内存中短暂判断
- 幂等命中后应区分“重复成功回放”“重复失败重试”“冲突重复请求”三种语义
- 幂等只保证同一运行时语义不被重复承接，不替代业务真相源自身的一致性校验

### 9.2 顺序控制

- `OA` 审批进展、`SAP` 高价值状态回写、部分企业微信事件流需要顺序校验
- 顺序控制基于 `binding_ref + direction + capability_type + event_sequence` 进行
- 缺失前序事件时，进入 `RECOVERY_REQUIRED` 或暂挂，不允许直接用最新状态覆盖全部历史
- 顺序冲突的修复边界在 `integration-hub` 内，通过重放、补齐或人工决议完成

### 9.3 限流与隔离

限流治理以“系统隔离、能力隔离、方向隔离”为原则。

- `rate_limit_bucket` 至少按 `system_name + direction + capability_type` 切分
- `OA` 审批桥接、企业微信消息、`SAP` 主数据交换应具备独立配额，避免单链路耗尽全部外部调用预算
- 单系统雪崩、回调风暴或票据异常时，必须能局部降级，不拖垮其他系统适配器
- 限流拒绝属于运行时显式结果，不允许静默丢弃请求或回调

## 10. 观察性与恢复边界

### 10.1 统一观察性

适配器运行时的观察性至少覆盖三类证据：

- 指标：按系统、方向、能力、状态统计成功率、延迟、重试率、积压量、人工介入量
- 审计：按 `trace_id`、对象锚点、绑定关系、操作主体追踪每次运行结论
- 日志：保留请求摘要、错误分层、恢复动作与限流决策，但不泄露敏感安全材料

统一要求如下：

- 必须能从 `contract_id`、`workflow_instance_id`、`document_id` 或外部对象标识反查全部适配器动作
- 必须区分“外部拒绝”“平台失败”“等待人工处理”“等待回调”四类运行语义
- 告警应以运行时归一结果为准，而不是以供应商原始错误文本直接驱动

### 10.2 恢复边界

适配器运行时允许的恢复动作仅包括：

- 自动重试
- 基于原上下文的安全重放
- 绑定修补后的重新承接
- 顺序补齐后的回放
- 对账差异修正后的再次提交
- 人工终止并保留审计结论

明确不属于运行时恢复边界的内容：

- 不回滚外部系统本体内部事务
- 不改造外部系统流程定义、账号体系或主数据结构
- 不替外部系统补做人工审批、人工建单或业务运营动作
- 不把跨系统最终一致性写成平台可完全控制的分布式事务

### 10.3 人工介入原则

- 人工介入是运行时正式能力，不是日志外的临时操作
- 人工介入必须绑定 `RecoveryTicket` 或等价恢复对象，并记录触发原因、操作人、处理结论与是否允许重放
- 人工决议只能修补平台侧绑定、映射、顺序和承接结论，不能伪造外部系统已成功执行的事实
