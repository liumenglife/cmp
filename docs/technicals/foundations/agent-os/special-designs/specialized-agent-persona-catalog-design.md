# Agent OS 子模块专项设计：专用 Agent 人格目录治理

## 1. 文档说明

本文档用于继续下沉 `agent-os` 主线中“专用 Agent 人格目录治理”能力，重点回答以下问题：

- `Agent OS` 如何把各类专用人格从零散 Prompt 补丁升级为正式配置资产与治理目录
- 人格元数据、适用任务域、工具边界、记忆边界、确认敏感度、委派偏好与模型能力偏好应如何统一表达
- 人格继承、覆盖、冲突、版本演进、停用替换应如何在 `agent-os` 内形成稳定规则

本文是以下文档的下游专项设计：

- [`agent-os Detailed Design`](../detailed-design.md)

`persona-kernel`、`prompt-assembly`、`tool-router`、`provider-abstraction`、`memory-governor`、`human-gate`、`delegation-coordinator` 是本文的强依赖或关键消费方：它们分别消费人格目录中的继承关系、人格补丁、工具域边界、模型能力偏好、记忆域默认值、确认敏感度与委派偏好。但这些关系属于 `agent-os` 内部协作关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不写具体 Prompt 正文、角色设定文案、语气示例或对话模板
- 不重写对外 `API` 路径、请求体、响应体、错误码与字段样例
- 不写配置文件格式、类名、函数签名、脚本、数据库迁移或源码目录组织
- 不把人格发布、灰度、替换写成实施计划、上线手册或运维步骤

## 2. 设计目标

- 让专用人格成为可发布、可审计、可回滚、可停用的正式治理资产，而不是开发期口头约定
- 让每个专用人格都能稳定回答“适用于什么任务、默认能做什么、默认不能做什么、默认如何协作”
- 让专用人格与 Prompt 分层、工具治理、Provider 路由、记忆治理、人工确认、委派调度保持同一套边界语言
- 让人格差异通过元数据与治理规则表达，而不是靠长文本 Prompt 隐式埋入
- 让人格目录采用“职责阶段轴 + 领域能力轴”的双轴模型，且职责阶段优先于领域能力

## 3. 治理范围与核心判断

`Agent OS` 中的专用人格，不应被理解为“写给模型的一段角色设定文案”，而应被理解为“受治理的人格配置资产”。

统一判断如下：

1. 专用人格是目录项，不是自由文本。
2. 专用人格建立在通用人格底座之上，不替换平台根约束与运行时框架。
3. 专用人格的正式价值在于收口任务域、默认能力边界和治理偏好，而不是收口文风。
4. 运行时消费的是“人格配置 + 人格补丁 + 当次装配快照”，而不是一份孤立的人设描述。

因此，`Agent OS` 至少需要治理以下三类对象：

### 3.1 人格目录项

人格目录项回答“平台里正式承认有哪些专用人格、各自归属什么能力域、当前处于什么生命周期状态”。

### 3.2 人格配置画像

人格配置画像回答“某个人格默认适用于什么任务、可使用哪些工具域、默认偏向什么记忆与确认策略、需要什么模型能力档位”。

### 3.3 人格版本与替换关系

人格版本与替换关系回答“这个人格当前是哪一版、适用于哪些任务、是否已废弃、由哪个继任人格承接”。

## 4. 专用人格目录模型

专用人格目录应以统一 `Persona Catalog` 视图治理，每个目录项都必须具备稳定元数据，而不能只保存名字与一段说明。

### 4.1 目录项最小元数据

每个专用人格目录项至少应表达以下字段语义：

- `persona_code`：人格正式稳定标识，供任务、运行、快照、审计与委派引用
- `persona_name`：面向管理与审查的名称
- `persona_status`：如 `DRAFT`、`ACTIVE`、`DEPRECATED`、`DISABLED`、`REPLACED`
- `persona_version`：当前正式版本标识
- `base_general_persona_code`：继承的通用人格底板
- `persona_domain`：所属任务域或专业能力域
- `persona_phase`：职责阶段轴，取值为 `EXPLORE`、`PLAN`、`IMPLEMENT`、`VERIFY`、`GUIDE`、`OPS`
- `owner_scope`：归属 `agent-os` 内部治理域，而不是某个业务模块私有配置
- `change_summary`：本版相对前版的治理差异摘要
- `replacement_persona_code`：停用或替换时的继任人格，可为空

本文统一使用 `persona_code` 作为人格目录、版本、治理、审计、任务、运行、快照与委派关系中的唯一正式标识。

### 4.2 双轴人格目录

专用人格目录采用双轴模型：职责阶段轴优先，领域能力轴收窄。职责阶段轴决定人格在运行时“处于哪类责任边界”，领域能力轴决定人格“面向哪类企业业务问题发挥能力”。

`persona_phase` 的阶段语义如下：

- `EXPLORE`：探索与证据收集阶段，默认只读，只能读取合同、文档、审批、签署、履约、风控与审计证据，不得执行写入或放行动作。
- `PLAN`：方案与路径规划阶段，生成可执行计划、分派建议、确认建议或验证计划，但不直接改变业务对象状态。
- `IMPLEMENT`：执行阶段，可在授权范围内执行业务配置或任务动作，例如发起审批补充、创建签署重试任务、标记履约风险待处理，但不得自证最终通过。
- `VERIFY`：独立验证阶段，默认只读，按 `Verification Agent` 边界检查业务证据包、对象状态和审计链，不消费实现者主观结论。
- `GUIDE`：业务用户引导阶段，面向业务用户、法务、审批运营或管理端用户解释状态、收集补充材料、引导确认路径，不替代正式确认。
- `OPS`：运维审计与恢复阶段，可在确认和审计约束下触发恢复、冻结、回滚、重试或接管动作；高风险恢复必须保留审计并按需升级确认。

阶段边界优先于领域能力：法务审查人格若处于 `EXPLORE` 或 `VERIFY`，仍默认只读；审批运营人格若处于 `IMPLEMENT`，可执行授权的审批补充动作但不得自证；运维审计人格若处于 `OPS`，可触发恢复但必须受审计、确认和权限边界约束。

领域能力轴只能收窄能力，不能放宽阶段边界。可发布的领域 persona 示例包括：法务审查、审批运营、签署运营、履约风控、文档治理、运维审计。领域 persona 只表达专业对象、证据偏好、术语理解和默认风险敏感度，不能把 `EXPLORE` 变成可写、不能把 `VERIFY` 变成实现、不能把 `OPS` 变成无确认恢复。

### 4.3 配置画像字段

除目录元数据外，每个人格还应具备正式配置画像，至少覆盖以下维度：

- 适用任务域
- 默认工具允许域与禁止域
- 默认记忆域与记忆注入偏好
- 默认确认敏感度
- 默认委派偏好
- 默认模型能力偏好
- 可覆盖位点与禁止覆盖位点

配置画像中的结果治理与风险治理统一按以下正式维度表达：

- 结果治理由“目标结果类型偏好”“领域术语理解与结果重心”“可覆盖位点中的结果表达差异”共同表达。
- 风险治理由“任务域中的风险语义”“默认工具允许域与禁止域”“默认确认敏感度”“默认委派偏好”共同表达。

### 4.4 目录项与 Prompt 补丁的关系

目录项本身不是 Prompt 正文；它只声明人格治理语义。真正进入 Prompt 分层体系的是与该目录项绑定的“专用人格补丁资产”。

统一关系如下：

- 人格目录项负责“声明这个人格是什么、能做什么、不能做什么”。
- Prompt 补丁负责“在静态底座之上如何表达差异化行为约束”。
- 目录项版本与 Prompt 补丁版本应可相互引用，但不能混成一个不可拆分对象。

因此，任务、运行、快照、委派、审计对象引用人格时，引用的始终是目录项正式主键 `persona_code`。

## 5. 适用任务域与匹配规则

专用人格目录必须显式回答“什么任务应该命中这个人格”，否则人格只会沦为手工指定标签。

### 5.1 任务域表达

每个专用人格至少应声明以下任务域信息：

- `task_type` 适配范围
- 目标结果类型偏好，如摘要、提取、审查、诊断、计划、委派子任务
- 主要对象域，如合同、文档、环境事件、平台治理对象、外部对接异常
- 风险语义，如偏保守、偏探索、偏证据优先、偏执行收敛

### 5.2 匹配原则

- 任务先匹配通用人格底座，再判断是否需要叠加专用人格
- 专用人格匹配优先基于任务域、对象域、风险语义与工具需求，而不是基于人类输入的模糊昵称
- 若多个专用人格都可命中，优先选择任务域更窄、工具边界更明确、风险偏好更贴近当前任务的一项
- 若当前任务无法被单一专用人格稳定覆盖，应优先转委派，而不是在同一运行里混装多个人格补丁

### 5.3 不匹配时的收口

- 没有合适专用人格时，回退到通用人格底座
- 命中候选之间冲突但无法安全自动收口时，优先保持通用人格或转委派，不强行拼接多个专用人格
- 未发布、已停用、已禁用人格不得参与正式匹配

## 6. 默认能力边界模型

专用人格目录的核心不是“有何专业描述”，而是“默认能力边界是什么”。

### 6.1 默认工具域

每个专用人格都应声明默认工具域，而不是只声明工具名白名单。工具域至少应表达：

- 允许调用的工具类型范围，如 `SEARCH`、`RULE`、`FILE`、`DB`、`HTTP`、`MODEL`、`INTERNAL_SERVICE`
- 禁止触达的高风险工具域或危险动作域
- 对可写工具、外部动作、批量动作是否默认拒绝、默认受限或默认需升级确认

统一原则如下：

- 人格目录声明的是默认能力域，不替代工具治理中的实际授权与四层漏斗判定
- 人格可以收窄默认可用工具域，但不能放宽平台根约束已禁止的工具边界
- 人格默认允许域只是候选授权输入，正式 `ToolGrant` 仍由任务上下文、风险等级与人工授权共同收口

### 6.2 默认记忆域

每个专用人格都应声明默认记忆消费边界，至少包括：

- 优先读取哪些作用域的记忆，如 `TASK`、`SESSION`、`AGENT`、`GLOBAL`
- 是否允许怀疑式记忆进入主注入层，或只能作为待验证观察
- 对长期经验与短期观察的默认权重偏好

统一原则如下：

- 人格只能影响记忆消费偏好，不能绕过记忆治理的可信度、新鲜度与冲突规则
- 高保守人格应优先消费已验证记忆，探索型人格可允许更多待验证观察，但必须显式带标签

### 6.3 默认确认敏感度

默认确认敏感度回答“该人格在面对不确定结果或副作用动作时，默认多早倾向于申请确认”。

建议统一表达为分层档位，而不是自由文本，例如：

- `LOW`：尽量由前三级自动收口，仅在高不确定高风险时进入人工确认
- `MEDIUM`：对正式结果放行、跨域写入、权限例外保持常规敏感
- `HIGH`：对冲突结论、高价值对象、副作用动作更早触发确认

该敏感度只影响默认倾向，不替代 `human-gate` 的正式触发规则。

### 6.4 默认委派偏好

默认委派偏好回答“当前人格更适合独立收敛，还是更适合尽早转给其他专用人格或发起并行子任务”。

建议至少表达以下语义：

- `SELF_CONTAINED`：优先本人格内收敛，除非明显超域
- `EARLY_SPECIALIZATION`：一旦遇到细分任务域，优先转专门人格
- `PARALLEL_EXPLORATION`：允许在受控配额下并行探索
- `HUMAN_FIRST_ON_CONFLICT`：冲突任务更偏向确认或接管，而非继续扩散委派

### 6.5 默认模型能力偏好

默认模型能力偏好不绑定具体厂商或模型名，而只表达能力画像偏好，例如：

- 质量优先还是成本优先
- 长上下文能力是否为常用偏好
- 结构化输出可靠度是否为硬需求
- 延迟敏感还是吞吐敏感

这类偏好是 `provider-routing-and-quota-design` 的输入之一，不是旁路指定具体 `Provider`。

## 7. 人格继承与覆盖边界

### 7.1 继承原则

专用人格必须建立在通用人格之上，且继承关系默认单向向下追加：

1. 平台根约束层
2. 运行时框架层
3. 通用人格层
4. 专用人格目录项声明的补丁层

专用人格只能在被允许的差异位点上追加或收窄，不能重写上层硬约束。

### 7.2 可覆盖位点

专用人格允许覆盖或细化的位点主要包括：

- 任务域适配范围
- 默认工具允许域的收窄
- 默认记忆域优先级
- 默认确认敏感度档位
- 默认委派偏好
- 默认模型能力偏好
- 领域术语理解与结果重心

### 7.3 禁止覆盖位点

以下内容不得由专用人格覆盖：

- 平台根约束中的安全、审计、身份和禁止行为底线
- 运行时框架中的 `QueryEngine Runtime Loop` 状态模型，以及 `P-A-O` 感知 / 行动 / 观察语义阶段
- 工具治理中已明确拒绝的危险动作边界
- 记忆治理中关于冲突、失效、过期和怀疑标签的基础规则
- 人工确认中“安全硬链路第 3 层的后段兜底而非默认主路径”的平台原则

## 8. 人格冲突规则

人格冲突治理的目标不是让多个专用人格同时混装，而是尽早收口为单一可解释选择、正式委派或安全回退。

### 8.1 冲突类型

- 多个人格同时命中同一任务域
- 一个人格的工具允许域与另一人格的确认敏感度倾向明显冲突
- 当前人格默认模型偏好与任务预算或合规边界冲突
- 父运行人格与拟委派子人格在目标责任上发生重叠或互相覆盖

### 8.2 收口原则

1. 平台根约束与运行时框架优先级最高。
2. 通用人格底座高于任何专用人格差异。
3. 任务域更窄、责任更单一的人格优先于覆盖面过宽的人格。
4. 更保守的工具边界与确认边界优先于更激进的默认偏好。
5. 无法安全自动收口时，优先委派或回退通用人格，不在同一运行里叠加多个竞争人格。

### 8.3 冲突后的运行语义

- 可自动收口的冲突，应在 `persona-kernel` 侧完成，不把人格冲突甩给模型自行理解
- 不可自动收口但确需专业差异时，应通过跨 Session 委派切换人格
- 冲突若将影响高风险动作、正式结果放行或跨模块写回，应允许进入人工确认或人工接管边界

## 9. 版本治理

### 9.1 版本对象

专用人格至少需要独立治理以下版本对象：

- 人格目录项版本
- 人格配置画像版本
- 关联 Prompt 补丁版本
- 适用任务域与冲突规则版本

### 9.2 版本原则

- 目录项版本回答“这个人格配置资产的正式定义是什么”
- Prompt 补丁版本回答“该人格如何进入静态底座分层”
- 两者可关联发布，但不应被压扁成一个不可拆版本号

### 9.3 变更类型

人格版本变更至少应区分：

- 元数据修订：不改变能力边界，仅修正说明、标签或引用
- 行为收窄：收紧工具域、记忆域、确认敏感度或委派边界
- 行为扩展：新增适用任务域或新增允许能力，但必须重新评估安全与治理影响
- 替代迁移：旧人格由新人格承接，目录项进入停用或替换态

### 9.4 发布与回滚原则

- 运行时只消费已发布人格版本，不消费草稿目录项
- 回滚优先针对最小受影响人格版本执行，不要求全目录同步回滚
- 既往运行快照必须保留其原始人格版本引用，不因回滚而改写

## 10. 停用与替换策略

### 10.1 停用触发场景

以下情况应支持人格停用或替换：

- 任务域已被更细颗粒人格拆分接管
- 原人格长期高冲突、高误用或高治理成本
- 原人格默认边界已不符合新的 Prompt、工具、记忆或确认治理规则
- 原人格关联的专业判断模型已被证明不再稳定适用

### 10.2 停用原则

- 停用不等于删除；已存在的人格引用、运行快照与审计证据必须保留可解释性
- 进入 `DEPRECATED` 后可继续被既往记录查询引用，但不应再作为默认命中人格
- 进入 `DISABLED` 后不得再被新任务匹配，仅保留追溯意义

### 10.3 替换原则

- 替换必须显式声明继任人格，而不是只修改同名人格内容
- 继任人格应说明与原人格在任务域、工具域、确认敏感度、委派偏好上的差异摘要
- 若替换会改变高风险动作边界或结果放行方式，应同步触发相关治理面的审查

## 11. 与其他专项设计的关系

### 11.1 与 Prompt 分层设计的关系

- 本文定义“人格目录如何成为正式配置资产”
- `prompt-layering-and-versioning-design` 定义“这些人格差异如何进入静态底座补丁与装配快照”

### 11.2 与工具治理设计的关系

- 本文定义人格的默认工具允许域与禁止域
- `tool-contract-and-sandbox-design` 定义正式授权、四层漏斗、沙箱和结果治理

### 11.3 与 Provider 路由设计的关系

- 本文定义人格默认模型能力偏好
- `provider-routing-and-quota-design` 定义这些偏好如何映射到能力画像、主备路由、降级与预算限制

### 11.4 与记忆治理设计的关系

- 本文定义人格默认记忆消费边界与偏好
- `memory-retrieval-and-expiration-design` 定义记忆如何检索、排序、冲突、失效与注入

### 11.5 与人工确认、委派设计的关系

- 本文定义人格默认确认敏感度与默认委派偏好
- `human-confirmation-and-console-design` 与 `delegation-scheduler-design` 分别定义这些偏好如何进入正式确认与调度链路

## 12. 本文结论

`Agent OS` 把专用人格做成正式配置资产与治理目录的关键，不是积累更多“角色文案”，而是建立稳定的人格目录语义：

- 以人格目录项、配置画像、版本与替换关系构成正式人格资产
- 以任务域、工具域、记忆域、确认敏感度、委派偏好、模型能力偏好定义人格默认能力边界
- 以“通用人格在上、专用人格在下”的继承模型守住覆盖边界
- 以冲突收口、版本治理、停用替换保证人格演进可解释、可回滚、可追溯

只有这样，`Agent OS` 才能把各类专用人格纳入统一运行时治理，而不是继续依赖零散 Prompt 文案、隐式经验和不可审计的人格切换。
