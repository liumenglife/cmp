# Agent OS 子模块专项设计：Auto Dream daemon 自演进资产管线与候选质量治理

## 1. 文档说明

本文档用于继续下沉 `agent-os` 主线中“`Auto Dream daemon` 的自演进资产管线”能力，但本文关注的不是算法实现，而是 `Agent OS` 如何把后台自动沉淀出的候选结论，治理为可验证、可灰度、可发布、可回滚的运行时经验资产。候选只是资产前态，正式出口是 `Memory Asset`、`Skill Asset` 或 `Pattern Asset`，不能把自动归纳结果直接写成平台正式真相。重点回答以下问题：

- `Auto Dream daemon` 产出的候选记忆、候选模式、候选优化结论、候选技能应如何收集、归一、聚类与进入候选池
- 候选质量应如何以统一维度建模，避免不同类型候选各自使用不可比较的判断语言
- 候选应如何在晋升、驳回、冻结、人工复核与历史审计之间形成稳定生命周期
- 该治理能力与记忆治理、Prompt 资产、人工确认、委派结果之间的边界应如何收口
- 已验证候选如何进入 `Memory Asset`、`Skill Asset`、`Pattern Asset` 的发布、回滚和退役链路

本文是以下文档的下游专项设计：

- [`agent-os Detailed Design`](../detailed-design.md)

`memory-governor`、`prompt-assembly`、`human-gate`、`delegation-coordinator`、`audit-observability` 是本文的强依赖或关键消费方：前两者分别消费已晋升候选形成正式记忆或正式策略资产，中间两者分别承接高风险候选复核与委派结果证据，最后一者承接候选全生命周期留痕。但这些关系属于 `agent-os` 内部协作关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不重写对外 `API` 路径、请求体、响应体、错误码与字段样例
- 不写聚类模型、排序模型、阈值函数、训练集、评测集或模型实验过程
- 不写作业编排脚本、数据库迁移、源码类图、消息主题或具体实现代码
- 不把候选处理流程写成实施计划、压测方案、上线手册或运营动作说明

## 2. 设计目标

- 让 `Auto Dream daemon` 只负责发现和整理候选，不直接把自动归纳结果写成平台正式真相
- 让候选从资产前态进入自演进资产管线，正式出口明确收敛为 `Memory Asset`、`Skill Asset`、`Pattern Asset`
- 让不同候选类型共享统一质量语言，避免“记忆候选看可信度、优化候选看感觉”的碎片化治理
- 让晋升建立在证据聚合、冲突检查、风险评估和可复用价值之上，而不是建立在单次运行表现之上
- 让驳回、冻结、人工复核和后续再验证都保留正式审计链，避免候选治理成为不可解释黑盒

## 3. 治理范围与候选分类

`Auto Dream daemon` 的治理对象不是“任何自动摘要”，而是后台整理后有潜力影响后续运行质量的候选结论。统一分为四类：

### 3.1 候选记忆

候选记忆用于表达可能值得沉淀为 `memory-governor` 正式对象的事实、模式、失败教训、工具偏好或约束摘要。它的目标是进入记忆治理链，而不是直接进入 Prompt 正文。

### 3.2 候选模式

候选模式用于表达跨运行重复出现的行为规律，例如某类任务更适合的工具组合、某类冲突的常见消解路径、某类委派切分的稳定收敛方式。它首先是运行时治理建议，不天然等于正式平台规则。

### 3.3 候选优化结论

候选优化结论用于表达对 Prompt 裁剪、上下文预算、路由偏好、确认敏感度、委派收敛策略等治理面的优化建议。它可以成为后续策略资产的输入，但不能由 `Auto Dream daemon` 直接发布为正式 Prompt 资产或正式调度策略。

### 3.4 候选技能

候选技能用于表达可复用工作流、步骤、坑点、验证方式和适用条件。它与记忆的边界是：记忆记录事实、偏好、动机和外部引用；技能记录“下次遇到同类任务应如何做”。`Auto Dream daemon` 可以产出候选技能，但不得直接发布为正式技能，必须经过质量画像、人工或治理复核和版本发布。

统一原则如下：

- 候选是治理对象，不是正式真相对象
- 候选可以被复用、被验证、被冻结，但在晋升前不得伪装成正式规则
- 候选的消费去向由类型决定，但质量判断语言保持统一
- 候选只代表资产前态；只有完成验证、灰度、发布和回滚入口登记后，才能成为正式资产

### 3.5 正式资产出口

自演进资产管线的正式出口分为三类：

- `Memory Asset`：沉淀已验证的业务事实、偏好、约束、失败教训或外部引用，例如某租户合同审批例外必须保留法务确认记录、某类签署渠道回执存在延迟窗口、某批履约逾期规则已经被人工确认。
- `Skill Asset`：沉淀可复用企业业务工作流经验，例如高风险条款复核流程、审批例外处理流程、签署失败处置流程、履约逾期风控模式。
- `Pattern Asset`：沉淀可复用运行策略，例如合同风险分析的路由偏好、审批异常的委派切法、签署失败的验证策略、文档比对后的回归检查策略。

`Skill Asset` 至少应包含以下结构语义：

- 元数据：`asset_id`、名称、领域、状态、责任域、来源候选簇、风险等级。
- 适用条件：任务类型、业务对象、租户范围、风险前提、禁止使用条件。
- 步骤：面向业务运行的可复用步骤，而不是实施动作说明。
- 坑点：常见误判、证据缺口、权限陷阱、人工确认遗漏点。
- 验证方法：需要哪些合同 / 文档版本差异、审批 / 签署 / 履约状态、审计事件和回归基线证明命中后可用。
- 版本号：资产版本、依赖的规则版本、替代关系和生效范围。
- 配套资源：业务规则引用、模板引用、检查清单、示例证据包入口。
- 回滚目标：回滚到的上一资产版本、需要撤销的命中记录、需重新验证的业务对象范围。

`Pattern Asset` 至少应包含以下结构语义：

- 可复用策略：候选策略的目标、约束、触发条件和停止条件。
- 路由偏好：哪些任务更适合法务审查、审批运营、签署运营、履约风控、文档治理或运维审计人格承接。
- 委派切法：如何按合同风险域、审批节点、签署渠道、履约责任或证据域拆分子运行。
- 验证策略：命中后应由哪些独立证据验证、是否需要 `Verification Agent`、失败如何进入回归基线。

## 4. 候选来源与收集入口

候选质量治理应建立在多来源证据汇聚之上，而不是单次运行回顾。`Auto Dream daemon` 的正式收集入口至少包括：

- 成功运行中的稳定结果摘要、工具链路摘要与结果回写后反馈
- 失败或部分成功运行中的失败路径、补救动作和异常聚类线索
- `delegation-coordinator` 回收的子运行结果、冲突回收结论与调度快照摘要
- `human-gate` 的批准、驳回、限次放行、要求修改与撤销结果
- `memory-governor` 的冲突扫描、失效、冻结、重复命中与再次验证结果
- Prompt 装配与工具路由中的预算命中、裁剪命中、降级命中与明显收益或退化信号

收集原则如下：

- 只收集已归一化的摘要与引用，不把原始 Prompt、完整模型输出或整包日志直接当候选正文
- 候选必须带来源对象引用，至少能追到任务、运行、结果、确认、委派或既有记忆对象
- 单次来源只能提供“证据片段”，不能单独决定最终晋升结论

## 5. 候选归一与聚类

`Auto Dream daemon` 不应把每次观察都保留为孤立候选，而应先做归一与聚类，把“相似结论的重复证据”收口为候选簇，再对候选簇做质量治理。

父文档中的正式承接应至少包括：`ao_auto_dream_job` 承接后台作业、`ao_dream_candidate_cluster` 承接候选簇生命周期、`ao_candidate_quality_profile` 承接统一质量画像；不再把候选直接表述成“整理后落 `ao_memory_item`”。

### 5.1 归一目标

归一阶段至少统一以下语义：

- 结论类型：记忆、模式、优化结论、技能
- 候选技能归一维度：工作流、步骤、坑点、验证方式与适用条件
- 主题范围：对象域、任务域、工具域、人格域或平台全局域
- 结论方向：正向推荐、负向禁止、条件性建议、待验证观察
- 证据引用：成功证据、失败证据、人工裁决证据、冲突证据

### 5.2 聚类主轴

候选聚类不只看语义相似度，还应同时考虑以下主轴：

- 是否指向同一治理主题或同一对象族
- 是否由相近任务类型、人格组合、工具路径或调度路径产生
- 是否共享相同结论方向，而不是把互相否定的观察聚成一个正向候选
- 是否处于相近策略版本、Prompt 版本、记忆版本背景下

### 5.3 聚类后的治理意义

聚类后的候选簇回答的不是“有没有人这么说过”，而是“是否已经积累到足以形成治理判断的重复证据”。因此：

- 晋升判断优先面向候选簇，而不是单条来源
- 单条强证据可以提高可信度，但仍要经过冲突与风险校验
- 同簇内部若长期出现互相抵触的证据，应优先进入冻结或待复核，而不是强行合并为单边结论

## 6. 统一质量模型

`Agent OS` 需要为四类候选建立统一的 `Candidate Quality Profile`。它不是单一分值，而是由多个维度共同组成的治理画像。核心维度如下：

### 6.1 可信度

可信度回答“这条候选在当前证据下有多大概率成立”。其判断重点包括：

- 来源是否来自成功运行、稳定工具结果或人工确认后的正式裁决
- 是否被多次独立运行重复支持，而不是只由单次偶发样本得出
- 证据链是否完整可追溯，是否能回到明确的来源对象与上下文
- 是否受失败链路、异常工具、上下文缺失或低质量观察污染

### 6.2 复用度

复用度回答“这条候选被晋升后，是否能在未来稳定产生收益”。其判断重点包括：

- 适用范围是单对象、单任务、单人格，还是可跨任务复用
- 是否能减少重复推理、重复工具调用、重复失败路径或重复人工确认
- 是否只是某次局部技巧，还是已经表现出跨场景稳定收益

### 6.3 新鲜度

新鲜度回答“这条候选现在是否仍贴近当前平台状态”。其判断重点包括：

- 最近验证时间、最近命中时间与最近冲突更新时间
- 其依赖的 Prompt 版本、工具策略、外部规则、对象事实是否已经漂移
- 结论是否强依赖短时上下文，若脱离原场景是否迅速失效

### 6.4 冲突度

冲突度回答“这条候选与现有正式真相或其他候选的冲突强度有多高”。其判断重点包括：

- 是否与已发布记忆、已发布 Prompt 资产、已确认人工裁决直接冲突
- 是否与同主题候选簇互相否定且暂时无法自动消解
- 冲突是否属于时序替换、范围差异，还是实质性语义矛盾

### 6.5 风险度

风险度回答“如果错误晋升这条候选，代价会有多大”。其判断重点包括：

- 是否会影响正式结果放行、权限边界、外部副作用或跨模块真相
- 是否会改变高频运行路径、Prompt 根约束或大范围调度策略
- 纠错成本是否高、回滚是否困难、误用后是否易扩散

## 7. 从评分到治理决策的统一语言

候选质量治理不应退化为“算一个总分然后自动处理”。更合适的方式是：先形成多维质量画像，再基于治理规则输出处理结论。

### 7.1 自动晋升

仅当候选同时满足以下特征时，才适合自动晋升到下游治理链：

- 可信度与复用度都达到高位
- 新鲜度仍在可接受窗口内
- 冲突度低，且不存在被正式裁决压制的事实
- 风险度处于可接受范围，错误晋升不会直接破坏高风险边界

这里的“自动晋升”并不意味着直接发布生产真相，而是自动进入其对应的正式治理入口：记忆候选进入 `memory-governor` 正式晋升面，优化候选进入策略评审面，模式候选进入运行时策略候选面。

### 7.2 自动驳回

出现以下情况时，应直接驳回：

- 证据明显不足，且长期未获得重复支持
- 已被人工明确驳回或被更高优先级正式真相覆盖
- 只在失败链路或异常上下文中出现，且无法证明具有独立价值
- 低复用且高噪声，继续保留只会污染候选池

驳回的含义是退出当前候选治理主路径，但保留必要审计记录，避免同样的低质量来源反复进入候选池而无人知晓。

### 7.3 冻结

冻结用于表达“这条候选现在不能晋升，但也不应直接丢弃”。典型场景包括：

- 可信度较高，但与新版本 Prompt、工具策略或正式记忆发生未决冲突
- 复用潜力明显，但当前风险度过高，必须等待人工复核或更多证据
- 同一主题出现多组互相竞争的候选簇，暂时无法安全选边

冻结的候选应退出自动晋升面，但保留重新解冻和再次验证的入口。

### 7.4 人工复核

以下候选不应靠后台自治收口，而应升级到人工复核：

- 涉及高风险动作、权限放大、结果放行策略或大范围 Prompt 约束调整
- 与人工已裁决结论存在显著冲突
- 若晋升将明显改变委派、确认、记忆或结果回写主链路

人工复核回答的是“是否允许进入下一治理层”，而不是让人工直接替代 `Auto Dream daemon` 做聚类和打分。

## 8. 候选生命周期与资产发布状态

候选生命周期建议统一为以下阶段：

1. `COLLECTED`：证据片段已被收集，但尚未完成聚类归一。
2. `CLUSTERED`：已归入候选簇，并形成初始质量画像。
3. `UNDER_REVIEW`：正在等待自动规则或人工复核收口。
4. `PROMOTED`：已进入对应正式治理链，并形成下游正式对象或正式候选入口。
5. `REJECTED`：已确认不再保留为活跃候选。
6. `FROZEN`：暂不允许晋升，但保留后续解冻和再验证可能。
7. `SUPERSEDED`：被更新鲜、更高质量的同主题候选簇替代。

这些状态应由父文档中的候选簇正式对象承接；质量画像的多维判断由独立质量画像对象承接，而不是塞回作业表或正式记忆表。

生命周期原则如下：

- `PROMOTED` 之后仍保留原候选簇，不因晋升而擦除历史判断依据
- `REJECTED` 与 `FROZEN` 都必须记录原因，不能只改状态不留解释
- `SUPERSEDED` 不等于错误，而是说明候选真相已由更新证据接管

候选进入正式资产管线后，应使用资产发布状态表达后续生命周期：

1. `CANDIDATE`：候选仍是资产前态，只能作为待验证输入。
2. `VERIFIED`：已通过独立证据验证，可进入受控灰度。
3. `CANARY`：在限定租户、限定业务对象或限定任务类型中灰度命中。
4. `PUBLISHED`：已发布为正式 `Memory Asset`、`Skill Asset` 或 `Pattern Asset`。
5. `ROLLED_BACK`：因误命中、失败率升高、证据冲突或业务风险被回滚到指定版本。
6. `RETIRED`：因规则失效、业务流程替换或长期无复用价值而退役。

示例：高风险条款复核流程可从候选技能进入 `Skill Asset`；审批例外处理中的委派切法可进入 `Pattern Asset`；签署失败处置中“渠道回执延迟不得直接判失败”的事实可进入 `Memory Asset`；履约逾期风控模式在误告警率升高时必须从 `PUBLISHED` 回滚到上一版本或转为 `RETIRED`。

## 9. 自演进资产指标

自演进资产是否有效，不能只看发布数量。正式指标至少包括：

- 复用率：资产在适用任务中的实际命中占比，以及命中是否集中在合同审查、审批、签署、履约、文档治理等预期场景。
- 命中后成功率：资产命中后任务通过验证、人工确认通过、业务对象状态正确收口的比例。
- 误命中率：资产被触发但后续证明不适用、造成错误建议或增加确认负担的比例。
- 回滚次数：资产因证据冲突、业务规则漂移、误放行或误告警被回滚的次数和影响范围。
- 基准测试结果：在固定合同 / 文档版本差异、审批异常、签署失败、履约逾期样本上的稳定表现。
- 人工否决率：资产命中后被法务、审批运营、签署运营、风控或运维审计人员否决的比例。

这些指标必须能回指资产版本、候选簇、命中任务、验证报告和审计事件，用于决定继续灰度、发布、回滚或退役。

## 10. 与其他治理面的边界

### 10.1 与记忆治理的边界

- `Auto Dream daemon` 负责发现、聚类和初筛候选
- `memory-governor` 负责把候选记忆接入正式记忆生命周期，包括失效、过期、撤销、遗忘与正式检索消费
- 候选质量高不等于已经是正式记忆；正式记忆仍需遵守记忆治理边界

### 10.2 与 Prompt 资产的边界

- 候选优化结论可以指出 Prompt 裁剪、预算、注入顺序或专用人格补丁的优化方向
- 但 `Auto Dream daemon` 不直接发布新 Prompt 版本，也不直接改写静态底座或动态注入资产
- 任何影响正式 Prompt 资产的候选，都只能作为 Prompt 治理输入，而不是旁路发布

### 10.3 与人工确认的边界

- 人工确认处理的是当前运行实例或当前治理判断的放行问题
- 候选质量治理处理的是“未来是否值得复用和沉淀”的平台长期问题
- 只有当候选晋升可能影响高风险边界或与人工裁决冲突时，才需要借助 `human-gate`

### 10.4 与委派结果的边界

- 委派结果是候选证据来源之一，不是天然更高等级的候选真相
- `delegation-coordinator` 负责子运行切分与回收，本文负责这些回收结果如何被后台整理后进入候选治理
- 委派链路中高质量模式可被沉淀为候选模式，但正式调度策略仍需独立治理

## 11. 审计与可追溯性

候选质量治理必须进入 `audit-observability` 统一审计链。至少以下动作应形成正式审计事件：

- 候选证据收集、归一、聚类与候选簇建立
- 质量画像生成、更新、降级与被新证据重算
- 晋升、驳回、冻结、解冻、替代与人工复核结论
- 候选被哪个下游治理面消费，以及消费后形成了什么正式对象或正式评审入口

系统至少应能追答以下问题：

- 某条正式记忆或正式策略最早来自哪个候选簇
- 某个候选为什么被晋升、驳回或冻结
- 候选使用了哪些来源证据，是否包含人工裁决或委派回收证据
- 候选质量判断是在什么 Prompt / 策略 / 记忆版本背景下做出的

## 12. 本文结论

`Auto Dream daemon` 的候选质量治理，本质上是一套自演进资产管线，而不是单一打分函数。其核心收口如下：

- 先把候选记忆、候选模式、候选优化结论、候选技能统一收口为正式候选对象，而不是让自动摘要直接进入平台真相
- 把候选明确定位为资产前态，正式出口收敛为 `Memory Asset`、`Skill Asset`、`Pattern Asset`
- 先聚类证据，再做质量画像，避免单次运行决定长期沉淀
- 以可信度、复用度、新鲜度、冲突度、风险度形成统一质量语言，用于驱动晋升、驳回、冻结与人工复核
- 把记忆治理、Prompt 资产、人工确认、委派结果都视为协作边界，而不是父子文档关系或旁路写入通道
- 让每一次候选治理判断都可追溯、可解释、可回查，确保 `Agent OS` 对后台自动沉淀出的候选结论具备正式质量治理能力
- 候选技能不得由 `Auto Dream daemon` 直接发布为正式技能，必须经过质量画像、复核与版本发布
