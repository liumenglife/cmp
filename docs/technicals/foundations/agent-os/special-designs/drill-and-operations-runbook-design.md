# Agent OS 子模块专项设计：演练与运维工作手册设计

## 1. 文档说明

本文档用于继续下沉 `agent-os` 主线中“正式演练场景与运维工作手册”能力，重点回答以下问题：

- `Agent OS` 如何把故障演练、恢复演练、人工接管与值守协同收口为正式运行保障资产
- 演练场景、故障注入范围、恢复对象、证据留存与恢复后核查应如何形成统一边界
- 运维工作手册应覆盖哪些正式运行职责，哪些内容应明确留在实施、排班或脚本层之外

本文是以下文档的下游专项设计：

- [`agent-os Detailed Design`](../detailed-design.md)

`runtime-core`、`tool-router`、`provider-abstraction`、`memory-governor`、`human-gate`、`delegation-coordinator`、`audit-observability`、`Auto Dream daemon` 是本文的强依赖或关键消费方：它们分别提供运行推进、工具与模型退化、记忆与候选治理、人工确认、委派回收、审计留痕与后台整理等正式运行对象，供演练与运维手册消费与约束。但这些关系属于 `agent-os` 内部协作关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不重写对外 `API` 路径、请求体、响应体、错误码与字段样例
- 不写具体 `shell` 脚本、压测命令、排班表、值班群编排或上线步骤
- 不把故障处理写成实施计划、人员组织方案、代码实现说明或平台部署手册
- 不写类名、函数签名、数据库迁移、消息主题、目录结构或源码细节

## 2. 设计目标

- 让演练与运维从“故障时临时查人、临时拼经验”升级为 `Agent OS` 正式承认的运行保障资产
- 让正式对象、正式状态、正式审计链成为演练和处置的共同语言，而不是依赖单点口头经验
- 让值守、接管、恢复、复盘和证据留存都围绕 `task`、`run`、`result`、`confirmation`、`delegation`、`memory`、`audit` 等正式对象展开
- 让演练边界与验证 / 性能专项互补：前者回答“运行中怎么保、出事后怎么收”，后者回答“发布前怎么证、容量上怎么判”

## 3. 运行保障资产的收口原则

`Agent OS` 不把演练与运维理解为上线后的附属文档，而是理解为正式运行治理的一部分。只要某项能力会影响运行连续性、故障收敛、责任切换、恢复可信度或事后可追溯性，就必须进入正式运行保障资产。

这里的“运行保障资产”不是泛指几篇说明文档，而是可以被识别、评审、启用、替换和停用的正式治理对象集合；四类资产只是分类面，真正被治理的是其中的单个资产条目。

统一收口为四类资产分类：

1. 演练场景资产：定义哪些正式失效、退化、冲突、暂停和接管场景必须被周期性验证。
2. 处置路径资产：定义当正式对象进入异常态时，应如何判断、升级、隔离、恢复或转人工。
3. 工作手册资产：定义值守角色、检查清单、交接边界、证据要求和恢复后核查口径。
4. 复盘沉淀资产：定义演练与真实故障如何回流到审计、候选治理、记忆治理与后续专项优化。

这四类资产共同回答的是“系统怎么被正式守住”，而不是“谁去手敲哪条命令”。

### 3.1 运行保障资产对象模型

运行保障资产的最小资产单元，是一条可独立治理的“运行保障资产条目”。每个条目都必须能单独回答“在什么条件下，由谁，针对哪些正式对象，按什么判定和证据要求执行什么动作”。

可被视为正式运行保障资产的对象，仅限以下几类条目：

- 演练场景模板条目：描述一个必须周期性验证的正式失效、退化、恢复或接管场景。
- 处置路径模板条目：描述某类异常态的判断条件、分流路径、升级边界、恢复边界与结束条件。
- 工作手册条目：描述某个值守动作、检查项、交接项、核查项或证据项的统一执行口径。
- 复盘沉淀条目：描述某类问题如何回流到后续治理面，形成稳定约束、补充检查项或新增演练要求。

按专题汇编的手册章节、演练包、值守清单或季度演练集，属于资产视图或资产发布包，可以作为引用入口，但不替代条目本身作为正式治理主对象。

### 3.2 运行保障资产准入与判定

任一条目要进入正式运行保障资产，至少必须满足以下准入条件：

- 有稳定的 `asset_id`、资产类型、版本号、状态与责任人，能够被审计和引用。
- 明确关联到正式对象、正式状态或正式能力面，而不是停留在泛化经验描述。
- 明确触发条件、适用范围、前置条件、执行动作、结束条件与所需证据。
- 明确由哪一层值守角色执行、哪一层角色批准升级、哪一层角色负责最终收口。
- 已经过一次正式评审；若是演练场景模板、处置路径模板或关键工作手册条目，还必须至少经过一次桌面推演、演练验证或真实事件验证，证明其判断链和证据链可执行。

一次演练模板或手册条目是否成为正式资产，不由文档作者单独决定，而由“治理裁决层提出 + 平台收口层批准”共同判定：治理裁决层确认该条目已能稳定指导判断与执行，平台收口层确认其适用范围、责任边界和证据要求已足以纳入正式运行治理面。涉及跨主线能力面时，还应征得对应能力负责人确认后方可生效。

### 3.3 运行保障资产生命周期边界

运行保障资产条目至少存在以下生命周期状态：`draft`、`candidate`、`active`、`superseded`、`retired`、`invalid`。

- `draft`：仅表示草拟内容，不能作为正式值守或演练依据。
- `candidate`：已进入评审或验证，但尚未成为正式执行基线。
- `active`：已通过准入，可作为正式演练、正式处置或正式值守依据。
- `superseded`：已有替代条目接管同一治理责任，原条目退出默认使用，但仍保留审计回指。
- `retired`：因能力下线、责任边界消失或场景不再成立而停用，不再要求继续维护。
- `invalid`：被证明判断错误、证据不足或与当前正式对象真相冲突，必须立即停止引用，并触发替换、回退或补建动作。

版本变更与失效边界如下：

- 不改变适用对象、责任边界、判定条件与证据要求的文字澄清，可视为同一资产的小版本更新。
- 只要触发条件、分流路径、恢复标准、证据要求、责任归属任一项发生变化，就必须形成新版本，并保留对上一版本的替代关系。
- 当条目已无法覆盖当前正式对象模型、当前状态机或当前治理链时，应判定为 `invalid`，而不是继续作为“参考说明”悬挂。
- 当新条目完整承接旧条目的治理责任且验证结论更可靠时，旧条目应转为 `superseded`；若原责任面整体取消，则转为 `retired`。

## 4. 正式演练场景边界

正式演练场景应围绕 `Agent OS` 的正式主链路与治理链路组织，而不是围绕底层中间件或单个机器故障零散罗列。

### 4.1 主链路演练场景

必须覆盖以下正式运行场景：

- 任务受理后无法稳定创建主运行，导致 `task_status` 与 `run_status` 脱节
- `QueryEngine Runtime Loop` 卡在 `CONTEXT_GOVERNING`、`TOOL_DECIDING`、`SANDBOX_EXECUTING`、`OBSERVATION_NORMALIZING` 任一阶段，形成僵尸运行或长时间无心跳运行
- 结果已生成但未能完成确认、放行或回写，导致正式结果悬空
- 环境事件持续涌入并改变运行判断，导致主运行反复重入或无法收敛

### 4.2 治理链路演练场景

必须覆盖以下治理场景：

- 工具调用命中权限拒绝、沙箱拒绝、配额命中或高风险动作升级确认
- 主 `Provider` 不可用、连续失败、结构化输出失稳或成本 / 速率异常，触发主备切换、降级或回退
- 记忆召回出现冲突、过期、误召回或候选污染，影响运行解释与结果可信度
- 委派子运行失联、重复回收、冲突回收或父运行长期等待子运行返回
- 人工确认长期未决、反复超时、裁决撤销或确认后恢复失败

### 4.3 恢复链路演练场景

必须覆盖以下恢复场景：

- 服务重启后，未完成运行能否依据正式对象恢复到正确等待态或推进态
- `WAITING_HUMAN_CONFIRMATION`、`WAITING_DELEGATION_RETURN`、`RUNNING` 三类关键中断态能否按对象真相恢复
- 缓存、锁、熔断、热点摘要等派生层全部丢失后，主状态能否从数据库真相源重建
- 部分对象恢复成功、部分对象未恢复成功时，系统能否显式收口为失败、降级或人工接管，而不是留下不可解释中间态

### 4.4 验证制度失效演练

必须覆盖以下验证制度失效场景，确保验证链不是实现链的自我背书：

- 实现者误自证：处理合同高风险条款后，实现 Agent 直接声明“已通过复核”，但缺少独立 `Verification Agent` 的只读验证报告。
- 验证 Agent 未独立：验证运行继承实现者完整上下文或主观结论，导致审批例外被错误视为已授权。
- 验证证据丢失：合同 / 文档版本差异、签署渠道回执、履约状态或审计事件缺失，验证结论仍被放行。
- 失败用例未入回归基线：签署状态误判、履约逾期未告警或高风险条款错误放行后，失败证据未进入后续回归基线。

演练通过标准是：系统能阻断自证放行，要求独立验证证据包，生成失败审计，并把失败用例写入回归基线入口。

### 4.5 工具安全拦截失败演练

必须覆盖以下工具安全场景，确保只读、可写、高风险动作边界真实有效：

- 危险业务动作绕过：审批例外、签署重试、合同状态放行或履约风险关闭绕过人工确认或权限拦截。
- 只读 Agent 获得写权限：`EXPLORE` 或 `VERIFY` 阶段人格意外获得审批处理、签署发起、履约状态变更等写权限。
- 沙箱拒绝未生效：受限工具仍能触达未授权租户、未授权合同版本或管理端配置。
- 权限分类器误放行：高风险条款放行、审批例外执行、签署状态改写被误判为低风险只读动作。

演练通过标准是：危险动作被拒绝或升级确认，拒绝原因可审计，业务对象状态未被无授权改写，误放行样本进入权限分类器回归。

### 4.6 压缩失败 / 记忆冲洗失败演练

必须覆盖以下上下文压缩与记忆冲洗场景，确保长链路运行不会丢失关键业务状态：

- Pre-compression memory flush 失败：合同审查、审批异常或签署排查进入压缩前，关键业务事实未先冲洗到正式记忆或证据摘要。
- 摘要丢失关键业务状态：压缩后遗漏高风险条款、审批例外原因、签署失败回执、履约逾期天数或责任方。
- 工具调用 / 结果对被压坏：合同版本比对结果、审批状态查询结果、签署渠道回执与后续结论脱钩，导致恢复后无法解释。

演练通过标准是：压缩前后对象状态、证据引用、工具观察和审计事件可互相回指；缺失时必须暂停放行并转补证或恢复路径。

### 4.7 自演进误发布演练

必须覆盖以下自演进资产误发布场景，确保候选不会污染正式运行：

- 错误 Skill 发布：错误的高风险条款复核流程发布为 `Skill Asset`，导致合同风险被错误放行。
- 错误 Pattern 污染调度：审批异常调查的错误委派切法发布为 `Pattern Asset`，导致审批例外被误执行或反复派发。
- Memory 误晋升：签署渠道回执延迟被误晋升为稳定事实，导致签署状态长期误判。
- 回滚失败：履约逾期风控模式误告警率升高后，无法回滚到上一资产版本或无法撤销命中影响。
- 误命中率升高：某项合同审查、审批、签署或履约资产在新业务规则下持续误命中，但未触发冻结、回滚或退役。

演练通过标准是：误发布资产可被定位到候选簇、资产版本和命中任务；系统能冻结命中、回滚版本、保留审计，并把误命中样本回流到自演进资产指标。

## 5. 故障注入范围

本文所说故障注入，是为了验证正式运行治理边界，而不是为了穷举所有基础设施事故。

### 5.1 允许纳入专项设计的注入面

- 运行时推进异常：状态迁移卡滞、检查点停更、循环次数异常增长、结果收敛失败
- 工具治理异常：契约失败、授权失败、沙箱拒绝、超时、重试耗尽、结果标准化失败
- `Provider` 异常：主路由失败、熔断触发、备路由接管失败、成本配额耗尽、能力退化
- 记忆异常：检索结果为空、冲突记忆上浮、陈旧记忆误注入、候选记忆误晋升
- 确认与委派异常：确认积压、确认过期、子运行失联、委派回收冲突、重复派发抑制失效
- 审计与恢复异常：关键审计缺失、恢复锚点不完整、指标信号存在但对象链回指断裂

### 5.2 不在本文展开的注入面

- 不展开基础设施级机器扩缩、容器编排、网络拓扑切换与具体中间件故障脚本
- 不展开压测流量制造、基准数据灌入、性能采样脚本和容量实验操作步骤
- 不把生产变更窗口、排班制度、公告流程和跨团队协作流程写成本文正文

## 6. 恢复演练对象与恢复判断

恢复演练必须以正式对象能否回到可解释状态为判断标准，而不是只看进程是否重启成功。

### 6.1 关键恢复对象

- `ao_agent_task`：任务主状态是否仍能表达任务终局与当前责任归属
- `ao_agent_run`：运行是否能恢复到正确阶段、正确等待态与正确失败语义
- `ao_agent_result`：结果是否保持正式结果壳层，不出现“模型原始输出冒充正式结果”
- `ao_human_confirmation`：待确认对象、已决策对象、过期对象能否恢复一致语义
- `ao_delegation`：父子运行关系、回收状态、合并状态能否恢复并继续追溯
- `ao_prompt_snapshot`、`ao_tool_invocation`、`ao_agent_audit_event`：证据链是否仍可支撑解释与恢复判断

### 6.2 恢复成立的正式标准

- 恢复后主状态以数据库正式对象为准，派生层可重建但不得反向改写真相源
- 恢复后每个待处理对象都必须有清晰下一动作：继续推进、等待人工、等待子运行、终止、降级或接管
- 恢复后不得出现“对象还在，但责任主体不明”“结果存在，但不可判断是否可放行”的悬空态
- 恢复后必须能通过审计事件与最近快照解释“恢复前发生了什么、恢复后为什么这么判”

## 7. 运维工作手册边界

运维工作手册在 `Agent OS` 中的职责，不是替代实现文档或人力制度，而是收口正式运行中必须被一致执行的治理动作。

工作手册应覆盖以下边界：

- 运行观察：看哪些正式对象、哪些状态、哪些异常聚类、哪些关键比率
- 异常分流：什么问题属于继续观察，什么问题必须进入告警、升级、确认或接管
- 恢复判断：什么时候允许自动恢复，什么时候必须人工复核后恢复
- 接管边界：什么时候由值守人继续等待，什么时候交给人工确认处理人，什么时候升级为人工接管
- 证据闭环：每次处置必须留下哪些审计、摘要、结论和后续动作锚点

工作手册不应覆盖以下内容：

- 不替代具体部署系统操作手册
- 不替代组织排班表、人员绩效或行政流程
- 不把临时技术实现细节写成平台长期正式边界

## 8. 值守分层与责任切换

`Agent OS` 的值守不应只有一个模糊“运维值班”角色，而应按判断职责分层，避免把所有问题都推给同一层处理。

建议至少划分三层值守责任：

1. 运行观察层：负责识别任务积压、异常态增长、关键等待态超时、恢复任务堆积、审计断链信号。
2. 治理裁决层：负责处理需要判断是否继续、是否降级、是否转确认、是否触发限流或人工接管的事项。
3. 平台收口层：负责处理跨对象、跨链路、跨恢复边界的系统性问题，并决定是否冻结某条能力路径、切换正式运行策略或暂停部分能力面。

分层原则如下：

- 观察层优先基于正式指标与对象状态发现问题，不直接越权改写治理结论
- 治理裁决层优先使用正式对象语义做收口，不通过口头约定跳过确认或审计
- 平台收口层处理的是系统性风险，不代替前两层做日常逐单判断

## 9. 检查清单

运行检查清单应围绕正式对象与正式边界组织，至少包括以下几类。

### 9.1 日常健康检查

- 任务受理量、运行推进量、终态分布是否与近期稳定基线显著偏离
- `WAITING_HUMAN_CONFIRMATION`、`WAITING_DELEGATION_RETURN`、长时间 `RUNNING` 是否异常积压
- `Provider` 主备切换、熔断次数、降级次数、配额命中是否持续升高
- 审计写入、关键快照、工具调用摘要是否完整可回指

### 9.2 异常时即时检查

- 问题是否局限于单一任务 / 单一人格 / 单一能力类型，还是已扩散到系统面
- 当前异常是否已有正式降级路径或人工确认 / 接管入口
- 是否存在结果悬空、确认悬空、委派悬空或恢复悬空对象
- 是否已有足够证据支持继续自动推进，否则是否应立即收口为暂停或接管

### 9.3 恢复后核查

- 恢复对象状态是否与审计链、等待态、结果态保持一致
- 先前积压对象是否重新进入稳定推进，而不是批量转为新的异常态
- 主备切换、降级、接管或回放动作是否留下正式审计与原因摘要
- 同类异常是否仍持续出现，若仍在持续则本次恢复只可视为临时收敛，不可视为正式关闭

## 10. 常见异常处置路径

本文不写具体操作命令，但必须收口异常判断路径。

### 10.1 运行卡滞

- 先判断是否为单运行卡滞还是阶段性系统卡滞
- 再判断最近检查点、最近工具调用、最近确认单、最近委派对象是否足以解释停滞
- 可恢复则按正式恢复锚点恢复；不可恢复则显式终止、重试或转人工接管

### 10.2 `Provider` 或工具能力退化

- 先判断是否已命中正式主备切换、降级或回退策略
- 若自动策略仍可收口，则值守重点转为观察退化范围与证据完整性
- 若自动策略也失效，必须升级为平台收口动作，例如限制高成本路径、暂停特定能力面或转人工优先路径

### 10.3 确认积压或超时

- 先区分是工作面拥堵、策略误触发过多，还是高风险事项真实增多
- 对已超时对象，不允许视为默认放行，只能终止、降级、重新确认或升级接管
- 若确认链长期成为瓶颈，应回流治理专项，而不是在运行期默认放宽确认边界

### 10.4 委派失联或冲突回收

- 先判断子运行是否仍可恢复、召回或重新派发
- 再判断父运行是否还能维持主线责任，还是已进入长期等待且缺少有效回收路径
- 若冲突结果已影响正式结果放行或危险动作，应升级到确认或接管边界，不允许伪装为已收敛

### 10.5 记忆污染或错误晋升

- 先阻断其继续进入主注入层或继续影响高频运行
- 再通过治理事件、来源对象与最近使用链排查影响范围
- 必要时冻结、撤销或降级相关记忆 / 候选，但不得无痕覆盖历史证据

## 11. 人工接管切换条件

人工接管不是常规便捷路径，而是自动运行与人工确认都不足以稳定收口时的正式切换语义。

至少出现以下情况时，应允许切换到人工接管：

- 自动恢复多次失败，且对象状态已无法保证继续自动推进的可信度
- 确认链反复超时或反复撤销，导致运行责任无法继续由自动路径承接
- 委派冲突、记忆冲突、结果冲突长期未解，且继续自动扩散只会放大不确定性
- `Provider`、工具、记忆、确认多条治理链同时退化，已经影响正式结果放行或正式回写可信度
- 审计证据不足以支撑继续自动运行，需要人工先完成责任收口与事实确认

切换原则如下：

- 接管前必须形成当前状态摘要、未完成事项摘要、风险摘要与证据引用
- 接管后仍以正式对象推进，不允许脱离 `Agent OS` 主对象体系形成黑盒人工处理
- 接管结束后必须把终态、补充证据与后续治理建议写回正式链路

## 12. 演练证据留存与复盘回流

演练若没有证据留存，就不能成为正式运行保障资产；证据包本身也是运行保障资产条目准入、续用、替换和停用的正式判定依据。

每次正式演练或正式故障处置，至少应留存以下证据：

- 演练 / 事件标识、触发时间、影响范围、涉及能力面与对象范围
- 关键状态迁移、降级动作、确认动作、接管动作、恢复动作的审计摘要
- 与本次演练直接相关的 `task_id`、`run_id`、`confirmation_id`、`delegation_id`、`trace_id` 等追踪锚点
- 恢复成立或恢复失败的判定依据
- 需要回流到记忆治理、候选治理、Prompt 治理、委派治理、确认治理或验证 / 性能专项的后续问题摘要
- 本次演练或事件对应的是新增资产、沿用资产、替换资产还是停用资产，以及对应 `asset_id` / 版本号 / 判定人

复盘回流原则如下：

- 真实故障与演练故障都应进入统一复盘语言，不分散为临时会议纪要
- 可沉淀的稳定教训，可进入 `Auto Dream daemon` 候选或正式记忆治理链，但不得跳过质量治理直接写成规则
- 若发现的是验证缺口或性能基线缺口，应回流到验证 / 性能专项，而不是在本文内扩写为测试方案

基于证据的资产治理收口规则如下：

- 若一次演练模板、处置路径或工作手册条目在正式演练 / 真实事件中已完整覆盖触发、判断、动作、证据和恢复核查链，且未暴露关键歧义，可由治理裁决层发起转正，由平台收口层判定进入 `active`。
- 若证据显示现行条目存在漏判、误判、责任不清、证据不足或恢复标准失真，应将现行条目标记为 `invalid` 或 `superseded`，不得继续以“经验可参考”名义视作正式资产。
- 若复盘结论只是局部文字澄清，不改变正式治理责任，可更新当前版本；若改变判断路径或责任边界，必须生成替代版本，并显式记录替换关系。
- 若某类场景连续多个周期不再存在，或对应能力面已退出正式运行范围，可由平台收口层判定条目转为 `retired`，并保留最后一次有效证据与停用原因。

## 13. 与验证 / 性能专项的边界

本文与“验证与性能设计”互为补充，但边界必须清晰。

- 验证 / 性能专项回答：正式验证面怎么定义、测试矩阵如何组织、负载模型怎么验证、退化如何判定。
- 本文回答：当系统进入正式运行后，哪些演练场景必须存在、故障如何收口、值守如何分层、接管如何切换、证据如何留存。
- 压测、基线、容量、回归判定属于验证 / 性能专项，不在本文展开。
- 真实运行中的故障演练、恢复演练、值守检查、异常处置与接管边界属于本文，不在验证 / 性能专项内展开。

## 14. 本文结论

`Agent OS` 把演练与运维工作收口为正式运行保障资产的关键，不是提前写好几份脚本，而是先定义一套稳定的运行保障语言：

- 以演练场景资产、处置路径资产、工作手册资产、复盘沉淀资产构成正式运行保障面
- 以任务、运行、结果、确认、委派、记忆、Prompt 快照、工具调用、审计事件作为演练与处置的共同对象语言
- 以值守分层、检查清单、异常处置路径、人工接管条件和恢复后核查项收口正式运行责任边界
- 以证据留存与复盘回流把演练和真实故障沉淀为后续治理输入，而不是停留在一次性操作经验

只有这样，`Agent OS` 才能把“演练”和“运维”从零散动作提升为平台正式运行保障能力。
