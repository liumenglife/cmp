# Agent OS 子模块专项设计：验证与性能设计

## 1. 文档说明

本文档用于继续下沉 `agent-os` 主线中“正式验证面与性能验证面”能力，重点回答以下问题：

- `Agent OS` 应如何定义正式验证层次、测试矩阵与通过标准，而不是只在实现阶段临时补测试
- `Agent OS` 的功能验证面、治理验证面、可观测性验证面与性能验证面应如何形成统一边界
- 性能基线、负载模型、容量边界、退化判定、回归策略与压测数据准备边界应如何正式收口

本文是以下文档的下游专项设计：

- [`agent-os Detailed Design`](../detailed-design.md)

`runtime-core`、`tool-router`、`provider-abstraction`、`memory-governor`、`human-gate`、`delegation-coordinator`、`audit-observability` 是本文的强依赖或关键消费方：前六者分别提供被验证的运行时主链路与治理能力，最后一者负责承接验证过程所需的证据、指标与退化信号。但这些关系属于 `agent-os` 内部协作关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不重写对外 `API` 路径、请求体、响应体、错误码与字段样例
- 不写具体测试代码、压测脚本、实现排期、人员分工或上线步骤
- 不把故障演练脚本、值守制度、告警处置、故障操作步骤写入本文
- 不写类名、函数签名、数据库迁移、目录结构或源码实现细节

## 2. 设计目标

- 让 `Agent OS` 的验证从“功能能跑”升级为“功能、治理、性能、可观测性都可正式验收”
- 让测试矩阵围绕正式对象、正式链路与正式风险面组织，而不是围绕零散接口或单点脚本组织
- 让性能验证聚焦运行时可承载性、治理链路稳定性与退化可判定性，而不是只看单次模型调用耗时
- 让后续实现、联调、验收和回归都使用同一套验证语言，避免每轮重新定义验证范围

## 3. 正式验证面与性能验证面的定义

`Agent OS` 的正式验证面，不是某一种测试类型，而是平台正式承认“哪些事实必须被验证”的边界集合。只要某条能力会影响任务受理、运行推进、结果放行、权限边界、审计留痕、资源消耗或恢复判断，就必须进入正式验证面。

统一划分为四个面：

1. 功能验证面：验证任务、运行、结果、工具、记忆、确认、委派等主链路是否按正式语义成立。
2. 治理验证面：验证权限、配额、确认、审计、冲突收口、隔离与降级是否按平台规则成立。
3. 可观测性验证面：验证日志、指标、审计事件、追踪锚点与退化信号是否足以支持解释、回归与恢复。
4. 性能验证面：验证在目标负载下，主链路、治理链路与后台链路是否满足正式性能基线与容量边界。

其中，前三个面共同构成“正式验证面”，第四个面构成“正式性能验证面”。二者关系是并列而非包含：功能正确不代表性能达标，性能达标也不代表治理闭环成立。

## 4. 验证层次与测试矩阵

`Agent OS` 的测试矩阵应按“验证层次 x 能力域”组织，而不是按单一测试框架组织。

### 4.1 验证层次

1. `L0` 静态与契约验证：对象模式、状态枚举、索引约束、版本引用、契约边界、幂等键与摘要对象的一致性。
2. `L1` 组件级验证：各内部能力域在独立输入下的收口语义是否成立，如 `prompt-assembly`、`tool-router`、`memory-governor`。
3. `L2` 主链路集成验证：围绕任务受理、`QueryEngine Runtime Loop`、结果生成、回写、确认、委派回收的正式链路验证。
4. `L3` 治理与异常验证：围绕权限拒绝、配额命中、降级回退、冲突记忆、确认超时、委派失败、恢复重试等边界验证。
5. `L4` 性能与容量验证：围绕延迟、吞吐、资源占用、排队、退化、恢复窗口与容量边界验证。

### 4.2 能力域矩阵

正式矩阵至少覆盖以下能力域：

- `task-ingress` 与任务幂等
- `persona-kernel` 与 `prompt-assembly`
- `runtime-core` 主循环
- `tool-router` 与 `provider-abstraction`
- `memory-governor` 与 `auto-dream-daemon`
- `human-gate`
- `delegation-coordinator`
- `audit-observability`
- 结果回写与恢复链路

矩阵使用原则如下：

- 每个能力域至少要有一条 `L1` 或以上验证，不允许存在“只有设计、没有验证入口”的能力域。
- 每条正式主链路至少覆盖一条成功路径、一条受控失败路径和一条治理暂停路径。
- 任何会改变正式状态或正式权限的能力，必须同时覆盖功能验证和治理验证，不能只测成功路径。
- 任何高频路径和高成本路径，必须进入 `L4` 性能验证。

正式验证必须引入对抗性的 `Verification Agent` 或等价独立验证角色：验证者与实现者隔离，实现 Agent 不得自证最终通过。验证者应主动构造失败、权限绕过、上下文污染、错误恢复、结果幻觉等场景；失败证据、错误输出和未通过用例必须保留，并进入后续回归基线。

### 4.3 运行时验证角色模型

运行时验证角色模型用于保证“完成业务动作的 Agent”和“判断业务动作是否成立的 Agent”在制度上分离。`Verification Agent` 是只读、对抗、独立上下文的验证角色，不是实现 Agent 的附属总结器。

`Verification Agent` 必须满足以下隔离要求：

- 只读：默认只能读取验证目标、业务证据包、对象状态、审计事件和运行输出，不得改写合同、审批、签署、履约、记忆或调度对象。
- 对抗：默认寻找失败证据、遗漏风险、权限绕过、证据断链和业务对象状态不一致，而不是为实现结果寻找解释。
- 独立上下文：不得继承实现 Agent 的完整长上下文、主观判断链、未验证推理或自我评价，只能消费正式传入的验证上下文来源。
- 独立工具集：工具授权由验证目标生成最小只读工具集，和实现 Agent 的可执行工具、业务动作工具、恢复工具分开授予。
- 独立审计：每次验证读取、比对、拒绝、通过和未覆盖风险都必须形成独立审计事件，不能合并进实现运行的普通观察。
- 独立回收格式：验证输出必须使用结构化回收格式，由父运行或结果放行链消费，不允许以自然语言口头通过替代正式验证结论。

实现 Agent 不得自证。实现 Agent 可以提供待验证对象、变更摘要和候选证据引用，但不得把“我已完成”“我认为通过”“我检查过”作为验证输入的结论来源。验证 Agent 不消费实现者主观结论，只消费企业业务证据包，例如：

- 业务对象变更差异：合同元数据、审批单、签署任务、履约计划、风险标签、管理端配置的前后状态。
- 合同 / 文档版本差异：条款文本、附件清单、批注、红线比对、版本号、来源引用与生效范围。
- 审批 / 签署 / 履约状态：审批节点、例外放行、签署渠道回执、履约里程碑、逾期风险状态与责任归属。
- 审计事件：工具调用摘要、人工确认裁决、权限拒绝、沙箱拒绝、恢复动作、结果回写与放行记录。
- 运行输出与对象状态：正式结果壳层、结构化字段、错误输出、当前数据库对象状态与可观测指标。

典型验证场景包括：合同起草后高风险条款是否仍被标记并进入复核、审批例外是否只在授权范围内执行、文档比对是否保留版本差异证据、签署失败是否未被误判为完成、履约逾期是否进入风控预警而非被结果摘要掩盖。

### 4.4 `verification_report` 字段语义

`verification_report` 是验证 Agent 的正式回收壳层，用于替代口头“验证通过”。其字段语义至少包括：

- `verification_target`：本次验证的业务目标、对象范围、期望状态与风险等级，例如某份合同审查结果放行、某个审批例外处理、某个签署失败恢复或某批履约逾期预警。
- `independent_context_sources`：验证 Agent 消费的独立上下文来源，包括业务对象快照、合同 / 文档版本差异、审计事件、运行输出、人工裁决和回归基线入口。
- `check_items`：逐项检查清单，表达每个检查项的对象、规则、证据入口、通过条件和实际观察。
- `failure_evidence`：不通过或存疑时的失败证据，必须回指具体业务对象、版本、状态、审计事件或输出片段。
- `uncovered_risks`：本次验证未覆盖但可能影响放行的风险，例如缺少签署渠道回执、缺少审批人裁决、缺少履约状态快照或证据链断点。
- `conclusion`：结构化结论，至少区分 `PASSED`、`FAILED`、`PARTIAL`、`INCONCLUSIVE`，并说明是否允许放行、需重试验证、需补证或需人工确认。
- `regression_baseline_entry`：本次验证通过或失败后进入回归基线的入口，包括失败用例、业务证据包引用、版本范围和后续回归触发条件。

## 5. 功能验证面

功能验证面的核心不是“功能点清单”，而是“正式对象如何闭环”。

### 5.1 任务与运行主链路

必须验证以下事实：

- 任务受理、幂等命中、风险初筛、运行创建与取消语义正确
- `QueryEngine Runtime Loop` 能够在成功、部分成功、失败、等待确认、等待委派返回等状态间正确推进
- 结果对象只由正式结果壳层对外暴露，不把模型原始输出误当正式结果

### 5.2 Prompt、工具与模型链路

必须验证以下事实：

- 通用人格先于专用人格，静态底座先于动态注入，装配顺序与版本引用可追溯
- 工具参数校验、工具结果标准化、模型能力抽象、主备切换与降级回退遵循统一语义
- 模型调用失败、工具拒绝、配额命中和降级回退能被运行时正确吸收，而不是把异常直接外泄为不可解释状态

### 5.3 记忆、确认与委派链路

必须验证以下事实：

- 记忆检索、裁剪、冲突标记、失效与晋升不会破坏当前运行结论的可解释性
- 人工确认能正确承接安全硬链路第 3 层的后段兜底，批准、驳回、要求修改、过期等决策会改变运行后续路径
- 委派的切分、子运行回收、冲突回收与父运行汇聚保持对象级可追踪关系

## 6. 治理验证面

治理验证面用于回答“系统是否按平台规则工作”，而不是只回答“是否产出结果”。

### 6.1 权限与安全边界

必须验证：

- 工具分级、白名单、资源域限制、沙箱限制和高风险动作隔离是否有效
- 专用人格不会绕过平台根约束、确认边界或审计要求
- 敏感读取、危险动作、权限升级和结果放行不会绕开 `human-gate` 或正式拒绝语义

### 6.2 配额、降级与恢复边界

必须验证：

- 租户、任务、人格、工具、`Provider` 级配额与速率限制命中后会返回显式治理结论
- 主备切换、降级、回退、重试与熔断能在正式边界内收口，不制造隐式旁路
- 僵尸运行、确认超时、委派失败、回写失败等异常能进入恢复或终止语义，而不是悬空

### 6.3 审计与一致性边界

必须验证：

- 任务、运行、结果、工具调用、确认、委派、记忆、候选质量治理都能形成正式审计链
- 审计事件、日志摘要、指标聚合与对象主键之间可以互相回指
- 恢复后数据库主状态、审计事实和对外可见状态之间不出现不可解释分叉

## 7. 性能基线与负载模型

性能验证不以“某一次模型很快”作为结论，而以“在正式负载模型下，整条运行时链路是否仍处于可控区间”作为结论。

### 7.1 性能基线对象

`Agent OS` 的性能基线至少按以下对象建立：

- 任务受理延迟与受理吞吐
- 运行调度延迟、队列等待时长与循环推进效率
- 工具调用开销、模型路由附加开销与降级切换开销
- 记忆检索时延与确认创建时延
- 委派创建、回收与合并时延
- 审计写入成功率、指标上报完整率与恢复判定时长

正式阈值应以上游需求基线与更严格口径为准；本文的职责是定义必须被验证的性能对象与判定语言，而不是重写招标或需求文档中的全部具体数值。

### 7.2 负载模型

正式性能验证至少覆盖四类负载模型：

1. 交互型负载：同步创建任务、查询运行、查询结果、处理确认，关注入口延迟与短时洪峰。
2. 长链路负载：多轮 `QueryEngine Runtime Loop`、多工具观察、模型重试、结果回写，关注长尾时延与队列堆积。
3. 治理型负载：配额命中、确认暂停、委派回收、冲突收口、审计写入高峰，关注治理链路是否拖垮主链路。
4. 后台型负载：记忆整理、`Auto Dream daemon`、索引更新、过期扫描、恢复任务，关注后台负载对前台运行的挤压。

### 7.3 容量边界

容量边界至少要正式回答以下问题：

- 单实例或单部署单元可稳定承接多少任务受理与运行推进压力
- 当确认、委派、记忆、审计等治理链路同时升高时，系统先在哪个环节出现瓶颈
- 当主 `Provider` 退化、主工具链抖动或后台整理积压时，主链路是否仍能维持最小可用能力
- 需要扩容的触发信号是什么，哪些指标上升意味着已经接近正式容量边界

## 8. 成功标准与退化判定

### 8.1 成功标准

正式验证通过，至少同时满足以下条件：

- 功能验证面中的正式主链路全部通过，且不存在未解释的状态分叉
- 治理验证面中的拒绝、暂停、降级、恢复、审计留痕全部有证据闭环
- 性能验证面在目标负载下达到正式基线，且没有出现因治理链路放大导致的系统性失稳
- 可观测性验证面能对每条失败结论、退化结论和容量告警给出可追溯证据

### 8.2 性能退化判定

性能退化不以单指标波动判定，而以“相对基线的持续偏离 + 对正式能力面的影响”联合判定。建议统一为三档：

- 观察级退化：核心指标相对最近稳定基线出现持续恶化，但尚未突破正式阈值；需要纳入回归观察。
- 告警级退化：核心延迟、吞吐、错误率、排队或资源占用已明显恶化，并开始影响部分正式链路；需要阻断发布或继续扩压。
- 失败级退化：已突破正式性能基线，或虽然未全部突破数值阈值，但已造成确认积压、委派失控、审计缺失、恢复失效等正式能力损坏；直接判定为不通过。

## 9. 回归策略与可观测性验证

### 9.1 回归策略

回归策略应按“变更影响面”分层触发：

- Prompt、人格、工具契约、记忆治理、确认治理、委派治理、路由策略等规则类变更，必须触发对应 `L1-L3` 回归。
- 运行时调度、缓存、锁、队列、恢复、审计、配额与模型路由变更，必须追加 `L4` 性能回归。
- 涉及多个治理面交叉的改动，必须至少回归一条端到端主链路和一条治理异常链路。

回归基线应采用最近一次正式通过的验证结果，而不是采用临时手工观察结果。

### 9.2 可观测性验证

可观测性本身必须被验证，至少包括：

- 每条正式主链路都能关联 `task_id`、`run_id`、`trace_id` 与关键治理对象主键
- 每次降级、熔断、配额命中、确认暂停、委派回收、恢复判定都能产出清晰指标或审计事件
- 压测与回归过程中能区分主链路退化、模型退化、治理退化和后台挤压，不把所有问题都归因为“慢”

## 10. 压测数据准备边界

压测数据准备只服务于负载真实性与边界验证，不应演化为实现或运维材料。其边界如下：

- 需要覆盖任务类型、对象类型、人格组合、工具族、确认类型、委派形态与记忆状态的代表性分布
- 需要区分只读链路、受控写入链路、高风险待确认链路和后台整理链路，避免所有流量都被简化成单一查询请求
- 需要提供可重复的基线数据集、扩容数据集和异常数据集，但不在本文规定生成脚本、导数流程或执行步骤
- 不允许使用会直接改变生产正式真相的数据准备方式；压测数据必须与正式生产数据隔离

## 11. 本文结论

`Agent OS` 的验证与性能设计，本质上是在实现前先定义“什么才算正式通过”。其核心收口如下：

- 以功能验证面、治理验证面、可观测性验证面和性能验证面共同定义 `Agent OS` 的正式验证边界
- 以 `L0-L4` 验证层次和能力域矩阵收口测试矩阵，而不是把验证散落为零碎脚本
- 以任务受理、运行推进、工具路由、记忆治理、人工确认、委派回收、审计恢复为主验证对象
- 以性能基线对象、四类负载模型、容量边界、退化判定和回归策略定义正式性能验证面
- 将演练脚本与运维手册明确留在后续独立专项设计，不在本文越界展开
