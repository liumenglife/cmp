# Agent OS 子模块专项设计：委派调度器

## 1. 文档说明

本文档用于继续下沉 `agent-os` 主线中“委派调度器”能力，重点回答以下问题：

- `Agent OS` 如何把多 Agent / 多子任务委派收口为可控调度能力，而不是放任并发扩散
- 委派对象、子任务切分、阶段推进、结果回收与冲突回收应如何形成统一治理语义
- 并行、串行、隔离配额、失败回退、重复委派抑制与人工接管升级边界应如何收口

本文是以下文档的下游专项设计：

- [`agent-os Detailed Design`](../detailed-design.md)

`runtime-core`、`persona-kernel`、`tool-router`、`memory-governor`、`human-gate`、`audit-observability` 是本文的强依赖或关键消费方：前两者决定何时需要委派与以何种人格边界发起子运行，中间两者分别提供执行观察与上下文摘要，后两者分别承接最终兜底确认与全量调度留痕。但这些关系属于 `agent-os` 内部协作关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不重写对外 `API` 路径、请求体、响应体、错误码与字段样例
- 不写线程池、队列分区、执行器类图、锁实现、数据库迁移脚本或消息主题细节
- 不把委派器写成实施计划、压测方案、上线手册、排班制度或运维脚本说明
- 不把某个具体模型、容器、调度框架或工作流引擎的私有实现直接写成平台标准

## 2. 设计目标

- 让委派成为 `Agent OS` 的正式调度能力，而不是主运行临时拼接的“开分身”技巧
- 让子任务切分、并行规模、上下文传递、结果回收和冲突回收都有显式边界
- 让自动自治保持默认主路径，人工接管只在自动路径无法继续安全收口时作为后段兜底
- 让并行受任务依赖、隔离配额和结果汇聚成本共同约束，而不是默认越多越好
- 让失败、重复派发、僵尸子运行和冲突结果都能回到统一恢复语义，而不是散落成例外处理

## 3. 调度定位与治理范围

`delegation-coordinator` 的职责不是简单“把任务分出去”，而是把主运行对外不可见的内部协作拆分，收口为一套可追踪、可约束、可回收的调度面。

委派采用 `Leader-Teammate` 模式：父运行是 `Leader`，负责目标拆解、授权快照、上下文裁剪、结果汇聚与最终业务裁决；子运行是 `Teammate`，负责在独立边界内完成局部目标、返回证据和风险，而不是替代父运行宣布主任务终局。

本文治理范围覆盖以下对象关系：

- 父运行决定是否发起委派
- 子运行以独立 `Session` 承接受限目标
- 多个子运行在并行或串行窗口内推进
- 回收结果进入父运行再判断是否合并、重派发、召回、回退或升级

统一原则如下：

- 委派是运行时调度动作，不是业务模块直接编排子任务的对外能力
- 子运行拥有独立执行责任，但不拥有脱离父运行目标的最终业务裁决权
- 委派必须以正式对象关系表达，不能依赖长上下文复制和隐式口头约定
- 调度优化目标是“更稳地收敛主任务”，不是“制造更多并行活动”
- 子运行必须具备独立上下文、独立工具集、独立权限和独立审计；父运行只授予最小必要材料，不共享无关观察或隐式权限

## 4. 委派对象模型

`Agent OS` 以“委派主对象 + 调度快照 + 回收结论”三层语义治理委派。

### 4.1 委派主对象

正式真相源仍以 `ao_delegation` 为主，但其内部语义应稳定表达以下维度：

- `delegation_goal`：本次委派真正要完成的局部目标，而不是泛化的“帮我做一下”
- `parent_run_id` 与 `delegated_run_id`：明确父子运行责任边界
- `delegated_agent_code`：子运行采用的专用能力域，但仍受通用人格底座约束
- `delegation_role`：表达本次运行在 `Leader-Teammate` 模式中的责任位置；父运行为 `Leader`，子运行为 `Teammate`
- `delegation_status`：表达创建、下发、执行、返回、合并、召回、失败、取消等正式调度态
- `merge_policy`：表达父运行如何消费子运行回执，而不是默认自动覆盖主结果
- `tool_grant_snapshot`：委派创建时授予子运行的工具域、只读 / 可写边界、禁止动作、确认要求和有效期快照
- `permission_scope_snapshot`：子运行可触达的业务对象、租户、合同 / 文档版本、审批 / 签署 / 履约范围和风险等级快照
- `memory_sync_policy`：定义父子运行之间哪些高价值业务事实、证据和冲突可以同步，哪些观察禁止同步

### 4.2 调度快照

每次委派判断都应形成调度快照语义，用于回答“为什么此刻允许派、派给谁、派多少个、按什么顺序回收”。快照至少应包含：

- 当前父运行阶段与待解决问题面
- 本次切分出的子任务集合摘要
- 任务之间的依赖关系判断
- 命中的并行上限、隔离配额与重复派发抑制结论
- 当前选择的回收策略与失败回退策略

调度快照是内部治理与审计锚点，不是新增对外接口。

### 4.3 回收结论

子运行返回后，父运行不直接把子结果视为主结果，而是先形成回收结论语义，至少表达：

- 父文档中的正式承接不能只保留 `returned_result_id`；应以独立回收结论对象或等价正式字段组，稳定承接 `SUCCEEDED`、`PARTIAL`、`FAILED`、`CONFLICTED`、`RECALLED` 等回收结论

- 已返回的是成功、部分成功、失败摘要还是冲突观察
- 该返回是否满足原委派目标
- 是否允许直接合并、需要再次验证、需要重派发、需要召回其他兄弟子运行或需要升级人工接管

## 5. 子任务切分原则

子任务切分的目标是把主运行当前无法高效收敛的问题拆成少数边界清晰的局部目标，而不是把主任务机械拆成任意碎片。

### 5.1 适合切分的场景

- 子问题拥有明确输入、明确产出，且回收后能被父运行独立判断
- 子问题需要专门人格、专门工具集或独立上下文窗口
- 子问题之间主要共享父目标，但不需要频繁共享中间思考过程
- 不切分会导致单运行上下文膨胀、人格漂移或工具观察相互污染

### 5.2 不应切分的场景

- 子问题之间强依赖上一子问题的即时中间判断
- 回收后仍必须逐条人工拼接才能形成任何有效结论
- 复制给子运行的上下文成本已经高于局部自治收益
- 当前问题只是单一工具调用或单一观察补充，不值得拉起独立子运行

### 5.3 切分粒度原则

- 一个子任务只承接一个清晰目标与一类主判断责任
- 切分优先按目标边界、证据域边界、风险边界拆，而不是按文件数、步骤数或想象中的并发度拆
- 父运行必须保留最终汇聚与裁决责任，不能把“最终怎么决定”整体外包给某个子运行

## 6. 调度阶段模型

委派调度建议统一为六个阶段，但每个阶段都允许短路收口，不要求机械跑满。

### 6.1 候选识别

父运行识别当前是否存在值得委派的局部问题，并先判断不委派是否也能在当前上下文内继续收敛。

### 6.2 切分建模

对候选问题形成子任务集合、依赖关系、建议人格、上下文摘要和目标产物类型。

### 6.3 调度裁决

结合任务依赖、隔离配额、预算压力、回收复杂度和重复派发风险，决定本轮采用：

- 不委派，继续由父运行处理
- 串行委派
- 有上限的并行委派
- 分批并行后再串行汇聚

### 6.4 子运行执行

子运行在独立 `Session` 中按受限目标推进，只消费被授予的最小必要上下文与能力边界。

### 6.5 回收汇聚

父运行收集子运行回执，对其做满足度判断、冲突扫描与合并决策。

### 6.6 收口恢复

根据回收结果恢复主运行：继续下一轮、结束、重派发、召回、降级或升级到人工接管。

## 7. 并行与串行策略

并行不是默认正确答案，而是受依赖关系、隔离成本和汇聚成本共同约束的受控选择。

### 7.1 优先串行的情况

- 后一子任务依赖前一子任务的正式观察或裁决
- 当前父运行需要先验证某个关键假设，再决定是否值得继续拆分
- 子结果之间存在显著覆盖关系，先拿到高价值结论即可淘汰后续子任务

### 7.2 允许并行的情况

- 子任务之间输入边界清晰，主要共享的是父目标而不是实时中间态
- 并行执行可以明显降低整体等待时长，且不会放大冲突回收成本
- 当前隔离配额、预算和结果汇聚能力都允许本轮扩散

### 7.3 分批并行策略

当任务既存在一定独立性，又存在后续汇聚依赖时，优先采用“分批并行”而非一次性全部放开：

- 第一批先覆盖最关键、最可能改写主判断的子任务
- 后续批次基于前批返回结果决定是否继续扩散
- 若前批已充分收口，后批应被抑制、召回或不再创建

这种策略保证并行建立在有效信息增益之上，而不是建立在盲目抢跑之上。

## 8. 隔离配额与上下文传递边界

### 8.1 隔离配额

委派调度至少受以下三类配额共同约束：

- 并发配额：当前任务、当前父运行、当前人格组合允许同时挂起多少个子运行
- 成本配额：本轮委派允许消耗多少模型、工具、检索、外部调用与等待预算
- 汇聚配额：父运行在本轮最多能有效消费多少个子结果而不导致回收失控

统一原则如下：

- 配额首先保护主任务收敛质量，其次才是吞吐
- 配额命中后应给出显式治理结论，如延后、分批、降级或拒绝扩散
- 子运行之间的隔离不仅是资源隔离，也包括风险隔离、证据域隔离与上下文污染隔离

### 8.2 上下文传递边界

父运行向子运行传递的内容应以“完成当前局部目标的最小必要材料”为准，通常包括：

- 局部目标说明
- 必要约束与禁止项
- 与目标直接相关的业务对象引用、观察摘要、证据引用和历史结论摘要
- 可用人格、工具、预算与确认边界摘要

默认不应直接传递：

- 父运行全部长上下文
- 与当前子目标无关的其他子任务中间观察
- 未经裁剪的敏感正文、原始 Prompt 正文或大批无关审计细节

边界原则是“让子运行足够完成任务，但不足以失控扩张”。

### 8.3 团队记忆同步边界

团队记忆同步服务于父子运行之间的业务事实闭环，而不是同步所有观察。同步内容必须经过价值与风险过滤。

允许同步的内容包括：

- 高价值业务事实：合同关键条款风险、审批例外原因、签署渠道回执、履约逾期状态、管理端配置约束。
- 证据：合同 / 文档版本差异、业务对象状态、审计事件、人工确认裁决、工具只读观察结果。
- 冲突：子运行之间对同一合同风险、审批节点、签署状态或履约责任的冲突判断。

默认不同步的内容包括：

- 与当前局部目标无关的观察、推测、过程性想法和未验证摘要。
- 父运行完整长上下文、其他子运行的无关中间结果、未授权敏感正文。
- 会扩大子运行权限或诱导其越过原委派目标的历史偏好。

团队记忆同步必须形成审计锚点，说明同步来源、同步原因、同步范围和消费方。同步后的事实仍需按记忆治理规则处理可信度、新鲜度和冲突，不因为来自 `Leader` 或 `Teammate` 就自动成为正式真相。

## 9. 结果汇聚与冲突回收

### 9.1 汇聚原则

父运行是唯一正式汇聚者。子运行只返回局部结论、证据摘要、失败观察或建议动作，不直接宣布主任务终局。

汇聚时至少区分以下几类结论：

- 可直接吸收的局部确定结果
- 需要与其他子结果交叉验证的结果
- 明确失败但对主运行有观察价值的结果
- 冲突未解、只能作为待裁决观察的结果

### 9.2 冲突回收原则

冲突回收不是异常尾巴，而是并行调度的正式组成部分。典型冲突包括：

- 多个子运行对同一对象给出不同判断
- 新返回结果推翻较早返回结果
- 子运行建议的后续动作彼此不兼容
- 子运行引用的证据域相互矛盾或可信度等级差异过大

统一处理原则如下：

- 父运行先按目标匹配度、证据质量、风险等级和新鲜度做自动收口
- 能被规则明确消解的冲突直接消解，不额外转人工
- 无法安全自动消解时，回收结论必须显式标记“冲突未解”，禁止伪装成已确定主结果
- 冲突未解且即将影响正式结果、危险动作或跨模块写回时，才升级到人工确认或人工接管边界
- 结果冲突也可以转交独立 `Verification Agent`，由其只读检查合同 / 文档版本差异、审批 / 签署 / 履约状态和审计证据后输出验证回收结论

### 9.3 召回与回收的关系

当某一子结果已经足以改写主判断时，父运行可以主动召回仍在执行但价值已下降的兄弟子运行，避免继续消耗预算并制造更多冲突回收成本。

### 9.4 企业业务委派示例

- 合同风险分析分派：`Leader` 将合同条款风险、附件一致性、历史履约风险拆给法务审查、文档治理、履约风控 `Teammate`，各自返回证据与风险等级，父运行统一汇聚放行建议。
- 审批异常调查：审批运营 `Teammate` 只读调查审批节点、例外原因和人工裁决证据；若需执行补充动作，必须回到父运行授权与确认边界。
- 签署失败排查：签署运营 `Teammate` 分析签署渠道回执、重试窗口和业务对象状态，避免把回执延迟误判为签署完成或永久失败。
- 履约逾期预警：履约风控 `Teammate` 检查里程碑、责任方、逾期天数和风险标签，父运行根据证据决定是否发起预警、确认或后续跟进。

## 10. 失败回退与重复委派抑制

### 10.1 失败回退

委派失败不等于主任务失败。父运行应先判断失败属于哪一类：

- 子任务目标本身不成立
- 子运行执行失败但可替代
- 当前并行窗口过大导致调度失稳
- 冲突长期无法收口

对应回退方向应优先包括：

- 改为父运行直接处理
- 缩小切分粒度后重新派发
- 从并行改为串行或分批并行
- 降级为非委派路径，如规则、检索、人工确认前置
- 在自动路径已明显失去收敛能力时升级人工接管

### 10.2 重复委派抑制

重复委派抑制用于避免父运行在相似上下文下反复派出等价子任务，导致预算空转与结果风暴。至少应基于以下维度判断“是否等价”：

- 相同父运行与相近阶段
- 相同子目标与相同对象引用范围
- 相同人格 / 工具边界与相同策略版本
- 前次回收尚未产生足以改变前提的新观察

统一原则如下：

- 等价子任务尚未回收时，不应再次派发平行副本
- 已失败的子任务只有在输入、策略或边界发生实质变化后，才值得重派发
- 重派发必须带着“为什么这次有机会更好”的新条件，而不是机械重试

## 11. 人工接管升级边界

人工接管是后段兜底，不是多 Agent 协作的默认组成部分。

只有出现以下情形时，才应从自动委派调度升级到人工接管：

- 多轮切分与回收后仍无法形成可执行主判断
- 冲突结果长期未解，且继续自动扩散只会放大不确定性
- 子运行持续失败、失联或僵死，自动恢复路径已失去可信度
- 当前结论即将影响高风险动作、正式写回或跨模块真相，但自动路径无法给出足够确定性

升级后的目标不是让人替代全部子运行思考，而是由人接手调度裁决责任：决定终止、缩圈、改派、人工完成或重新放回自动链路。

## 12. 与其他边界的关系

### 12.1 与 `API Design` 的边界

- 对外 `API` 只暴露委派相关资源与粗粒度状态
- 本文定义的是委派调度的内部治理语义，不是接口字段或状态枚举说明

### 12.2 与实现层的边界

- 本文回答的是“如何受控调度”，不是“底层用什么队列、线程池、锁或调度框架实现”
- 具体执行器、抢占策略、消息投递、心跳探测与恢复作业实现细节应在实现阶段再下沉

### 12.3 与人工确认边界的关系

- 人工确认处理的是少数自动无法安全裁决的委派冲突或例外
- 人工接管处理的是自动调度整体已无法稳定收敛的情况
- 两者都属于后段兜底，不改变“优先自动自治”的主路径

## 13. 本文结论

本设计将 `Agent OS` 的多 Agent / 多子任务协作收口为正式调度能力：

- 以委派主对象、调度快照、回收结论建立稳定对象模型
- 以候选识别、切分建模、调度裁决、子运行执行、回收汇聚、收口恢复形成统一阶段语义
- 以任务依赖、隔离配额、上下文边界和汇聚成本共同约束并行规模
- 以冲突回收、失败回退、重复委派抑制和人工接管升级边界保证调度可控
