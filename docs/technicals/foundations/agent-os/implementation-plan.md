## 2. 计划边界与原则

本计划只覆盖 `Agent OS` 子模块的实施推进顺序、任务依赖、联调准备、测试验收
与上线准备，不替代需求、架构、接口或详细设计文档。

实施过程中遵循以下原则：

- `Agent OS` 是围绕 `QueryEngine` 的薄 Harness 运行时，建设重点先落在最小可执行闭环，
  不先围绕单一模型能力、任务 CRUD 或 Prompt 拼接做功能拼接。
- 模型只是工具，不是运行时中心；业务模块只能接入 `Agent OS` 的统一能力边界，
  不把业务能力直接绑定到具体模型 / `Provider`。
- 通用 Agent 人格先于专用 Agent 人格；专用人格建立在通用人格之上，不另起一套
  运行时底板。
- 提示词工程采用“静态底座 + 动态注入”，且静态底座中的输出约束适用于全部
  Agent。
- AI 输入既包括业务输入，也包括环境输入；实施顺序必须先让两类输入都能进入
  同一 `QueryEngine` 闭环，再补外围治理能力。
- 多 Agent 协作采用跨 Session 委派，不在同一长上下文内通过频繁换人格替代正式
  委派链路。
- 记忆、`Auto Dream`、人工确认、委派、审计都依赖统一任务对象和运行对象，
  不应在主链未稳定前并行散建。
- 使用总平台既定正式文档链路，不在本计划中引入新的平行规格体系。

## 3. 前置条件

### 3.1 运行环境与基础设施准备

- `Agent OS` 所在开发、测试、联调、预发环境已按
  `Docker Compose / 企业内网` 基线具备稳定部署条件。
- `MySQL`、`Redis`、统一日志采集、任务执行环境、配置注入机制与对象存储引用能力
  已可用，能够支撑任务、运行、结果、审计、记忆摘要与异步后台作业的正式链路。
- 平台统一鉴权、组织架构、权限底座、审计中心、任务中心、配置中心的最小能力
  已可复用，避免 `Agent OS` 在实现阶段私建第二套用户、权限、日志或任务机制。
- 与文件、数据库、检索、规则、HTTP 调用等平台内工具能力的最小接入方式已明确，
  但不要求在前置阶段一次性接完全部工具。

### 3.2 模型 / Provider 配置前置

- `Agent OS` 需要的模型能力类别、`Provider` 接入方式、凭据注入方式、配额边界、
  超时与失败回退策略已在环境侧具备配置条件。
- 至少准备一条可用于联调的模型能力接入路径，但正式口径保持模型可替换、
  `Provider` 可替换，不把某个供应商当作模块成立前提。
- 模型调用所需密钥、网关地址、网络白名单、审计标识与调用成本观测方式已具备
  最小落地条件。
- 如果需要结构化提取、嵌入、重排或长上下文推理等不同能力，应先确认能力分类与
  配置边界，而不是把“一个默认模型”扩写成全部运行时能力。

### 3.3 工具接入前置

- 统一工具注册、权限分级、调用摘要留痕与超时隔离规则已明确，至少能支撑只读类
  工具先接入。
- 文件、数据库、检索、规则、外部 API 等工具接入前，需先确认调用身份、可访问
  范围、失败回写方式和审计字段，不允许业务模块绕过 `Agent OS` 直接把工具能力
  暴露为模型外挂。
- 环境输入接入所需的错误事件、告警事件、外部回调异常、工具失败事件上报入口
  已具备最小可用路径。

### 3.4 人工确认链路准备

- 管理端或等价操作端已具备人工确认待办承接能力，至少可处理结果放行、危险动作
  授权、委派批准或驳回等确认场景。
- 人工确认处理人与业务对象、权限范围、确认意见、过期策略和回写规则已明确，
  避免把人工确认做成临时消息通知后人工口头确认。
- 结果释放与人工确认之间的关系已收口为正式状态转换，不允许出现“结果已对业务
  可见，但确认单仍未处理”的失配状态。

### 3.5 审计、日志与任务链路准备

- `Agent OS` 全链路所需的追踪标识、任务对象主键、运行对象主键、审计事件落点和
  异常日志采集方式已确定。
- 统一任务中心能够承接 `AgentTask`、运行恢复、人工确认等待、委派回收、
  `Auto Dream` 后台作业等异步任务语义。
- 平台已具备按任务、运行、确认、委派、工具调用、环境事件维度查询日志与审计
  记录的最小能力。
- 联调环境与预发环境中的日志和审计字段口径保持一致，避免上线前再整体补埋点。

## 4. 总体阶段划分

本模块按五个阶段推进：

1. 最小 `Harness Kernel`
2. 工具契约与沙箱
3. 上下文与记忆治理
4. 验证 / 安全 / 治理
5. 多 Agent 与自演进

阶段之间以前一阶段稳定产物为下一阶段输入，不建议跳阶段并行铺开。

## 5. 分阶段实施计划

### 5.1 阶段一：最小 `Harness Kernel`

#### 目标

- 建立最小 `QueryEngine` 闭环，而不是只交付任务 CRUD、状态查询或 Prompt 字符串拼接。
- 让业务输入与环境输入都能进入同一 `Task Ingress -> Prompt Assembly -> Provider Budget Check -> Model Call -> Observation Normalize -> State Checkpoint -> Termination Check` 主链。
- 形成可追踪、可恢复、可审计的薄 Harness 内核，为后续工具、上下文、记忆、安全、验证和委派提供统一挂载点。
- 至少以合同管理平台任务作为端到端验收锚点，覆盖合同、审批、文档、签署、履约、风控、业务用户、管理端和运维审计中的核心对象引用。

#### 核心产出

- `AgentTask`、`AgentRun`、`AgentResult`、`PromptSnapshot`、最小 `Checkpoint` 与最小 `AuditEvent` 可用。
- 最小 `QueryEngine` 可以完成至少一轮 `while loop`：接收任务、装配受控 Prompt、检查 Provider 预算、调用模型、归一观察、写检查点、判断终止。
- 业务模块输入与环境输入的统一受理、幂等校验、状态推进与结果查询链路可用。
- 任务受理、运行开始、运行结束、失败回收和结果出站的统一审计与日志基础可用。
- 面向后续工具调用、上下文治理、记忆、人工确认和委派的挂点已预留，但不在本阶段展开外围能力。

#### 前置依赖

- 环境部署、基础配置、统一任务中心、统一审计中心已具备最小可复用条件。
- `Agent OS` 的架构边界、API 边界和详细设计中的主对象边界已可支撑正式实现。
- 业务模块和环境事件侧至少各具备一条可联调的入站路径。

#### 完成标志

- 业务模块可以通过统一任务入口提交任务并稳定查询运行状态与结果。
- 环境输入可以通过统一事件入口进入主链路，而不是旁路写日志后脱离运行时。
- 任意一次任务执行都具备任务主键、运行主键、结果主键、Prompt 快照、检查点和追踪标识。
- 最小 `QueryEngine` 闭环可以在没有完整工具体系和长期记忆的情况下完成受控问答或摘要任务。
- 失败任务不会留下“任务已成功、运行未结束”或“结果已出站、审计未落库”的
  结构性不一致状态。

#### 独立验证门禁

- 验证角色使用独立上下文读取接口契约、运行输出、审计事件和检查点，不消费实现 Agent 的主观结论。
- 必须验证最小闭环覆盖成功、模型失败、预算拒绝、外部取消、最大循环和结果满足契约。
- 端到端业务验收样例：业务用户提交合同风险提示任务，`QueryEngine` 生成受控结果，管理端可查询运行状态，运维审计可回放任务受理、运行检查点和结果出站摘要。
- 实现 Agent 不得自证最终通过；通过结论必须由独立验证角色基于证据包给出。

### 5.2 阶段二：工具契约与沙箱

#### 目标

- 在最小 `Harness Kernel` 稳定后，建立统一工具契约、工具发现、沙箱执行和结果归一能力。
- 让模型、检索、规则、文件、数据库、外部 API 都作为 `QueryEngine` 可治理工具接入，而不是各自绕过运行时。
- 先接只读工具和模型工具，再接受控写入与高风险副作用工具。
- 优先覆盖合同文档读取、条款抽取、审批流查询、签署状态查询、履约节点查询、风控规则命中和审计证据导出等企业业务工具。

#### 核心产出

- `Tool Registry`、`ToolSearch`、工具元数据、工具授权、工具定义快照和工具调用 envelope 可用。
- `Sandbox Executor` 支持只读、受控写入、高风险副作用的权限分级、超时、拒绝和审计。
- `Observation Normalize` 能把工具成功、失败、超时、沙箱拒绝和输出卸载结果转成统一观察。
- 模型 / `Provider` 作为工具能力纳入调用摘要、预算、失败和降级记录。
- `Guard Chain` 的确定性规则层接入工具执行前置链路，为后续 LLM Judge 和人工确认预留挂点。

#### 前置依赖

- 阶段一的最小 `QueryEngine` 闭环、运行对象、Prompt 快照、检查点和审计事件已稳定可用。
- 至少一条模型能力接入路径与若干平台内只读工具接入方式已准备完成。
- 工具权限、沙箱策略、输出卸载和审计字段口径已收口。

#### 完成标志

- `QueryEngine` 可以通过工具契约完成模型调用、只读工具调用、失败观察和结果回写。
- 沙箱拒绝不会被模型重试绕过，必须形成观察、失败码、人工确认或终止结果。
- 工具输出超长时以摘要 + 引用进入观察，不直接撑爆 Prompt。
- 业务模块不直接接触完整工具 schema、Provider 私参或工具内部参数。

#### 独立验证门禁

- 验证角色必须构造工具成功、工具失败、工具超时、沙箱拒绝、输出卸载和模型降级用例。
- 验证工具调用与工具结果在审计链中成对存在。
- 端到端业务验收样例：合同审查任务调用合同文档读取和条款抽取工具，命中风控规则后形成证据引用，沙箱拒绝越权读取其他合同附件并写入审计。
- 实现 Agent 不得自证最终通过；工具安全与沙箱门禁必须由独立验证角色给出证据结论。

### 5.3 阶段三：上下文与记忆治理

#### 目标

- 建立 `Context Governor`，让上下文窗口成为主动治理资源，而不是被动拼接容器。
- 将 Prompt 装配、静态底座、动态注入、微压缩、事实压缩、完整压缩、自动熔断和记忆召回接入 `QueryEngine` 每轮循环。
- 建立记忆候选、主动召回、压缩前记忆冲洗、失效和怀疑式校验的最小治理链路。
- 让合同版本差异、审批例外规则、签署异常处理经验、履约风险模式和法务审查口径以受控摘要进入 `Task Brief`、`Governance Brief` 或 `Memory Brief`，而不是塞入完整原文。

#### 核心产出

- `Context Governor` 内部模块可用：微压缩、事实压缩、完整压缩、自动熔断、工具调用 / 结果成对保留、压缩前记忆冲洗。
- `Prompt Assembly` 支持静态底座版本、专用人格补丁、动态注入摘要、缓存前缀策略和 Prompt 快照引用。
- `Memory Intake`、`Active Recall`、候选记忆可信度、过期 / 冻结 / 撤销基础能力可用。
- 上下文预算估算与 Provider 预算门联动，超预算时能裁剪、压缩、降级或终止。
- 运行结果、失败观察和高价值事实可进入候选记忆，但不会无差别自动晋升为长期事实。

#### 前置依赖

- 阶段一至阶段二的最小闭环、工具契约、沙箱、观察归一和审计链路已稳定可用。
- Prompt、上下文预算、记忆对象、工具结果卸载和 Provider 预算字段已在详细设计中收口。
- 日志与审计链路已可承接上下文治理快照、压缩结果、记忆冲洗和失败原因摘要。

#### 完成标志

- 长任务可以在上下文接近阈值时按顺序触发微压缩、事实压缩、完整压缩或熔断，而不是无限追加历史。
- 工具调用 / 结果对不会被压缩破坏；破坏时必须阻断 Prompt 组装并生成失败诊断。
- 压缩前记忆冲洗失败不会被吞掉，必须生成补偿作业、降级或人工接管。
- 业务模块仍只消费任务、运行和结果摘要，不接触 Prompt 正文、Memory 原文或子 Session 完整上下文。

#### 独立验证门禁

- 验证角色必须覆盖 `CONTEXT_BUDGET_EXCEEDED`、`TOOL_PAIR_BROKEN`、`MEMORY_FLUSH_FAILED` 和 Provider 预算不足。
- 验证长上下文任务经过压缩后仍能保留关键事实、工具配对和检查点恢复能力。
- 端到端业务验收样例：长合同及多份补充协议经过压缩后，仍保留关键条款风险、审批例外、签署状态和履约节点引用，业务用户看到摘要，运维审计能追溯裁剪原因。
- 实现 Agent 不得自证最终通过；上下文和记忆治理必须由独立验证角色用运行回放证据确认。

### 5.4 阶段四：验证 / 安全 / 治理

#### 目标

- 在最小内核、工具沙箱、上下文记忆治理稳定后，补齐验证、安全、人工确认、审计、指标和成本治理。
- 让高风险动作、结果放行、Provider 成本、运行回放和失败诊断都进入正式治理平面。
- 建立实现 Agent 与验证 Agent 的职责隔离，防止实现者自证。

#### 核心产出

- `Guard Chain` 完整接入确定性规则、LLM Judge、人工确认和运行时沙箱。
- `Human Confirmation Gate` 可处理结果放行、危险动作授权、权限升级和确认超时恢复。
- 运维审计 API 可查询审计、指标、运行回放、失败诊断、成本配额和验证报告。
- `Verification Agent` 可基于证据包、合同版本差异、审批 / 签署 / 履约状态、配置差异、审计事件和运行输出给出独立结论。
- 成本配额、Provider 熔断、速率限制、降级策略和回归基线可用。

#### 前置依赖

- 阶段一至阶段三的 `QueryEngine`、工具沙箱、上下文治理、记忆候选、Prompt 快照和审计对象已稳定可复用。
- 人工确认承接端、权限校验、确认回写规则、审计查询和指标采集已准备完成。
- 独立验证角色具备只读工具集、独立上下文和独立报告出口。

#### 完成标志

- 高风险结果或动作在未确认前不会直接进入业务正式链路。
- 人工确认超时、沙箱拒绝、Provider 预算不足和验证失败都有明确状态、失败码和恢复策略。
- 运维审计 API 可以回放关键运行轨迹，但不泄露完整 Prompt、Provider 私参或 Memory 原文。
- 独立验证角色可以阻断发布或回归合并，且其结论可审计。

#### 独立验证门禁

- 验证角色必须覆盖 `SANDBOX_REJECTED`、`HUMAN_CONFIRMATION_TIMEOUT`、预算耗尽、验证失败和审计证据缺失。
- 正式验收必须引用验证报告、失败证据、未覆盖风险和回归基线入口。
- 端到端业务验收样例：高金额合同命中高风险条款后暂停结果放行，法务在管理端确认处理意见，验证角色基于合同版本差异、审批状态、风控证据和审计事件出具结论。
- 实现 Agent 不得自证最终通过；安全、治理和验证门禁只能由独立验证角色放行。

### 5.5 阶段五：多 Agent 与自演进

#### 目标

- 在单 Agent `QueryEngine` 治理闭环稳定后，接入跨 Session 委派、多 Agent 协作和自演进资产管线。
- 让 `Auto Dream`、候选记忆、候选技能、候选模式都进入可验证、可灰度、可回滚的资产发布流程。
- 收口跨模块联调、委派演练、自演进误发布演练、审计检查、运行指标与上线切换准备。

#### 核心产出

- `Cross-Session Delegation Coordinator` 支持父运行、子运行、独立 Session、最小上下文、回收摘要和合并策略。
- `Auto Dream` 后台作业可生成候选记忆、候选技能和候选模式，但必须经过验证、灰度、发布和回滚链路。
- `Memory / Skill / Pattern Asset` 发布管线可用，具备复用率、命中后成功率、误命中率和回滚次数指标。
- 面向业务模块、环境事件、工具系统、模型 / `Provider`、人工确认端、委派子运行和自演进资产的联调清单与执行记录完成。
- 测试清单、验收用例、上线检查清单、灰度策略和回退预案完成。

#### 前置依赖

- 前四个阶段的核心能力已经完成并通过独立验证门禁。
- 需要外部条件的模型能力联调资源已经就绪，且其接入配置已按抽象边界完成落地。
- 人工确认承接端、业务验收口径、审计检查项、安全检查项和自演进回滚口径已统一。

#### 完成标志

- 主链场景能够覆盖“任务受理 -> `QueryEngine` 循环 -> 工具 / 沙箱 -> 上下文 / 记忆治理 -> 结果生成 / 人工确认 / 委派回收 -> 审计回查”。
- 至少一条业务输入链路和一条环境输入链路可以稳定完成端到端联调。
- 模型切换或 `Provider` 切换不会改变业务接入方式，只影响内部配置与运行策略。
- 上线准备清单明确，具备环境校验、配置校验、权限校验、审计校验、指标校验和回退
  方案。
- 自演进资产误发布、误命中和回滚失败都有演练记录和阻断策略。

#### 独立验证门禁

- 验证角色必须覆盖父子运行隔离、委派结果冲突、团队记忆同步边界、自演进误发布和资产回滚。
- 最终上线验收必须由独立验证角色出具 `PASS` 或 `PASS_WITH_NOTES` 结论。
- 端到端业务验收样例：合同审查主 Agent 委派法务风险、签署异常和履约逾期三个子运行，回收冲突证据后由验证角色裁决，自演进资产仅在灰度和回滚链路可用后发布。
- 实现 Agent 不得自证最终通过；上线前最终通过结论必须来自独立验证报告。

## 6. 模块推进顺序

`Agent OS` 的推进顺序应保持为：最小 `Harness Kernel` -> 工具契约与沙箱 -> 上下文与记忆治理 -> 验证 / 安全 / 治理 -> 多 Agent 与自演进。

原因如下：

- 先做最小 `Harness Kernel`，是为了先证明 `QueryEngine` 的 while loop、预算门、观察、检查点和终止条件成立。没有这条闭环，任务 CRUD 和 Prompt 拼接都只是外围壳层。
- 再做工具契约与沙箱，是为了让模型、检索、规则、文件、数据库和外部 API 都以统一工具语义接入，而不是让模型调用先污染业务接口与运行日志。
- 再做上下文与记忆治理，是因为工具观察和模型响应开始积累后，必须通过 `Context Governor`、记忆冲洗和压缩机制保护主循环质量。
- 再做验证 / 安全 / 治理，是为了在核心执行链稳定后补齐人工确认、验证角色、运行回放、失败诊断、成本配额和企业审计门禁。
- 最后做多 Agent 与自演进，是因为委派和自演进资产都消费稳定运行证据，必须建立在父子运行关系、证据包、回归基线和回滚机制之上。

## 7. 外部联调准备

本模块只对真正需要外部条件的部分提出联调准备要求。

### 7.1 模型能力联调准备

- 需准备至少一个可用的模型能力接入环境，用于验证统一模型抽象、调用超时、失败
  回退、配额约束和结果摘要留痕。
- 联调准备应以“模型能力类别 + 统一抽象配置”组织，不以单一模型名称作为正式文档
  主语义。
- 若存在多个 `Provider` 候选，应至少验证配置切换、失败降级和审计字段一致性，而非
  只验证首选 `Provider` 的 happy path。

### 7.2 外部工具或 API 联调准备

- 只有当 `Agent OS` 需要调用平台外部 API 或外部知识能力时，才需要提前准备对应的
  测试地址、签名方式、网络白名单和回调接收条件。
- 外部工具接入必须先完成调用身份、权限范围、超时策略与失败落审计规则确认。
- 平台内文件、数据库、检索、规则等能力属于平台内部工具接入，不作为外部联调资源
  单独前置。

### 7.3 人工确认协同准备

- 如人工确认涉及平台管理端之外的业务审批人、法务或运营角色，应在联调前明确待办
  入口、通知方式、超时策略和处理回执规则。
- 人工确认链路只要求确认承接端与权限边界就绪，不要求绑定单一消息通道或单一外部
  通知产品。

## 8. 测试、验收与上线准备

### 8.1 测试重点

- 主链正确性测试：覆盖任务受理、运行推进、结果生成、失败回收、取消、重试与状态
  一致性。
- Prompt 与人格测试：覆盖通用人格继承、专用人格挂载、静态底座约束不失效、动态
  注入摘要正确装配。
- 工具与模型抽象测试：覆盖工具调用摘要、模型切换、`Provider` 切换、超时、失败、
  降级与幂等行为。
- 阶段三至少覆盖 `QueryEngine` 主循环状态迁移与中止条件验证，不能只验证单轮模型调用或工具调用 happy path。
- 治理能力测试：覆盖记忆召回与失效、`Auto Dream` 候选生成、人工确认批准 / 驳回 /
  退回修改、跨 Session 委派与结果合并。
- 审计与可观测测试：覆盖任务、运行、工具、确认、委派、环境事件的追踪字段、日志
  查询和审计回查。
- 实现完成后必须由独立验证角色执行正式验证；实现 Agent 不得自证最终通过，未通过用例、失败证据和错误输出需进入回归基线。

### 8.2 验收重点

- 验证 `Agent OS` 已成为平台 AI 能力底座 / 运行时操作系统，而不是若干业务模块各自
  直连模型的集合。
- 验证模型只是工具，不是运行时中心；业务模块面向的是统一任务与结果接口，而非
  单一模型 SDK。
- 验证通用人格先于专用人格、静态底座输出约束适用于全部 Agent、提示词工程采用
  静态底座 + 动态注入。
- 验证 AI 输入同时覆盖业务输入与环境输入，多 Agent 协作采用跨 Session 委派。
- 验证高风险结果与关键动作必须经过人工确认，且全过程可审计。

### 8.3 上线准备

- 形成上线前配置核对清单，覆盖模型 / `Provider` 配置、工具白名单、权限范围、
  确认策略、委派策略、记忆治理策略、告警阈值和日志留存策略。
- 形成灰度发布与回退策略，确保模型能力异常、外部工具失败或人工确认端不可用时，
  不会破坏主业务流程的可恢复性。
- 形成容量与成本观测基线，覆盖任务量、运行耗时、工具超时、模型调用量、确认积压和
  委派积压。
- 形成审计抽查与上线后巡检清单，确保结果释放、危险动作、委派回收和记忆沉淀都能
  被回查。
