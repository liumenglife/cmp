# 组织架构 / 权限 / 统一认证主线专项设计：外部身份协议治理

## 1. 文档说明

本文档用于继续下沉 `identity-access` 主线中“外部身份协议治理”能力，重点回答以下问题：

- 企业微信、`SSO`、`LDAP` 的协议字段差异应如何在平台内统一承接
- 签名校验、票据校验、回调可信性与重放防护应如何在身份接入壳层内收口
- 外部身份在映射为平台主体之前，应如何完成指纹归一、冲突冻结、人工处理与审计留痕

本文是以下文档的下游专项设计：

- [`identity-access Detailed Design`](../detailed-design.md)

`integration-hub`、审计中心、通知中心是本专项设计的强依赖或关键消费方：前者负责外部连接与适配承接，后者分别消费身份异常事实与人工处理结果。但这些关系属于跨模块协作或依赖关系，不属于本文档的父文档关系，也不构成本文档的上游来源。

本文不展开以下内容：

- 不重写企业微信、`SSO`、`LDAP` 对外 `API` 路径、参数样例、错误码与联调脚本
- 不写身份绑定、用户主档、组织主档、角色授权的物理表 `DDL` 或索引细节
- 不把协议接入流程写成实施计划、排期拆分、运维手册或上线步骤
- 不写具体 `SDK` 调用方式、类图、配置键名或最终代码实现

## 2. 设计目标

- 为企业微信、`SSO`、`LDAP` 建立统一协议接入壳层，避免不同来源各自穿透到主体映射与会话建立逻辑中
- 在不抹平协议差异的前提下，收口最小标准化字段集、可信性校验边界与回调承接语义
- 保证所有外部身份进入平台前，都必须经过签名或票据校验、身份指纹归一、冲突冻结判断与审计留痕
- 明确“协议治理”与“平台主体治理”的边界：外部身份只影响进入平台的凭据与映射候选，不改写平台主体真相
- 为失效、解绑、重新绑定和人工处理提供统一状态边界，避免静默覆盖历史绑定关系

## 3. 适用范围与治理边界

本专项设计覆盖企业微信网页登录 / 应用登录、平台接入的通用 `SSO`、以及基于目录认证结果承接的 `LDAP` 身份进入链路。

统一边界如下：

- 本设计只治理“外部身份如何被平台可信承接”，不治理外部系统自身账号生命周期
- 本设计的终点是“生成可进入主体映射判断的标准化身份上下文”，不是直接落平台主体主数据
- 平台主体是否创建、是否归并到既有 `User`、是否授予角色，仍由 `identity-access` 其他内部能力负责
- `integration-hub` 只负责连接器、协议适配器、外部网络交互与通用验签 / 换票基座，不持有外部身份协议治理主对象
- `identity-access` 负责 `protocol_exchange_id`、标准化身份上下文、绑定预检查、冲突冻结、人工处置与会话准入语义

### 3.1 正式承接对象

外部身份协议治理不再停留在概念层，正式承接对象统一为：

- `ia_protocol_exchange`：协议交换主对象，承接 `protocol_exchange_id`、`idempotency_key`、来源消息键、可信校验结论、重放拒绝与冻结后重试约束
- `ia_identity_binding_precheck`：绑定预检查对象，承接身份指纹、绑定候选、冲突冻结判断与 `session_gate_result`
- `ia_identity_manual_disposition`：人工处置历史对象，承接冻结、解冻、解绑、重绑、撤销、批准重试等人工决定

统一要求如下：

- 重复回调、自动重试、恢复和人工介入都必须围绕 `protocol_exchange_id` 追踪
- 预检查判断必须有正式记录，不能只存在瞬时内存或日志中
- 冻结后的重试是否允许，必须以前一条人工处置或交换状态为准，不能靠调用方自行猜测

## 4. 统一协议接入壳层

### 4.1 分层模型

统一协议接入壳层拆为五层：

1. `IngressReceiver`：接收浏览器跳转、服务端回调、目录认证承接结果或换票请求
2. `TrustVerifier`：执行签名校验、票据校验、时效校验、来源校验与重放检查
3. `ProtocolNormalizer`：把企业微信、`SSO`、`LDAP` 的协议结果转换为平台标准化身份上下文
4. `BindingPrecheck`：在进入主体映射前执行身份指纹归一、历史绑定查询、冲突冻结判断与幂等去重
5. `SessionExchangeGate`：基于预检查结果决定是否允许换取平台受控会话、进入人工处理或直接拒绝

五层职责必须分离：

- 接入壳层负责“可信承接与标准化”，不直接解释角色、组织、数据权限
- `TrustVerifier` 负责判断“这次外部身份结果是否可信”，不负责决定“它最终是谁”
- `BindingPrecheck` 负责给主体映射提供稳定输入，不直接做静默自动归并

### 4.2 统一主链路

统一主链路固定为以下 8 步：

1. 接收外部请求、回调或中间认证结果，并生成 `protocol_exchange_id`
2. 识别 `provider`、入口类型、租户上下文与预期 `redirect` / 回调场景
3. 执行签名、票据、时效、来源与重放校验
4. 按 `provider` 画像解析原始字段，并形成标准化身份上下文
5. 计算身份指纹并检查最近幂等记录、已有绑定和冲突组状态
6. 若满足换票条件，则换取平台受控短期登录结果；若存在冲突，则冻结并进入人工处理
7. 若允许建立平台会话，则只生成会话候选上下文，不绕过正式主体映射边界
8. 记录审计事实、可信结论、协议版本与最终处置状态

其中第 1、5、6、8 步的正式落点分别是 `ia_protocol_exchange`、`ia_identity_binding_precheck`、`ia_identity_manual_disposition` 与 `ia_identity_audit_event`，避免协议治理语义只停留在运行时链路。

## 5. Provider 差异抽象

### 5.1 统一画像对象

三类来源统一抽象为 `ProviderProtocolProfile`，至少包含以下属性：

- `provider_code`：`WECOM`、`SSO`、`LDAP`
- `entry_mode`：浏览器重定向、服务端回调、目录结果承接、后台换票
- `trust_mode`：签名校验、票据校验、目录绑定校验、组合校验
- `identity_key_strategy`：来源内稳定身份键的抽取规则
- `ticket_strategy`：是否需要一次或多次换票、票据失效窗口与续期策略
- `callback_mode`：同步回调、异步通知或无回调
- `allowed_clock_skew`：时钟偏差容忍窗口
- `replay_protection_mode`：`nonce`、一次性票据、请求号或组合键重放防护

### 5.2 各来源的差异重点

- 企业微信：重点在 `code` 换票、服务端票据时效、回调来源校验、应用上下文与企业主体边界
- `SSO`：重点在认证断言或票据的可信签发方校验、回调重定向完整性、状态参数绑定与一次性换票约束
- `LDAP`：重点在目录认证承接结果的可信来源、目录属性标准化、账号禁用 / 失效状态承接与重新校验窗口

统一要求如下：

- 平台不要求三类来源暴露完全相同的原始字段，但必须都能产出统一标准化字段集
- 来源差异只能留在 `ProviderProtocolProfile` 与解析器内部，不允许向主体映射或会话层泄漏为三套判断逻辑

## 6. 最小标准化字段集

### 6.1 标准化身份上下文

协议治理阶段统一产出 `ExternalIdentityEnvelope`，最少包含以下字段：

- `provider`
- `provider_tenant_ref`
- `provider_app_ref`
- `external_identity_key`
- `external_login_name`
- `external_display_name`
- `external_union_ref`
- `external_org_ref`
- `external_email`
- `external_mobile`
- `external_status`
- `credential_type`
- `credential_issued_at`
- `credential_expires_at`
- `verification_result`
- `verification_evidence_ref`
- `raw_payload_ref`
- `protocol_exchange_id`

字段使用原则如下：

- `external_identity_key` 是来源内稳定身份键，优先用于绑定唯一性判断
- `external_login_name`、`external_email`、`external_mobile` 只能作为指纹辅助因子，不能单独替代稳定身份键
- `external_union_ref` 用于承接跨应用或跨入口的统一身份线索；若来源不支持，可为空
- `external_org_ref` 只表达来源侧组织线索，不直接写入平台组织真相
- `raw_payload_ref` 只保存原始报文引用或脱敏摘要，不在业务日志中保存原文

### 6.2 身份指纹归一

在进入平台主体映射前，必须生成 `identity_fingerprint`。其输入优先级如下：

1. `provider + provider_tenant_ref + external_identity_key`
2. `provider + provider_tenant_ref + external_union_ref`
3. `provider + provider_tenant_ref + external_login_name`
4. `external_email`、`external_mobile` 仅作为辅助证据，不可单独构成正式主指纹

归一原则如下：

- 指纹只服务绑定预检查、冲突聚类与人工处理辅助，不等于平台最终 `user_id`
- 同一来源主指纹变化时，必须视为重新识别事件，不能无痕覆盖旧指纹
- 不同来源即使辅助字段一致，也不能仅凭手机号或邮箱自动判定同一平台主体

## 7. 签名、票据与可信性校验边界

### 7.1 签名 / 票据校验责任

统一协议接入壳层必须完成以下校验：

- 签名是否合法、签名材料是否命中当前允许版本
- 票据是否由受信任来源签发、是否处于有效期内、是否已被使用或撤销
- `state`、`nonce`、回调请求号、一次性票据等是否与发起链路匹配
- 回调来源、租户、应用标识、重定向目标是否与预期上下文一致

校验边界如下：

- 校验通过，只表示“协议层可信”，不表示“可以自动映射为某个平台主体”
- 校验失败，必须在协议层直接拒绝，不进入主体映射、组织挂接或角色处理流程
- 平台不对外部票据长期保真负责，只在受控时效窗口内承认其可换票价值

### 7.2 回调可信性判断

回调可信性统一输出为以下状态：

- `TRUSTED`
- `REJECTED_SIGNATURE`
- `REJECTED_TICKET`
- `REJECTED_REPLAY`
- `REJECTED_CONTEXT_MISMATCH`
- `WAIT_MANUAL_REVIEW`

进入 `WAIT_MANUAL_REVIEW` 的典型场景包括：

- 签名本身合法，但回调关联的上下文缺失或已被人工冻结
- 票据处于边界失效窗口，平台无法自动确认是否可承接
- 同一 `protocol_exchange_id` 下出现多个来源声明不一致的身份结果

## 8. 回调承接、幂等与换票 / 换会话

### 8.1 回调承接原则

- 回调接口只承接认证结果与可信性判断，不承接组织全量同步、角色写入或业务数据落库
- `protocol_exchange_id` 表示一次正式受理的协议交换对象；首次承接时生成，后续重复回调、自动重试和人工介入都必须复用该对象追踪、冻结和恢复
- 单次回调、自动重试、人工补录都属于追加到该交换对象上的事件或状态推进，不单独升级为新的协议交换主对象；来源请求号、回调号或票据摘要继续作为幂等识别输入
- 同一回调重复到达时，只允许复用既有处理结论或进入补偿，不允许二次创建新的绑定候选
- 同一 `protocol_exchange_id` 一旦进入 `FROZEN`、`MANUAL_ONLY` 或 `TERMINATED`，后续自动回调只能追加到既有交换记录，不允许绕过冻结语义重开新链路

### 8.2 幂等键与去重范围

协议层幂等最小键统一为：

- `provider + provider_tenant_ref + protocol_message_key + credential_digest`

其中：

- `protocol_message_key` 可来自回调事件号、断言 `id`、请求号或目录承接批次号
- `credential_digest` 用于区分同一消息号下票据或断言内容不同的异常重放

幂等约束如下：

- 幂等命中后，只允许更新时间戳、审计引用和重复次数，不允许重跑主体映射副作用
- 已进入人工冻结的记录，即使回调再次到达，也不得自动转为成功
- 重放防护的正式拒绝结果必须写回 `ia_protocol_exchange.verification_result=REJECTED_REPLAY`，不能只留在网关或适配器日志里

### 8.3 换票与换会话边界

统一将“换票”和“换会话”分成两段：

1. 协议换票：外部 `code`、断言、目录认证结果换成平台认可的 `ExternalIdentityEnvelope`
2. 平台换会话：在绑定预检查通过后，把标准化身份上下文换成平台受控会话候选

边界要求如下：

- 协议换票失败，不得生成任何平台登录成功语义
- 平台换会话只在 `verification_result=TRUSTED` 且未命中冲突冻结时继续
- 平台会话建立依赖主体映射最终结论；若主体映射需要人工确认，则只保留待处理交换记录，不签发正式会话
- 若 `ia_protocol_exchange.retry_policy_status` 已是 `FROZEN_NO_RETRY`、`MANUAL_ONLY` 或 `TERMINATED`，则任何自动换会话尝试都必须被拒绝

## 9. 冲突冻结与人工处理

### 9.1 冲突冻结条件

出现以下任一情况时，必须冻结，不允许自动覆盖：

- 同一 `provider + external_identity_key` 已绑定到不同平台主体
- 同一外部身份指纹命中多个活跃候选主体，且无单一确定规则可收口
- 外部身份历史绑定为已解绑 / 已撤销，但本次回调试图直接恢复到不同主体
- 回调可信但来源侧关键身份键发生漂移，且无法证明属于同一自然人
- 外部状态显示已失效、禁用或租户不一致，但本地仍存在活跃绑定尝试换会话

### 9.2 人工处理入口

人工处理入口只解决以下问题：

- 选择绑定到哪个平台主体
- 确认是否维持冻结、执行解绑、重新绑定或彻底撤销
- 查看原始证据摘要、历史绑定链路、最近回调与可信性结论

人工处理入口不是父文档或独立业务模块关系，它只是 `identity-access` 在协议治理阶段的受控处置面。

人工处置的正式输出不是页面动作本身，而是 `ia_identity_manual_disposition` 追加记录；后续是否允许继续重试、解绑或重绑，都必须读取该历史对象。

### 9.3 处置结果约束

- 人工确认前，不允许自动补签平台会话
- 人工重新绑定必须保留旧绑定历史、旧指纹历史和处置原因
- 人工撤销后，同一外部身份再次进入时仍需重新走可信校验与预检查链路

## 10. 审计留痕与恢复边界

### 10.1 必须留痕的事件

- 外部请求进入、回调接收、协议换票成功 / 失败
- 签名校验失败、票据失效、重放拒绝、上下文不匹配
- 身份指纹生成、绑定预检查命中、冲突冻结、人工解冻
- 解绑、重新绑定、失效继承、重新登录后的再次承接

### 10.2 审计最小证据

每条审计事实至少应能回溯：

- `provider`
- `protocol_exchange_id`
- `verification_result`
- `identity_fingerprint`
- `precheck_id`
- `disposition_id`
- `binding_status_before` / `binding_status_after`
- `operator_type`：系统自动或人工处置
- `raw_payload_ref`

### 10.3 恢复边界

- 审计与交换事实恢复后，可以重放“预检查判断”，但不能伪造历史登录成功结果
- 原始票据失效后，不允许仅凭历史审计记录重新签发平台会话
- 人工冻结、解绑、撤销等状态必须优先于任何自动重试任务
- 恢复时必须先恢复 `ia_protocol_exchange`、`ia_identity_binding_precheck`、`ia_identity_manual_disposition` 的最后正式状态，再决定是否允许继续承接

## 11. 失效、解绑与重新绑定边界

### 11.1 失效

- 外部身份票据失效，只影响当前换票结果，不直接删除既有绑定历史
- 外部账号被禁用或租户关系失效时，绑定关系进入受控失效态，阻止继续换会话
- 平台会话失效后，下一次登录仍需重新完成协议可信校验，不允许继承旧回调结论

### 11.2 解绑

- 解绑表示平台不再承认该外部身份与当前主体的活跃绑定关系
- 解绑后保留历史链路、历史指纹、历史可信结论与操作原因
- 已解绑记录再次登录时，默认按“新承接 + 历史关联可见”的方式处理，而不是直接恢复活跃绑定

### 11.3 重新绑定

- 重新绑定必须重新经过可信校验与绑定预检查
- 若重新绑定目标与历史主体不同，必须进入人工确认
- 重新绑定成功后，应形成新的绑定版本或新的状态转换记录，不覆盖旧事实

## 12. 本文结论

企业微信、`SSO`、`LDAP` 的外部身份进入平台时，必须先经过统一协议接入壳层完成签名 / 票据校验、回调可信性判断、标准化字段归一、身份指纹生成与绑定预检查，再决定是否进入平台会话换取与主体映射。

外部身份协议治理的正式边界停留在“为平台主体映射提供可信、可审计、可冻结的输入”，并由 `ia_protocol_exchange`、`ia_identity_binding_precheck`、`ia_identity_manual_disposition` 承接协议交换、预检查和人工处置历史；它不把外部协议差异直接扩散到平台主体真相、组织真相或授权真相内部。
