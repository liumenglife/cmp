## 2. 计划边界与原则

- 本计划只覆盖组织架构 / 权限 / 统一认证主线的平台侧实施安排，
  不承担 `SSO`、`LDAP`、企业微信等外部系统本体改造。
- 平台只存在一套正式的用户、组织、角色、权限真相；业务模块不得各自
  维护一套用户、组织、权限真相。
- `SSO`、`LDAP`、企业微信只影响身份进入平台和映射过程，不改写平台主体真相。
- 审批节点必须绑定部门、人员或组织规则；流程、合同、文档等主线只能
  消费本主线给出的统一身份、组织与授权边界。
- 数据权限是正式一等能力，必须独立建设、独立验证，不能退化为业务列表
  的附带过滤条件。
- 管理端可按部门、人员做解密下载授权，但授权配置、命中判定与审计归因
  必须复用本主线统一口径。
- 本计划遵循“先统一主体，再组织与角色权限，再数据权限，再跨主线授权
  命中与审计闭环，最后联调验收上线”的推进顺序，避免先做局部授权、后补
  主体真相导致返工。

## 3. 前置条件

### 3.1 统一认证 / `LDAP` / 企业微信身份映射前置条件

- 已明确平台统一主体模型与外部身份映射边界，确认平台主体是正式真相，
  外部身份只作为映射输入。
- 已取得最小可联调的 `SSO`、`LDAP`、企业微信测试接入条件，包括测试账号、
  测试应用、必要凭证、回调地址、可信域名或网络连通条件。
- 已确认同一自然人可能来自多个认证源时的归并规则、冲突处理方式与人工
  处理责任边界，避免联调时把外部账号直接写成平台正式主体。
- 已确认企业微信组织 / 人员同步只作为身份映射与成员挂接输入，不作为
  平台组织真相替代来源。

### 3.2 组织架构、角色、权限、数据权限、审计前置条件

- 已明确平台存在多组织、部门、人员、角色、权限、数据权限的统一治理边界。
- 已确认审批节点必须绑定部门、人员或组织规则，流程引擎不会自建一套审批
  参与人真相。
- 已确认菜单权限、功能权限、数据权限三类授权边界，以及它们与统一身份、
  组织上下文的关联方式。
- 已确认关键身份与授权事件需要进入统一审计，包括登录、映射冲突、组织变更、
  角色授权、数据权限命中、解密下载授权命中与高敏拒绝事件。
- 已确认合同、文档、流程、通知、统计等业务模块后续均以本主线为唯一授权
  判定来源，不保留模块内长期私有权限真相。

### 3.3 流程引擎、文档中心、加密下载授权、集成主线接入前置条件

- 流程引擎已明确审批节点绑定、组织规则解析、候选人选择与授权校验均依赖
  本主线，不在流程侧复制组织与权限真相。
- 文档中心已明确文件访问、预览、下载、协作等权限需要复用本主线统一身份
  与授权结果，而不是维护独立文档用户体系。
- 已明确管理端可按部门、人员做解密下载授权，且加密文档主线只负责受控执行、
  任务处理与留痕，不负责组织授权真相。
- 集成主线已明确 `SSO`、`LDAP`、企业微信等外部接入由统一适配层承接，
  本主线负责其进入平台后的身份归一、组织挂接与授权解释。
- 上述主线接入边界未明确前，不进入跨主线正式联调，只进行样例数据和最小
  连通性准备。

## 4. 总体阶段划分

本主线按五个阶段推进。

1. 统一主体与身份映射基础能力阶段
2. 组织架构与角色 / 权限基础能力阶段
3. 数据权限与授权判定能力阶段
4. 跨主线授权命中与审计闭环阶段
5. 联调、验收与上线准备阶段

阶段之间按严格依赖推进；后一阶段可以预研，但不得跳过前一阶段完成标志
直接进入正式交付。

## 5. 分阶段实施计划

### 5.1 第一阶段：统一主体与身份映射基础能力

#### 目标

- 先建立平台统一主体、统一会话与统一身份映射入口，确保平台内只存在一套
  正式用户真相。
- 收口 `SSO`、`LDAP`、企业微信进入平台后的身份归一规则，为后续组织挂接、
  角色授权、数据权限和审计归因提供稳定主体基础。

#### 核心产出

- 平台统一主体、会话、身份绑定与映射冲突处理的正式实施基线。
- `SSO`、`LDAP`、企业微信进入平台后的统一承接路径与最小联调清单。
- 身份映射异常、冲突冻结、人工确认与审计留痕的基础闭环。
- 面向其他模块可复用的统一用户上下文与主体引用口径。

#### 前置依赖

- `SSO`、`LDAP`、企业微信测试接入条件已完成最小收集。
- 本主线与集成主线之间的接入边界已明确。
- 平台已确认主体归一优先于外部账号原样透传。

#### 完成标志

- 登录、会话、外部身份换取与主体归一都回到同一平台主体口径。
- 同一自然人来自多个认证源时，不再在平台内形成多份正式账号真相。
- 身份映射冲突可识别、可冻结、可审计，而不是静默覆盖。
- 后续组织、角色、权限建设可以直接建立在统一主体之上。

### 5.2 第二阶段：组织架构与角色 / 权限基础能力

#### 目标

- 在统一主体成立后，建立平台统一组织树、成员挂接、角色、菜单权限与功能权限
  的正式治理边界。
- 为审批节点绑定部门、人员或组织规则提供稳定组织真相，为管理端与业务端的
  基础权限控制提供统一入口。

#### 核心产出

- 多组织、部门、成员归属、主部门 / 兼职部门等组织基础能力。
- 角色、菜单权限、功能权限的统一授权载体与授权维护路径。
- 审批节点绑定部门、人员或组织规则所依赖的组织解析基础能力。
- 面向合同、流程、文档、通知等模块的组织查询与权限摘要供给能力。

#### 前置依赖

- 第一阶段统一主体与身份映射基础能力完成。
- 审批节点绑定组织主数据的约束已固定。
- 流程引擎、文档中心、管理端均已确认不自建组织与基础权限真相。

#### 完成标志

- 平台组织、部门、人员归属关系具备统一维护与统一引用路径。
- 角色、菜单权限、功能权限已从业务模块局部判断收口为平台统一能力。
- 审批节点可以稳定绑定部门、人员或组织规则，且来源统一。
- 管理端、业务端、接口层对“谁能进入哪里、执行什么动作”不再各自解释。

### 5.3 第三阶段：数据权限与授权判定能力

#### 目标

- 在统一主体和基础权限建立后，把数据权限提升为正式一等能力，形成与组织、
  角色、业务上下文联动的统一授权判定能力。
- 让合同列表、审批待办、文档访问、授权管理范围等场景共用同一套数据范围解释。

#### 核心产出

- 数据权限范围定义、生效规则、查询能力与统一判定入口。
- 面向合同、流程、文档、管理操作的统一授权判定机制。
- 解密下载授权配置与命中判定所依赖的数据权限收口能力。
- 批量授权检查、授权依据摘要与拒绝原因输出能力。

#### 前置依赖

- 第二阶段组织架构与角色 / 权限基础能力完成。
- 合同、流程、文档等主线已明确各自资源对象与业务上下文的引用方式。
- 管理端解密下载授权已确认按部门、人员授权，并受统一数据权限约束。

#### 完成标志

- 数据权限已具备独立配置、独立查询、独立验证路径。
- 合同、流程、文档、管理授权等场景可以复用统一授权判定，而不是各自拼装
  私有过滤逻辑。
- 解密下载授权命中已能结合部门、人员、组织上下文与业务对象统一判断。
- 数据权限不再被实现为某个列表页的附属筛选条件。

### 5.4 第四阶段：跨主线授权命中与审计闭环

#### 目标

- 在统一授权判定成立后，补齐与流程引擎、文档中心、加密下载授权、通知、
  审计、集成主线之间的正式消费链路。
- 把“能判断授权”提升为“授权能被跨主线稳定命中、被追踪、被审计、被追溯”。

#### 核心产出

- 流程引擎审批节点选人、待办可见、审批动作校验的统一授权接入闭环。
- 文档中心预览、下载、协作与加密下载授权命中的统一授权接入闭环。
- 高敏授权命中、越权拒绝、身份冲突与授权变更的审计视图与追踪链路。
- 与通知、集成主线联动的主体归因、授权结果传递与异常告警基础能力。

#### 前置依赖

- 第三阶段数据权限与授权判定能力完成。
- 流程引擎、文档中心、加密文档、通知、审计、集成主线已具备最小可接入点。
- 关键高敏动作与关键审计事件范围已明确。

#### 完成标志

- 流程、文档、解密下载等关键场景均通过本主线命中统一授权结果。
- 高敏动作与关键拒绝事件可按统一主体、组织上下文与授权依据追溯。
- 业务模块不再保留绕过本主线的临时授权旁路。
- 身份流、授权流、审计流形成主线级闭环。

### 5.5 第五阶段：联调、验收与上线准备

#### 目标

- 将前四阶段形成的身份、组织、授权与审计能力收敛为可交付状态。
- 完成外部身份映射联调、跨主线授权联调、场景验收、上线切换与回退准备。

#### 核心产出

- `SSO`、`LDAP`、企业微信最小联调清单与通过记录。
- 覆盖登录、映射、组织挂接、角色授权、数据权限、解密下载授权命中、
  审批授权校验等关键场景的测试结果与验收结论。
- 生产环境认证配置、组织初始化、角色授权初始化、审计与告警配置、
  发布窗口与回退方案。
- 上线后首轮身份映射核查、授权核查、异常值守与问题升级路径。

#### 前置依赖

- 前四阶段完成标志全部达成。
- 外部联调窗口、测试账号、网络连通、回调配置与跨主线接入点均已落实。
- 统一主体、组织、角色权限、数据权限、审计视图在测试环境已稳定运行。

#### 完成标志

- 本主线具备正式上线条件。
- 外部身份进入平台、跨主线授权命中与审计追溯均已形成可执行闭环。
- 已形成第一版可用于生产推进的主线级实施基线。

## 6. 模块推进顺序

本主线推进顺序固定为：统一主体 -> 组织与角色权限 -> 数据权限 ->
跨主线授权命中与审计闭环。

原因如下：

- 先统一主体，是因为平台只允许一套正式用户真相；如果主体未先收口，后续组织挂接、
  角色授权、数据范围与审计归因都会建立在不稳定账号之上，后面必然返工。
- 再做组织与角色权限，是因为审批节点必须绑定部门、人员或组织规则，且菜单权限、
  功能权限都依赖统一组织与主体上下文；没有组织与角色权限，平台只能做到登录，
  不能做到正式可用的访问控制。
- 然后做数据权限，是因为数据权限必须建立在稳定主体、组织归属、角色授权之上。
  如果跳过前两步直接做数据范围，只会把数据权限写成各业务模块自己的筛选规则，
  无法成为正式一等能力。
- 最后补齐跨主线授权命中与审计闭环，是因为流程、文档、加密下载、通知、审计
  都必须消费前面已经稳定下来的主体、组织、角色与数据权限能力。过早联动会把
  临时口径扩散到多个主线，后续难以统一回收。

对应模块实施顺序建议如下：

1. 统一主体、会话、身份映射与冲突处理骨架
2. 组织架构、成员挂接、角色、菜单权限、功能权限
3. 数据权限、统一授权判定、解密下载授权命中
4. 流程引擎、文档中心、加密下载、通知、审计、集成主线接入与闭环
5. 联调验证、验收收口与上线准备

## 7. 外部联调准备

### 7.1 `SSO` 最小联调准备

- 测试环境、测试账号、票据换取方式、回调地址、可信域名与网络访问条件已确认。
- 已准备最小登录成功、登录失败、票据失效、主体冲突四类联调样例。
- 已明确平台侧只承接认证结果与身份映射，不承担 `SSO` 本体改造。

### 7.2 `LDAP` 最小联调准备

- 已确认目录访问方式、测试目录账号、最小字段集、连通条件与查询频率边界。
- 已准备目录用户存在、目录用户禁用、组织信息变更、映射冲突四类样例。
- 已明确 `LDAP` 只影响身份映射与成员挂接输入，不直接改写平台主体与组织真相。

### 7.3 企业微信最小联调准备

- 测试企业、测试应用、测试成员、回调配置、可信域名与测试终端准备完成。
- 已准备授权登录、组织 / 人员同步、身份映射成功、身份映射冲突四类样例。
- 已明确企业微信影响身份映射与移动入口承接，不改写平台正式主体真相。

### 7.4 联调准备边界

- 联调准备只覆盖平台侧身份映射、组织挂接与授权接入所需的条件、样例与环境。
- 不展开外部认证系统、目录系统或企业微信本体改造计划。
- 如外部系统侧存在改造需求，应登记为外部依赖，不并入本主线实施责任。

## 8. 测试、验收与上线准备

### 8.1 测试重点

- 主体测试：验证平台只有一套正式用户真相，同一自然人不会因多认证源形成多套主体。
- 映射测试：验证 `SSO`、`LDAP`、企业微信进入平台后的映射、冲突冻结、人工处理与
  审计留痕。
- 组织与权限测试：验证组织挂接、角色授权、菜单权限、功能权限与审批节点绑定的
  一致性。
- 数据权限测试：验证合同、流程、文档、管理操作等场景的数据范围判定一致性。
- 授权命中测试：验证解密下载授权、审批动作校验、待办可见范围、文档访问权限等
  关键命中场景。
- 审计测试：验证登录、映射冲突、授权变更、数据权限命中、高敏拒绝事件均可追溯。

### 8.2 验收重点

- 平台只存在一套正式的用户、组织、角色、权限真相。
- 审批节点必须绑定部门、人员或组织规则，且流程引擎不再维护第二套选人真相。
- 管理端可按部门、人员做解密下载授权，且授权命中可追踪、可审计。
- `SSO`、`LDAP`、企业微信影响身份映射，但不改写平台主体真相。
- 数据权限已成为正式一等能力，并被合同、流程、文档、管理场景统一消费。
- 业务模块不存在长期绕过本主线维护用户、组织、权限真相的实现路径。

### 8.3 上线准备

- 确认生产环境认证配置、回调地址、可信域名、目录连接、凭证引用与网络策略
