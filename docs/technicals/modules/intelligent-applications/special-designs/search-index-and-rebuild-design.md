# 检索 / OCR / AI 业务应用主线专项设计：搜索索引与重建治理

## 1. 文档说明

本文档用于继续下沉 `intelligent-applications` 主线中“搜索索引文档结构、稳定召回、快照治理、重建触发与权限裁剪”能力，重点回答以下问题：

- 哪些正式来源可以进入搜索输入层，以及每类来源的可检索边界是什么
- 搜索索引文档、分层读模型、主键体系、来源锚点和结果快照如何设计
- 精准、模糊、筛选、排序、分页、导出、聚合与权限裁剪如何形成统一查询模型
- 索引刷新、重建、重放、补数、版本切换和可重建机制如何闭环
- 搜索如何向 `OCR`、AI 上下文装配、合同侧视图提供稳定输入，而不把候选集升级为真相源
- 并发、幂等、缓存、降级、失败恢复、监控指标、审计与验收口径如何收口

本文是以下文档的下游专项设计：

- [`intelligent-applications Detailed Design`](../detailed-design.md)

本文不展开以下内容：

- 不写具体搜索引擎产品选型、分词插件安装脚本或集群部署参数
- 不写 `OCR` 结果内部结构、AI Prompt 模板或合同侧字段级回写规则的专项细节
- 不写排序算法训练、向量参数调优、模型评测脚本或实施排期
- 不把历史方案、兼容口径或“后续再决定”写入正式设计

## 2. 目标与边界

### 2.1 设计目标

- 让搜索成为 `intelligent-applications` 主线的稳定输入层，只输出可追溯、可裁剪、可重建的候选集
- 让合同主档、文档中心、`OCR` 和条款库进入统一索引治理，阶段三后再按正式放行条件增量接入 AI 可检索摘要
- 让索引文档、结果快照、重建任务和权限裁剪拥有稳定主键与版本语义，避免 AI 和业务视图消费漂移结果
- 让文档版本切换、合同摘要变化、条款启停变化和 AI 结果失效都能异步刷新且可补数、可重放、可切换
- 让搜索引擎失效时仍保持真相源不受损，并能通过正式来源重建读模型与结果集

### 2.2 边界原则

- 合同主档是业务真相源；搜索只索引 `cc_contract_summary`、台账投影和稳定引用，不解释合同主状态最终语义。
- 文档中心是文件真相源；搜索只索引 `document_version_id` 对应的受控文本与摘要，不保存私有文件副本。
- `OCR` 结果是受控派生结果；只有 `READY` / `PARTIAL` 且允许用于搜索的结果才能进入索引输入层。
- 条款库是正式资源库；搜索只能消费正式启用版本或显式允许暴露的受控条款版本快照。
- AI 应用结果不属于阶段二搜索主链输入；如阶段三后需要增量接入，只允许暴露稳定摘要与证据锚点，不允许把未放行草稿、模型原始长文本或无来源结论直接入索引。
- 搜索结果、搜索快照、导出包都是读模型或派生产物，必须可失效、可重建、可审计，不能升级为新真相源。

## 3. 输入来源与准入治理

### 3.1 正式输入来源

搜索输入按“阶段二主链输入 + 阶段三后可扩展输入”收口：

| 来源类型 | 所属阶段 | 上游对象 | 最小锚点 | 可检索内容 | 不可检索内容 |
| --- | --- | --- | --- | --- | --- |
| 合同主档 | 阶段二主链 | `cc_contract_master`、`cc_contract_summary`、`cc_contract_ledger_projection` | `contract_id` | 合同编号、名称、分类、主体展示、摘要、风险摘要、台账筛选字段 | 合同内部临时编辑态、未发布草稿字段、只用于写模型的中间状态 |
| 文档中心 | 阶段二主链 | `dc_document_asset`、`dc_document_version`、`dc_capability_binding` | `document_asset_id`、`document_version_id` | 文档标题、文档角色、版本摘要、页级引用、能力挂接状态 | 私有文件副本、未激活版本的默认曝光、脱离版本锚点的裸文本 |
| `OCR` 派生结果 | 阶段二主链 | `ia_ocr_result` | `ocr_result_id`、`document_version_id` | 全文引用、页级片段、版面块摘要、字段候选摘要、语言片段 | 未通过质量门槛的原始引擎载荷、仅供内部调试的低质量噪声 |
| 条款库 | 阶段二主链 | `cc_clause_version`、语义标签 | `clause_version_id` | 标题、正文摘要、语义标签、风险标签、适用范围、多语言摘要 | 停用版本、草稿试写内容、未过适用范围校验的资源 |
| AI 应用结果摘要 | 阶段三后增量接入 | `ia_ai_application_result` | `result_id`、原始 `contract_id` / `document_version_id` / `semantic_ref_id` | 已稳定结果的摘要、标题、证据片段、回指来源、结果状态摘要 | 模型原始回复全文、未确认草稿、无来源回答、待人工确认前的自动结论 |

### 3.2 AI 可检索摘要边界

AI 结果不属于阶段二正式输入。若阶段三后按增量能力接入搜索，只允许按以下边界暴露：

- 只允许 `result_status in (READY, PARTIAL)` 的稳定结果进入索引扩展链路。
- 可检索对象只保留 `title`、`summary_snippet`、`evidence_digest`、`source_ref_list`、`risk_level`、`result_locale` 等稳定摘要字段。
- AI 问答类结果若无法稳定回指合同、文档页或条款版本，则禁止进入索引。
- 高风险结果在人工确认前只允许以“待确认候选”身份出现在运营或审计查询中，不进入业务默认检索曝光。

### 3.3 输入归一与准入链路

1. 上游对象发生正式状态变化，发布 `SearchSourceRefreshRequest`。
2. `search-runtime` 按来源类型读取正式对象、能力挂接状态、版本状态和权限域摘要。
3. 归一为统一 `SearchSourceEnvelope`，计算 `source_version_digest`。
4. 执行来源准入校验：对象存在、状态允许、权限域完整、锚点可回指、摘要可生成。
5. 通过校验的来源进入索引构建任务；未通过的来源写审计并进入 `SKIPPED` / `FAILED` 分流。

## 4. 索引文档结构与分层读模型

### 4.1 索引分层

搜索读模型按“源层 -> 文档层 -> 结果层 -> 快照层”组织：

- `SearchSourceEnvelope`：统一来源归一层，表达“这条可索引输入是谁、来自哪、当前摘要版本是什么”。
- `SearchDocument`：面向引擎的索引文档层，表达“引擎实际检索什么字段”。
- `SearchResultView`：查询结果投影层，表达“业务侧最终看到什么结构”。
- `SearchResultSnapshot`：短期快照层，表达“某次查询在当时当刻得到的稳定候选集”。

其中，`SearchDocument` 只服务索引和召回，`SearchResultSnapshot` 只服务查询稳定性与 AI 上下文装配，两者都不是正式业务对象。快照运行时可持有查询摘要、范围摘要、权限摘要和代次信息，但这些摘要不要求逐项下沉为 `ia_search_result_set` 当前正式字段。

### 4.2 统一索引文档模型

索引文档统一命名为 `SearchDocument`，最小结构如下：

| 字段 | 说明 |
| --- | --- |
| `search_doc_id` | 搜索文档主键，稳定表达一次来源对象在某个摘要版本下的索引身份 |
| `doc_type` | `CONTRACT` / `DOCUMENT` / `CLAUSE`，阶段三后可增量扩展 `AI_RESULT` |
| `source_object_id` | 来源对象主键，如 `contract_id`、`document_version_id`、`clause_version_id` |
| `source_anchor_json` | 稳定锚点，至少包含跳转对象、页号、片段主键或语义引用主键 |
| `source_version_digest` | 本次索引输入的版本摘要，用于刷新判重、切换和补数 |
| `contract_id` | 业务归属锚点，可为空 |
| `document_asset_id` / `document_version_id` | 文件归属锚点，可为空 |
| `semantic_ref_id` | 条款语义引用锚点，可为空 |
| `title_text` | 标题字段 |
| `body_text` | 检索主文本 |
| `keyword_text` | 关键词归一字段，承接编号、别名、标签、术语 |
| `filter_payload_json` | 面向筛选和聚合的稳定字段，如分类、状态、组织、语言、风险等级 |
| `sort_payload_json` | 面向排序的稳定字段，如更新时间、签署日期、置信度、风险等级 |
| `locale_code` | 主语言 |
| `visibility_scope_json` | 文档 / 合同 / 条款 / AI 结果的可见范围摘要 |
| `exposure_status` | `ACTIVE` / `STALE` / `HIDDEN` / `DELETED` |
| `rebuild_generation` | 当前索引代次 |
| `indexed_at` | 最近一次成功入索引时间 |

### 4.3 主键与来源锚点规则

主键与锚点统一遵循以下规则：

- `search_doc_id` 由 `doc_type + source_object_id + source_version_digest` 生成，保证同一来源对象的不同摘要版本不会互相覆盖。
- `source_object_id` 永远指向正式对象或正式派生对象主键，不允许使用缓存键、导出键或临时查询键充当来源身份。
- `source_anchor_json` 必须能把命中结果回指到合同详情、文档页、条款版本，或在后续扩展场景下回指 AI 结果详情。
- 对文档和 `OCR` 命中，锚点最小粒度应到 `document_version_id + page_no + citation_ref`；对条款命中，最小粒度应到版本级；对 AI 结果扩展命中，必须同时回指其原始证据来源。

### 4.4 分类型文档口径

- `ContractSearchDocument`：承接合同编号、名称、相对方、分类、摘要、风险摘要、台账筛选字段。
- `DocumentSearchDocument`：承接文档标题、文档角色、版本摘要、`OCR` 文本片段、页级命中锚点。
- `ClauseSearchDocument`：承接条款标题、条款正文摘要、语义标签、风险标签、适用范围和多语言摘要。
- `AiResultSearchDocument`：仅作为阶段三后扩展对象，承接已稳定 AI 结果的摘要、证据摘要、来源回指和风险提示，不承接原始模型长文。

### 4.5 结果快照与过期治理

搜索结果快照统一落到 `ia_search_result_set` 和缓存层，治理规则如下：

- 结果集主键 `result_set_id` 表达一次查询的稳定候选集，不复用业务对象主键。
- `ia_search_result_set` 当前正式承接字段以父文档定义为准，至少包括 `search_query_id`、`result_status`、`item_payload_json`、`facet_payload_json`、`total`、`ranking_profile_code`、`expires_at`、`cache_hit_flag`。
- 查询摘要、范围摘要、权限摘要和快照代次属于结果快照运行机制，可通过查询主表快照、缓存键、审计事件和任务上下文共同承接，但不在本文强制要求新增为 `ia_search_result_set` 当前正式字段。
- 结果快照默认短生命周期，用于分页复用、导出复用、AI 上下文装配复用和审计回放。
- 快照过期只影响查询复用，不影响索引文档本身；快照失效后允许按同一查询重算。
- 当权限域、来源摘要版本或查询条件变化导致原快照不再可信时，直接生成新快照，旧快照进入 `EXPIRED`。

## 5. 查询模型与稳定召回

### 5.1 查询归一模型

统一查询对象为 `SearchIntent`，至少包含以下维度：

- `query_text`：原始查询词
- `normalized_query_text`：归一化后的精确匹配词
- `fuzzy_query_text`：模糊匹配词
- `scope_list`：`CONTRACT` / `DOCUMENT` / `CLAUSE`，阶段三后可增量扩展 `AI_RESULT`
- `filter_payload_json`：组织、状态、分类、语言、风险等级、时间区间、文档角色等
- `sort_by` / `sort_order`
- `page` / `page_size`
- `export_flag`
- `aggregate_fields`

### 5.2 查询能力

搜索正式支持以下能力：

- 精准查询：面向合同编号、条款编码、文件编号、业务标签等稳定键。
- 模糊查询：面向标题、摘要、正文片段、关键词、相对方展示名等全文字段。
- 筛选：面向合同分类、组织、文档角色、语言、资源状态、风险等级、时间区间等结构化字段。
- 排序：面向得分、更新时间、签署日期、创建时间、风险等级、质量分、命中位置等。
- 分页：基于稳定快照分页，不允许同一结果集翻页过程中候选顺序漂移。
- 导出：导出基于已有结果快照，不重新临时跑一套未审计的查询。
- 聚合：支持合同状态、分类、语言、风险等级、组织维度的分面统计。
- 权限裁剪：在召回前后均执行，但以召回前范围裁剪为主，召回后只做最后兜底剔除。

### 5.3 稳定召回规则

- 先范围裁剪，再召回，再排序，再快照持久化。
- 同一查询在同一查询条件、同一权限快照语义、同一索引代次下必须得到稳定候选顺序。
- AI 结果类召回只作为阶段三后的增强召回层，不能压过合同、文档、条款等正式来源的稳定锚点结果。
- 命中片段必须返回 `source_anchor_json` 和 `source_version_digest`，不返回无锚点高亮文本。

### 5.4 导出与聚合边界

- 导出对象只允许来自已落库的 `result_set_id`，导出内容受同一次权限裁剪约束。
- 导出包只包含结果摘要、来源锚点和允许外露的字段；敏感正文仍按权限二次裁剪。
- 聚合统计以权限裁剪后的候选集为基础，不允许基于全量索引返回越权计数。

## 6. 刷新、重建、重放与版本切换

### 6.0 引擎适配职责与契约

搜索引擎内部实现可替换，但必须先收口到统一适配层。最小适配职责如下：

- `SearchEngineAdapter`：统一封装索引写入、批量删除、别名切换、查询执行、聚合查询和健康检查。
- `SearchDocumentMapper`：把 `SearchSourceEnvelope` 映射为引擎侧字段，不把引擎私有字段泄漏给业务模型。
- `SearchRebuildSwitcher`：承接双代构建、别名切换、回退切换和观察窗口治理。

适配契约最小要求如下：

- 上游只提交统一 `SearchDocument` 与统一查询模型，不感知引擎特定索引名、分词器名或厂商 DSL。
- 适配层必须支持 `upsert`、`mark-stale`、`bulk-rebuild`、`switch-generation`、`search`、`aggregate` 六类能力。
- 引擎切换不得改变 `ia_search_query`、`ia_search_result_set`、审计事件和来源锚点口径。

### 6.1 刷新触发源

以下事件必须触发搜索刷新：

- 合同主档摘要、分类主链、台账投影或风险摘要变化
- 文档中心 `current_version_id` 切换、文档标题变化、文档角色变化、能力挂接状态变化
- `OCR` 结果从不可用变为可用，或旧结果被 `SUPERSEDED`
- 条款版本启停、适用范围变化、语义标签变化、多语言摘要变化
- 阶段三后如接入 AI 结果索引，则其结果状态变化、结果失效、证据来源失效或摘要版本变化也必须触发扩展刷新
- 权限域、组织归属、密级标签变化导致可见范围变化

### 6.2 刷新类型

- 增量刷新：单来源对象变化后，仅重建其对应 `search_doc_id` 集合。
- 范围重建：单合同、单文档、单条款族或单组织范围的批量重建。
- 全量重建：新索引代次初始化、引擎切换、字段结构升级、严重数据漂移修正时触发。
- 结果快照重放：不重建索引，只按既有 `result_set_id`、查询快照语义和审计记录重放结果视图。
- 补数：历史漏刷、脏数据修正、失败任务补齐时，按来源范围补建缺失文档。

### 6.3 版本切换与双代运行

索引版本切换统一采用“双代构建 + 别名切换”策略：

1. 新建 `rebuild_generation`，在后台构建新一代索引文档。
2. 构建期间旧代继续承接查询，确保业务查询不被阻断。
3. 新代完成抽样校验、计数校验、权限校验后，切换查询别名到新代。
4. 旧代保留短观察窗口，仅用于审计回放与快速回退。
5. 观察窗口结束后，旧代进入清理流程。

### 6.4 文档版本切换治理

当文档中心主版本切换时，搜索侧必须同步执行：

- 旧 `document_version_id` 对应 `DocumentSearchDocument` 和 `OCR` 派生文档转为 `STALE`
- 新版本索引任务进入优先队列
- 旧结果快照若命中旧版本，则保留审计可查但退出默认业务曝光
- AI 上下文装配读取搜索结果时，默认只消费当前 `ACTIVE` / `READY` 代次

### 6.5 重放与补数规则

- 重放面向“证明当时查到了什么”，必须依赖 `result_set_id`、当时的查询快照语义和审计记录，不能直接读取当前最新索引代。
- 补数面向“把该有的索引补回来”，必须重新读取正式来源生成新的 `search_doc_id`，不从旧快照反推正式文档。
- 同一来源对象允许多次补数，但只允许一个当前 `ACTIVE` 代次对外默认曝光。

## 7. 权限裁剪与数据安全

### 7.1 权限裁剪链路

权限裁剪分三层执行：

1. 查询前范围裁剪：根据用户、角色、部门、数据范围、合同归属和文档访问权限，先收缩 `scope_list`。
2. 召回后对象裁剪：对召回候选逐条校验 `visibility_scope_json`、文档密级、条款适用范围，以及阶段三后 AI 结果扩展对象的来源可见性。
3. 返回前字段裁剪：对可见对象继续裁剪可外露字段，尤其是正文片段、后续扩展 AI 摘要和敏感标签。

### 7.2 权限口径

- 合同级对象按合同数据权限、组织范围和状态可见性裁剪。
- 文档级对象按文档中心访问权限、密级、文档角色和版本状态裁剪。
- 条款对象按资源启停状态、适用范围、组织范围和语种暴露范围裁剪。
- 阶段三后若接入 AI 结果对象，则按来源对象权限与结果状态双重裁剪；来源不可见则结果必不可见。

### 7.3 快照与导出安全

- 查询受理、结果快照、导出与 AI 消费必须复用同一权限快照语义，具体承接可通过查询主表、结果集、缓存键和审计事件共同完成。
- 若用户权限变化导致旧快照越权，旧快照立即进入 `EXPIRED`，不得继续复用。
- 导出任务复用搜索快照时，必须再次校验导出动作权限，避免“可看不可导”的口径冲突。

## 8. 上下游契约边界

### 8.1 与 `OCR` 的契约

- 搜索只消费 `ia_ocr_result` 的稳定结果对象，不直连引擎原始回调载荷。
- `OCR` 需向搜索提供 `ocr_result_id`、`document_version_id`、`content_fingerprint`、页级引用和质量发布标记。
- 搜索不会回改 `OCR` 结果状态；若索引失败，只创建搜索补偿任务。

### 8.2 与 AI 上下文装配的契约

- AI 上下文装配只消费 `SearchResultSnapshot` 或稳定 `SearchResultView`，不直接访问引擎实时召回结果。
- 搜索需输出：稳定排序候选、来源锚点、摘要片段、语言标记、权限快照和结果代次。
- AI 上下文装配不得要求搜索返回无锚点全文；如需更长正文，应按锚点二次回源读取正式来源。

### 8.3 与合同侧视图的契约

- 合同侧列表和详情页可消费搜索提供的结果视图、聚合计数和跳转锚点，但不得把搜索结果当作合同主状态解释器。
- 搜索需为合同侧提供稳定跳转对象：`contract_id`、`document_asset_id`、`document_version_id`、`page_no`、`semantic_ref_id`。
- 合同侧筛选面板复用搜索聚合时，以搜索快照为读模型，不回写合同写模型。

### 8.4 与文档中心的契约

- 文档中心通过 `dc_capability_binding(capability_type=SEARCH)` 维护文档版本与搜索挂接状态。
- 搜索成功后回写 `result_ref`、`result_checksum`、`last_synced_at` 到挂接摘要，不回写索引明细到文档中心私有表。
- 文档中心版本切换后，搜索必须尊重 `STALE` 语义，不继续把旧主版本当成默认曝光对象。

## 9. 并发、幂等、缓存、降级与失败恢复

### 9.1 并发控制

- 来源级锁：`doc_type + source_object_id + rebuild_generation`，避免同一来源被并发重复构建。
- 查询级锁：`normalized_query_text + scope_list + filter_payload_json + sort_by + sort_order + page + page_size + export_flag + aggregate_fields + permission_scope_digest + rebuild_generation`，避免同一查询在短时间内重复生成多个结果快照。
- 别名切换锁：全局单写，保证索引代次切换原子化。

### 9.2 幂等规则

- 刷新任务以 `source_object_id + source_version_digest + rebuild_generation` 命中幂等。
- 查询受理以 `normalized_query_text + scope_list + filter_payload_json + sort_by + sort_order + page + page_size + export_flag + aggregate_fields + permission_scope_digest + rebuild_generation` 命中幂等快照。
- 导出任务以 `result_set_id + export_profile_code` 命中幂等。

### 9.3 缓存策略

- `Redis` 仅缓存热点查询结果、分面统计、短期导出准备态和权限裁剪中间结果。
- 缓存失效后必须能从索引文档或正式来源重建，不能成为唯一结果来源。
- 缓存键统一带查询条件摘要、权限范围摘要和索引代次，避免串读旧权限结果。

### 9.4 降级策略

- 搜索引擎不可用时，允许降级为合同台账和结构化字段的有限查询。
- `OCR` 文本未就绪时，文档类查询可退化为文档标题、角色、版本摘要查询。
- 阶段三后如接入 AI 结果索引，其索引不可用时，禁止用未审计的实时模型结果替代搜索候选。
- 降级只影响召回深度，不影响合同主档、文档中心和条款库真相成立。

### 9.5 失败恢复

- 索引构建失败：保留来源对象和旧索引代次，记录失败码并重试或转补数。
- 结果快照失败：不影响索引文档；允许重新生成快照。
- 别名切换失败：继续保留旧代对外服务，回滚新代曝光。
- 全量重建中断：按 `rebuild_generation` 断点续跑，不重新清空正式来源。

## 10. 监控指标与审计

### 10.1 监控指标

- 索引指标：文档总量、按类型文档量、增量刷新延迟、全量重建耗时、别名切换耗时。
- 查询指标：查询 QPS、P95 / P99 时延、快照命中率、缓存命中率、导出成功率。
- 质量指标：零结果率、无锚点结果拦截数、权限剔除率、旧版本命中率、AI 结果越权拦截数。
- 恢复指标：失败任务堆积量、补数积压量、重建失败率、回退次数。

### 10.2 审计事件

以下动作必须写审计：

- 查询受理、权限裁剪、结果快照生成、导出生成
- 来源对象刷新、重建触发、补数执行、别名切换、回退切换
- 阶段三后如接入 AI 结果索引，其越权剔除、字段裁剪、快照过期

审计记录至少包含：操作者 / 系统、查询摘要、权限摘要、来源范围、索引代次、结果集主键、失败码和追踪号。

## 11. 验收清单

### 11.1 输入与索引验收

- 合同主档、文档中心、`OCR`、条款库都能进入阶段二统一索引治理链路；AI 可检索摘要只作为阶段三后增量接入能力。
- 任一索引文档都能回指到稳定来源锚点，不存在无主文本条目。
- 文档主版本切换后，旧版本默认曝光退出，新版本能异步接管索引曝光。

### 11.2 查询与权限验收

- 精准、模糊、筛选、排序、分页、导出、聚合能力口径一致。
- 权限裁剪先于快照持久化执行，导出与翻页不会复用越权结果。
- AI 结果不可见时，其来源命中也不会被错误放大为可见摘要。

### 11.3 重建与恢复验收

- 增量刷新、范围重建、全量重建、补数和结果快照重放均可独立执行。
- 引擎切换或索引结构升级时，可通过双代构建完成无中断切换。
- 搜索索引、缓存和结果快照全部可以从正式来源重新构建。

### 11.4 安全与审计验收

- 所有导出、快照动作都有审计记录；阶段三后如接入 AI 摘要暴露，也必须具备对应审计记录。
- 权限变化后旧快照能够失效，不能继续被业务侧或 AI 上下文装配复用。
- 搜索失效或降级不会破坏合同主档、文档中心和条款库的真相恢复能力。

## 12. 本文结论

本文将搜索能力收口为“稳定输入层 + 可重建索引层 + 可审计快照层”的三层结构：

- 输入来源只承接正式对象和受控派生结果
- 索引文档只承接稳定锚点、摘要版本和检索字段
- 查询结果只承接权限裁剪后的稳定候选集
- 快照、缓存、导出和 AI 消费全部以可失效、可重建、可审计为前提

该口径与 `intelligent-applications` 主线既有 `Detailed Design` 保持一致，并为后续 AI 上下文装配、合同侧视图和运维治理提供正式搜索输入基线。
