# 加密算法与密钥层级专项设计

## 1. 文档定位

本文档是 `encrypted-document` 子模块的**加密算法与密钥层级治理设计**，承接 [`detailed-design.md`](../detailed-design.md) 第 12 节中"加密算法选择、密钥层级、密钥轮换、介质保护和托管实现"的专项下沉。

本文档只回答以下问题：
- 加密算法如何治理：对象是谁、稳定锚点在哪里、版本如何归属、最小治理单元是什么
- 密钥层级如何划分：治理边界在哪里、轮换策略由谁负责
- 介质保护和密钥托管：治理对象是什么、与 `contract-core`、`document-center` 的边界如何划分
- 算法/密钥/轮换策略的审计最小单元是什么

本文档**不承担**以下内容：
- 不展开 API 路径、请求/响应结构、错误码
- 不输出 DDL、表结构、索引细节
- 不编写算法代码、密钥生成代码、介质读写代码
- 不包含实施排期、工时估算、负责人分配
- 不写成运维手册、部署方案、故障排查指南

## 2. 加密算法治理

### 2.1 治理对象

加密算法治理的**一级对象**是 `encryption-profile`（加密方案），每个方案封装一组算法选择、参数配置与安全基线。

治理对象层级：

| 层级 | 对象 | 说明 |
| --- | --- | --- |
| 一级 | `encryption-profile` | 加密方案，是算法治理的最小单元 |
| 二级 | `algorithm-spec` | 算法规格，描述具体算法、模式、参数 |
| 三级 | `parameter-binding` | 参数绑定，将算法规格与具体执行参数关联 |

`encryption-profile` 不持有密钥材料，只持有"使用什么算法、以什么参数、达到什么安全基线"的策略描述。

### 2.2 稳定锚点

加密算法治理的**稳定锚点**是 `profile-code`（方案编码），其稳定性约束如下：

- `profile-code` 是跨模块、跨版本的稳定引用键，文档中心、合同主档、审计链路均通过 `profile-code` 引用加密方案。
- `profile-code` 一旦发布，不允许修改其核心算法标识；若需变更算法，只能创建新 `profile-code`。
- `profile-code` 与 `document-center` 的松耦合：文档中心只引用 `profile-code`，不感知算法规格内部细节。
- `profile-code` 与 `contract-core` 的松耦合：合同主档不持有加密算法状态，仅通过 `document_asset_id` 间接关联到 `encryption-profile`。

### 2.3 版本归属

加密算法版本归属规则：

| 版本维度 | 归属主体 | 说明 |
| --- | --- | --- |
| `profile-version` | `encrypted-document` 子模块 | 加密方案自身的版本演进，由加密模块独立治理 |
| `algorithm-version` | `encrypted-document` 子模块 | 算法规格版本，随安全评估或合规要求演进 |
| `binding-version` | `encrypted-document` 子模块 | 参数绑定版本，跟随方案版本联动 |
| `document-version` | `document-center` | 文档版本，加密模块只引用，不拥有 |
| `contract-version` | `contract-core` | 合同版本，加密模块不直接访问 |

版本归属的核心原则：**加密算法版本由加密模块自治，文档版本和合同版本由各自归属模块治理，通过 `profile-code` 建立跨模块关联**。

### 2.4 最小治理单元

加密算法的最小治理单元是 **`encryption-profile` + `profile-version` 的组合**，具备以下治理属性：

- 可独立启用/停用
- 可独立审计（谁在何时启用了哪个方案）
- 可独立评估（安全基线是否达标）
- 可独立下线（标记废弃，不影响历史加密结果的可解密性）

最小治理单元**不包含**具体密钥材料、具体文档对象、具体合同对象，保持策略层与执行层分离。

## 3. 密钥层级治理

### 3.1 密钥层级结构

密钥层级按"平台级 → 方案级 → 文档级"三级组织，各级治理边界如下：

```
平台级密钥（platform-key）
  └── 方案级密钥（profile-key / key-encryption-key）
        └── 文档级密钥（document-key / data-encryption-key）
```

| 层级 | 密钥类型 | 治理主体 | 说明 |
| --- | --- | --- | --- |
| 平台级 | 主密钥 / 根密钥 | 平台运维 / 安全管理员 | 保护方案级密钥，不直接加密业务数据 |
| 方案级 | 方案密钥 / KEK | `encrypted-document` | 保护文档级密钥，按 `encryption-profile` 组织 |
| 文档级 | 文档密钥 / DEK | `encrypted-document` | 加密具体文档内容，按文档版本独立生成 |

### 3.2 治理边界

密钥层级的治理边界：

**`encrypted-document` 负责的边界：**
- 方案级密钥的生成、存储、轮换、归档
- 文档级密钥的派生、封装、存储引用、销毁
- 密钥与 `encryption-profile` 的绑定关系
- 密钥生命周期状态机（激活、轮换中、已归档、已销毁）

**`encrypted-document` 不负责的边界：**
- 平台级主密钥的托管方式（由平台运维/安全管理员治理，加密模块只消费其保护能力）
- 文档内容本身（由 `document-center` 治理）
- 合同业务上下文（由 `contract-core` 治理）

**`document-center` 的边界：**
- 持有文档对象、版本链、存储定位
- 通过 `encryption-profile-code` 引用加密方案
- 不持有任何密钥材料，不感知密钥层级结构

**`contract-core` 的边界：**
- 持有合同主档、业务状态、生命周期
- 不直接参与密钥层级治理
- 通过 `document_asset_id` → `encryption-profile-code` 间接关联

### 3.3 密钥轮换策略治理责任

密钥轮换策略按层级划分治理责任：

| 轮换类型 | 治理主体 | 触发条件 | 影响范围 |
| --- | --- | --- | --- |
| 平台级主密钥轮换 | 平台运维 / 安全管理员 | 合规要求、安全事件、定期轮换策略 | 全平台方案级密钥均需重新封装 |
| 方案级密钥轮换 | `encrypted-document` | 方案配置、密钥年龄、安全策略 | 该方案下全部文档级密钥需重新封装 |
| 文档级密钥轮换 | `encrypted-document` | 文档新版本、密钥泄露事件、按需轮换 | 仅影响单个文档版本 |

轮换策略的治理原则：
- **轮换不破坏历史可解密性**：旧密钥不下线，进入归档状态，保留至所有依赖文档不再需要解密
- **轮换是版本化操作**：每次轮换生成新版本密钥，旧版本标记为归档，不物理删除
- **轮换与文档版本解耦**：文档新版本可使用新密钥，历史版本继续使用原密钥，互不影响

## 4. 介质保护与密钥托管治理

### 4.1 介质保护治理对象

介质保护治理对象是**加密文档的存储引用与存储介质访问凭证**，不涉及文档内容本身。

| 治理对象 | 归属主体 | 说明 |
| --- | --- | --- |
| 存储定位引用 | `document-center` | 文档中心持有文件对象的存储位置真相 |
| 介质访问凭证 | `encrypted-document` | 加密模块持有解密所需的最小访问凭证 |
| 受控介质标签 | `encrypted-document` | 标记介质是否受控、加密状态、访问限制 |

介质保护**不包含**文档内容明文、文档版本内容、临时解密缓冲区——这些属于执行层，不在治理设计范围内。

### 4.2 密钥托管治理对象

密钥托管治理对象是**密钥材料的存储形式与访问控制策略**，不涉及密钥生成算法或加密运算。

| 治理对象 | 归属主体 | 说明 |
| --- | --- | --- |
| 密钥存储形式 | `encrypted-document` | 密钥如何持久化、如何被主密钥保护 |
| 密钥访问策略 | `encrypted-document` | 谁可以在什么条件下访问哪些密钥 |
| 密钥托管快照 | `encrypted-document` | 密钥托管状态的可审计快照 |

密钥托管与 `document-center` 的边界：
- `document-center` 不持有密钥托管任何状态
- `document-center` 只通过 `encryption-profile-code` 间接感知"该文档是否受加密治理"
- 密钥托管变更不影响文档中心的文件对象真相

密钥托管与 `contract-core` 的边界：
- `contract-core` 不直接参与密钥托管治理
- 合同主档通过 `document_asset_id` → `encryption-profile-code` 间接关联
- 密钥托管状态变更不反向改写合同主档状态

### 4.3 跨模块引用关系

```
contract-core (合同主档)
  └── contract_id
        └── document-center (文档资产)
              └── document_asset_id
                    └── encrypted-document (加密方案引用)
                          └── encryption-profile-code
                                └── encrypted-document (密钥层级)
                                      └── platform-key / profile-key / document-key
```

引用关系原则：
- **自上而下引用**：合同 → 文档 → 加密方案 → 密钥层级
- **禁止反向依赖**：密钥层级不持有合同主档引用，不持有文档内容引用
- **松耦合**：`profile-code` 是唯一跨模块稳定键，各模块不直接依赖对方内部模型

## 5. 审计最小单元

### 5.1 算法审计最小单元

算法审计最小单元是 **`encryption-profile` + `profile-version` + `algorithm-spec`**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 方案编码 | `profile-code` |
| 方案版本 | `profile-version` |
| 算法标识 | `algorithm-spec-id` |
| 变更类型 | 新增 / 启用 / 停用 / 参数调整 / 版本升级 |
| 变更主体 | 操作人 / 系统 / 策略触发 |
| 变更时间 | 操作时间戳 |
| 影响范围 | 受影响的文档范围（不展开到具体文档列表，只记录范围描述） |

### 5.2 密钥审计最小单元

密钥审计最小单元是 **密钥层级 + 密钥版本 + 操作类型**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 密钥层级 | 平台级 / 方案级 / 文档级 |
| 密钥标识 | 密钥 ID（非密钥材料） |
| 密钥版本 | 密钥版本号 |
| 操作类型 | 生成 / 轮换 / 归档 / 销毁 / 访问 / 封装 / 解封 |
| 操作主体 | 操作人 / 系统 / 自动策略 |
| 操作时间 | 操作时间戳 |
| 关联对象 | 关联的 `profile-code`（不展开到具体文档或合同） |

### 5.3 轮换策略审计最小单元

轮换策略审计最小单元是 **轮换策略 + 执行记录 + 影响范围**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 轮换策略编码 | `rotation-policy-code` |
| 轮换层级 | 平台级 / 方案级 / 文档级 |
| 执行触发源 | 手动 / 定时 / 事件驱动 |
| 执行状态 | 准备 / 执行中 / 完成 / 失败 / 回滚 |
| 执行时间 | 开始时间 / 完成时间 |
| 影响密钥数量 | 受影响的密钥数量汇总（不包含密钥材料） |
| 失败原因 | 失败时的原因编码 |

### 5.4 审计关联约束

- 审计事件只记录治理动作，不记录加密/解密执行细节（执行细节由 `ed_encryption_audit_event` 记录）
- 审计事件中的密钥标识不包含密钥材料本身，只保留密钥 ID 和版本号
- 审计事件与 `contract-core`、`document-center` 的关联通过 `profile-code` 间接建立，不直接引用合同或文档内部状态
- 审计事件保留期限按合规要求设定，与密钥生命周期解耦
