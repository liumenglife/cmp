# 明文导出包专项设计

## 1. 文档定位

本文档是 `encrypted-document` 子模块的**明文导出包治理设计**，承接 [`detailed-design.md`](../detailed-design.md) 第 12 节中"明文导出包格式、水印、脱敏选项、文件名规范"的专项下沉。

本文档只回答以下问题：
- 明文导出包如何治理：对象是谁、稳定锚点在哪里、版本如何归属、最小治理单元是什么
- 水印选项的治理边界在哪里、责任如何划分
- 脱敏选项的治理边界在哪里、责任如何划分
- 文件名规范的治理对象是什么、与导出包的关系如何
- 与第一、二份专项设计的引用关系
- 与 `contract-core`、`document-center` 的边界如何划分

本文档**不承担**以下内容：
- 不展开 API 路径、请求/响应结构、错误码
- 不输出 DDL、表结构、索引细节
- 不编写导出包格式代码、水印注入代码、脱敏处理代码
- 不包含实施排期、工时估算、负责人分配
- 不写成运维手册、部署方案、故障排查指南
- 不描述具体包格式代码、水印渲染代码、脱敏算法代码

## 2. 明文导出包治理

### 2.1 治理对象

明文导出包治理的**一级对象**是 `export-package`（导出包），每个导出包封装一次解密下载作业的受控导出结果。

治理对象层级：

| 层级 | 对象 | 说明 |
| --- | --- | --- |
| 一级 | `export-package` | 导出包，是明文导出治理的最小单元 |
| 二级 | `package-spec` | 包规格，描述导出包的格式约束、组成结构 |
| 三级 | `package-metadata` | 包元数据，记录导出包的溯源、时效、归属信息 |

`export-package` 不持有明文文件内容本身，只持有"某次下载作业导出的受控结果是什么格式、包含什么元数据、受什么约束"的治理描述。

### 2.2 稳定锚点

明文导出包治理的**稳定锚点**是 `package-id`（导出包标识），其稳定性约束如下：

- `package-id` 是跨模块、跨版本的稳定引用键，文档中心、合同主档、审计链路均通过 `package-id` 引用导出包。
- `package-id` 一旦生成，不允许修改其核心属性（关联作业、导出时间、导出主体）；若需变更导出条件，只能创建新的导出作业生成新包。
- `package-id` 与 `document-center` 的松耦合：文档中心不持有导出包状态，只通过 `document_version_id` 间接感知"该文档是否有历史导出记录"。
- `package-id` 与 `contract-core` 的松耦合：合同主档不持有导出包状态，仅通过 `document_asset_id` → `package-id` 间接关联到导出历史。

### 2.3 版本归属

明文导出包版本归属规则：

| 版本维度 | 归属主体 | 说明 |
| --- | --- | --- |
| `package-version` | `encrypted-document` 子模块 | 导出包自身的版本演进，由加密模块独立治理 |
| `spec-version` | `encrypted-document` 子模块 | 包规格版本，随导出要求或合规要求演进 |
| `metadata-version` | `encrypted-document` 子模块 | 元数据版本，跟随包规格联动 |
| `document-version` | `document-center` | 文档版本，加密模块只引用，不拥有 |
| `contract-version` | `contract-core` | 合同版本，加密模块不直接访问 |
| `job-version` | `encrypted-document` | 下载作业版本，导出包依附于作业存在 |

版本归属的核心原则：**导出包版本由加密模块自治，文档版本和合同版本由各自归属模块治理，通过 `package-id` 建立跨模块关联**。

### 2.4 最小治理单元

明文导出包的最小治理单元是 **`export-package` + `package-spec` 的组合**，具备以下治理属性：

- 可独立溯源（关联到特定下载作业、特定主体、特定时间）
- 可独立审计（谁在何时导出了哪个文档的明文版本）
- 可独立失效（下载地址过期、导出包标记过期，不影响文档中心正式文件）
- 可独立追踪（通过 `job_id` 关联完整导出链路）

最小治理单元**不包含**具体明文文件内容、具体文件存储路径、具体下载地址——这些属于执行层，不在治理设计范围内。

## 3. 水印选项治理

### 3.1 治理边界

水印选项治理的**核心对象**是 `watermark-policy`（水印策略），定义导出包中是否嵌入水印、水印内容来源、水印呈现方式。

水印选项治理边界：

**`encrypted-document` 负责的边界：**
- 水印策略的治理规则（是否启用、水印类型、内容模板）
- 水印与导出包的绑定关系（哪个导出包应用哪套水印策略）
- 水印内容的数据源治理（可引用哪些上下文字段作为水印内容）
- 水印策略的审计要求（谁在何时对哪个导出包应用了什么水印）

**`encrypted-document` 不负责的边界：**
- 水印的具体渲染实现（由执行层或前端负责）
- 水印的视觉设计、字体、颜色、透明度、位置算法
- 水印内容的生成逻辑（如用户姓名、部门、时间戳的具体拼接方式）

**`document-center` 的边界：**
- 持有文档对象、版本链、存储定位
- 不感知水印策略内部细节
- 只通过 `package-id` 间接感知"该文档的导出包是否包含水印"

**`contract-core` 的边界：**
- 持有合同主档、业务状态、生命周期
- 不直接参与水印策略治理
- 通过 `document_asset_id` → `package-id` 间接关联

### 3.2 水印责任划分

水印选项的责任划分：

| 责任维度 | `encrypted-document` 负责 | `document-center` 负责 | `contract-core` 负责 |
| --- | --- | --- | --- |
| 水印策略定义 | ✅ 是否启用、模板配置 | ❌ 不感知 | ❌ 不感知 |
| 水印内容数据源 | ✅ 定义可引用的上下文字段 | ❌ 不提供水印专用接口 | ✅ 提供合同上下文字段 |
| 水印与导出包绑定 | ✅ 记录绑定关系 | ❌ 不持有绑定状态 | ❌ 不持有绑定状态 |
| 水印渲染执行 | ❌ 不负责（执行层负责） | ❌ 不负责 | ❌ 不负责 |
| 水印审计 | ✅ 记录水印应用事件 | ❌ 不记录 | ❌ 不记录 |

### 3.3 水印治理约束

水印选项必须满足以下治理约束：

| 约束维度 | 治理要求 |
| --- | --- |
| 内容可追溯 | 水印内容必须包含可追溯到导出主体的信息（如用户ID、部门） |
| 时效可判定 | 水印必须包含导出时间，便于判断导出文件的新鲜度 |
| 来源可溯源 | 水印必须能追溯到原文档、原合同，不形成孤立明文文件 |
| 策略可审计 | 水印的启用/禁用、模板变更必须形成治理事件 |
| 内容不篡改 | 水印一旦应用，治理层不提供修改或移除水印的接口 |

## 4. 脱敏选项治理

### 4.1 治理边界

脱敏选项治理的**核心对象**是 `desensitization-policy`（脱敏策略），定义导出包中哪些字段需要脱敏、脱敏规则、脱敏后的呈现方式。

脱敏选项治理边界：

**`encrypted-document` 负责的边界：**
- 脱敏策略的治理规则（是否启用、脱敏字段范围、脱敏算法类型）
- 脱敏策略与导出包的绑定关系（哪个导出包应用哪套脱敏策略）
- 脱敏策略的审计要求（谁在何时对哪个导出包应用了什么脱敏规则）

**`encrypted-document` 不负责的边界：**
- 脱敏的具体执行算法（由执行层负责）
- 脱敏后的数据格式、编码、结构变换细节
- 脱敏规则的语法解析或表达式求值

**`document-center` 的边界：**
- 持有文档对象、版本链、存储定位
- 不感知脱敏策略内部细节
- 只通过 `package-id` 间接感知"该文档的导出包是否经过脱敏处理"

**`contract-core` 的边界：**
- 持有合同主档、业务状态、生命周期
- 不直接参与脱敏策略治理
- 通过 `document_asset_id` → `package-id` 间接关联
- 可提供合同字段作为脱敏候选字段的数据源

### 4.2 脱敏责任划分

脱敏选项的责任划分：

| 责任维度 | `encrypted-document` 负责 | `document-center` 负责 | `contract-core` 负责 |
| --- | --- | --- | --- |
| 脱敏策略定义 | ✅ 字段范围、算法类型 | ❌ 不感知 | ❌ 不感知 |
| 脱敏字段数据源 | ✅ 定义可脱敏的字段范围 | ✅ 提供文档字段结构 | ✅ 提供合同字段结构 |
| 脱敏与导出包绑定 | ✅ 记录绑定关系 | ❌ 不持有绑定状态 | ❌ 不持有绑定状态 |
| 脱敏执行 | ❌ 不负责（执行层负责） | ❌ 不负责 | ❌ 不负责 |
| 脱敏审计 | ✅ 记录脱敏应用事件 | ❌ 不记录 | ❌ 不记录 |

### 4.3 脱敏治理约束

脱敏选项必须满足以下治理约束：

| 约束维度 | 治理要求 |
| --- | --- |
| 字段可配置 | 脱敏字段范围必须可配置，不允许硬编码脱敏字段列表 |
| 策略可组合 | 支持多套脱敏策略，不同导出场景可应用不同策略 |
| 执行可追溯 | 脱敏执行必须可追溯到具体的策略版本和配置快照 |
| 策略可审计 | 脱敏的启用/禁用、字段变更必须形成治理事件 |
| 原文不留存 | 脱敏执行后，治理层不保留原文与脱敏结果的映射关系 |

## 5. 文件名规范治理

### 5.1 治理对象

文件名规范治理的**核心对象**是 `filename-policy`（文件名策略），定义导出包中明文文件的命名规则、组成元素、编码约束。

| 治理对象 | 归属主体 | 说明 |
| --- | --- | --- |
| `filename-policy` | `encrypted-document` | 文件名策略，定义命名规则、组成元素、约束 |
| `filename-template` | `encrypted-document` | 文件名模板，定义可引用的变量和格式 |
| `filename-metadata` | `encrypted-document` | 文件名元数据，记录某次导出使用的文件名和模板版本 |

文件名规范**不包含**文件扩展名与格式关联、文件路径生成逻辑、文件系统兼容性处理——这些属于执行层，不在治理设计范围内。

### 5.2 治理约束

文件名规范必须满足以下治理约束：

| 约束维度 | 治理要求 |
| --- | --- |
| 可追溯性 | 文件名必须包含可追溯到原文档的标识（如文档ID或版本号） |
| 可识别性 | 文件名必须包含导出时间或导出批次信息，避免同名覆盖 |
| 可审计性 | 文件名生成规则变更必须形成治理事件 |
| 编码合规 | 文件名必须符合同一编码规范（治理层定义规范，执行层负责实现） |
| 不与真相冲突 | 导出文件名不替代文档中心正式文件名，两者各自独立治理 |

### 5.3 与导出包的关系

文件名规范与导出包的关系：

- 每个 `export-package` 必须关联一个 `filename-policy`
- 文件名在导出包生成时按模板和上下文变量生成
- 文件名变更不影响导出包的其他治理属性（水印、脱敏等独立治理）
- 文件名历史记录通过 `package-metadata` 中的 `export_filename` 字段追溯

## 6. 与第一、二份专项设计的引用关系

本文档与 [`crypto-algorithm-and-key-hierarchy-design.md`](crypto-algorithm-and-key-hierarchy-design.md) 和 [`controlled-read-handle-design.md`](controlled-read-handle-design.md) 的引用关系：

### 6.1 架构层级关系

```
加密算法与密钥层级专项设计（第一份）
  └── encryption-profile（加密方案）
        └── controlled-read-handle-design（第二份）
              └── read-handle（读取句柄）
                    └── plaintext-export-package-design（本文档）
                          └── export-package（导出包）
                                └── watermark-policy（水印策略，本文档）
                                └── desensitization-policy（脱敏策略，本文档）
                                └── filename-policy（文件名策略，本文档）
```

### 6.2 治理依赖关系

| 本文档依赖前两份的内容 | 说明 |
| --- | --- |
| `encryption-profile` | 导出包是解密下载的结果，依赖加密方案定义的原始保护边界 |
| `read-handle` | 导出包是受控读取的延伸结果，读取句柄决定访问模式，导出包决定输出形式 |
| `stream-segment-policy` | 导出包生成可能依赖流式读取能力，但治理层只关注输出结果，不关注流式细节 |
| `temporary-cache-policy` | 导出包生成可能使用临时缓存，但治理层只关注最终产物，不关注缓存细节 |

### 6.3 治理边界划分

- **第一份专项设计**负责：加密算法选择、密钥层级划分、密钥轮换策略、介质保护和密钥托管
- **第二份专项设计**负责：读取句柄治理、流式分段策略治理、临时缓存介质治理
- **本文档**负责：导出包治理、水印选项治理、脱敏选项治理、文件名规范治理
- **三个文档的共同约束**：都不持有文档明文内容、都不编写实现代码、都保持与 `document-center` 和 `contract-core` 的松耦合

## 7. 与 `contract-core`、`document-center` 的边界说明

### 7.1 与 `document-center` 的边界

| 边界维度 | `document-center` 负责 | `encrypted-document` 负责 |
| --- | --- | --- |
| 文件真相 | 持有文档资产、版本链、存储定位 | 不持有文件真相，只引用 `document_asset_id` |
| 版本管理 | 管理文档版本的新增、切换、失效 | 只感知当前版本，不管理版本生命周期 |
| 导出包状态 | 不持有导出包任何状态 | 持有导出包的治理状态、元数据、关联策略 |
| 文件名真相 | 持有文档中心正式文件名 | 只定义导出包的命名规范，不替代正式文件名 |
| 水印/脱敏 | 不参与水印/脱敏治理 | 负责定义水印/脱敏策略，执行层负责具体实现 |

引用关系：`document-center` ← `document_asset_id` ← `encrypted-document`（导出包）

### 7.2 与 `contract-core` 的边界

| 边界维度 | `contract-core` 负责 | `encrypted-document` 负责 |
| --- | --- | --- |
| 合同主档 | 持有合同业务状态、生命周期 | 不持有合同主档，只引用 `contract_id` |
| 业务权限 | 管理合同的业务级权限 | 只读取权限结果，不管理业务权限 |
| 组织归属 | 管理合同归属部门、负责人 | 只读取组织快照，不管理组织关系 |
| 导出溯源 | 不直接持有导出包引用 | 导出包通过 `contract_id` 溯源到合同，但不反向改写合同主档 |
| 水印内容源 | 可提供合同字段作为水印内容源 | 定义水印可引用的合同字段范围，不负责内容生成 |

引用关系：`contract-core` ← `contract_id` ← `encrypted-document`（导出包）

### 7.3 跨模块引用原则

- **自上而下引用**：合同 → 文档 → 加密方案 → 读取句柄 → 导出包
- **禁止反向依赖**：导出包不持有合同主档引用，不持有文档内容引用
- **松耦合**：`package-id`、`handle-id`、`profile-code` 是跨模块稳定键，各模块不直接依赖对方内部模型
- **治理隔离**：`contract-core` 不感知导出包的存在，`document-center` 只通过引用感知导出历史

## 8. 审计最小单元

### 8.1 导出包审计最小单元

导出包审计最小单元是 **`export-package` + `package-spec` + `job_id`**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 导出包标识 | `package-id` |
| 包规格标识 | `package-spec-id` |
| 关联作业 | `job_id`（非作业执行细节） |
| 变更类型 | 生成 / 交付 / 过期 / 失效 |
| 变更主体 | 操作人 / 系统 |
| 变更时间 | 操作时间戳 |
| 影响范围 | 受影响的文档与合同（不展开到具体文件内容） |

### 8.2 水印审计最小单元

水印审计最小单元是 **`watermark-policy` + 应用记录**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 水印策略编码 | `watermark-policy-code` |
| 水印类型 | 文本水印 / 图片水印 / 二维码水印 |
| 内容模板版本 | 水印内容模板版本号 |
| 应用类型 | 启用 / 禁用 / 模板变更 / 绑定到导出包 |
| 应用主体 | 操作人 / 系统 / 策略触发 |
| 应用时间 | 操作时间戳 |
| 关联导出包 | `package-id`（不展开到具体文档或合同） |

### 8.3 脱敏审计最小单元

脱敏审计最小单元是 **`desensitization-policy` + 应用记录**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 脱敏策略编码 | `desensitization-policy-code` |
| 脱敏字段范围 | 脱敏字段列表（不记录具体字段值） |
| 脱敏算法类型 | 脱敏算法标识（不记录算法实现） |
| 应用类型 | 启用 / 禁用 / 字段变更 / 绑定到导出包 |
| 应用主体 | 操作人 / 系统 / 策略触发 |
| 应用时间 | 操作时间戳 |
| 关联导出包 | `package-id`（不展开到具体文档或合同） |

### 8.4 文件名审计最小单元

文件名审计最小单元是 **`filename-policy` + 生成记录**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 文件名策略编码 | `filename-policy-code` |
| 文件名模板版本 | 模板版本号 |
| 生成类型 | 模板变更 / 应用到导出包 / 规范启用 / 规范禁用 |
| 生成主体 | 操作人 / 系统 / 策略触发 |
| 生成时间 | 操作时间戳 |
| 关联导出包 | `package-id`（不展开到具体文档或合同） |

### 8.5 审计关联约束

- 审计事件只记录治理动作，不记录导出包生成执行细节（执行细节由 `ed_encryption_audit_event` 记录）
- 审计事件中的导出包标识不包含文件内容或存储路径，只保留 `package-id` 和状态
- 审计事件与 `contract-core`、`document-center` 的关联通过 `package-id` 间接建立，不直接引用合同或文档内部状态
- 审计事件保留期限按合规要求设定，与导出包生命周期解耦
