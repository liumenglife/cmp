# 脱敏策略与二次存储专项设计

## 1. 文档定位

本文档是 `encrypted-document` 子模块的**脱敏策略与二次存储治理设计**，承接 [`detailed-design.md`](../detailed-design.md) 第 12 节中"搜索/AI 消费文本的脱敏策略与二次存储治理"的专项下沉。

本文档只回答以下问题：
- 脱敏策略如何治理：对象是谁、稳定锚点在哪里、版本如何归属、最小治理单元是什么
- 二次存储如何治理：边界在哪里、责任如何划分、与原始密文的关系是什么
- 搜索/AI 消费方如何通过脱敏文本与二次存储获得受控访问
- 与前四份专项设计的引用关系
- 与 `contract-core`、`document-center` 的边界如何划分

本文档**不承担**以下内容：
- 不展开 API 路径、请求/响应结构、错误码
- 不输出 DDL、表结构、索引细节
- 不编写脱敏算法代码、正则替换代码、掩码处理代码
- 不包含二次存储的具体技术方案（如搜索引擎索引结构、向量数据库配置）
- 不包含实施排期、工时估算、负责人分配
- 不写成运维手册、部署方案、故障排查指南
- 不描述具体脱敏规则实现、文本替换实现、索引构建实现

## 2. 脱敏策略治理

### 2.1 治理对象

脱敏策略治理的**一级对象**是 `desensitization-policy`（脱敏策略），每个策略封装一组针对特定消费方（搜索、AI）的文本脱敏规则与安全基线。

治理对象层级：

| 层级 | 对象 | 说明 |
| --- | --- | --- |
| 一级 | `desensitization-policy` | 脱敏策略，是脱敏治理的最小单元 |
| 二级 | `policy-spec` | 策略规格，描述脱敏维度、保留字段、替换规则 |
| 三级 | `policy-binding` | 策略绑定，将脱敏策略与消费方（搜索/AI）关联 |

`desensitization-policy` 不持有明文或密文内容，不持有二次存储实体，只持有"为哪个消费方、按什么维度、保留哪些信息、以什么规则脱敏"的治理描述。

### 2.2 稳定锚点

脱敏策略治理的**稳定锚点**是 `policy-code`（策略编码），其稳定性约束如下：

- `policy-code` 是跨模块、跨版本的稳定引用键，搜索服务、AI 服务、审计链路均通过 `policy-code` 引用脱敏策略。
- `policy-code` 一旦发布，不允许修改其核心脱敏维度（如哪些字段必须脱敏、哪些信息必须保留）；若需变更脱敏范围，只能创建新 `policy-code`。
- `policy-code` 与 `document-center` 的松耦合：文档中心只持有原始密文，不感知脱敏策略内部规则，脱敏由加密模块在受控读取后执行。
- `policy-code` 与 `contract-core` 的松耦合：合同主档不持有脱敏策略状态，脱敏策略仅通过 `document_asset_id` 间接关联到合同上下文。

### 2.3 版本归属

脱敏策略版本归属规则：

| 版本维度 | 归属主体 | 说明 |
| --- | --- | --- |
| `policy-version` | `encrypted-document` 子模块 | 脱敏策略自身的版本演进，由加密模块独立治理 |
| `spec-version` | `encrypted-document` 子模块 | 策略规格版本，随合规要求或隐私政策演进 |
| `binding-version` | `encrypted-document` 子模块 | 策略绑定版本，跟随策略版本联动 |
| `document-version` | `document-center` | 文档版本，加密模块只引用，不拥有 |
| `contract-version` | `contract-core` | 合同版本，加密模块不直接访问 |

版本归属的核心原则：**脱敏策略版本由加密模块自治，文档版本和合同版本由各自归属模块治理，通过 `policy-code` 建立跨模块关联**。

### 2.4 最小治理单元

脱敏策略的最小治理单元是 **`desensitization-policy` + `policy-spec` 的组合**，具备以下治理属性：

- 可独立启用/停用（关联消费方授权生效/失效）
- 可独立审计（谁在何时定义了哪个脱敏策略、应用于哪个消费方）
- 可独立评估（脱敏强度是否合规、是否过度保留敏感信息）
- 可独立下线（标记废弃，不影响历史二次存储的可审计性）

最小治理单元**不包含**具体脱敏算法实现、具体文本替换逻辑、具体正则规则——这些属于执行层，不在治理设计范围内。

## 3. 二次存储治理

### 3.1 治理对象

二次存储治理的**一级对象**是 `secondary-storage`（二次存储实体），每个实体封装一份脱敏后文本在特定消费方（搜索索引、AI 向量库等）中的存储治理描述。

治理对象层级：

| 层级 | 对象 | 说明 |
| --- | --- | --- |
| 一级 | `secondary-storage` | 二次存储实体，是存储治理的最小单元 |
| 二级 | `storage-mapping` | 存储映射，描述脱敏文本与原始密文的关联关系 |
| 三级 | `storage-lifecycle` | 存储生命周期，描述二次存储的创建、更新、过期、清理规则 |

`secondary-storage` 不持有原始密文或明文，不持有脱敏策略本身，只持有"为哪个消费方、关联哪个原始文档、按什么生命周期、存储什么脱敏结果"的治理描述。

### 3.2 稳定锚点

二次存储治理的**稳定锚点**是 `storage-id`（存储标识），其稳定性约束如下：

- `storage-id` 是跨模块、跨版本的稳定引用键，搜索服务、AI 服务、审计链路均通过 `storage-id` 引用二次存储实体。
- `storage-id` 与原始密文的关联通过 `document_version_id` + `policy-code` 建立，不直接持有密文内容或解密密钥。
- `storage-id` 与 `document-center` 的松耦合：文档中心只持有原始密文，二次存储由加密模块管理，文档中心不感知二次存储存在。
- `storage-id` 与 `contract-core` 的松耦合：合同主档不持有二次存储状态，二次存储仅通过 `contract_id` 间接关联到合同上下文。

### 3.3 版本归属

二次存储版本归属规则：

| 版本维度 | 归属主体 | 说明 |
| --- | --- | --- |
| `storage-version` | `encrypted-document` 子模块 | 二次存储实体自身的版本演进，由加密模块独立治理 |
| `mapping-version` | `encrypted-document` 子模块 | 存储映射版本，随文档版本或策略变更联动 |
| `lifecycle-version` | `encrypted-document` 子模块 | 生命周期版本，随合规要求或存储策略演进 |
| `document-version` | `document-center` | 文档版本，二次存储只引用，不拥有 |
| `contract-version` | `contract-core` | 合同版本，二次存储不直接访问 |

版本归属的核心原则：**二次存储版本由加密模块自治，文档版本和合同版本由各自归属模块治理，通过 `storage-id` 建立跨模块关联**。

### 3.4 最小治理单元

二次存储的最小治理单元是 **`secondary-storage` + `storage-lifecycle` 的组合**，具备以下治理属性：

- 可独立创建/更新/过期/清理（按生命周期规则自动或手动触发）
- 可独立审计（谁在何时为哪个文档创建了哪类二次存储）
- 可独立评估（存储必要性、保留期限是否合规）
- 可独立下线（标记废弃，不影响原始密文的可恢复性）

最小治理单元**不包含**具体存储技术方案（如搜索引擎索引结构、向量数据库配置）、具体文本格式——这些属于执行层，不在治理设计范围内。

### 3.5 治理边界与责任划分

二次存储的治理边界：

**`encrypted-document` 负责的边界：**
- 二次存储实体的治理规则（创建条件、更新触发、过期策略、清理规则）
- 二次存储与脱敏策略的绑定关系（哪个脱敏结果存储到哪个消费方）
- 二次存储与原始密文的关联关系（通过 `document_version_id` + `policy-code` 建立映射）
- 二次存储的生命周期治理（存活时间、刷新触发、废弃条件）
- 二次存储的审计要求（谁在何时为哪个文档创建了哪类二次存储）

**`encrypted-document` 不负责的边界：**
- 二次存储的具体技术实现（搜索索引构建、向量嵌入生成、数据库写入）
- 二次存储的查询接口（由消费方自行提供）
- 二次存储的性能优化（索引调优、向量检索优化）

**消费方（搜索服务、AI 服务）的边界：**
- 持有二次存储的具体实现（索引、向量库、缓存等）
- 通过 `storage-id` 引用二次存储实体，不感知脱敏策略内部规则
- 负责二次存储的查询、检索、推理等使用场景
- 不负责脱敏策略定义、不负责原始密文解密

**`document-center` 的边界：**
- 持有文档对象、版本链、密文存储定位
- 不感知二次存储存在，只通过 `document_version_id` 被间接引用
- 文档版本切换时，不直接触发二次存储更新，由加密模块监听版本变更事件后决策

**`contract-core` 的边界：**
- 持有合同主档、合同状态、合同生命周期
- 不感知二次存储存在，只通过 `contract_id` 被间接引用
- 合同状态变更时，不直接触发二次存储清理，由加密模块监听状态变更事件后决策

## 4. 搜索/AI 消费方适配治理

### 4.1 消费方接入模式

搜索/AI 消费方通过以下模式获得受控访问：

```
document-center（原始密文）
      ↓ 受控解密（通过 controlled-read-handle）
encrypted-document（明文临时读取）
      ↓ 应用 desensitization-policy
encrypted-document（脱敏文本）
      ↓ 写入 secondary-storage
消费方（搜索索引 / AI 向量库）
```

### 4.2 消费方治理边界

**搜索服务：**
- 引用 `policy-code` 获取适用于搜索的脱敏策略
- 持有搜索索引形式的二次存储
- 不感知原始密文、不持有解密密钥
- 只通过 `storage-id` 关联脱敏文本与搜索结果

**AI 服务：**
- 引用 `policy-code` 获取适用于 AI 消费的脱敏策略
- 持有向量嵌入或结构化文本形式的二次存储
- 不感知原始密文、不持有解密密钥
- 只通过 `storage-id` 关联脱敏文本与推理上下文

### 4.3 消费方与加密模块的边界

- 加密模块负责：脱敏策略定义、受控解密、脱敏执行、二次存储治理
- 消费方负责：二次存储的具体使用（搜索、推理）、消费方内部的权限控制
- 两者通过 `policy-code` 和 `storage-id` 建立松耦合关联

## 6. 与 `contract-core`、`document-center` 的边界说明

### 6.1 与 `document-center` 的边界

| 维度 | `document-center` 负责 | `encrypted-document` 负责 |
| --- | --- | --- |
| 文档存储 | 原始密文存储、版本链管理 | 不持有原始密文，只引用 `document_version_id` |
| 受控解密 | 不负责解密，只提供密文读取接口 | 通过受控读取句柄获取临时明文访问 |
| 脱敏处理 | 不感知脱敏策略 | 定义脱敏策略、执行脱敏、生成脱敏文本 |
| 二次存储 | 不感知二次存储存在 | 管理二次存储实体、生命周期、映射关系 |
| 版本变更事件 | 发布文档版本变更事件 | 监听事件，决策是否更新二次存储 |

### 6.2 与 `contract-core` 的边界

| 维度 | `contract-core` 负责 | `encrypted-document` 负责 |
| --- | --- | --- |
| 合同主档 | 合同信息、状态管理、生命周期 | 不持有合同主档，只引用 `contract_id` |
| 状态变更事件 | 发布合同状态变更事件 | 监听事件，决策是否清理二次存储 |
| 脱敏策略 | 不感知脱敏策略 | 定义脱敏策略，与合同无直接状态耦合 |
| 二次存储 | 不感知二次存储存在 | 管理二次存储，通过 `contract_id` 间接关联 |

## 7. 治理闭环

脱敏策略与二次存储的治理闭环：

1. **策略定义**：`encrypted-document` 定义 `desensitization-policy`，绑定到消费方
2. **受控解密**：通过受控读取句柄获取临时明文访问
3. **脱敏执行**：应用 `desensitization-policy`，生成脱敏文本
4. **二次存储**：创建 `secondary-storage`，关联 `policy-code` 与 `document_version_id`
5. **消费方使用**：搜索/AI 服务通过 `storage-id` 使用脱敏文本
6. **生命周期管理**：按 `storage-lifecycle` 自动过期/清理
7. **审计追踪**：全链路审计 `policy-code`、`storage-id`、操作主体、时间戳

治理闭环的核心原则：**脱敏文本只在受控解密后生成，二次存储只持有脱敏结果，不持有原始密文或明文，所有操作可审计、可追踪、可恢复（以原始密文为真相源）**。
