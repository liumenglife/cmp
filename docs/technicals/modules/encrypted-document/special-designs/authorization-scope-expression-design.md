# 授权范围表达式专项设计

## 1. 文档定位

本文档是 `encrypted-document` 子模块的**授权范围表达式治理设计**，承接 [`detailed-design.md`](../detailed-design.md) 第 12 节中"授权范围表达式是否扩展到更细粒度字段、标签或业务规则"的专项下沉。

本文档只回答以下问题：
- 授权范围表达式如何治理：对象是谁、稳定锚点在哪里、版本如何归属、最小治理单元是什么

- 与 `contract-core`、`document-center` 的边界如何划分

本文档**不承担**以下内容：
- 不展开 API 路径、请求/响应结构、错误码
- 不输出 DDL、表结构、索引细节
- 不编写表达式解析代码、字段匹配代码、标签计算代码、规则引擎代码

## 2. 授权范围表达式治理

### 2.1 治理对象

授权范围表达式治理的**一级对象**是 `scope-expression`（范围表达式），每个表达式封装一次授权规则的适用范围判定逻辑。

治理对象层级：

| 层级 | 对象 | 说明 |
| --- | --- | --- |
| 一级 | `scope-expression` | 范围表达式，是授权范围治理的最小单元 |
| 二级 | `expression-spec` | 表达式规格，描述表达式的语法类型、匹配维度、求值约束 |
| 三级 | `expression-binding` | 表达式绑定，将表达式与具体授权规则关联 |

`scope-expression` 不持有授权规则本身，不持有文档内容，只持有"某次授权适用于哪些对象、按什么维度匹配、以什么规则过滤"的治理描述。

### 2.2 稳定锚点

授权范围表达式治理的**稳定锚点**是 `expression-code`（表达式编码），其稳定性约束如下：

- `expression-code` 是跨模块、跨版本的稳定引用键，文档中心、合同主档、审计链路均通过 `expression-code` 引用范围表达式。
- `expression-code` 一旦发布，不允许修改其核心匹配维度；若需变更适用范围，只能创建新 `expression-code`。
- `expression-code` 与 `document-center` 的松耦合：文档中心只引用 `expression-code` 作为授权范围标识，不感知表达式内部匹配逻辑。
- `expression-code` 与 `contract-core` 的松耦合：合同主档不持有范围表达式状态，仅通过 `authorization_id` → `expression-code` 间接关联到授权范围。

### 2.3 版本归属

授权范围表达式版本归属规则：

| 版本维度 | 归属主体 | 说明 |
| --- | --- | --- |
| `expression-version` | `encrypted-document` 子模块 | 范围表达式自身的版本演进，由加密模块独立治理 |
| `spec-version` | `encrypted-document` 子模块 | 表达式规格版本，随授权策略或合规要求演进 |
| `binding-version` | `encrypted-document` 子模块 | 表达式绑定版本，跟随表达式版本联动 |
| `document-version` | `document-center` | 文档版本，加密模块只引用，不拥有 |
| `contract-version` | `contract-core` | 合同版本，加密模块不直接访问 |
| `authorization-version` | `encrypted-document` | 授权规则版本，表达式依附于授权规则存在 |

版本归属的核心原则：**范围表达式版本由加密模块自治，文档版本和合同版本由各自归属模块治理，通过 `expression-code` 建立跨模块关联**。

### 2.4 最小治理单元

授权范围表达式的最小治理单元是 **`scope-expression` + `expression-spec` 的组合**，具备以下治理属性：

- 可独立启用/停用（关联授权规则生效/失效）
- 可独立审计（谁在何时定义了哪个范围表达式）
- 可独立评估（表达式覆盖范围是否合规、是否过度授权）
- 可独立下线（标记废弃，不影响历史授权作业的可审计性）

最小治理单元**不包含**具体文档字段值、具体标签计算结果、具体业务规则执行细节——这些属于执行层，不在治理设计范围内。

## 3. 细粒度字段扩展治理

### 3.1 治理边界

细粒度字段扩展治理的**核心对象**是 `field-scope-policy`（字段范围策略），定义授权范围是否扩展到文档字段、合同字段或其他业务字段，以及字段匹配的治理规则。

细粒度字段扩展治理边界：

**`encrypted-document` 负责的边界：**
- 字段范围策略的治理规则（是否允许按字段过滤、可引用哪些字段类别）
- 字段范围与授权规则的绑定关系（哪个授权规则应用哪套字段范围策略）
- 字段范围的数据源治理（可引用哪些上下文字段作为匹配条件）
- 字段范围的审计要求（谁在何时对哪个授权规则应用了什么字段范围）

**`encrypted-document` 不负责的边界：**
- 字段值的具体提取实现（由执行层或文档中心负责）
- 字段匹配的具体算法（等于、包含、正则、范围比较等由执行层负责）
- 字段结构的解析逻辑（如 JSON 路径、XML 路径、表格列映射）

**`document-center` 的边界：**
- 持有文档对象、版本链、存储定位
- 可提供文档元数据字段作为字段范围的候选数据源
- 不感知字段范围策略内部细节，只通过 `expression-code` 间接感知"该文档是否在某个字段范围内"

**`contract-core` 的边界：**
- 持有合同主档、业务字段、生命周期
- 可提供合同字段作为字段范围的候选数据源
- 不直接参与字段范围策略治理
- 通过 `authorization_id` → `expression-code` 间接关联

### 3.2 字段扩展责任划分

细粒度字段扩展的责任划分：

| 责任维度 | `encrypted-document` 负责 | `document-center` 负责 | `contract-core` 负责 |
| --- | --- | --- | --- |
| 字段范围策略定义 | ✅ 是否启用、可引用字段类别 | ❌ 不感知 | ❌ 不感知 |
| 字段数据源注册 | ✅ 定义可引用的字段范围 | ✅ 提供文档元数据字段清单 | ✅ 提供合同业务字段清单 |
| 字段与授权绑定 | ✅ 记录绑定关系 | ❌ 不持有绑定状态 | ❌ 不持有绑定状态 |
| 字段值提取执行 | ❌ 不负责（执行层负责） | ✅ 可提供文档字段值提取能力 | ✅ 可提供合同字段值查询能力 |
| 字段匹配执行 | ❌ 不负责（执行层负责） | ❌ 不负责 | ❌ 不负责 |
| 字段范围审计 | ✅ 记录字段范围应用事件 | ❌ 不记录 | ❌ 不记录 |

### 3.3 字段扩展治理约束

细粒度字段扩展必须满足以下治理约束：

| 约束维度 | 治理要求 |
| --- | --- |
| 字段可配置 | 可引用的字段范围必须可配置，不允许硬编码字段列表 |
| 数据源可溯源 | 字段值必须能追溯到其来源模块（文档中心或合同主档） |
| 过度授权防护 | 字段范围策略必须可评估是否构成过度授权，不允许无边界字段匹配 |
| 策略可审计 | 字段范围的启用/禁用、字段变更必须形成治理事件 |
| 执行不留存 | 字段匹配执行后，治理层不保留字段值与匹配结果的映射关系 |

## 4. 标签扩展治理

### 4.1 治理边界

标签扩展治理的**核心对象**是 `tag-scope-policy`（标签范围策略），定义授权范围是否扩展到标签体系、标签分类、标签继承关系，以及标签匹配的治理规则。

标签扩展治理边界：

**`encrypted-document` 负责的边界：**
- 标签范围策略的治理规则（是否允许按标签过滤、可引用哪些标签体系）
- 标签范围与授权规则的绑定关系（哪个授权规则应用哪套标签范围策略）
- 标签范围的数据源治理（可引用哪些标签作为匹配条件）
- 标签范围的审计要求（谁在何时对哪个授权规则应用了什么标签范围）

**`encrypted-document` 不负责的边界：**
- 标签的具体计算实现（由执行层或标签治理模块负责）
- 标签继承逻辑（父子标签、部门标签、角色标签等由标签体系负责）
- 标签的存储与索引实现（由标签治理模块负责）

**`document-center` 的边界：**
- 持有文档对象、版本链、存储定位
- 可提供文档标签作为标签范围的候选数据源
- 不感知标签范围策略内部细节，只通过 `expression-code` 间接感知"该文档是否具有某个标签"

**`contract-core` 的边界：**
- 持有合同主档、业务标签、生命周期
- 可提供合同标签作为标签范围的候选数据源
- 不直接参与标签范围策略治理
- 通过 `authorization_id` → `expression-code` 间接关联

### 4.2 标签扩展责任划分

标签扩展的责任划分：

| 责任维度 | `encrypted-document` 负责 | `document-center` 负责 | `contract-core` 负责 |
| --- | --- | --- | --- |
| 标签范围策略定义 | ✅ 是否启用、可引用标签体系 | ❌ 不感知 | ❌ 不感知 |
| 标签数据源注册 | ✅ 定义可引用的标签范围 | ✅ 提供文档标签清单 | ✅ 提供合同标签清单 |
| 标签与授权绑定 | ✅ 记录绑定关系 | ❌ 不持有绑定状态 | ❌ 不持有绑定状态 |
| 标签计算执行 | ❌ 不负责（标签体系负责） | ✅ 可提供文档标签查询能力 | ✅ 可提供合同标签查询能力 |
| 标签匹配执行 | ❌ 不负责（执行层负责） | ❌ 不负责 | ❌ 不负责 |
| 标签范围审计 | ✅ 记录标签范围应用事件 | ❌ 不记录 | ❌ 不记录 |

### 4.3 标签扩展治理约束

标签扩展必须满足以下治理约束：

| 约束维度 | 治理要求 |
| --- | --- |
| 标签可配置 | 可引用的标签范围必须可配置，不允许硬编码标签列表 |
| 标签体系隔离 | 文档标签与合同标签必须分体系治理，不允许混用标签空间 |
| 标签继承可追踪 | 标签继承关系必须可追踪，不允许隐式标签匹配 |
| 策略可审计 | 标签范围的启用/禁用、标签变更必须形成治理事件 |
| 标签不扩散 | 标签范围策略不允许自动扩散到未授权的标签体系 |

## 5. 业务规则扩展治理

### 5.1 治理边界

业务规则扩展治理的**核心对象**是 `rule-scope-policy`（规则范围策略），定义授权范围是否扩展到业务规则、规则类型、规则优先级，以及规则匹配的治理规则。

业务规则扩展治理边界：

**`encrypted-document` 负责的边界：**
- 规则范围策略的治理规则（是否允许按业务规则过滤、可引用哪些规则类型）
- 规则范围与授权规则的绑定关系（哪个授权规则应用哪套规则范围策略）
- 规则范围的数据源治理（可引用哪些业务规则作为匹配条件）
- 规则范围的审计要求（谁在何时对哪个授权规则应用了什么规则范围）

**`encrypted-document` 不负责的边界：**
- 业务规则的具体执行实现（由执行层或规则引擎负责）
- 规则优先级计算（由规则引擎负责）
- 规则的条件求值（由规则引擎负责）

**`document-center` 的边界：**
- 持有文档对象、版本链、存储定位
- 不直接参与规则范围策略治理
- 只通过 `expression-code` 间接感知"该文档是否符合某个业务规则"

**`contract-core` 的边界：**
- 持有合同主档、业务规则、生命周期
- 可提供合同业务规则作为规则范围的候选数据源
- 不直接参与规则范围策略治理
- 通过 `authorization_id` → `expression-code` 间接关联
- 可提供规则引擎能力作为业务规则的数据源

### 5.2 业务规则扩展责任划分

业务规则扩展的责任划分：

| 责任维度 | `encrypted-document` 负责 | `document-center` 负责 | `contract-core` 负责 |
| --- | --- | --- | --- |
| 规则范围策略定义 | ✅ 是否启用、可引用规则类型 | ❌ 不感知 | ❌ 不感知 |
| 规则数据源注册 | ✅ 定义可引用的规则范围 | ❌ 不提供文档规则 | ✅ 提供合同业务规则清单 |
| 规则与授权绑定 | ✅ 记录绑定关系 | ❌ 不持有绑定状态 | ❌ 不持有绑定状态 |
| 规则执行 | ❌ 不负责（规则引擎负责） | ❌ 不负责 | ✅ 可提供规则引擎能力 |
| 规则匹配执行 | ❌ 不负责（执行层负责） | ❌ 不负责 | ❌ 不负责 |
| 规则范围审计 | ✅ 记录规则范围应用事件 | ❌ 不记录 | ❌ 不记录 |

### 5.3 业务规则扩展治理约束

业务规则扩展必须满足以下治理约束：

| 约束维度 | 治理要求 |
| --- | --- |
| 规则可配置 | 可引用的业务规则范围必须可配置，不允许硬编码规则列表 |
| 规则类型隔离 | 不同类型业务规则必须分类型治理，不允许混用规则空间 |
| 规则优先级可追踪 | 规则优先级必须可追踪，不允许隐式优先级覆盖 |
| 策略可审计 | 规则范围的启用/禁用、规则变更必须形成治理事件 |
| 规则不越界 | 规则范围策略不允许越过加密模块职责边界，去判定非安全治理类的业务规则 |

## 6. 与前三份专项设计的引用关系

本文档与 [`crypto-algorithm-and-key-hierarchy-design.md`](crypto-algorithm-and-key-hierarchy-design.md)、[`controlled-read-handle-design.md`](controlled-read-handle-design.md) 和 [`plaintext-export-package-design.md`](plaintext-export-package-design.md) 的引用关系：

### 6.1 架构层级关系

```
加密算法与密钥层级专项设计（第一份）
  └── encryption-profile（加密方案）
        └── controlled-read-handle-design（第二份）
              └── read-handle（读取句柄）
                    └── plaintext-export-package-design（第三份）
                          └── export-package（导出包）
                                └── authorization-scope-expression-design（本文档）
                                      └── scope-expression（范围表达式）
                                            └── field-scope-policy（字段范围策略，本文档）
                                            └── tag-scope-policy（标签范围策略，本文档）
                                            └── rule-scope-policy（规则范围策略，本文档）
```

### 6.2 治理依赖关系

| 本文档依赖前三份的内容 | 说明 |
| --- | --- |
| `encryption-profile` | 范围表达式服务于解密下载授权，依赖加密方案定义的原始保护边界 |
| `read-handle` | 范围表达式决定谁能获得受控读取，读取句柄是受控访问的载体 |
| `export-package` | 范围表达式决定谁能获得解密下载导出包，导出包是授权的最终产物 |
| `authorization` | 范围表达式是授权规则的核心匹配条件，依附于授权规则存在 |

### 6.3 治理边界划分

- **第一份专项设计**负责：加密算法选择、密钥层级划分、密钥轮换策略、介质保护和密钥托管
- **第二份专项设计**负责：读取句柄治理、流式分段策略治理、临时缓存介质治理
- **第三份专项设计**负责：导出包治理、水印选项治理、脱敏选项治理、文件名规范治理
- **本文档**负责：范围表达式治理、细粒度字段扩展治理、标签扩展治理、业务规则扩展治理
- **四个文档的共同约束**：都不持有文档明文内容、都不编写实现代码、都保持与 `document-center` 和 `contract-core` 的松耦合

## 7. 与 `contract-core`、`document-center` 的边界说明

### 7.1 与 `document-center` 的边界

| 边界维度 | `document-center` 负责 | `encrypted-document` 负责 |
| --- | --- | --- |
| 文件真相 | 持有文档资产、版本链、存储定位 | 不持有文件真相，只引用 `document_asset_id` |
| 版本管理 | 管理文档版本的新增、切换、失效 | 只感知当前版本，不管理版本生命周期 |
| 文档元数据 | 持有文档元数据字段 | 只引用元数据字段作为范围表达式候选数据源 |
| 文档标签 | 持有文档标签体系 | 只引用文档标签作为范围表达式候选数据源 |
| 范围表达式状态 | 不持有范围表达式任何状态 | 持有范围表达式的治理状态、规格、绑定关系 |
| 字段/标签匹配 | 可提供字段值/标签查询能力 | 负责定义字段/标签范围策略，执行层负责具体匹配 |

引用关系：`document-center` ← `document_asset_id` ← `encrypted-document`（范围表达式）

### 7.2 与 `contract-core` 的边界

| 边界维度 | `contract-core` 负责 | `encrypted-document` 负责 |
| --- | --- | --- |
| 合同主档 | 持有合同业务状态、生命周期 | 不持有合同主档，只引用 `contract_id` |
| 业务权限 | 管理合同的业务级权限 | 只读取权限结果，不管理业务权限 |
| 组织归属 | 管理合同归属部门、负责人 | 只读取组织快照，不管理组织关系 |
| 合同字段 | 持有合同业务字段 | 只引用合同字段作为范围表达式候选数据源 |
| 合同标签 | 持有合同业务标签 | 只引用合同标签作为范围表达式候选数据源 |
| 业务规则 | 持有合同业务规则、规则引擎 | 只引用业务规则作为范围表达式候选数据源 |
| 范围表达式状态 | 不持有范围表达式任何状态 | 持有范围表达式的治理状态、规格、绑定关系 |
| 字段/标签/规则匹配 | 可提供字段值/标签/规则查询能力 | 负责定义字段/标签/规则范围策略，执行层负责具体匹配 |

引用关系：`contract-core` ← `contract_id` ← `encrypted-document`（范围表达式）

### 7.3 跨模块引用原则

- **自上而下引用**：合同 → 文档 → 加密方案 → 读取句柄 → 导出包 → 范围表达式
- **禁止反向依赖**：范围表达式不持有合同主档引用，不持有文档内容引用
- **松耦合**：`expression-code`、`package-id`、`handle-id`、`profile-code` 是跨模块稳定键，各模块不直接依赖对方内部模型
- **治理隔离**：`contract-core` 不感知范围表达式的存在，`document-center` 只通过引用感知某文档是否在某个范围内
- **数据源与服务分离**：`contract-core` 和 `document-center` 可提供字段/标签/规则作为数据源，但不参与范围表达式的治理决策

### 8.5 审计关联约束

- 审计事件只记录治理动作，不记录表达式求值执行细节（执行细节由 `ed_encryption_audit_event` 记录）
- 审计事件中的表达式标识不包含表达式内容本身，只保留 `expression-code` 和版本号
- 审计事件与 `contract-core`、`document-center` 的关联通过 `expression-code` 间接建立，不直接引用合同或文档内部状态
- 审计事件保留期限按合规要求设定，与范围表达式生命周期解耦
- 字段值、标签值、规则结果不在审计事件中记录，只记录其类别和版本
