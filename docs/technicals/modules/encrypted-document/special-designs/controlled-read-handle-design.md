# 受控读取句柄专项设计

## 1. 文档定位

本文档是 `encrypted-document` 子模块的**受控读取句柄治理设计**，承接 [`detailed-design.md`](../detailed-design.md) 第 12 节中"受控读取句柄的具体协议、流式分段策略和临时缓存介质实现"的专项下沉。

本文档只回答以下问题：
- 受控读取句柄如何治理：对象是谁、稳定锚点在哪里、版本如何归属、最小治理单元是什么
- 流式分段策略的治理边界在哪里
- 临时缓存介质的治理对象是什么、责任边界如何划分
- 与第一份专项设计（加密算法与密钥层级）的引用关系
- 与 `contract-core`、`document-center` 的边界如何划分

本文档**不承担**以下内容：
- 不展开 API 路径、请求/响应结构、错误码
- 不输出 DDL、表结构、索引细节
- 不编写句柄协议代码、流式读取代码、缓存读写代码
- 不包含实施排期、工时估算、负责人分配
- 不写成运维手册、部署方案、故障排查指南
- 不描述具体协议代码实现、流式传输代码实现、缓存介质代码实现

## 2. 受控读取句柄治理

### 2.1 治理对象

受控读取句柄治理的**一级对象**是 `read-handle`（读取句柄），每个句柄封装一次受控访问的授权、边界与生命周期。

治理对象层级：

| 层级 | 对象 | 说明 |
| --- | --- | --- |
| 一级 | `read-handle` | 读取句柄，是受控读取治理的最小单元 |
| 二级 | `handle-policy` | 句柄策略，描述句柄的权限范围、消费模式、时效约束 |
| 三级 | `access-ticket` | 访问票据，是句柄的可验证短期凭证 |

`read-handle` 不持有解密密钥材料，不持有文档明文内容，只持有"谁可以在什么条件下以什么模式访问多久"的治理描述。

### 2.2 稳定锚点

受控读取句柄治理的**稳定锚点**是 `handle-id`（句柄标识），其稳定性约束如下：

- `handle-id` 是跨模块、跨版本的稳定引用键，文档中心、合同主档、审计链路均通过 `handle-id` 引用读取句柄。
- `handle-id` 一旦生成，不允许修改其核心属性（主体、文档、场景）；若需变更访问条件，只能创建新 `handle-id`。
- `handle-id` 与 `document-center` 的松耦合：文档中心只引用 `handle-id` 作为受控读取入口，不感知句柄内部策略细节。
- `handle-id` 与 `contract-core` 的松耦合：合同主档不持有读取句柄状态，仅通过 `document_asset_id` 间接关联到 `read-handle`。

### 2.3 版本归属

受控读取句柄版本归属规则：

| 版本维度 | 归属主体 | 说明 |
| --- | --- | --- |
| `handle-version` | `encrypted-document` 子模块 | 句柄自身的版本演进，由加密模块独立治理 |
| `policy-version` | `encrypted-document` 子模块 | 句柄策略版本，随安全策略或合规要求演进 |
| `ticket-version` | `encrypted-document` 子模块 | 票据版本，跟随句柄版本联动 |
| `document-version` | `document-center` | 文档版本，加密模块只引用，不拥有 |
| `contract-version` | `contract-core` | 合同版本，加密模块不直接访问 |

版本归属的核心原则：**读取句柄版本由加密模块自治，文档版本和合同版本由各自归属模块治理，通过 `handle-id` 建立跨模块关联**。

### 2.4 最小治理单元

受控读取句柄的最小治理单元是 **`read-handle` + `handle-policy` 的组合**，具备以下治理属性：

- 可独立批准/拒绝
- 可独立审计（谁在何时获得了哪个文档的受控访问）
- 可独立撤销（标记为REVOKED，不影响历史访问记录）
- 可独立追踪（通过 `trace-id` 关联完整访问链路）

最小治理单元**不包含**具体解密密钥、具体文档明文、具体消费方内部状态，保持治理层与执行层分离。

## 3. 流式分段策略治理

### 3.1 治理边界

流式分段策略治理的**核心对象**是 `stream-segment-policy`（流式分段策略），定义如何将加密文档内容分段、按什么粒度解密、以什么顺序交付给消费方。

流式分段策略治理边界：

**`encrypted-document` 负责的边界：**
- 分段大小的治理策略（最小段、最大段、推荐段）
- 分段解密的顺序策略（顺序、随机、按需）
- 分段与消费模式的匹配策略（流式、预览令牌、临时文本、内部句柄）
- 分段失败时的治理策略（重试、降级、终止）

**`encrypted-document` 不负责的边界：**
- 具体流式传输协议实现（由消费方或服务框架负责）
- 网络层分片、TCP 窗口、HTTP 分块传输细节
- 消费方内部的缓冲区管理、渲染逻辑

**`document-center` 的边界：**
- 持有文档对象、版本链、存储定位
- 不感知流式分段策略内部细节
- 只通过 `handle-id` 间接感知"该文档当前是否允许流式受控访问"

**`contract-core` 的边界：**
- 持有合同主档、业务状态、生命周期
- 不直接参与流式分段策略治理
- 通过 `document_asset_id` → `handle-id` 间接关联

### 3.2 分段策略治理约束

流式分段策略必须满足以下治理约束：

| 约束维度 | 治理要求 |
| --- | --- |
| 最小分段 | 不允许单个分段超过治理阈值，防止大段明文驻留 |
| 解密隔离 | 每个分段独立解密，不跨段缓存明文 |
| 消费模式匹配 | 不同 `consumption_mode` 对应不同分段策略，不混用 |
| 失败域隔离 | 单个分段失败不影响已交付分段的治理记录 |
| 审计覆盖 | 每个分段的交付必须可追踪，不形成审计盲区 |

## 4. 临时缓存介质治理

### 4.1 治理对象

临时缓存介质治理对象是**受控读取过程中的临时明文驻留与访问凭证**，不涉及文档内容本身。

| 治理对象 | 归属主体 | 说明 |
| --- | --- | --- |
| 临时缓冲区引用 | `encrypted-document` | 标记临时明文的存在周期、范围、访问限制 |
| 缓存访问凭证 | `encrypted-document` | 临时缓存的读取凭证，有时效性和单次使用约束 |
| 受控介质标签 | `encrypted-document` | 标记介质是否为临时缓存、是否受控、清理状态 |

临时缓存介质**不包含**文档内容明文、消费方内部缓存、长期存储介质——这些属于执行层或消费方内部，不在治理设计范围内。

### 4.2 责任边界

临时缓存介质的责任边界：

**`encrypted-document` 的责任：**
- 定义临时缓存的生命周期策略（最长时间、最大次数、清理触发条件）
- 定义临时缓存的访问约束（谁可以访问、访问频次、并发限制）
- 定义临时缓存的清理责任（自动过期、主动清理、残留检测）
- 记录临时缓存的治理事件（创建、访问、过期、清理）

**`encrypted-document` 不负责：**
- 消费方内部的缓冲区管理
- 操作系统级缓存、文件系统缓存、浏览器缓存
- 消费方对临时明文的后续处理

**`document-center` 的边界：**
- 不持有临时缓存任何状态
- 不感知临时缓存的存在与生命周期
- 只通过 `handle-id` 间接感知"该文档当前是否有活跃的受控读取"

**`contract-core` 的边界：**
- 不直接参与临时缓存介质治理
- 合同主档不持有临时缓存引用
- 临时缓存状态变更不反向改写合同主档状态

### 4.3 临时缓存治理约束

临时缓存介质必须满足以下治理约束：

| 约束维度 | 治理要求 |
| --- | --- |
| 生命周期 | 临时缓存不得超过句柄有效期，句柄过期后缓存必须进入清理流程 |
| 访问隔离 | 临时缓存只能由持有有效 `access-ticket` 的主体访问 |
| 残留清理 | 必须有自动清理机制，不允许临时缓存无限期残留 |
| 审计覆盖 | 临时缓存的创建、访问、清理必须形成治理事件 |
| 明文不落盘 | 临时缓存不允许持久化为新的文件真相源 |

## 5. 与第一份专项设计的引用关系

本文档与 [`crypto-algorithm-and-key-hierarchy-design.md`](crypto-algorithm-and-key-hierarchy-design.md) 的引用关系：

### 5.1 架构层级关系

```
加密算法与密钥层级专项设计（第一份）
  └── encryption-profile（加密方案）
        └── read-handle（读取句柄，本文档）
              └── stream-segment-policy（流式分段策略，本文档）
              └── temporary-cache-policy（临时缓存策略，本文档）
```

### 5.2 治理依赖关系

| 本文档依赖第一份的内容 | 说明 |
| --- | --- |
| `encryption-profile` | 读取句柄必须引用加密方案，确定使用什么算法保护的内容 |
| `profile-code` | 句柄的稳定锚点依赖于加密方案的稳定锚点 |
| 密钥层级 | 句柄不直接持有密钥，但解密动作依赖密钥层级的治理能力 |
| 介质保护 | 临时缓存介质的治理依赖于第一份文档中介质保护的治理边界 |

### 5.3 治理边界划分

- **第一份专项设计**负责：加密算法选择、密钥层级划分、密钥轮换策略、介质保护和密钥托管
- **本文档**负责：读取句柄治理、流式分段策略治理、临时缓存介质治理
- **两个文档的共同约束**：都不持有文档明文内容、都不编写实现代码、都保持与 `document-center` 和 `contract-core` 的松耦合

## 6. 与 `contract-core`、`document-center` 的边界说明

### 6.1 与 `document-center` 的边界

| 边界维度 | `document-center` 负责 | `encrypted-document` 负责 |
| --- | --- | --- |
| 文件真相 | 持有文档资产、版本链、存储定位 | 不持有文件真相，只引用 `document_asset_id` |
| 版本管理 | 管理文档版本的新增、切换、失效 | 只感知当前版本，不管理版本生命周期 |
| 读取入口 | 提供文档的原始存储访问能力 | 提供受控读取句柄作为安全访问入口 |
| 流式传输 | 可提供底层存储读取能力 | 定义流式分段的安全治理策略 |
| 缓存管理 | 管理自身的存储缓存 | 管理受控读取过程中的临时缓存介质 |

引用关系：`document-center` ← `document_asset_id` ← `encrypted-document`

### 6.2 与 `contract-core` 的边界

| 边界维度 | `contract-core` 负责 | `encrypted-document` 负责 |
| --- | --- | --- |
| 合同主档 | 持有合同业务状态、生命周期 | 不持有合同主档，只引用 `contract_id` |
| 业务权限 | 管理合同的业务级权限 | 只读取权限结果，不管理业务权限 |
| 组织归属 | 管理合同归属部门、负责人 | 只读取组织快照，不管理组织关系 |
| 时间线摘要 | 记录合同时间线事件 | 只向时间线写入安全事件摘要 |
| 读取场景 | 不直接参与读取场景定义 | 定义 `access_scene` 与合同上下文的关联 |

引用关系：`contract-core` ← `contract_id` ← `encrypted-document`

### 6.3 跨模块引用原则

- **自上而下引用**：合同 → 文档 → 加密方案 → 读取句柄
- **禁止反向依赖**：读取句柄不持有合同主档引用，不持有文档内容引用
- **松耦合**：`handle-id` 和 `profile-code` 是跨模块稳定键，各模块不直接依赖对方内部模型
- **治理隔离**：`contract-core` 不感知读取句柄的存在，`document-center` 只通过引用感知受控读取状态

## 7. 审计最小单元

### 7.1 句柄审计最小单元

句柄审计最小单元是 **`read-handle` + `handle-policy` + `access-ticket`**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 句柄标识 | `handle-id` |
| 句柄策略 | `handle-policy-id` |
| 票据标识 | `access-ticket`（非票据内容） |
| 变更类型 | 创建 / 批准 / 拒绝 / 撤销 / 过期 / 消费完成 |
| 变更主体 | 操作人 / 系统 / 内部服务 |
| 变更时间 | 操作时间戳 |
| 影响范围 | 受影响的文档与场景（不展开到具体分段或缓存细节） |

### 7.2 流式分段审计最小单元

流式分段审计最小单元是 **`stream-segment-policy` + 分段交付记录**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 分段策略编码 | `segment-policy-code` |
| 消费模式 | `consumption_mode` |
| 分段序号 | 分段在流中的序号（不记录分段内容） |
| 交付状态 | 成功 / 失败 / 超时 |
| 交付时间 | 分段交付时间戳 |
| 关联句柄 | `handle-id`（不展开到具体文档或合同） |

### 7.3 临时缓存审计最小单元

临时缓存审计最小单元是 **临时缓存介质 + 生命周期事件**，审计必须覆盖：

| 审计要素 | 说明 |
| --- | --- |
| 缓存标识 | 缓存引用 ID（非缓存内容） |
| 缓存类型 | 缓冲区 / 临时文件 / 内存区域 |
| 生命周期事件 | 创建 / 访问 / 过期 / 清理 |
| 事件主体 | 操作人 / 系统 / 内部服务 |
| 事件时间 | 事件时间戳 |
| 关联句柄 | `handle-id`（不展开到具体文档或合同） |

### 7.4 审计关联约束

- 审计事件只记录治理动作，不记录流式传输执行细节（执行细节由 `ed_encryption_audit_event` 记录）
- 审计事件中的句柄标识不包含票据内容或缓存内容本身，只保留 ID 和状态
- 审计事件与 `contract-core`、`document-center` 的关联通过 `handle-id` 间接建立，不直接引用合同或文档内部状态
- 审计事件保留期限按合规要求设定，与句柄生命周期解耦
