## 2. 计划边界与原则

- 合同主档是业务真相源。本计划按“先让后续业务围绕同一 `contract_id`
  成立，再让周边能力消费稳定结果”的顺序推进，不允许履约、变更、终止、
  归档各自长出新的合同主档。
- 文档中心是文件真相源。履约凭证、补充协议、终止协议、归档稿、归档封包等
  文件对象都通过文档中心治理，本业务组只维护业务引用、稳定摘要和回写关系。
- 流程引擎通过 `contract_id` 绑定合同，不拥有合同主档；需要审批的变更、终止、
  归档动作都建立在这一前提上推进。
- 履约、变更、终止、归档围绕同一 `contract_id` 运行。变更、终止是在原合同
  主链上推进，不创建新的合同主档；归档记录属于归档模块真相，但不能替代合同
  主档业务真相。
- 搜索、AI、审计、通知只消费稳定摘要、里程碑与时间线，而不是直接消费各模块
  私有过程数据。本业务组的实施顺序必须先把稳定消费面做成，再接周边消费。
- 本业务组优先形成阶段闭环，再进入跨模块联调。未完成上游真相源和摘要回写前，
  不把搜索、AI、通知等消费链路提前做成事实来源。

## 3. 前置条件

### 3.1 合同主档、文档中心、流程引擎前置条件

- 合同管理本体已形成正式 `contract_id`、合同生命周期主状态、统一时间线入口和
  受控回写边界，能够为本业务组提供稳定主链入口。
- 文档中心已具备正式文件写入、主版本引用、稳定读取和受控摘要回写能力，能够
  承接履约凭证、补充协议、终止材料、归档材料和归档产物的统一治理。
- 流程引擎已具备按 `contract_id` 发起、绑定、回写审批结果的正式入口；流程实例
  只表达审批运行时真相，不替代合同主档对业务主状态的解释权。
- 合同管理本体与本业务组之间的回写口径已统一，至少明确哪些动作回写主状态、
  哪些动作只回写阶段摘要、里程碑和时间线。

### 3.2 签章完成后生效 / 履约起点前置条件

- 签章结果回写链路已可稳定判断合同是否进入生效阶段，避免履约模块在合同尚未
  生效时提前创建正式履约主记录。
- 合同主档已能输出“已生效 / 可进入履约”的稳定准入判断，作为履约启动、变更
  准入、终止准入和归档准入的统一基础。
- 履约起点规则已统一：签章完成并满足生效条件后，履约围绕同一 `contract_id`
  启动；若合同尚未生效，则不进入正式履约链路。
- 履约启动所需的合同上下文、正文引用、附件引用和责任归属信息已可从合同主档与
  文档中心稳定取得，不要求履约模块重复建档。

### 3.3 搜索、AI、审计、通知等消费前置条件

- 搜索、AI、审计、通知已明确自身是消费方，不反向定义履约、变更、终止、归档
  的业务真相。
- 本业务组稳定摘要、里程碑与时间线的统一口径已在
  [`Detailed Design`](./detailed-design.md) 中收口，能够作为周边消费的唯一正式输入。
- 审计中心、通知中心、统一任务中心和异常恢复能力已具备复用条件，保证后续摘要
  刷新、事件分发、索引刷新和通知补偿不需要本业务组私建第二套基础设施。
- 搜索和 AI 所需的受控文档引用、合同上下文、摘要字段和权限边界已准备完成，
  避免周边能力直接跨读过程主表或文档底层对象。

## 4. 总体阶段划分

本业务组按五个阶段推进：

1. 履约与稳定摘要基础能力
2. 变更与终止能力
3. 归档与生命周期收口能力
4. 周边消费挂接能力
5. 联调、验收与上线准备

总体顺序遵循“先履约，再变更 / 终止，再归档，再周边消费，最后整体联调与上线
准备”的节奏。

原因不是把四类模块机械串行，而是先建立合同生效后的默认主路径和稳定消费面，
再在该主路径上叠加需要审批和收口的后续动作，最后再让搜索、AI、审计、通知等
外围能力消费已经稳定下来的摘要、里程碑与时间线。

## 5. 分阶段实施计划

### 5.1 阶段一：履约与稳定摘要基础能力

**目标**

- 先建立合同生效后的默认主路径，让后续业务从“签章完成”顺利进入“履约可跟踪”。
- 先把本业务组的稳定摘要、里程碑和时间线基础能力做起来，为后续变更、终止、
  归档以及周边消费提供统一落点。

**核心产出**

- 履约聚合记录、履约节点和履约进展跟踪的基础闭环
- 围绕同一 `contract_id` 的履约摘要、风险摘要、里程碑和时间线基础能力
- 履约与合同主档之间的受控回写入口，能够回写阶段摘要和关键进展
- 履约与文档中心之间的正式文件引用能力，能够稳定引用履约凭证和执行材料

**前置依赖**

- 合同主档、签章结果回写和生效判断已可稳定使用
- 文档中心已具备正式文件写入与稳定引用能力
- 统一任务中心、审计中心和权限体系已可复用

**完成标志**

- 已生效合同能够围绕同一 `contract_id` 启动正式履约链路
- 履约节点和履约摘要能够稳定沉淀，不存在脱离合同主档的第二份业务身份
- 合同主档可以读取本业务组输出的履约摘要、里程碑和时间线结果
- 搜索、AI、审计、通知后续所需的稳定消费面已经具备基础形态

### 5.2 阶段二：变更与终止能力

**目标**

- 在履约主路径已经成立后，补齐原合同主链上的变更和终止能力。
- 让变更、终止都建立在既有合同上下文和履约上下文之上，而不是各自开新主档。

**核心产出**

- 变更申请、变更审批、变更结果回写和变更历史沉淀的基础闭环
- 终止申请、终止审批、终止结果回写和善后摘要沉淀的基础闭环
- 变更、终止与流程引擎的正式挂接能力，流程仍通过 `contract_id` 绑定合同
- 变更、终止完成后对合同主档和生命周期摘要的受控回写能力

**前置依赖**

- 阶段一已形成稳定的履约基础能力、稳定摘要和时间线能力
- 流程引擎已具备按 `contract_id` 启动并承接结果回写的正式入口
- 文档中心已可承接补充协议、变更说明、终止协议、终止材料等正式文件引用

**完成标志**

- 变更、终止都在原合同主链上推进，不创建新的合同主档
- 变更、终止结果可以稳定回写合同主档当前阶段摘要和时间线
- 流程引擎只承接审批运行时，不接管合同主档业务真相
- 履约、变更、终止之间已经形成清晰的先后与并存边界，后续归档可以基于稳定结果
  判断准入条件

### 5.3 阶段三：归档与生命周期收口能力

**目标**

- 在履约、变更、终止结果都可稳定沉淀后，再建立归档模块的正式收口能力。
- 让归档围绕同一 `contract_id` 收集完整输入集、形成归档记录，并回写合同主档
  归档摘要，而不是把归档记录做成新的业务真相源。

**核心产出**

- 归档记录、归档输入集校验、归档状态跟踪和归档结果沉淀能力
- 归档与文档中心之间的正式引用能力，能够收集正文、附件、履约凭证、变更材料、
  终止材料和归档产物引用
- 归档与合同主档之间的受控回写能力，能够回写归档摘要、归档里程碑和归档引用
- 生命周期收口判断能力，能够明确哪些合同已满足归档条件、哪些仍处于未收口状态

**前置依赖**

- 阶段一、阶段二已完成履约、变更、终止的稳定摘要和时间线沉淀
- 文档中心已能稳定提供归档输入集和归档产物的正式引用
- 权限、审计、任务中心已可承接归档创建、归档重建和归档回查相关动作

**完成标志**

- 归档记录已作为归档模块过程真相成立，但不会替代合同主档业务真相
- 归档入口能够围绕同一 `contract_id` 组织输入集和收口动作
- 合同主档已能读取本业务组输出的归档摘要和归档里程碑
- 履约、变更、终止、归档四类能力已经围绕同一生命周期主链形成完整后续业务闭环

### 5.4 阶段四：周边消费挂接能力

**目标**

- 在本业务组内部主链和稳定消费面已经成立后，再挂接搜索、AI、审计、通知等周边
  消费能力。
- 确保周边能力消费的是稳定摘要、里程碑与时间线，而不是本业务组私有过程表。

**核心产出**

- 搜索消费生命周期稳定摘要和里程碑的正式入口
- AI 消费生命周期摘要、时间线和受控文档引用的正式入口
- 审计串联履约、变更、终止、归档关键动作与回写结果的正式链路
- 通知围绕同一 `contract_id` 触达履约提醒、变更待办、终止审批、归档完成等
  稳定事件的正式链路

**前置依赖**

- 阶段一到阶段三已形成稳定摘要、里程碑、时间线和受控引用边界
- 搜索、AI、审计、通知相关底座能力已具备最小可联调入口
- 权限和审计规则已能约束周边消费只读取允许暴露的稳定信息

**完成标志**

- 搜索、AI、审计、通知已围绕同一 `contract_id` 消费本业务组稳定结果
- 周边能力没有直接跨读履约、变更、终止、归档私有过程真相
- 生命周期摘要和时间线已经成为合同详情、搜索、AI、审计、通知的统一正式输入
- 周边消费失败不会破坏本业务组过程真相和合同主档回写结果

### 5.5 阶段五：联调、验收与上线准备

**目标**

- 在内部主链和周边消费链路都成立后，统一收口跨模块联调、验收样例、上线检查和
  回退准备。
- 把本业务组从“功能已成型”推进到“可以进入正式发布窗口”。

**核心产出**

- 面向合同管理本体、文档中心、流程引擎、搜索、AI、审计、通知的联调清单
- 覆盖履约、变更、终止、归档主链及异常链的验收样例
- 上线前配置检查、数据检查、回写检查、任务检查和回退预案
- 生命周期业务组运行手册、问题定位入口和异常补偿入口

**前置依赖**

- 前四个阶段核心能力均已闭环
- 真正需要外部条件的联调资源已准备完成
- 关键跨模块接口和回写口径已进入可冻结状态

**完成标志**

- 可覆盖“生效 -> 履约 -> 变更 / 终止 -> 归档 -> 查询 / 审计回查 / 通知”的正式
  验证链路
- 本业务组与合同主档、文档中心、流程引擎之间的主链回写已经通过联调验证
- 周边消费链路已通过权限、审计、失败补偿和回查验证
- 上线检查项、回退步骤和问题处置入口已明确，可进入正式发布窗口

## 6. 模块推进顺序

本业务组按“先履约，再变更 / 终止，再归档，再周边消费”的顺序推进，原因如下：

1. 先做履约，是因为履约是合同在签章完成并满足生效条件后的默认主路径。
   如果履约基础能力和稳定摘要没有先成立，后续变更、终止只能在缺少执行上下文、
   缺少履约风险信息、缺少里程碑沉淀的情况下推进，容易让后续模块各自维护一份
   过程解释口径。

2. 再做变更 / 终止，是因为它们都建立在原合同主链和履约上下文之上。
   变更需要知道当前有效业务状态和履约进展，终止需要知道当前履约情况、善后事项
   和是否存在未收口事项；两者都不应在履约主路径尚未成立时先行展开。

3. 然后做归档，是因为归档是生命周期收口动作，而不是中途动作。
   归档输入集要依赖正文、履约凭证、变更材料、终止材料和当前稳定摘要。如果在
   履约、变更、终止结果尚未稳定前就推进归档，只会把未定稿过程数据过早固化成
   归档结果。

4. 最后接周边消费，是因为搜索、AI、审计、通知只应消费稳定摘要、里程碑与时间线。
   如果这些消费能力提前接入，它们会反向倒逼过程表和中间状态成为事实来源，破坏
   “合同主档是业务真相源、文档中心是文件真相源”的正式边界。

这一顺序的核心目的，是先把后续业务组做成围绕同一 `contract_id` 运行的正式主链，
再把外部消费和增强能力挂上去，而不是反过来让消费方定义后续业务口径。

## 7. 外部联调准备

本业务组只对真正需要外部条件的部分做联调准备说明，不把本业务组整体写成外部
系统依赖。

### 7.1 `OA` 审批主路径联调准备

- 变更、终止、归档中凡默认走 `OA` 主审批路径的场景，需要先冻结流程模板、发起方式、
  回调方式和结果回传窗口。
- 准备至少一组覆盖变更审批、终止审批或归档审批的真实联调样例，验证发起、回调、
  审批结果承接、摘要回写和异常补偿。
- 准备重复回调、乱序回调、失败重发和人工补发样例，验证流程引擎和合同主档之间的
  回写不会被异常联调数据破坏。

### 7.2 外部消息触达联调准备

- 如果通知链路需要企业微信等外部消息通道参与联调，应准备测试账号、测试应用和
  最小组织样例，用于验证履约提醒、变更待办、终止审批提醒和归档完成通知。
- 外部消息通道只验证触达结果，不作为业务状态真相来源；通知失败不得改变履约、
  变更、终止、归档正式状态。

### 7.3 联调前内部先决准备

- 准备覆盖履约、变更、终止、归档全链路的合同样例、文件样例和回写对账清单
- 准备生命周期摘要对账清单、时间线对账清单和权限 / 审计联调用例
- 准备跨模块异常回放样例，覆盖流程回写失败、文件引用缺失、通知失败、索引失败等
  场景

## 8. 测试、验收与上线准备

### 8.1 测试重点

- 履约测试：生效准入、履约启动、履约节点推进、履约凭证引用、履约风险和履约完成
- 变更测试：变更申请、流程挂接、变更结果回写、变更历史保留和变更后有效摘要
- 终止测试：终止申请、流程挂接、终止结果回写、善后摘要和终止后限制
- 归档测试：归档准入、输入集校验、归档记录生成、归档产物引用和归档摘要回写
- 消费测试：搜索、AI、审计、通知是否只消费稳定摘要、里程碑与时间线
- 异常测试：重复提交、重复回调、乱序回调、摘要刷新失败、通知失败、索引失败、
  权限拒绝和回查补偿

### 8.2 验收重点

- 能证明合同主档是业务真相源，履约、变更、终止、归档都围绕同一 `contract_id`
  运行，没有创建新的合同主档
- 能证明文档中心是文件真相源，本业务组只保存业务引用、稳定摘要和受控回写结果
- 能证明流程引擎通过 `contract_id` 绑定合同，不拥有合同主档
- 能证明变更、终止是在原合同主链上推进，不是新建合同业务主记录
- 能证明归档记录属于归档模块真相，但不能替代合同主档业务真相
- 能证明搜索、AI、审计、通知只消费稳定摘要、里程碑与时间线，而不是消费模块私有
  过程数据

### 8.3 上线准备

- 配置准备：流程承接策略、通知开关、生命周期摘要刷新任务、索引刷新任务、审计
  规则和归档策略开关
- 数据准备：联调遗留数据清理、测试归档记录清理、生命周期摘要初始化与索引初始化
- 运行准备：失败补偿任务、回写重试机制、日志检索入口、审计回查入口和告警规则
- 发布准备：灰度顺序、回退条件、回退步骤、上线观察窗口和问题升级路径
