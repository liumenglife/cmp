## 2. 履约节点类型字典的引用依赖边界

履约节点类型字典不属于本文的直接治理对象，其定义、归属、稳定锚点、版本归属与最小治理单元，统一以前一份专项设计
[`fulfillment-rules-and-risk-scoring-design.md`](fulfillment-rules-and-risk-scoring-design.md)
为唯一真相源。

本文只在以下场景引用该字典，不重复定义：

- 时间线事件字典需要识别哪些履约动作会投影为生命周期事件时，可引用第一份专项中既定的 `node_type` 分类。
- 通知模板需要区分履约提醒、逾期提醒、节点完成通知等场景时，可消费第一份专项已定义的履约节点语义。
- 文档引用角色字典、流程用途字典与履约场景发生交叉时，只允许引用第一份专项中的既有节点类型，不在本文新增履约节点类型治理条目。

本文的所有字典与通知规则，均不得重新声明履约节点类型字典的锚点、版本归属或最小治理单元。

## 3. 文档引用角色字典治理设计

### 3.1 治理对象

文档引用角色字典治理的**直接对象**是：

| 治理对象 | 说明 | 不治理内容 |
| --- | --- | --- |
| 角色编码 | 角色唯一标识、编码规范、命名约束 | 角色实例数据、角色的使用统计 |
| 角色名称 | 中文名称、英文名称、简称 | 角色的多语言翻译资源文件 |
| 角色分类 | 业务分类、技术分类、分类层级 | 分类的业务统计分析报表 |
| 角色业务语义 | 适用场景、不适用的场景、业务约束 | 角色的具体权限配置 |
| 角色与主文档的关联约束 | 哪些角色可以作为主文档引用 | 主文档判定的具体逻辑 |

文档引用角色字典治理的**间接约束对象**是：

- `cl_lifecycle_document_ref` 的 `role_code` 字段含义、枚举值域、校验约束
- `cl_lifecycle_summary` 中摘要块的文档引用角色聚合约束
- `cl_lifecycle_timeline_event` 中文档里程碑事件的 `event_type` 与角色的关联约束
- `document-center` 的文档引用消费约束

### 3.2 稳定锚点

文档引用角色字典的**稳定锚点**选择如下：

| 稳定锚点 | 说明 | 选择原因 |
| --- | --- | --- |
| `role_code` | 角色编码，字典的核心标识 | 角色编码是文档引用、摘要聚合、时间线事件的共同分类键 |
| `contract_id` | 合同主档主键，角色作用的根本业务实体 | 合同主档是业务真相源，角色必须在合同上下文中消费 |
| `document_id` | 文档中心文档主键，角色绑定的文档对象 | 文档中心是文件真相源，角色必须绑定到具体文档 |
| `business_scenario` | 业务场景（履约/变更/终止/归档），角色分类的维度 | 不同业务场景可能有不同的角色定义 |

**稳定锚点原则**：文档引用角色字典只能引用上述稳定锚点作为输入，不得引用以下不稳定锚点：

- 流程引擎的节点 ID、任务 ID（流程实例是外部对象）
- 文档中心的具体文件版本 ID（版本是外部对象的属性，不是角色本身的属性）
- 临时计算字段、缓存字段、非持久化投影字段
- 具体文档引用实例的 ID（实例是运行时对象）

### 3.3 版本归属

文档引用角色字典的版本归属按以下原则划分：

| 规则类型 | 定义权归属 | 消费方 | 变更权限 | 版本化方式 |
| --- | --- | --- | --- | --- |
| 角色编码与名称 | `contract-lifecycle` 模块 | 文档引用、摘要、时间线、搜索、AI | 模块级变更，需同步更新摘要与时间线消费面 | 随模块版本演进，不单独版本化 |
| 角色分类 | `contract-lifecycle` 模块 | 文档引用、通知、AI | 模块级变更，需评估对通知的连锁影响 | 随模块版本演进，不单独版本化 |
| 角色业务语义 | `contract-lifecycle` 模块 | 文档引用、变更影响、终止善后、归档 | 模块级变更，需评估对变更、终止、归档的连锁影响 | 随模块版本演进，不单独版本化 |
| 角色与主文档关联 | `contract-lifecycle` 模块 | 文档引用、`contract-core` 主文档引用 | 模块级变更，需评估对合同主档的连锁影响 | 随模块版本演进，不单独版本化 |

**版本归属原则**：

- 文档引用角色字典属于 `contract-lifecycle` 模块的内部治理资产，不属于 `document-center` 的文档元数据，也不属于 `contract-core` 的主文档角色。
- 文档引用角色字典变更不需要变更合同主档、文档中心、流程引擎的版本，但可能需要刷新 `cl_lifecycle_summary` 和 `cl_lifecycle_timeline_event` 的消费面。
- 文档引用角色字典不对外暴露为独立 API 资源，只通过文档引用、摘要、时间线间接消费。
- 文档引用角色字典的变更需同步通知变更影响与回写专项设计（第二份专项设计），确保字段映射一致性。

### 3.4 最小治理单元

文档引用角色字典的**最小治理单元**是：

- **单个角色编码**：每个 `role_code` 编码是一个最小治理单元，包含编码、名称、分类、业务语义、适用场景、与主文档的关联。
- **角色分类条目**：每个分类层级是一个最小治理单元，包含分类编码、名称、业务语义。
- **角色与主文档关联条目**：每个"角色编码 + 业务场景"组合是一个最小治理单元。

**最小治理单元原则**：

- 不允许将多个角色合并为一个"通用角色"，每个角色必须有独立的治理条目。
- 不允许将角色分类与角色编码合并为同一个治理单元，分类是分类，编码是编码。
- 不允许将文档引用角色字典与流程用途字典合并为同一个治理单元，参见第 4 节。

## 4. 流程用途字典治理设计

### 4.1 治理对象

流程用途字典治理的**直接对象**是：

| 治理对象 | 说明 | 不治理内容 |
| --- | --- | --- |
| 用途编码 | 用途唯一标识、编码规范、命名约束 | 用途实例数据、用途的使用统计 |
| 用途名称 | 中文名称、英文名称、简称 | 用途的多语言翻译资源文件 |
| 用途分类 | 业务分类、技术分类、分类层级 | 分类的业务统计分析报表 |
| 用途业务语义 | 适用场景、不适用的场景、业务约束 | 用途的具体流程配置 |
| 用途与流程定义的关联约束 | 哪些流程定义适用于该用途 | 流程定义的具体配置、节点配置 |

流程用途字典治理的**间接约束对象**是：

- `cl_lifecycle_process_ref` 的 `purpose_code` 字段含义、枚举值域、校验约束
- `cl_lifecycle_summary` 中摘要块的流程用途聚合约束
- `cl_lifecycle_timeline_event` 中流程里程碑事件的 `event_type` 与用途的关联约束
- `workflow-engine` 的流程定义消费约束

### 4.2 稳定锚点

流程用途字典的**稳定锚点**选择如下：

| 稳定锚点 | 说明 | 选择原因 |
| --- | --- | --- |
| `purpose_code` | 用途编码，字典的核心标识 | 用途编码是流程引用、摘要聚合、时间线事件的共同分类键 |
| `contract_id` | 合同主档主键，用途作用的根本业务实体 | 合同主档是业务真相源，用途必须在合同上下文中消费 |
| `workflow_definition_id` | 流程定义主键，用途绑定的定义级对象 | 流程用途治理的是"哪类流程可承接哪种用途"，必须锚定可长期复用的流程定义，而不是某次运行实例 |
| `business_scenario` | 业务场景（变更/终止/归档），用途分类的维度 | 不同业务场景有不同的流程用途定义 |

**稳定锚点原则**：流程用途字典只能引用上述稳定锚点作为输入，不得引用以下不稳定锚点：

- `workflow_instance_id`、`lifecycle_process_ref_id` 等流程实例或流程引用记录 ID（实例与引用记录都属于运行时对象）
- 流程引擎的内部节点 ID、任务 ID（流程引擎不拥有合同主档，其内部结构是外部对象）
- 文档中心的具体文件版本 ID（文件是外部对象）
- 临时计算字段、缓存字段、非持久化投影字段

进一步约束如下：

- 流程用途字典的治理锚点是 `purpose_code + business_scenario + workflow_definition_id` 这一组定义级语义，不是某一条流程实例记录。
- 流程实例只是在 `cl_lifecycle_process_ref` 中承接"某个业务事项实际发起了哪次流程"的运行时事实，用于流程跟踪、摘要投影和时间线消费。
- 因此，流程用途字典回答的是"什么用途可绑定什么流程定义"，流程实例回答的是"这次业务实际跑了哪个实例"，二者必须严格分层，不得混作同一锚点。

### 4.3 版本归属

流程用途字典的版本归属按以下原则划分：

| 规则类型 | 定义权归属 | 消费方 | 变更权限 | 版本化方式 |
| --- | --- | --- | --- | --- |
| 用途编码与名称 | `contract-lifecycle` 模块 | 流程引用、摘要、时间线、搜索、AI | 模块级变更，需同步更新摘要与时间线消费面 | 随模块版本演进，不单独版本化 |
| 用途分类 | `contract-lifecycle` 模块 | 流程引用、通知、AI | 模块级变更，需评估对通知的连锁影响 | 随模块版本演进，不单独版本化 |
| 用途业务语义 | `contract-lifecycle` 模块 | 流程引用、变更影响、终止善后、归档 | 模块级变更，需评估对变更、终止、归档的连锁影响 | 随模块版本演进，不单独版本化 |
| 用途与流程定义关联 | `contract-lifecycle` 模块 | 流程引用、`workflow-engine` 流程定义 | 模块级变更，需评估对流程引擎的连锁影响 | 随模块版本演进，不单独版本化 |

**版本归属原则**：

- 流程用途字典属于 `contract-lifecycle` 模块的内部治理资产，不属于 `workflow-engine` 的流程定义字典，也不属于 `contract-core` 的业务状态字典。
- 流程用途字典变更不需要变更合同主档、文档中心、流程引擎的版本，但可能需要刷新 `cl_lifecycle_summary` 和 `cl_lifecycle_timeline_event` 的消费面。
- 流程用途字典不对外暴露为独立 API 资源，只通过流程引用、摘要、时间线间接消费。
- 流程用途字典的变更需同步通知变更影响与回写专项设计（第二份专项设计）、终止善后与访问控制专项设计（第三份专项设计）、归档封包与借阅归还专项设计（第四份专项设计），确保各场景流程用途一致。

### 4.4 最小治理单元

流程用途字典的**最小治理单元**是：

- **单个用途编码**：每个 `purpose_code` 编码是一个最小治理单元，包含编码、名称、分类、业务语义、适用场景、与流程定义的关联。
- **用途分类条目**：每个分类层级是一个最小治理单元，包含分类编码、名称、业务语义。
- **用途与流程定义关联条目**：每个"用途编码 + 业务场景"组合是一个最小治理单元。

**最小治理单元原则**：

- 不允许将多个用途合并为一个"通用用途"，每个用途必须有独立的治理条目。
- 不允许将用途分类与用途编码合并为同一个治理单元，分类是分类，编码是编码。
- 不允许将流程用途字典与文档引用角色字典合并为同一个治理单元，用途是流程的用途，角色是文档的角色，二者不可混淆。

## 5. 时间线事件字典治理设计

### 5.1 治理对象

时间线事件字典治理的**直接对象**是：

| 治理对象 | 说明 | 不治理内容 |
| --- | --- | --- |
| 事件类型编码 | 事件类型唯一标识、编码规范、命名约束 | 事件实例数据、事件类型的使用统计 |
| 事件类型名称 | 中文名称、英文名称、简称 | 事件类型的多语言翻译资源文件 |
| 事件类型分类 | 业务分类（里程碑/状态变化/关键动作）、分类层级 | 分类的业务统计分析报表 |
| 事件类型业务语义 | 适用场景、不适用的场景、业务约束、是否里程碑 | 事件类型的具体处理逻辑 |
| 事件类型与来源表的关联约束 | 事件来源于哪个子域主表 | 来源主表的具体字段映射 |

时间线事件字典治理的**间接约束对象**是：

- `cl_lifecycle_timeline_event` 的 `event_type` 字段含义、枚举值域、校验约束
- `cl_lifecycle_summary` 中摘要块的里程碑事件聚合约束
- 各子域主表（履约/变更/终止/归档）与时间线事件的生成约束
- 搜索、AI、审计、通知对时间线事件的消费约束

### 5.2 稳定锚点

时间线事件字典的**稳定锚点**选择如下：

| 稳定锚点 | 说明 | 选择原因 |
| --- | --- | --- |
| `event_type` | 事件类型编码，字典的核心标识 | 事件类型是时间线、摘要、搜索、AI、通知的共同分类键 |
| `contract_id` | 合同主档主键，事件作用的根本业务实体 | 合同主档是业务真相源，事件必须围绕同一 `contract_id` 运行 |
| `source_table` | 事件来源子域主表，事件来源的治理维度 | 事件是子域主表动作的投影，必须明确来源 |
| `is_milestone` | 是否里程碑事件，事件分类的重要维度 | 里程碑是摘要、时间线、通知的重要分类维度 |

**稳定锚点原则**：时间线事件字典只能引用上述稳定锚点作为输入，不得引用以下不稳定锚点：

- 流程引擎的节点 ID、任务 ID（流程实例是外部对象）
- 文档中心的具体文件版本 ID（文件是外部对象）
- 临时计算字段、缓存字段、非持久化投影字段
- 具体事件实例的 ID（实例是运行时对象）
- 各子域主表的具体记录 ID（记录是运行时对象，不是事件类型的属性）

### 5.3 版本归属

时间线事件字典的版本归属按以下原则划分：

| 规则类型 | 定义权归属 | 消费方 | 变更权限 | 版本化方式 |
| --- | --- | --- | --- | --- |
| 事件类型编码与名称 | `contract-lifecycle` 模块 | 时间线、摘要、搜索、AI、审计、通知 | 模块级变更，需同步更新摘要与时间线消费面 | 随模块版本演进，不单独版本化 |
| 事件类型分类 | `contract-lifecycle` 模块 | 时间线、摘要、通知、AI | 模块级变更，需评估对摘要、通知的连锁影响 | 随模块版本演进，不单独版本化 |
| 事件类型业务语义 | `contract-lifecycle` 模块 | 时间线、各子域主表、变更影响、终止善后、归档 | 模块级变更，需评估对各子域的连锁影响 | 随模块版本演进，不单独版本化 |
| 事件类型与来源表关联 | `contract-lifecycle` 模块 | 时间线、各子域主表 | 模块级变更，需评估对各子域主表的连锁影响 | 随模块版本演进，不单独版本化 |

**版本归属原则**：

- 时间线事件字典属于 `contract-lifecycle` 模块的内部治理资产，不属于 `contract-core` 的业务事件字典，也不属于 `workflow-engine` 的流程事件字典。
- 时间线事件字典变更不需要变更合同主档、文档中心、流程引擎的版本，但可能需要刷新 `cl_lifecycle_summary`、`cl_lifecycle_timeline_event`、以及各子域主表的事件生成逻辑。
- 时间线事件字典不对外暴露为独立 API 资源，只通过时间线、摘要、搜索、AI、审计、通知间接消费。
- 时间线事件字典的变更需同步通知前四份专项设计，确保各子域的事件生成与事件类型定义一致：
  - 履约规则与风险评分专项设计（第一份）：履约里程碑事件定义
  - 变更影响与回写专项设计（第二份）：变更里程碑事件定义
  - 终止善后与访问控制专项设计（第三份）：终止里程碑事件定义
  - 归档封包与借阅归还专项设计（第四份）：归档里程碑事件定义

### 5.4 最小治理单元

时间线事件字典的**最小治理单元**是：

- **单个事件类型编码**：每个 `event_type` 编码是一个最小治理单元，包含编码、名称、分类、业务语义、是否里程碑、来源表。
- **事件类型分类条目**：每个分类层级（里程碑/状态变化/关键动作）是一个最小治理单元。
- **事件类型与来源表关联条目**：每个"事件类型 + 来源表"组合是一个最小治理单元。

**最小治理单元原则**：

- 不允许将多个事件类型合并为一个"通用事件类型"，每个事件类型必须有独立的治理条目。
- 不允许将事件类型分类与事件类型编码合并为同一个治理单元，分类是分类，编码是编码。
- 不允许将时间线事件字典与各子域过程表合并为同一个治理单元，事件是投影，过程是事实，二者不可混淆。

## 6. 通知模板治理设计

### 6.1 治理边界

通知模板的治理边界如下：

| 治理边界内（属于通知模板治理） | 治理边界外（不属于通知模板治理） |
| --- | --- |
| 通知场景定义（什么业务动作触发通知） | 具体合同生命周期过程触发的通知实例 |
| 通知模板结构（模板包含哪些占位符、哪些字段） | 通知模板的渲染代码、模板引擎配置 |
| 通知模板与事件类型的映射关系 | 通知发送的调度逻辑、重试逻辑 |
| 通知模板与摘要块的关联关系 | 通知模板的数据库存储实现 |
| 通知模板在摘要、时间线、审计中的呈现约束 | 通知模板的 API 接口设计 |
| 通知消费方约束（哪些角色接收什么通知） | 通知渠道配置、消息推送服务配置 |

**治理边界原则**：

- 通知模板治理的是"通知场景、模板结构、映射关系、消费方约束"，不是"怎么渲染通知、怎么发送通知"。
- 通知模板不替代时间线事件，通知模板是对时间线事件的消费投影，不是事件本身。
- 通知模板不拥有合同主档，通知模板只通过 `cl_lifecycle_summary` 和 `cl_lifecycle_timeline_event` 消费。
- 通知模板不拥有流程引擎的审批结论，流程审批结果是通知的输入，不是通知模板的直接治理对象。
- 通知模板不拥有文档中心的文件对象，文件引用只作为通知模板的间接引用，不作为模板治理的直接对象。

### 6.2 责任划分

通知模板的责任划分如下：

| 责任类型 | 归属方 | 说明 |
| --- | --- | --- |
| 通知场景定义权 | `contract-lifecycle` 模块 | 定义什么业务动作触发什么通知场景 |
| 通知模板结构定义权 | `contract-lifecycle` 模块 | 定义通知模板包含哪些占位符、哪些字段 |
| 通知模板与事件映射权 | `contract-lifecycle` 模块 | 定义什么事件类型对应什么通知模板 |
| 通知模板执行权 | `contract-lifecycle` 模块内部服务 | 按治理定义的场景、模板、映射执行通知，但不拥有模板渲染实现细节的治理权 |
| 通知消费权 | 通知中心、摘要域、时间线域、审计 | 消费稳定后的通知结果，不重新渲染通知 |
| 通知解释权 | `contract-lifecycle` 模块 | 对通知场景、模板结构、映射关系的业务含义、变更影响、消费影响拥有解释权 |
| 合同主档引用权 | `contract-core` 模块 | 通知中可能引用合同主档信息，但合同主档不负责通知模板定义 |
| 文档引用权 | `document-center` 模块 | 通知中可能引用文档信息，但文档中心不负责通知模板定义 |
| 流程引用权 | `workflow-engine` 模块 | 通知中可能引用流程信息，但流程引擎不负责通知模板定义 |

**责任划分原则**：

- `contract-lifecycle` 模块拥有通知模板的定义权和解释权，但不拥有通知发送渠道——通知通过平台通知中心发送。
- 通知模板不拥有合同主档，合同主档信息只作为通知模板的引用内容，不作为模板治理的直接对象。
- 通知模板不拥有流程引擎的审批节点，流程审批结果是通知的输入，不是通知模板的治理对象。
- 通知模板不拥有文档中心的文件对象，文件引用只作为通知模板的间接引用，不作为模板治理的直接对象。
- 通知模板的变更需同步通知前四份专项设计，确保通知场景与各子域业务动作一致。

### 6.3 最小治理单元

通知模板的**最小治理单元**是：

- **单个通知场景**：每个通知场景（如"履约节点即将到期"）是一个最小治理单元，包含场景编码、触发条件、模板结构、消费方。
- **通知模板与事件类型映射条目**：每个"通知场景 + 事件类型"组合是一个最小治理单元。
- **通知消费方条目**：每个"通知场景 + 消费方角色"组合是一个最小治理单元。

**最小治理单元原则**：

- 不允许将多个通知场景合并为一个"通用通知场景"，每个场景必须有独立的治理条目。
- 不允许将通知模板与通知发送合并为同一个治理单元，模板是模板，发送是发送。
- 不允许将通知模板与时间线事件字典合并为同一个治理单元，模板是消费，事件是来源，二者不可混淆。

## 7. 字典与通知的协同治理

### 7.1 字典之间的引用关系

| 字典 A | 字典 B | 引用关系 | 说明 |
| --- | --- | --- | --- |
| 文档引用角色字典 | 时间线事件字典 | 角色编码 → 事件类型 | 文档引用动作生成对应时间线事件 |
| 流程用途字典 | 时间线事件字典 | 用途编码 → 事件类型 | 流程动作生成对应时间线事件 |
| 时间线事件字典 | 通知模板 | 事件类型 → 通知场景 | 时间线事件触发对应通知场景 |

**协同治理原则**：

- 字典之间的引用关系必须明确、稳定、可追溯，不得存在循环依赖或模糊引用。
- 字典变更时，必须评估对引用方和被引用方的连锁影响。
- 字典之间的引用关系属于各字典治理的一部分，在各字典章节中已明确。

### 7.2 字典与通知的消费链路

```text
文档引用角色字典 → 文档引用动作 → 时间线事件字典 → 通知模板 → 通知中心
流程用途字典 → 流程动作 → 时间线事件字典 → 通知模板 → 通知中心
```

**消费链路原则**：

- 字典是治理定义，动作是运行时事实，事件是动作投影，通知是事件消费。
- 字典治理不负责动作执行、事件生成、通知发送，只负责定义"什么字典产生什么动作、生成什么事件、触发什么通知"。
- 通知模板治理不负责事件生成，只负责定义"什么事件触发什么通知"。

### 7.3 字典与通知的版本协同

当字典发生变更时，需同步评估以下影响：

| 变更类型 | 需同步评估的字典/通知 | 需同步通知的专项设计 |
| --- | --- | --- |
| 文档引用角色字典变更 | 时间线事件字典、通知模板 | 第二份（变更影响与回写） |
| 流程用途字典变更 | 时间线事件字典、通知模板 | 第二份（变更影响与回写）、第三份（终止善后与访问控制）、第四份（归档封包与借阅归还） |
| 时间线事件字典变更 | 通知模板 | 第一份、第二份、第三份、第四份 |
| 通知模板变更 | 无（通知模板是消费方） | 第一份、第二份、第三份、第四份 |

另：若第一份专项调整履约节点类型字典，本文只需评估时间线事件字典与通知模板的消费影响，不接管该字典本身的治理定义。

**版本协同原则**：

- 字典与通知的版本协同由 `contract-lifecycle` 模块统一负责，不分散到各消费方。
- 字典与通知的变更需在主模块版本中统一发布，不单独发布字典或通知模板。
- 字典与通知的变更需在所有相关专项设计中同步更新，确保治理一致性。

## 8. 与 `contract-core`、`document-center`、`workflow-engine` 的边界说明

### 8.1 与 `contract-core` 的边界

| 本模块治理对象 | `contract-core` 的边界 | 边界说明 |
| --- | --- | --- |
| 文档引用角色字典 | 不拥有合同主档主文档角色 | 文档引用角色是生命周期过程的文档角色，不是合同主档的主文档角色 |
| 流程用途字典 | 不拥有合同主档状态字典 | 流程用途是生命周期流程的分类，不是合同主档状态的分类 |
| 时间线事件字典 | 不拥有合同主档事件字典 | 时间线事件是生命周期过程的投影，不是合同主档本身的事件 |
| 通知模板 | 不拥有合同主档通知模板 | 通知模板是生命周期过程的通知，不是合同主档本身的通知 |

**边界原则**：

- `contract-core` 负责合同主档的本体治理，包括合同类型、合同状态、合同字段等。
- `contract-lifecycle` 负责合同后续业务过程的治理，包括文档引用角色、流程用途、时间线事件和通知模板；履约节点类型字典另由第一份专项设计承接。
- 二者通过 `LifecycleIntegrationFacade` 和 `LifecycleWritebackService` 进行集成，不交叉治理。

### 8.2 与 `document-center` 的边界

| 本模块治理对象 | `document-center` 的边界 | 边界说明 |
| --- | --- | --- |
| 文档引用角色字典 | 不拥有文档元数据字典 | 文档引用角色是业务过程中的角色，不是文档本身的元数据 |
| 通知模板（可能引用文档） | 不拥有文档通知模板 | 通知模板只引用文档信息，不拥有文档的治理权 |

**边界原则**：

- `document-center` 负责文档对象的本体治理，包括文档类型、文档版本、文档元数据等。
- `contract-lifecycle` 负责业务过程中文档引用的角色治理，不拥有文档对象本身。
- 二者通过 `cl_lifecycle_document_ref` 进行集成，文档中心是文件真相源。

### 8.3 与 `workflow-engine` 的边界

| 本模块治理对象 | `workflow-engine` 的边界 | 边界说明 |
| --- | --- | --- |
| 流程用途字典 | 不拥有流程定义字典 | 流程用途是业务过程中流程的分类，不是流程引擎的流程定义分类 |
| 时间线事件字典（流程相关） | 不拥有流程事件字典 | 时间线事件是业务过程对流程动作的投影，不是流程引擎本身的事件 |
| 通知模板（可能引用流程） | 不拥有流程通知模板 | 通知模板只引用流程信息，不拥有流程的治理权 |

**边界原则**：

- `workflow-engine` 负责流程定义的本体治理，包括流程类型、流程节点、流程表单等。
- `contract-lifecycle` 负责业务过程中流程用途的治理，不拥有流程定义本身。
- 二者通过 `cl_lifecycle_process_ref` 进行集成，流程引擎通过 `contract_id` 绑定合同。

## 9. 本文结论

本专项设计已经明确以下治理基线：

- 文档引用角色字典、流程用途字典、时间线事件字典的治理对象、稳定锚点、版本归属、最小治理单元
- 通知模板的治理边界、责任划分、最小治理单元
- 字典之间的引用关系、字典与通知的消费链路、版本协同原则
- 第一份专项设计中履约节点类型字典与本文治理对象之间的引用依赖边界
- 与 `contract-core`、`document-center`、`workflow-engine` 的边界划分
- 与前四份专项设计的引用关系和协同治理要求

本文档是治理设计，不是实现设计。所有字典与通知的具体实现、存储、查询、渲染、发送均不在本文档范围内。
