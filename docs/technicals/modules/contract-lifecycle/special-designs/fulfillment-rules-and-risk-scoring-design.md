## 2. 履约规则治理设计

### 2.1 治理对象

履约规则治理的**直接对象**是：

| 治理对象 | 说明 | 不治理内容 |
| --- | --- | --- |
| 履约节点类型字典 | 节点类型编码、名称、分类、业务语义、适用场景 | 节点实例数据、节点执行日志 |
| 逾期判定规则 | 逾期判定基准、宽限期、例外条件、生效范围 | 具体节点的逾期计算逻辑代码 |
| 履约状态流转规则 | 节点状态、聚合状态、完成判定条件的治理约束 | 状态机实现代码、状态持久化 |
| 履约责任归属规则 | 节点责任人、组织单元、变更规则的治理约束 | 具体责任人分配算法 |

履约规则治理的**间接约束对象**是：

- `cl_performance_record` 与 `cl_performance_node` 的字段含义、枚举值域、状态语义
- `cl_lifecycle_summary` 中履约摘要块的结构约束
- `cl_lifecycle_timeline_event` 中履约里程碑事件的生成约束

### 2.2 稳定锚点

履约规则的**稳定锚点**选择如下：

| 稳定锚点 | 说明 | 选择原因 |
| --- | --- | --- |
| `contract_id` | 合同主档主键，履约规则作用的根本业务实体 | 合同主档是业务真相源，履约必须围绕同一 `contract_id` 运行 |
| `node_type` | 履约节点类型编码，规则治理的最小分类维度 | 节点类型是履约规则分类、逾期判定、风险评分的稳定分类键 |
| `performance_status` / `node_status` | 履约聚合状态与节点状态，规则治理的稳定状态维度 | 状态是逾期判定、风险评分、责任归属的稳定输入维度 |
| `planned_at` / `due_at` | 计划时间与到期时间，逾期判定的稳定时间基准 | 时间是逾期判定规则的核心输入，必须是稳定字段 |

**稳定锚点原则**：履约规则只能引用上述稳定锚点作为输入，不得引用以下不稳定锚点：

- 流程引擎的节点 ID、任务 ID（流程实例是外部对象）
- 文档中心的具体文件版本 ID（文件是外部对象）
- 临时计算字段、缓存字段、非持久化投影字段

### 2.3 版本归属

履约规则的版本归属按以下原则划分：

| 规则类型 | 定义权归属 | 消费方 | 变更权限 | 版本化方式 |
| --- | --- | --- | --- | --- |
| 履约节点类型字典 | `contract-lifecycle` 模块 | 履约聚合、节点明细、摘要、时间线、搜索、AI | 模块级变更，需同步更新摘要与时间线消费面 | 随模块版本演进，不单独版本化 |
| 逾期判定规则 | `contract-lifecycle` 模块 | 履约节点、履约聚合、风险评分、通知、AI | 模块级变更，需评估对风险评分、通知的连锁影响 | 随模块版本演进，不单独版本化 |
| 履约状态流转规则 | `contract-lifecycle` 模块 | 履约聚合、摘要、时间线、合同主档回写 | 模块级变更，需评估对合同主档回写的连锁影响 | 随模块版本演进，不单独版本化 |
| 履约责任归属规则 | `contract-lifecycle` 模块 | 履约节点、通知、审计 | 模块级变更，需同步组织架构底座 | 随模块版本演进，不单独版本化 |

**版本归属原则**：

- 履约规则属于 `contract-lifecycle` 模块的内部治理资产，不属于 `contract-core` 的合同主档规则，也不属于 `workflow-engine` 的流程规则。
- 履约规则变更不需要变更合同主档、文档中心、流程引擎的版本，但可能需要刷新 `cl_lifecycle_summary` 和 `cl_lifecycle_timeline_event` 的消费面。
- 履约规则不对外暴露为独立 API 资源，只通过履约聚合、节点、摘要、时间线间接消费。

### 2.4 最小治理单元

履约规则的**最小治理单元**是：

- **履约节点类型**：每个 `node_type` 编码是一个最小治理单元，包含类型编码、名称、分类、业务语义、适用场景。
- **逾期判定规则**：按"节点类型 + 状态条件"组合为最小治理单元，例如"交付类节点、已逾期、未完成"的判定规则。
- **履约状态流转规则**：按"状态来源 + 目标状态"组合为最小治理单元，例如"节点状态 → 聚合状态"的映射规则。
- **履约责任归属规则**：按"节点类型 + 责任维度"组合为最小治理单元，例如"付款类节点、组织单元归属"规则。

**最小治理单元原则**：

- 不允许将多个节点类型的规则合并为一个"通用规则"，每个节点类型必须有独立的治理条目。
- 不允许将逾期判定与风险评分合并为同一个治理单元，逾期判定是事实判断，风险评分是评估判断。
- 不允许将履约责任归属与流程引擎的审批节点责任人混为同一个治理单元，履约责任归属是业务规则，流程节点责任人是流程规则。

## 3. 风险评分治理设计

### 3.1 治理边界

风险评分的治理边界如下：

| 治理边界内（属于风险评分治理） | 治理边界外（不属于风险评分治理） |
| --- | --- |
| 风险等级枚举定义（`risk_level` 的值域） | 具体合同履约记录的风险等级计算结果 |
| 风险评分维度定义（评分从哪些维度来） | 评分算法的代码实现 |
| 风险评分与履约状态的映射关系 | 评分的实时计算调度逻辑 |
| 风险等级变更的事件生成规则 | 风险等级字段的数据库存储实现 |
| 风险评分结果在摘要、时间线、通知中的呈现约束 | 风险评分的 API 接口设计 |

**治理边界原则**：

- 风险评分治理的是"评分维度、等级定义、映射关系、呈现约束"，不是"怎么算分"。
- 风险评分不替代履约状态，风险评分是对履约状态的评估投影，不是履约状态本身。
- 风险评分不拥有合同主档，风险评分结果只通过 `cl_lifecycle_summary` 和 `cl_lifecycle_timeline_event` 消费。

### 3.2 责任划分

风险评分的责任划分如下：

| 责任类型 | 归属方 | 说明 |
| --- | --- | --- |
| 风险维度定义权 | `contract-lifecycle` 模块 | 定义从哪些履约维度（节点逾期、进度延迟、责任缺失等）参与评分 |
| 风险等级定义权 | `contract-lifecycle` 模块 | 定义风险等级编码、名称、业务语义 |
| 风险评分执行权 | `contract-lifecycle` 模块内部服务 | 按治理定义的维度和等级执行评分，但不拥有评分算法实现细节的治理权 |
| 风险评分消费权 | 摘要域、时间线域、通知、AI | 消费稳定后的风险等级，不重新计算风险 |
| 风险评分解释权 | `contract-lifecycle` 模块 | 对风险等级的业务含义、变更影响、消费影响拥有解释权 |
| 风险评分审计权 | 平台审计中心 | 对风险等级变更动作进行审计留痕 |

**责任划分原则**：

- `contract-lifecycle` 模块拥有风险评分的定义权和解释权，但不拥有合同主档的修改权——风险评分结果通过 `LifecycleIntegrationFacade` 回写合同主档。
- 风险评分不拥有流程引擎的审批结论，流程审批结果是履约的输入，不是风险评分的直接治理对象。
- 风险评分不拥有文档中心的文件对象，文件引用只作为风险评分的间接引用，不作为评分治理的直接对象。

### 3.3 与履约规则的关系

风险评分与履约规则的治理关系：

- 履约规则是风险评分的**输入治理约束**：逾期判定规则的结果、履约状态流转规则的结果，是风险评分的输入维度之一。
- 风险评分不修改履约规则：风险评分只能读取履约规则的稳定输出，不能反向修改履约节点类型、逾期判定规则、履约状态流转规则。
- 履约规则变更可能触发风险评分的重新评估：履约规则变更后，已存在的履约记录和节点可能需要重新评分，但这属于补偿和恢复边界，不属于风险评分治理本身。

## 4. 与周边模块的边界说明

### 4.1 与 `contract-core` 的边界

| 边界维度 | `contract-core` 负责 | `contract-lifecycle` 履约规则与风险评分负责 |
| --- | --- | --- |
| 合同主档 | 持有 `contract_id`、合同身份、生命周期主状态、主状态流转规则 | 不持有合同主档，只通过 `contract_id` 引用合同 |
| 履约状态回写 | 提供受控回写接口，接收履约摘要、履约阶段状态 | 通过 `LifecycleIntegrationFacade` 调用回写接口，回写履约摘要、阶段状态、风险等级摘要 |
| 生命周期映射 | 持有生命周期主状态与周边状态的映射规则 | 履约状态流转规则不与生命周期主状态映射规则混淆，履约状态是周边状态 |
| 风险等级呈现 | 合同主档可以持有履约风险等级摘要引用 | 风险评分不直写合同主档，只通过摘要回写间接呈现 |

**边界原则**：履约规则与风险评分不拥有合同主档，不修改合同主档的身份、分类、主状态。`contract-core` 不拥有履约节点类型、逾期判定、风险评分维度，这些属于 `contract-lifecycle` 的治理资产。

### 4.2 与 `document-center` 的边界

| 边界维度 | `document-center` 负责 | `contract-lifecycle` 履约规则与风险评分负责 |
| --- | --- | --- |
| 文件对象 | 持有文件对象主记录、版本链、存储定位、加密状态 | 不持有文件对象，只通过 `cl_lifecycle_document_ref` 引用文件 |
| 履约凭证 | 文件对象可以表达履约凭证、补充协议等 | 履约节点类型字典不包含文件格式规则、文件内容判定规则，文件只作为履约节点的引用 |
| 风险评分依据 | 文件内容是 AI 风险识别的输入之一 | 风险评分维度不包含"文件内容审查"作为直接评分维度，文件内容审查属于 AI 辅助能力，不是履约规则治理的一部分 |

**边界原则**：履约规则与风险评分不拥有文件对象，不定义文件格式、文件内容、文件版本规则。文件引用只作为履约节点的业务语义附件，不作为履约规则或风险评分的直接治理对象。

### 4.3 与 `workflow-engine` 的边界

| 边界维度 | `workflow-engine` 负责 | `contract-lifecycle` 履约规则与风险评分负责 |
| --- | --- | --- |
| 流程定义 | 持有流程定义、节点定义、审批结论 | 不持有流程定义，只通过 `cl_lifecycle_process_ref` 引用流程实例 |
| 履约审批 | 变更审批、终止审批、归档审批的流程实例 | 履约节点类型不包含"审批节点类型"，审批流程是外部对象，不是履约节点 |
| 逾期判定 | 流程任务可以有到期时间 | 履约逾期判定规则不引用流程任务的到期时间，只引用履约节点的 `planned_at` / `due_at` |
| 风险评分 | 流程审批结论可以作为履约的输入 | 风险评分不引用流程节点状态、流程任务状态，只引用履约状态流转后的稳定结果 |

**边界原则**：履约规则与风险评分不拥有流程引擎，不引用流程节点 ID、任务 ID、流程定义 ID 作为规则输入。流程审批结果是履约的输入，但不是履约规则或风险评分的稳定锚点。

## 5. 治理真理与稳定性原则

### 5.1 治理真理

履约规则与风险评分的治理真理：

1. **履约规则是 `contract-lifecycle` 的内部治理资产**，不属于合同主档、不属于流程引擎、不属于文档中心。
2. **风险评分是对履约状态的评估投影**，不替代履约状态，不拥有合同主档。
3. **稳定锚点只引用 `contract_id`、节点类型、状态字段、时间字段**，不引用流程实例、文件版本、临时计算字段。
4. **最小治理单元必须独立、不可合并**，节点类型、逾期判定、状态流转、责任归属必须分治。
5. **规则变更必须评估消费面影响**，摘要、时间线、通知、AI 的连锁影响必须同步评估。

### 5.2 稳定性原则

履约规则与风险评分的稳定性原则：

- 节点类型编码一旦发布，不得随意删除或重用，只能新增或标记为废弃。
- 风险等级编码一旦发布，不得随意删除或重用，只能新增或标记为废弃。
- 逾期判定规则的宽限期、例外条件变更，必须评估对已存在履约记录的影响。
- 履约规则与风险评分的变更，必须同步更新 `cl_lifecycle_summary` 和 `cl_lifecycle_timeline_event` 的消费面约束。

## 6. 本文结论

履约规则与风险评分的治理设计已明确为：

- 履约规则治理对象是节点类型字典、逾期判定规则、状态流转规则、责任归属规则，最小治理单元按节点类型和规则类型分治。
- 履约规则的稳定锚点是 `contract_id`、节点类型、状态字段、计划/到期时间，不引用流程实例或文件版本。
- 履约规则版本归属 `contract-lifecycle` 模块，随模块版本演进，不单独版本化，不对外暴露为独立 API 资源。
- 风险评分治理边界是维度定义、等级定义、映射关系、呈现约束，不治理评分算法实现。
- 风险评分责任划分明确：定义权和解释权归 `contract-lifecycle` 模块，执行权归模块内部服务，消费权归摘要、时间线、通知、AI。
- 与 `contract-core` 的边界：履约规则不拥有合同主档，风险评分只通过摘要回写间接呈现。
- 与 `document-center` 的边界：履约规则不拥有文件对象，文件只作为履约节点的业务语义附件。
- 与 `workflow-engine` 的边界：履约规则不引用流程实例，流程审批结果是履约输入但不是规则稳定锚点。
