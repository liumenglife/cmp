## 2. 变更影响治理设计

### 2.1 治理对象

变更影响治理的**直接对象**是：

| 治理对象 | 说明 | 不治理内容 |
| --- | --- | --- |
| 变更影响评估块结构 | `impact_scope_json` 的顶层结构、一级键、类型约束 | 影响评估的具体计算结果、评估算法实现 |
| 变更影响范围条目 | 影响范围条目的类型编码、业务语义、影响维度 | 具体变更记录的影响评估结果数据 |
| 变更影响等级定义 | 影响等级编码、名称、业务语义、适用场景 | 影响等级的实时计算结果 |
| 变更生效日期规则 | 生效日期计算基准、生效条件、例外场景 | 具体变更记录的生效日期计算逻辑 |

变更影响治理的**间接约束对象**是：

- `cl_contract_change` 的 `impact_scope_json`、`effective_date`、`result_version_no` 字段含义与约束
- `cl_lifecycle_summary` 中变更摘要块的结构约束
- `cl_lifecycle_timeline_event` 中变更里程碑事件的生成约束
- `LifecycleIntegrationFacade` 向 `contract-core` 回写的内容约束

### 2.2 稳定锚点

变更影响的**稳定锚点**选择如下：

| 稳定锚点 | 说明 | 选择原因 |
| --- | --- | --- |
| `contract_id` | 合同主档主键，变更影响评估的根本业务实体 | 合同主档是业务真相源，变更必须围绕同一 `contract_id` 运行 |
| `change_id` | 变更事项主键，变更影响评估的作用对象 | 一次变更对应一条 `ContractChange`，影响评估依附于变更事项 |
| `change_type` | 变更类型编码，影响评估分类的稳定维度 | 变更类型是影响范围、影响等级的稳定分类键 |
| `impact_scope_json` | 影响评估块，承载变更影响范围结构化表达 | 影响评估块是变更影响治理的核心载体 |
| `effective_date` | 变更生效日期，影响评估的时间基准 | 生效日期是影响范围、回写时机的重要输入 |

**稳定锚点原则**：变更影响评估只能引用上述稳定锚点作为输入，不得引用以下不稳定锚点：

- 流程引擎的节点 ID、任务 ID（流程实例是外部对象）
- 文档中心的具体文件版本 ID（文件是外部对象）
- 临时计算字段、缓存字段、非持久化投影字段
- 合同主档中不属于变更影响评估结果的字段

### 2.3 版本归属

变更影响的版本归属按以下原则划分：

| 规则类型 | 定义权归属 | 消费方 | 变更权限 | 版本化方式 |
| --- | --- | --- | --- | --- |
| 变更影响评估块结构 | `contract-lifecycle` 模块 | 变更事项、摘要、时间线、搜索、AI | 模块级变更，需同步更新摘要与时间线消费面 | 随模块版本演进，不单独版本化 |
| 变更影响范围条目 | `contract-lifecycle` 模块 | 变更事项、摘要、通知、AI | 模块级变更，需评估对摘要、通知的连锁影响 | 随模块版本演进，不单独版本化 |
| 变更影响等级定义 | `contract-lifecycle` 模块 | 变更事项、摘要、时间线、合同主档回写 | 模块级变更，需评估对合同主档回写的连锁影响 | 随模块版本演进，不单独版本化 |
| 变更生效日期规则 | `contract-lifecycle` 模块 | 变更事项、摘要、回写时机 | 模块级变更，需评估对回写时机的影响 | 随模块版本演进，不单独版本化 |

**版本归属原则**：

- 变更影响评估属于 `contract-lifecycle` 模块的内部治理资产，不属于 `contract-core` 的合同主档规则，也不属于 `workflow-engine` 的流程规则。
- 变更影响评估变更不需要变更合同主档、文档中心、流程引擎的版本，但可能需要刷新 `cl_lifecycle_summary`、`cl_lifecycle_timeline_event`、以及合同主档的回写内容。
- 变更影响评估不对外暴露为独立 API 资源，只通过变更事项、摘要、时间线间接消费。

### 2.4 最小治理单元

变更影响的**最小治理单元**是：

- **变更影响范围条目**：每个影响范围条目类型编码是一个最小治理单元，包含类型编码、名称、业务语义、影响维度。
- **变更影响等级**：每个影响等级编码是一个最小治理单元，包含等级编码、名称、业务语义、适用场景。
- **变更生效日期规则**：按"变更类型 + 生效条件"组合为最小治理单元，例如"金额变更、生效后次日"的规则。

**最小治理单元原则**：

- 不允许将多个影响范围条目类型合并为一个"通用影响范围"，每个影响维度必须有独立的治理条目。
- 不允许将影响等级与回写规则合并为同一个治理单元，影响等级是评估结果，回写规则是应用规则。
- 不允许将变更影响评估与履约规则合并为同一个治理单元，参见[第一份专项设计](fulfillment-rules-and-risk-scoring-design.md)。

## 3. 字段级映射治理设计

### 3.1 治理边界

字段级映射的治理边界如下：

| 治理边界内（属于字段级映射治理） | 治理边界外（不属于字段级映射治理） |
| --- | --- |
| 字段映射关系定义（变更源字段 → 合同主档目标字段） | 具体变更记录的字段映射执行结果 |
| 字段映射条件约束（什么条件下触发映射） | 映射算法的代码实现 |
| 字段映射后的取值范围校验规则 | 字段映射的实时计算调度逻辑 |
| 字段映射与变更影响等级的关联关系 | 字段映射的数据库存储实现 |
| 字段映射结果在摘要、时间线、通知中的呈现约束 | 字段映射的 API 接口设计 |

**治理边界原则**：

- 字段级映射治理的是"映射关系、条件约束、校验规则、呈现约束"，不是"怎么执行映射"。
- 字段级映射不替代变更影响评估，字段级映射是变更影响评估结果的执行落地，不是评估本身。
- 字段级映射不拥有合同主档，映射结果只通过 `LifecycleIntegrationFacade` 回写合同主档。

### 3.2 责任划分

字段级映射的责任划分如下：

| 责任类型 | 归属方 | 说明 |
| --- | --- | --- |
| 字段映射关系定义权 | `contract-lifecycle` 模块 | 定义变更源字段与合同主档目标字段的映射关系 |
| 字段映射条件定义权 | `contract-lifecycle` 模块 | 定义什么条件下触发字段映射 |
| 字段映射执行权 | `contract-lifecycle` 模块内部服务 | 按治理定义的映射关系和条件执行映射，但不拥有映射算法实现细节的治理权 |
| 字段映射消费权 | 摘要域、时间线域、通知、AI | 消费稳定后的字段映射结果，不重新计算映射 |
| 字段映射解释权 | `contract-lifecycle` 模块 | 对字段映射关系的业务含义、变更影响、消费影响拥有解释权 |
| 合同主档回写权 | `contract-core` 模块 | 提供受控回写接口，接收字段映射结果 |
| 字段映射审计权 | 平台审计中心 | 对字段映射变更动作进行审计留痕 |

**责任划分原则**：

- `contract-lifecycle` 模块拥有字段映射的定义权和解释权，但字段映射结果通过 `LifecycleIntegrationFacade` 回写合同主档，不直接修改合同主档。
- 字段映射不拥有流程引擎的审批结论，流程审批结果是变更的前置条件，不是字段映射的直接治理对象。
- 字段映射不拥有文档中心的文件对象，文件引用只作为变更影响的间接引用，不作为字段映射的直接治理对象。

### 3.3 与第一份专项设计的关系

字段级映射与履约规则与风险评分的治理关系：

- 履约规则与风险评分是变更影响评估的**输入治理约束之一**：履约状态、风险等级可以作为变更影响评估的补充输入维度。
- 字段级映射不修改履约规则与风险评分：字段级映射只能读取履约和风险评分的稳定输出，不能反向修改履约节点类型、逾期判定规则、风险评分维度。
- 履约规则变更可能触发变更影响评估的重新评估：履约规则变更后，已存在的变更记录可能需要重新评估影响，但这属于补偿和恢复边界，不属于字段级映射治理本身。

## 4. 回写规则治理设计

### 4.1 回写边界

回写规则的治理边界如下：

| 回写边界内（属于回写规则治理） | 回写边界外（不属于回写规则治理） |
| --- | --- |
| 回写内容定义（回写什么字段、什么条件下回写） | 回写接口的调用代码实现 |
| 回写时机定义（审批通过后、生效后、人工确认后） | 回写调用的调度逻辑 |
| 回写失败处理规则（重试、补偿、人工介入） | 回写失败的具体重试算法 |
| 回写结果在摘要、时间线中的呈现约束 | 回写接口的 API 设计 |
| 回写版本号（`result_version_no`）的递增规则 | 回写版本号的数据库存储实现 |

**回写边界原则**：

- 回写规则治理的是"回写什么、何时回写、失败怎么办、如何呈现"，不是"怎么调用回写接口"。
- 回写规则不拥有合同主档，回写结果只是对合同主档的更新操作，合同主档的真相源仍在 `contract-core`。
- 回写失败不回滚变更事项本身，变更事项的状态是独立真相，回写失败只影响合同主档的一致性，通过补偿机制修复。

### 4.2 回写内容约束

回写内容的治理约束如下：

| 回写内容 | 是否回写 | 说明 |
| --- | --- | --- |
| 最新变更摘要引用 | 是 | 回写到合同主档的变更摘要字段 |
| 变更后的生效日期 | 是 | 回写到合同主档的生效日期字段 |
| 变更后的合同金额 | 条件回写 | 仅当变更影响范围包含金额变更时回写 |
| 变更后的合同期限 | 条件回写 | 仅当变更影响范围包含期限变更时回写 |
| 变更后的合同状态 | 条件回写 | 变更完成后更新合同主档的生命周期状态 |
| 变更详情全文 | 否 | 不集中回写到合同主档，只通过摘要引用 |
| 变更文档全文 | 否 | 通过 `cl_lifecycle_document_ref` 引用，不回写合同主档 |
| 履约状态、风险等级 | 否 | 这些是履约规则与风险评分的治理对象，不属于变更回写 |

**回写内容原则**：

- 只回写稳定结果：最新变更摘要、生效日期、条件性金额/期限、变更后状态。
- 不回写详情：变更详情保留在 `cl_contract_change` 和 `impact_scope_json` 中，合同主档只持有摘要引用。
- 不回写文件：变更协议等文件通过文档引用挂接，不集中在合同主档字段中。

### 4.3 回写责任划分

回写规则的责任划分如下：

| 责任类型 | 归属方 | 说明 |
| --- | --- | --- |
| 回写内容定义权 | `contract-lifecycle` 模块 | 定义回写什么、什么条件下回写 |
| 回写时机定义权 | `contract-lifecycle` 模块 | 定义审批通过后、生效后等回写时机 |
| 回写执行权 | `LifecycleIntegrationFacade` | 负责调用 `contract-core` 的受控回写接口 |
| 回写接口提供权 | `contract-core` 模块 | 提供受控回写接口，接收回写内容 |
| 回写失败处理权 | `contract-lifecycle` 模块 | 负责回写失败的重试、补偿、人工介入策略 |
| 回写结果消费权 | 摘要域、时间线域、通知 | 消费回写完成后的稳定状态 |
| 回写审计权 | 平台审计中心 | 对回写动作、失败、补偿进行审计留痕 |

## 5. 与周边模块的边界说明

### 5.1 与 `contract-core` 的边界

| 边界维度 | `contract-core` 负责 | `contract-lifecycle` 变更影响与回写负责 |
| --- | --- | --- |
| 合同主档 | 持有 `contract_id`、合同身份、生命周期主状态、主状态流转规则 | 不持有合同主档，只通过 `contract_id` 引用合同 |
| 变更回写 | 提供受控回写接口，接收变更摘要、生效日期、条件性金额/期限、变更后状态 | 通过 `LifecycleIntegrationFacade` 调用回写接口，回写变更结果 |
| 变更影响评估 | 不持有变更影响评估块、影响范围条目、影响等级 | 变更影响评估块 `impact_scope_json` 是 `contract-lifecycle` 的内部治理资产 |
| 字段级映射 | 不持有变更源字段到合同主档字段的映射关系 | 字段级映射关系是 `contract-lifecycle` 的治理资产 |

**边界原则**：变更影响与回写不拥有合同主档，不修改合同主档的身份、分类、主状态。`contract-core` 不拥有变更影响评估块、字段级映射关系，这些属于 `contract-lifecycle` 的治理资产。

### 5.2 与 `document-center` 的边界

| 边界维度 | `document-center` 负责 | `contract-lifecycle` 变更影响与回写负责 |
| --- | --- | --- |
| 文件对象 | 持有文件对象主记录、版本链、存储定位 | 不持有文件对象，只通过 `cl_lifecycle_document_ref` 引用文件 |
| 变更协议 | 文件对象可以表达变更协议、补充协议 | 变更影响评估块不包含文件格式规则、文件内容判定规则，文件只作为变更影响的间接引用 |
| 回写内容 | 合同主档可以持有变更协议文件引用 | 回写内容不包含文件本体，只包含变更摘要引用 |

**边界原则**：变更影响与回写不拥有文件对象，不定义文件格式、文件内容、文件版本规则。文件引用只作为变更影响评估的业务语义附件，不作为变更影响或字段级映射的直接治理对象。

### 5.3 与 `workflow-engine` 的边界

| 边界维度 | `workflow-engine` 负责 | `contract-lifecycle` 变更影响与回写负责 |
| --- | --- | --- |
| 流程定义 | 持有流程定义、节点定义、审批结论 | 不持有流程定义，只通过 `cl_lifecycle_process_ref` 引用流程实例 |
| 变更审批 | 变更审批的流程实例 | 变更影响评估不包含"审批节点类型"，审批流程是外部对象，不是变更影响评估的直接输入 |
| 回写时机 | 流程审批结论是回写时机的条件之一 | 回写时机定义引用流程审批结果，但不把流程节点 ID 作为稳定锚点 |
| 影响评估 | 流程审批结论可以作为变更的前置条件 | 变更影响评估不引用流程节点状态、流程任务状态，只引用审批完成后的稳定结果 |

**边界原则**：变更影响与回写不拥有流程引擎，不引用流程节点 ID、任务 ID、流程定义 ID 作为治理输入。流程审批结果是变更的前置条件，但不是变更影响评估或字段级映射的稳定锚点。

## 6. 治理真理与稳定性原则

### 6.1 治理真理

变更影响与回写的治理真理：

1. **变更影响评估是 `contract-lifecycle` 的内部治理资产**，不属于合同主档、不属于流程引擎、不属于文档中心。
2. **字段级映射是对变更影响评估结果的执行落地**，不替代变更影响评估，不拥有合同主档。
3. **回写规则只定义回写什么、何时回写、失败怎么办**，不拥有合同主档，回写结果通过 `LifecycleIntegrationFacade` 执行。
4. **稳定锚点只引用 `contract_id`、变更 ID、变更类型、影响评估块、生效日期**，不引用流程实例或文件版本。
5. **最小治理单元必须独立、不可合并**，影响范围条目、影响等级、生效日期规则必须分治。
6. **变更影响评估变更必须评估消费面影响**，摘要、时间线、通知、AI、合同主档回写的连锁影响必须同步评估。

### 6.2 稳定性原则

变更影响与回写的稳定性原则：

- 变更类型编码一旦发布，不得随意删除或重用，只能新增或标记为废弃。
- 影响等级编码一旦发布，不得随意删除或重用，只能新增或标记为废弃。
- 影响范围条目类型编码一旦发布，不得随意删除或重用，只能新增或标记为废弃。
- 字段映射关系变更，必须评估对已存在变更记录的影响。
- 回写规则变更，必须评估对合同主档、摘要、时间线、通知的连锁影响。
- 变更影响与回写的变更，必须同步更新 `cl_lifecycle_summary`、`cl_lifecycle_timeline_event` 的消费面约束，必要时触发合同主档的补偿回写。

## 7. 本文结论

变更影响与回写的治理设计已明确为：

- 变更影响治理对象是影响评估块结构、影响范围条目、影响等级定义、生效日期规则，最小治理单元按影响维度分治。
- 变更影响的稳定锚点是 `contract_id`、变更 ID、变更类型、影响评估块、生效日期，不引用流程实例或文件版本。
- 变更影响版本归属 `contract-lifecycle` 模块，随模块版本演进，不单独版本化，不对外暴露为独立 API 资源。
- 字段级映射治理边界是映射关系、条件约束、校验规则、呈现约束，不治理映射算法实现。
- 字段级映射责任划分明确：定义权和解释权归 `contract-lifecycle` 模块，执行权归模块内部服务，消费权归摘要、时间线、通知、AI，回写权归 `contract-core` 模块。
- 回写规则治理边界是回写内容、回写时机、回写失败处理、回写呈现约束，不治理回写接口调用实现。
- 回写内容只回写稳定结果：最新变更摘要、生效日期、条件性金额/期限、变更后状态，不回写详情全文和文件本体。
- 与 `contract-core` 的边界：变更影响与回写不拥有合同主档，只通过 `LifecycleIntegrationFacade` 回写变更结果。
- 与 `document-center` 的边界：变更影响与回写不拥有文件对象，文件只作为变更影响的业务语义附件。
- 与 `workflow-engine` 的边界：变更影响与回写不引用流程实例，审批结果是变更的前置条件但不是稳定锚点。
