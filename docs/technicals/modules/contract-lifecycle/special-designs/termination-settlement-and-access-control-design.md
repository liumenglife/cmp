## 2. 终止善后清单治理设计

### 2.1 治理对象

终止善后清单治理的**直接对象**是：

| 治理对象 | 说明 | 不治理内容 |
| --- | --- | --- |
| 终止善后清单结构 | 善后清单的顶层结构、条目分类、完成标准 | 具体终止记录的善后执行结果数据 |
| 善后事项类型字典 | 善后事项类型编码、名称、分类、业务语义、适用场景 | 具体善后事项的实例数据 |
| 善后完成判定规则 | 善后完成判定基准、例外条件、生效范围 | 具体终止记录的善后完成计算逻辑 |
| 善后责任归属规则 | 善后责任人、组织单元、变更规则的治理约束 | 具体责任人分配算法 |

终止善后清单治理的**间接约束对象**是：

- `cl_contract_termination` 的 `post_action_status`、`settlement_summary` 字段含义与约束
- `cl_lifecycle_summary` 中终止摘要块的结构约束
- `cl_lifecycle_timeline_event` 中终止里程碑事件的生成约束

### 2.2 稳定锚点

终止善后的**稳定锚点**选择如下：

| 稳定锚点 | 说明 | 选择原因 |
| --- | --- | --- |
| `contract_id` | 合同主档主键，终止善后作用的根本业务实体 | 合同主档是业务真相源，终止必须围绕同一 `contract_id` 运行 |
| `termination_id` | 终止事项主键，终止善后清单的作用对象 | 一次终止对应一条 `ContractTermination`，善后清单依附于终止事项 |
| `termination_type` | 终止类型编码，善后清单分类的稳定维度 | 终止类型是善后事项分类、完成判定的稳定分类键 |
| `termination_status` | 终止状态，善后清单生效的状态基准 | 状态是善后判定、权限限制的稳定输入维度 |
| `post_action_status` | 善后状态，善后清单完成情况的稳定表达 | 善后状态是结算规则、权限限制的重要输入 |

**稳定锚点原则**：终止善后清单只能引用上述稳定锚点作为输入，不得引用以下不稳定锚点：

- 流程引擎的节点 ID、任务 ID（流程实例是外部对象）
- 文档中心的具体文件版本 ID（文件是外部对象）
- 临时计算字段、缓存字段、非持久化投影字段
- 履约、变更模块的私有过程状态（除非已沉淀为稳定摘要）

### 2.3 版本归属

终止善后清单的版本归属按以下原则划分：

| 规则类型 | 定义权归属 | 消费方 | 变更权限 | 版本化方式 |
| --- | --- | --- | --- | --- |
| 终止善后清单结构 | `contract-lifecycle` 模块 | 终止事项、摘要、时间线、搜索、AI | 模块级变更，需同步更新摘要与时间线消费面 | 随模块版本演进，不单独版本化 |
| 善后事项类型字典 | `contract-lifecycle` 模块 | 终止事项、摘要、通知、AI | 模块级变更，需评估对摘要、通知的连锁影响 | 随模块版本演进，不单独版本化 |
| 善后完成判定规则 | `contract-lifecycle` 模块 | 终止事项、摘要、时间线、结算规则、权限限制 | 模块级变更，需评估对结算、权限的连锁影响 | 随模块版本演进，不单独版本化 |
| 善后责任归属规则 | `contract-lifecycle` 模块 | 终止事项、通知、审计 | 模块级变更，需同步组织架构底座 | 随模块版本演进，不单独版本化 |

**版本归属原则**：

- 终止善后清单属于 `contract-lifecycle` 模块的内部治理资产，不属于 `contract-core` 的合同主档规则，也不属于 `workflow-engine` 的流程规则。
- 终止善后清单变更不需要变更合同主档、文档中心、流程引擎的版本，但可能需要刷新 `cl_lifecycle_summary`、`cl_lifecycle_timeline_event` 的消费面。
- 终止善后清单不对外暴露为独立 API 资源，只通过终止事项、摘要、时间线间接消费。

### 2.4 最小治理单元

终止善后的**最小治理单元**是：

- **善后事项类型**：每个善后事项类型编码是一个最小治理单元，包含类型编码、名称、分类、业务语义、适用场景。
- **善后完成判定规则**：按"终止类型 + 善后事项类型 + 完成条件"组合为最小治理单元，例如"到期终止、文件归档、已完成"的判定规则。
- **善后责任归属规则**：按"善后事项类型 + 责任维度"组合为最小治理单元，例如"结算确认、财务部门归属"规则。

**最小治理单元原则**：

- 不允许将多个善后事项类型合并为一个"通用善后清单"，每个善后事项类型必须有独立的治理条目。
- 不允许将善后完成判定与结算规则合并为同一个治理单元，善后完成判定是事实判断，结算规则是业务规则。
- 不允许将善后责任归属与流程引擎的审批节点责任人混为同一个治理单元，善后责任归属是业务规则，流程节点责任人是流程规则。
- 不允许将终止善后清单与履约节点类型混为同一个治理单元，参见[第一份专项设计](fulfillment-rules-and-risk-scoring-design.md)。
- 不允许将终止善后清单与变更影响评估混为同一个治理单元，参见[第二份专项设计](change-impact-and-write-back-design.md)。

## 3. 结算规则治理设计

### 3.1 治理边界

结算规则的治理边界如下：

| 治理边界内（属于结算规则治理） | 治理边界外（不属于结算规则治理） |
| --- | --- |
| 结算项目类型定义（结算项目编码、名称、分类） | 具体终止记录的结算执行结果 |
| 结算规则定义（结算基准、计算维度、例外条件） | 结算算法的代码实现 |
| 结算与终止状态的映射关系 | 结算的实时计算调度逻辑 |
| 结算结果在摘要、时间线、通知中的呈现约束 | 结算的数据库存储实现 |
| 结算结果对权限限制的影响规则 | 结算的 API 接口设计 |

**治理边界原则**：

- 结算规则治理的是"结算项目、结算规则、映射关系、呈现约束"，不是"怎么算钱"。
- 结算规则不替代终止状态，结算规则是对终止后业务状态的经济性评估，不是终止状态本身。
- 结算规则不拥有合同主档，结算结果只通过 `cl_lifecycle_summary` 和 `cl_lifecycle_timeline_event` 消费。

### 3.2 责任划分

结算规则的责任划分如下：

| 责任类型 | 归属方 | 说明 |
| --- | --- | --- |
| 结算项目定义权 | `contract-lifecycle` 模块 | 定义结算项目类型、名称、分类、业务语义 |
| 结算规则定义权 | `contract-lifecycle` 模块 | 定义结算基准、计算维度、例外条件 |
| 结算执行权 | `contract-lifecycle` 模块内部服务 | 按治理定义的规则执行结算，但不拥有结算算法实现细节的治理权 |
| 结算结果消费权 | 摘要域、时间线域、通知、AI | 消费稳定后的结算结果，不重新计算结算 |
| 结算结果解释权 | `contract-lifecycle` 模块 | 对结算结果的业务含义、变更影响、消费影响拥有解释权 |
| 结算结果回写权 | `LifecycleIntegrationFacade` | 负责将结算结果回写到合同主档摘要（如适用） |
| 结算审计权 | 平台审计中心 | 对结算动作、结果变更进行审计留痕 |

**责任划分原则**：

- `contract-lifecycle` 模块拥有结算规则的定义权和解释权，但结算结果通过 `LifecycleIntegrationFacade` 回写合同主档（如适用），不直接修改合同主档核心字段。
- 结算规则不拥有流程引擎的审批结论，流程审批结果是终止的前置条件，不是结算规则的直接治理对象。
- 结算规则不拥有文档中心的文件对象，文件引用只作为结算的间接引用，不作为结算规则的直接治理对象。

### 3.3 与前两份专项设计的关系

结算规则与履约规则、变更影响的治理关系：

- 履约规则与风险评分是结算规则的**间接输入治理约束之一**：履约状态、风险等级可以作为结算规则的补充输入维度，例如逾期履约可能影响结算扣款。
- 变更影响评估是结算规则的**间接输入治理约束之一**：变更结果可能影响结算基准，例如变更后的金额是结算的重要输入。
- 结算规则不修改履约规则与风险评分：结算规则只能读取履约和风险评分的稳定输出，不能反向修改履约节点类型、逾期判定规则、风险评分维度。
- 结算规则不修改变更影响评估：结算规则只能读取变更影响评估的稳定输出，不能反向修改变更影响范围、影响等级。

## 4. 终止后权限限制治理设计

### 4.1 治理边界

终止后权限限制的治理边界如下：

| 治理边界内（属于权限限制治理） | 治理边界外（不属于权限限制治理） |
| --- | --- |
| 权限限制项目定义（限制项目编码、名称、分类） | 具体合同终止后的权限限制执行结果 |
| 权限限制触发规则（什么条件下触发什么限制） | 权限检查的代码实现 |
| 权限限制与终止状态的映射关系 | 权限系统的实时检查调度逻辑 |
| 权限限制结果在摘要、时间线、通知中的呈现约束 | 权限限制的数据库存储实现 |
| 权限限制对结算、归档的影响规则 | 权限限制的 API 接口设计 |

**治理边界原则**：

- 权限限制治理的是"限制项目、触发规则、映射关系、呈现约束"，不是"怎么检查权限"。
- 权限限制不替代终止状态，权限限制是对终止后合同访问能力的约束，不是终止状态本身。
- 权限限制不拥有认证系统，权限限制规则只通过 `contract-core` 和平台权限底座间接执行。

### 4.2 责任划分

权限限制的责任划分如下：

| 责任类型 | 归属方 | 说明 |
| --- | --- | --- |
| 权限限制项目定义权 | `contract-lifecycle` 模块 | 定义权限限制项目类型、名称、分类、业务语义 |
| 权限限制触发规则定义权 | `contract-lifecycle` 模块 | 定义什么条件下触发什么权限限制 |
| 权限限制执行权 | `contract-core` 模块 + 平台权限底座 | 按治理定义的规则执行权限检查，但 `contract-lifecycle` 不拥有权限检查实现细节的治理权 |
| 权限限制消费权 | 合同详情、搜索、AI、通知 | 消费稳定后的权限限制结果，不重新检查权限 |
| 权限限制解释权 | `contract-lifecycle` 模块 | 对权限限制的业务含义、变更影响、消费影响拥有解释权 |
| 权限限制回写权 | `LifecycleIntegrationFacade` | 负责将权限限制结果回写到合同主档或权限系统 |
| 权限限制审计权 | 平台审计中心 | 对权限限制动作、结果变更进行审计留痕 |

**责任划分原则**：

- `contract-lifecycle` 模块拥有权限限制的定义权和解释权，但权限限制的执行权归属 `contract-core` 模块和平台权限底座，因为权限限制最终体现在合同访问能力的约束上。
- 权限限制不拥有流程引擎的审批结论，流程审批结果是终止的前置条件，不是权限限制的直接治理对象。
- 权限限制不拥有文档中心的文件对象，但文件访问限制是权限限制的重要组成部分，文件访问限制的执行权归属文档中心和权限底座。

### 4.3 与前两份专项设计的关系

权限限制与履约规则、变更影响的治理关系：

- 履约规则与风险评分是权限限制的**间接输入治理约束之一**：履约状态可能影响终止后的权限限制，例如严重履约风险可能导致更严格的权限限制。
- 变更影响评估是权限限制的**间接输入治理约束之一**：变更结果可能影响权限限制，例如重大变更可能导致终止后不同的权限限制策略。
- 权限限制不修改履约规则与风险评分：权限限制只能读取履约和风险评分的稳定输出，不能反向修改履约节点类型、逾期判定规则、风险评分维度。
- 权限限制不修改变更影响评估：权限限制只能读取变更影响评估的稳定输出，不能反向修改变更影响范围、影响等级。

## 5. 与周边模块的边界说明

### 5.1 与 `contract-core` 的边界

| 边界维度 | `contract-core` 负责 | `contract-lifecycle` 终止善后与访问控制负责 |
| --- | --- | --- |
| 合同主档 | 持有 `contract_id`、合同身份、生命周期主状态、主状态流转规则 | 不持有合同主档，只通过 `contract_id` 引用合同 |
| 终止状态回写 | 提供受控回写接口，接收终止摘要、终止时间、善后状态、权限限制摘要 | 通过 `LifecycleIntegrationFacade` 调用回写接口，回写终止结果、善后状态、权限限制摘要 |
| 权限限制执行 | 持有合同访问权限、数据权限规则，执行权限限制 | 定义权限限制项目、触发规则，但不执行权限检查本身 |
| 生命周期映射 | 持有生命周期主状态与周边状态的映射规则 | 终止善后状态不与生命周期主状态映射规则混淆，终止善后是周边状态 |
| 结算结果呈现 | 合同主档可以持有结算结果摘要引用（如适用） | 结算规则不直写合同主档核心字段，只通过摘要回写间接呈现 |

**边界原则**：终止善后与访问控制不拥有合同主档，不修改合同主档的身份、分类、主状态。`contract-core` 不拥有终止善后清单、结算规则、权限限制规则，这些属于 `contract-lifecycle` 的治理资产。

### 5.2 与 `document-center` 的边界

| 边界维度 | `document-center` 负责 | `contract-lifecycle` 终止善后与访问控制负责 |
| --- | --- | --- |
| 文件对象 | 持有文件对象主记录、版本链、存储定位、访问权限 | 不持有文件对象，只通过 `cl_lifecycle_document_ref` 引用文件 |
| 善后文件清单 | 文件对象可以表达善后需要的文件材料 | 终止善后清单不包含文件格式规则、文件内容判定规则，文件只作为善后事项的引用 |
| 文件访问限制 | 文档中心持有文件访问权限，执行文件访问限制 | 权限限制可以包含文件访问限制规则，但执行权归属文档中心 |
| 结算依据 | 文件内容可以作为结算依据之一 | 结算规则不将"文件内容审查"作为直接结算维度，文件内容审查属于 AI 辅助能力，不是终止善后治理的一部分 |

**边界原则**：终止善后与访问控制不拥有文件对象，不定义文件格式、文件内容、文件版本规则。文件引用只作为终止善后事项的业务语义附件，不作为终止善后或结算规则的直接治理对象。

### 5.3 与 `workflow-engine` 的边界

| 边界维度 | `workflow-engine` 负责 | `contract-lifecycle` 终止善后与访问控制负责 |
| --- | --- | --- |
| 流程定义 | 持有流程定义、节点定义、审批结论 | 不持有流程定义，只通过 `cl_lifecycle_process_ref` 引用流程实例 |
| 终止审批 | 终止审批的流程实例 | 终止善后清单不包含"审批节点类型"，审批流程是外部对象，不是善后清单的直接输入 |
| 善后完成判定 | 流程审批结论可以作为善后的输入 | 善后完成判定不引用流程节点状态、流程任务状态，只引用终止完成后的稳定结果 |
| 权限限制触发 | 流程审批结论可以触发权限限制 | 权限限制不引用流程节点状态、流程任务状态，只引用终止状态变更后的稳定结果 |

**边界原则**：终止善后与访问控制不拥有流程引擎，不引用流程节点 ID、任务 ID、流程定义 ID 作为规则输入。流程审批结果是终止的前置条件，但不是终止善后清单或结算规则的稳定锚点。

## 6. 治理真理与稳定性原则

### 6.1 治理真理

终止善后与访问控制的治理真理：

1. **终止善后清单是 `contract-lifecycle` 的内部治理资产**，不属于合同主档、不属于流程引擎、不属于文档中心。
2. **结算规则是对终止后业务状态的经济性评估**，不替代终止状态，不拥有合同主档。
3. **权限限制是对终止后合同访问能力的约束**，执行权归属 `contract-core` 和平台权限底座，定义权归属 `contract-lifecycle`。
4. **稳定锚点只引用 `contract_id`、终止 ID、终止类型、终止状态、善后状态**，不引用流程实例或文件版本。
5. **最小治理单元必须独立、不可合并**，善后事项类型、结算项目、权限限制项目必须分治。
6. **终止善后、结算、权限限制变更必须评估消费面影响**，摘要、时间线、通知、AI、合同主档回写的连锁影响必须同步评估。
7. **终止善后与访问控制不拥有合同主档、不拥有流程引擎、不拥有文件对象**，只通过稳定引用和受控接口与周边模块协作。

### 6.2 稳定性原则

终止善后与访问控制的稳定性原则：

- 善后事项类型编码一旦发布，不得随意删除或重用，只能新增或标记为废弃。
- 结算项目编码一旦发布，不得随意删除或重用，只能新增或标记为废弃。
- 权限限制项目编码一旦发布，不得随意删除或重用，只能新增或标记为废弃。
- 善后完成判定规则的例外条件变更，必须评估对已存在终止记录的影响。
- 结算规则的基准、计算维度变更，必须评估对已存在终止记录的结算结果影响。
- 权限限制触发规则变更，必须评估对已终止合同的访问能力影响。
- 终止善后与访问控制的变更，必须同步更新 `cl_lifecycle_summary`、`cl_lifecycle_timeline_event` 的消费面约束，必要时触发合同主档的补偿回写。

## 7. 本文结论

终止善后与访问控制的治理设计已明确为：

- 终止善后清单治理对象是善后清单结构、善后事项类型字典、善后完成判定规则、善后责任归属规则，最小治理单元按善后事项类型和规则类型分治。
- 终止善后的稳定锚点是 `contract_id`、终止 ID、终止类型、终止状态、善后状态，不引用流程实例或文件版本。
- 终止善后版本归属 `contract-lifecycle` 模块，随模块版本演进，不单独版本化，不对外暴露为独立 API 资源。
- 结算规则治理边界是结算项目、结算规则、映射关系、呈现约束，不治理结算算法实现。
- 结算规则责任划分明确：定义权和解释权归 `contract-lifecycle` 模块，执行权归模块内部服务，消费权归摘要、时间线、通知、AI，回写权归 `LifecycleIntegrationFacade`。
- 权限限制治理边界是限制项目、触发规则、映射关系、呈现约束，不治理权限检查实现。
- 权限限制责任划分明确：定义权和解释权归 `contract-lifecycle` 模块，执行权归 `contract-core` 模块和平台权限底座，消费权归合同详情、搜索、AI、通知，回写权归 `LifecycleIntegrationFacade`。
- 与前两份专项设计的关系：履约规则、变更影响是结算规则和权限限制的间接输入，但结算和权限限制不修改履约规则和变更影响。
- 与 `contract-core` 的边界：终止善后与访问控制不拥有合同主档，只通过 `LifecycleIntegrationFacade` 回写终止结果、善后状态、结算结果、权限限制摘要。
- 与 `document-center` 的边界：终止善后与访问控制不拥有文件对象，文件只作为善后事项的业务语义附件，文件访问限制的执行权归属文档中心。
- 与 `workflow-engine` 的边界：终止善后与访问控制不引用流程实例，审批结果是终止的前置条件但不是稳定锚点。
