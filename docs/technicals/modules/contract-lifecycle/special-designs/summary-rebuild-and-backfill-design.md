## 2. 摘要重建治理设计

### 2.1 治理对象

摘要重建治理的**直接对象**是：

| 治理对象 | 说明 | 不治理内容 |
| --- | --- | --- |
| 生命周期摘要兼容口径 | `cl_lifecycle_summary` 各摘要槽位的稳定语义、兼容边界、升级约束 | 摘要表 DDL、字段落库实现 |
| 摘要重建范围定义 | 哪些合同、哪些阶段、哪些摘要槽位允许进入重建 | 重建任务的脚本参数实现 |
| 摘要来源映射关系 | 履约、变更、终止、归档、时间线到摘要的稳定投影关系 | 来源数据抓取代码、聚合算法代码 |
| 差异分类口径 | “缺失、过期、不兼容、需人工确认”四类差异的治理定义 | 具体差异比对程序 |
| 回填验收口径 | 回填后如何判定摘要可用、可消费、可继续联调 | 联调脚本、验收脚本实现 |

摘要重建治理的**间接约束对象**是：

- `cl_lifecycle_summary` 中履约、变更、终止、归档、风险、待办、里程碑等摘要槽位的稳定语义
- `cl_lifecycle_timeline_event` 作为摘要重建的稳定事件投影来源之一的使用边界
- `contract-core` 合同主档消费的生命周期摘要引用与回写节奏
- 搜索、AI、审计、通知对生命周期摘要稳定消费面的依赖约束

### 2.2 稳定锚点

摘要重建的**稳定锚点**选择如下：

| 稳定锚点 | 说明 | 选择原因 |
| --- | --- | --- |
| `contract_id` | 生命周期摘要归属的根合同实体 | 所有后续业务都围绕同一 `contract_id` 运行，重建必须回到同一合同主链 |
| `source_resource_type + source_resource_id` | 摘要来源记录的稳定业务身份 | 摘要只能从正式来源资源投影，不能从临时查询结果反推 |
| `summary_version` | 当前摘要结构与消费面的稳定兼容版本标识 | 兼容治理必须区分“旧摘要仍可读”与“新摘要需重建”，且该标识不能被单次写回序号污染 |
| `current_stage` / `stage_status` | 生命周期当前阶段与状态投影 | 外部消费面首先依赖阶段与状态，重建必须保持阶段语义稳定 |
| `milestone_code + occurred_at` | 里程碑级事件的稳定摘要锚点 | 里程碑是摘要、时间线、通知的共同稳定语义，不应由脚本自行发明 |

**稳定锚点原则**：摘要重建只能引用上述稳定锚点作为输入，不得引用以下不稳定锚点：

- 流程引擎内部节点 ID、任务 ID、临时审批路径
- 文档中心文件内容快照、临时导出结果、非正式下载副本
- 搜索索引、通知投递记录、AI 推断结果等下游投影数据
- 缓存字段、运行时拼装字段、一次性报表字段

### 2.3 版本归属

摘要版本兼容的归属按以下原则划分：

这里先明确唯一语义：`summary_version` 只表示**生命周期摘要结构与消费契约版本**，用于回答“当前摘要是否仍兼容既有消费面、是否需要按新结构重建”。

如需表达“第几次正式写回”或“哪一次摘要快照更新”，必须使用独立的**写回序号 / 快照序号**概念；它服务于去重、幂等、补偿和先后判断，不得再复用 `summary_version`。

| 治理项 | 定义权归属 | 消费方 | 变更权限 | 版本化方式 |
| --- | --- | --- | --- | --- |
| 生命周期摘要结构版本 | `contract-lifecycle` 模块 | 合同详情、搜索、AI、审计、通知、`contract-core` 摘要回写 | 模块级变更，需评估全部消费面 | 随模块兼容口径演进，使用稳定 `summary_version` 标识 |
| 生命周期摘要写回序号 | 生命周期摘要当前写入链路 | 去重、消费幂等、补偿放行、主档回写先后判断 | 每次正式写回自然推进，不承载结构兼容判断 | 使用独立写回序号 / 快照序号表达，不与 `summary_version` 混用 |
| 来源映射兼容口径 | `contract-lifecycle` 模块 | 摘要重建、回填验收、跨模块联调 | 模块级变更，需评估前五份专项设计输出影响 | 随模块版本演进，不单独对外版本化 |
| 回填批次归属 | 本模块当前治理批次 | 数据修复任务、验收、审计 | 仅在确认范围与验收口径后变更 | 以批次治理记录归档，不形成独立对外协议版本 |
| 差异分类口径 | `contract-lifecycle` 模块 | 执行方、验收方、审计方 | 模块级变更，需同步验收口径 | 随模块治理口径演进 |

**版本归属原则**：

- 生命周期摘要兼容规则属于 `contract-lifecycle` 的内部治理资产，不属于 `contract-core` 的合同主档版本规则，也不属于 `document-center` 的文档版本规则。
- `summary_version` 只回答结构兼容与消费契约问题，不回答“这是不是最新一次写回”。
- 写回序号只回答先后次序、幂等与补偿问题，不得被消费方解释为结构升级信号。
- 摘要版本升级可以要求重建摘要，但不能倒逼重写来源事实；来源事实仍以履约、变更、终止、归档正式记录为准。
- 回填批次是治理归档单元，不等于独立业务版本；其职责是记录一次受控修复的边界，而不是创造新的业务真相。

### 2.4 最小治理单元

摘要重建的**最小治理单元**是：**单合同、单摘要槽位、单摘要版本**。

展开约束如下：

- 单合同：任何重建都必须明确落在某一个 `contract_id` 上，不允许以“整表一起修”替代合同级归属。
- 单摘要槽位：履约、变更、终止、归档、风险、待办、里程碑等摘要块分治，不允许把所有摘要变化混成一个不可解释的大修复。
- 单摘要版本：每一次重建都必须明确对应的 `summary_version`，避免旧摘要、新摘要、过渡摘要混写；如需记录同版本内的多次写回，另用写回序号表达。

**最小治理单元原则**：

- 不允许把“单合同多摘要槽位”默认提升为最小治理单元，除非这些槽位共享同一来源变更且验收口径一致。
- 不允许把“单批次”视为最小治理单元；批次只是组织方式，不是治理颗粒度。
- 不允许把下游索引刷新、通知补发、AI 重跑视为摘要重建单元，它们是消费补偿，不是摘要真相修复本身。

## 3. 回填流程治理设计

### 3.1 治理边界

回填流程的治理边界如下：

| 治理边界内（属于回填治理） | 治理边界外（不属于回填治理） |
| --- | --- |
| 回填范围声明：哪些合同、哪些摘要槽位、哪些版本差异进入本次回填 | 具体脚本参数、脚本运行命令 |
| 来源事实校验：确认回填依赖的正式来源是否完整、可解释、可追溯 | 来源事实的补录实现、外部系统补数实现 |
| 差异分类：区分可自动重建、可自动回填、需人工确认、禁止自动处理 | 差异比对程序、SQL 修复细节 |
| 回填准入与冻结：回填期间哪些写操作需要冻结或受控串行 | 平台发布流程、运维窗口编排 |
| 回填验收：摘要一致性、版本一致性、消费面一致性是否达标 | 具体验收脚本、报表实现 |

**回填边界原则**：

- 回填治理修复的是摘要投影与稳定消费面，不修复来源事实本身。
- 如果来源事实缺失、冲突或被人工改写，回填必须停在人工确认，不允许脚本凭推断补造业务事实。
- 回填可以刷新摘要引用、阶段投影、里程碑快照和受控消费面，但不能绕过 `contract-core`、`document-center`、`workflow-engine` 的正式边界去篡改它们的主数据。

### 3.2 责任划分

回填流程的责任划分如下：

| 责任类型 | 归属方 | 说明 |
| --- | --- | --- |
| 摘要兼容口径定义权 | `contract-lifecycle` 模块 | 定义 `summary_version`、来源映射、差异分类和可回填范围 |
| 写回序号治理口径 | 生命周期摘要写入链路 | 定义正式写回的先后判断、幂等和补偿边界，但不定义结构兼容 |
| 来源事实提供权 | `contract-core`、`document-center`、`workflow-engine` 及本模块各子域 | 提供各自边界内的正式事实，不对外转移真相归属 |
| 回填执行权 | `contract-lifecycle` 模块受控执行单元 | 按既定口径执行重建与回填，但不拥有改写周边模块真相的权限 |
| 回填准入批准权 | 模块治理负责人 + 相关边界模块确认 | 当回填触及跨模块消费面时，需共同确认边界与风险 |
| 回填验收权 | `contract-lifecycle` 模块 + 下游消费方代表 | 验收摘要可读性、消费一致性、跨模块引用稳定性 |
| 审计留痕权 | 平台审计中心 | 记录范围声明、执行批次、差异结论、人工确认和最终结果 |

**责任划分原则**：

- `contract-lifecycle` 拥有摘要兼容和回填口径的定义权，但不拥有跨模块来源事实的解释权。
- `contract-core`、`document-center`、`workflow-engine` 只对各自正式事实负责，不为生命周期摘要投影错误背书。
- 下游消费方可以参与验收，但不能反向定义摘要真相；搜索、AI、通知、审计仍然只是消费方。

### 3.3 治理阶段

回填流程在治理层应固定为以下五个阶段：

1. 范围冻结：明确回填合同范围、摘要槽位范围、版本差异范围，并锁定验收口径。
2. 来源核验：确认履约、变更、终止、归档、时间线以及周边引用的正式来源完整可追溯。
3. 摘要重建：按稳定锚点重建目标摘要，不直接覆盖未确认的来源冲突。
4. 差异处置：把差异分为可自动回填、需人工确认、禁止自动处理三类，禁止一刀切覆盖。
5. 验收放行：确认 `summary_version`、写回序号推进关系、阶段投影、里程碑快照、合同主档引用和下游消费面一致后再放行。

这些阶段是治理顺序，不等于实现流水线，也不等于运维操作手册。

## 5. 与周边模块的边界说明

### 5.1 与 `contract-core` 的边界

| 边界维度 | `contract-core` 负责 | `contract-lifecycle` 摘要重建与回填负责 |
| --- | --- | --- |
| 合同主档真相 | 持有 `contract_id`、合同主状态、合同本体字段、主时间线入口 | 不持有合同主档，只围绕 `contract_id` 重建生命周期摘要投影 |
| 摘要回写入口 | 提供受控摘要回写与主档挂接边界 | 只在摘要通过验收后按受控边界回写，不直接改写合同本体事实 |
| 状态冲突解释 | 解释合同主状态为何成立 | 解释生命周期摘要为何需要重建，但不能覆盖主状态真相 |

**边界原则**：摘要回填可以修复合同主档消费的生命周期摘要引用，但不能把 `contract-core` 降级为摘要镜像存储。

### 5.2 与 `document-center` 的边界

| 边界维度 | `document-center` 负责 | `contract-lifecycle` 摘要重建与回填负责 |
| --- | --- | --- |
| 文件对象与版本链 | 持有文档主记录、版本链、存储定位、受控访问能力 | 只消费正式文档引用和稳定文件语义，不重建文件版本链 |
| 归档封包与正式材料引用 | 保障引用对象存在且可追溯 | 只把正式引用投影到生命周期摘要，不补造缺失文件 |
| 文件级修复 | 处理文件缺失、版本异常、引用失效等文件真相问题 | 遇到文件真相缺口时停止自动回填，转人工确认或上游修复 |

**边界原则**：摘要重建可以因文档引用变化而刷新摘要，但不能把文档中心的文件缺失问题伪装成摘要兼容问题处理掉。

### 5.3 与 `workflow-engine` 的边界

| 边界维度 | `workflow-engine` 负责 | `contract-lifecycle` 摘要重建与回填负责 |
| --- | --- | --- |
| 审批运行时真相 | 持有流程定义、实例、任务、审批结论 | 只消费已沉淀的流程结论和稳定用途引用 |
| 流程状态补齐 | 保障流程实例结论可追溯 | 只把正式结论投影进摘要，不重算审批路径 |
| 异常流程解释 | 解释撤回、驳回、重提等流程事实 | 只解释这些事实在摘要中的投影，不反向改写流程实例 |

**边界原则**：摘要回填可以同步流程结论在生命周期摘要中的最终投影，但不能根据摘要倒推或改写流程引擎内部运行时事实。

## 6. 与批次规划承接关系

[`implementation-batch-plan.md`](../../../implementation-batch-plan.md) 已明确：`contract-lifecycle` 属于第三批能力，且进入全面实现前建议先补一轮“接口、状态与挂载点收口”。

本文与该批次规划的承接关系如下：

- 本文属于 `contract-lifecycle` 进入第三批实现前的治理收口内容之一，负责把摘要兼容、重建边界、回填责任和验收口径先固定下来。
- 本文不替代批次顺序判断，不重新定义第三批位置，也不展开排期；批次顺序仍以 `implementation-batch-plan.md` 为唯一正式依据。
- 本文为第三批内部的实现准备提供前置约束：只有当摘要重建与回填边界清晰，`contract-lifecycle` 才能稳定消费第二批沉淀出的合同状态、文档版本链和审批结果。
- 本文同时为第五批综合联调提供一致性前提，确保搜索、AI、审计、通知等消费方看到的是可重建、可验证、可追溯的生命周期摘要，而不是临时拼装结果。

## 7. 治理真理与稳定性原则

### 7.1 治理真理

摘要重建与回填的治理真理：

1. **生命周期摘要是稳定消费面，不是来源事实本身。**
2. **摘要只能从正式来源资源重建，不能从下游投影反推业务真相。**
3. **摘要版本兼容属于 `contract-lifecycle` 的治理责任，不属于周边模块的责任转嫁项。**
4. **回填修复的是投影与引用，不是借机重写历史业务事实。**
5. **最小治理单元必须落到单合同、单摘要槽位、单摘要版本，不能默认扩大到整批整库。**

### 7.2 稳定性原则

摘要重建与回填的稳定性原则：

- `summary_version` 一旦发布，必须保持可解释的兼容关系，不得把不同语义的摘要混用为同一版本。
- 写回序号必须与 `summary_version` 解耦；同一 `summary_version` 下可以存在多次正式写回，但这些写回不能被描述成多个结构版本。
- 摘要槽位的稳定语义一旦发布，不得随意复用旧字段承载新含义；如语义变化，必须通过新版本治理。
- 回填前必须先确认来源事实完整性；来源事实不稳时，回填不得先行。
- 回填后必须验证合同主档引用、时间线里程碑和下游消费面的一致性，不能只验证摘要表本身。
- 任一自动回填都必须留下可追溯的范围声明、差异分类和验收结论。

## 8. 本文结论

摘要重建与回填的治理设计已明确为：

- 摘要重建治理对象是兼容口径、重建范围、来源映射、差异分类和回填验收，不治理脚本实现。
- 稳定锚点是 `contract_id`、来源资源身份、`summary_version`、阶段状态和里程碑事件，不引用流程内部 ID、文件内容快照或下游投影数据。
- 版本归属在 `contract-lifecycle` 模块内部，回填批次只是治理归档单元，不创造新的业务真相版本。
- `summary_version` 已收口为摘要结构与消费契约版本；写回序号另行表达幂等、补偿与先后判断。
- 最小治理单元是单合同、单摘要槽位、单摘要版本，避免一次回填无边界扩散。
- 回填流程治理边界与责任划分已经明确：定义权归 `contract-lifecycle`，来源事实归各边界模块，验收和审计独立受控。
- 与前五份专项设计的关系已经收口为“消费其稳定输出，不重定义其治理对象”。
- 与 `contract-core`、`document-center`、`workflow-engine` 的边界已经明确：摘要可重建，但周边模块正式真相不可被摘要回填替代。
- 与 `implementation-batch-plan.md` 的关系已经明确：本文是 `contract-lifecycle` 在第三批全面实现前的一项治理收口，而不是新的排期文档。
