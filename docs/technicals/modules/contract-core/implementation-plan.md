## 2. 计划边界与原则

- 合同主档是业务真相源，本计划按“先让合同主档成立，再让周边能力围绕
  `contract_id` 挂接”的顺序推进，不允许周边模块各自长出合同真相源。
- 文档中心是文件真相源，合同管理本体只承接正文、附件、签章稿、归档稿等
  文件引用与摘要回写，不复制文件版本链真相。
- 流程引擎通过 `contract_id` 绑定合同，不拥有合同主档；默认主审批路径仍优先
  走 `OA`，但平台流程引擎是一期开箱可用的正式能力。
- 模板库 / 条款库是正式能力资源，不按“资料先挂上、后续再治理”的方式推进；
  条款库从第一版开始就按后续合同 AI 的重要底座建设。
- 多语言是正式能力，不是可选备注；实施顺序中必须把主档、模板、条款、台账
  展示、搜索输入和 AI 输入的多语言落点纳入正式推进范围。
- 周边模块挂接遵循“先主链闭环、再摘要回写、再增强消费”的顺序，避免周边先行
  反向固化合同状态口径。
- 外部联调只写真正依赖外部条件的部分；企业微信需要测试账号，自研加密软件和
  电子签章不依赖外部测试账号。

## 3. 前置条件

### 3.1 合同主档、模板库、条款库、起草入口前置条件

- 合同主档的统一身份、分类主链、生命周期主状态、摘要回写入口和按
  `contract_id` 组织的主链路口径已在
  [`Architecture Design`](./architecture-design.md) 与
  [`Detailed Design`](./detailed-design.md) 中稳定收口。
- 起草入口已明确围绕 `contract_id` 组织，不再走脱离合同主档的独立合同真相。
- 模板库与条款库的资源边界、版本边界、启停边界和起草引用边界已稳定，能够支持
  “基于模板起草”“空白起草”“引用条款起草”三类主入口。
- 模板库与条款库的启用版本、适用范围和治理责任已明确，避免进入开发后仍以
  临时文档或附件代替正式模板 / 条款资源。
- 条款库的稳定编码、版本与适用范围口径已准备完成，确保后续 AI、比对、推荐、
  风险提示都消费正式条款资源，而不是消费散落草稿。

### 3.2 文档中心、流程引擎、签章、归档、搜索、AI 挂接前置条件

- 文档中心至少已具备正式文件写入、主版本引用、稳定读取和摘要回写入口，能够承接
  主正文、附件、签章稿、归档稿的文件真相治理。
- 流程引擎已具备按 `contract_id` 启动流程、生成统一审批摘要和回写合同状态的正式
  入口；`OA` 主路径与平台承接路径的判断口径已统一。
- 签章模块、归档模块、搜索模块和 AI 能力都已明确只消费合同主档摘要、文档中心
  正式版本或正式启用的模板 / 条款版本，不直接定义新的合同一级对象。
- 摘要回写规则已明确：周边模块只回写阶段摘要、关键里程碑、受控引用键和审计信息，
  不直接硬覆盖合同主档状态。
- 搜索和 AI 的挂接已明确只作为消费与增强链路，不反向成为业务真相源。

### 3.3 多语言、审计、权限、主数据前置条件

- 多语言基础能力已可支撑中文、英文、西文的正式编码和展示，不把翻译文本当作
  状态值、分类值或主键值。
- 组织架构主数据已可稳定提供部门、人员和组织规则，并支撑审批节点绑定、数据归属、
  权限判断和通知路由。
- 权限体系已具备合同查看、编辑、起草、模板管理、条款管理、审批发起、签章发起、
  归档发起和高敏动作控制能力。
- 审计中心已具备关键动作留痕能力，至少覆盖建稿、主状态切换、审批回写、签章回写、
  归档回写、搜索 / AI 受控采纳等关键动作。
- 统一任务中心和异常恢复能力已可复用，保证摘要刷新、索引刷新、AI 作业、文档回写、
  回调补偿等动作不需要合同管理本体私建第二套任务体系。

## 4. 总体阶段划分

本模块按五个阶段推进：

1. 合同主档与起草入口基础能力
2. 模板库、条款库与多语言基础能力
3. 合同主链与文档中心 / 流程引擎闭环
4. 周边模块挂接与摘要回写
5. 联调、验收与上线准备

阶段之间遵循“先合同主档、再正式资源、再主链闭环、再周边挂接、最后总体验收”
的顺序。前一阶段的完成标志，是后一阶段进入实现、联调和验收准备的前置门槛。

## 5. 分阶段实施计划

### 5.1 阶段一：合同主档与起草入口基础能力

**目标**

- 先建立合同主档作为业务真相源的最小可运行闭环。
- 先让“合同是谁、如何创建、由谁归属、处于什么阶段”成立，再接模板、条款和周边。

**核心产出**

- 合同主档创建、编辑、查询、台账基础读模型和详情聚合骨架
- 围绕 `contract_id` 的起草入口骨架，支持空白起草和基础建稿确认
- 合同主状态、阶段摘要和统一时间线入口
- 合同主档与组织归属、权限控制、审计留痕的基础闭环

**前置依赖**

- 总平台共享底座、组织权限、审计中心、任务中心已可复用
- 子模块 [`Architecture Design`](./architecture-design.md)、
  [`API Design`](./api-design.md)、[`Detailed Design`](./detailed-design.md)
  已完成主档、起草和摘要边界收口

**完成标志**

- 合同可以在平台内稳定生成唯一 `contract_id` 并形成正式主档
- 合同台账和详情可以围绕同一主档读取，不存在第二份合同业务主记录
- 起草入口已围绕合同主档工作，不再需要脱离主档维护独立合同真相
- 主档创建、编辑、查看和关键状态动作已进入权限与审计链路

### 5.2 阶段二：模板库、条款库与多语言基础能力

**目标**

- 在合同主档已经成立后，再把模板库、条款库和多语言能力补成正式上游资源。
- 让模板 / 条款真正成为可治理、可启用、可被起草和后续 AI 消费的正式能力。

**核心产出**

- 模板库的资源、版本、启停和适用范围基础能力
- 条款库的资源、版本、启停、风险标签和适用范围基础能力
- 模板绑定条款、起草引用模板 / 条款、多语言文本装配的基础闭环
- 主档、模板、条款、台账展示的多语言基础能力

**前置依赖**

- 阶段一已完成合同主档和起草入口基础闭环
- 多语言基础设施已可提供稳定语义编码与展示文本读取能力
- 权限与审计已能区分模板管理、条款管理和普通合同操作

**完成标志**

- 模板库 / 条款库均以正式资源方式可创建、启用、停用和引用
- 起草入口可稳定选择模板、引用条款，并保留来源关系
- 多语言已覆盖主档标题、模板、条款和台账展示，不是后加备注字段
- 条款库已具备后续合同 AI 消费所需的正式资源基础，而不是附件资料区

### 5.3 阶段三：合同主链与文档中心 / 流程引擎闭环

**目标**

- 在主档、模板、条款都成立后，打通合同主链与文档中心、流程引擎的正式闭环。
- 让合同从“可建稿”进入“可写正文、可发审批、可统一回写状态”的业务主链。

**核心产出**

- 主正文、附件与合同主档的稳定绑定关系
- 文档中心回写文件摘要、版本摘要、加密摘要后的合同侧接收能力
- 默认 `OA` 主路径与平台流程引擎正式承接能力下的统一审批发起入口
- 审批摘要、状态映射、时间线回写和异常补偿基础闭环

**前置依赖**

- 阶段一、阶段二已形成稳定的主档、起草、模板、条款与多语言基础
- 文档中心已具备正式文件写入、主版本引用和摘要回写能力
- 流程引擎已具备按 `contract_id` 绑定合同并输出统一审批摘要的能力
- `OA` 主路径与平台承接路径的口径已统一，不要求合同侧适配两套业务语义

**完成标志**

- 合同正文与附件进入文档中心后，合同侧只维护正式引用与摘要，不复制文件真相
- 审批发起和回写均围绕 `contract_id` 完成，流程引擎通过 `contract_id` 绑定合同，
  不拥有合同主档
- 默认主审批路径仍优先走 `OA`，平台流程引擎也已具备一期正式承接能力
- 合同状态推进通过统一映射与摘要回写完成，不存在周边模块直接硬改主档状态

### 5.4 阶段四：周边模块挂接与摘要回写

**目标**

- 在合同主链闭环后，再接入签章、履约、变更、终止、归档、搜索和 AI 等周边能力。
- 让周边模块围绕同一 `contract_id` 消费、回写和增强，而不是各自定义合同主语义。

**核心产出**

- 签章、履约、变更、终止、归档的准入校验与摘要回写能力
- 搜索消费合同摘要、多语言文本和文档中心受控文本的正式入口
- AI 消费合同主档、模板版本、条款版本、文档摘要和详情聚合上下文的正式入口
- 周边模块统一回写里程碑、摘要、引用键和审计信息的规则

**前置依赖**

- 阶段三已形成稳定的主档、文件引用、审批摘要和状态回写闭环
- 签章、履约、归档、搜索、AI 等模块已具备最小可联调入口
- 审计中心、任务中心和权限体系已能承接高敏操作、异步作业和异常补偿

**完成标志**

- 周边模块均围绕同一 `contract_id` 挂接，不生成第二合同真相源
- 合同侧只接收受控摘要、里程碑和稳定引用键，不吸收周边私有过程明细
- 搜索与 AI 已建立正式消费链路，但不会反向改写合同主档真相
- 归档、签章、履约、变更、终止结果都能回写合同主档摘要与时间线

### 5.5 阶段五：联调、验收与上线准备

**目标**

- 在模块能力成型后，收口跨模块联调、验收样例、上线检查与回退准备。
- 把“模块可开发”推进到“模块可交付上线”。

**核心产出**

- 面向文档中心、流程引擎、签章、归档、搜索、AI、组织权限和审计链路的联调清单
- 覆盖主档、模板 / 条款、多语言、审批双路径、摘要回写和周边挂接的验收样例
- 上线前配置检查、数据检查、审计检查、索引与任务检查、回退预案
- 模块运行手册、问题定位入口和异常处理入口

**前置依赖**

- 前四个阶段的核心功能闭环均已形成
- 需要外部条件的联调资源已就绪，尤其是企业微信测试账号与对应环境
- `OA` 联调窗口、文档中心联调窗口和关键周边模块联调窗口已可用

**完成标志**

- 主链场景能够覆盖“起草 -> 正文入库 -> 审批 -> 签章 / 生效 -> 履约 / 变更 /
  终止 / 归档 -> 查询 / 审计回查”的正式验证链路
- 多语言、模板 / 条款、审批双路径、周边摘要回写均完成联调验证
- 上线检查项、回退步骤和问题处置入口已明确
- 模块具备进入正式发布窗口的条件

## 6. 模块推进顺序

本模块按“先主档、再模板 / 条款、再文档与审批闭环、再周边挂接”的顺序推进，
原因如下：

1. 先做合同主档，是因为合同主档是业务真相源。
   如果主档未先成立，模板、审批、签章、归档、搜索、AI 都会各自沉淀业务身份和
   状态口径，最终形成多份合同真相。

2. 再做模板库 / 条款库与起草配套能力，是因为它们是合同创建前的正式能力资源。
   先把资源库治理好，后续起草、审查、推荐和 AI grounding 才有统一输入；否则只会
   让合同起草长期依赖临时模板和散落条款。

3. 然后做文档中心 / 流程引擎闭环，是因为合同主链需要同时绑定文件真相源和审批
   运行时真相。
   在这一阶段收口“文档中心是文件真相源”“流程引擎通过 `contract_id` 绑定合同，
   不拥有合同主档”，才能稳定形成主正文、审批摘要和状态回写的统一闭环。

4. 最后接周边模块，是因为签章、履约、变更、终止、归档、搜索和 AI 都应建立在
   已经稳定的合同主链之上。
   如果这些能力先行，极易把周边过程表、索引或 AI 结果误做成合同一级真相。

这一顺序的核心目的，是先把合同管理本体做成“拥有统一业务真相和统一主链”的正式
能力，再把周边能力逐步挂上去，而不是反过来被周边能力定义主链。

## 7. 外部联调准备

本模块只对真正需要外部条件的部分做联调准备说明，不把平台内子模块能力写成外部账号
依赖。

### 7.1 `OA` 主路径联调准备

- 确认 `OA` 流程模板、发起方式、回调方式、鉴权方式和状态回传窗口已冻结到可联调版本
- 准备至少一条默认走 `OA` 的真实合同审批样例，验证发起、回调、摘要回写和异常补偿
- 准备重复回调、乱序回调、失败重发和人工补发样例，验证合同主档状态不会被异常回调
  破坏

### 7.2 企业微信联调准备

- 准备企业微信测试账号、测试应用和最小联调组织样例
- 准备登录、身份同步、消息触达等与合同待办、通知、审批提醒相关的联调路径

### 7.3 不作为外部测试账号依赖的联调项

- 自研加密软件属于平台内挂接能力，不作为外部测试账号前置条件
- 自研电子签章属于平台内正式能力，不作为外部测试账号前置条件
- 文档中心、流程引擎、搜索、归档、AI 与合同管理本体的联调重点是平台内摘要回写、
  权限链路、任务链路和审计链路，不额外引入外部账号要求

### 7.4 联调前应先准备的内容

- 准备覆盖主档、模板 / 条款、正文、附件、审批、签章、归档、多语言的联调数据集
- 准备合同状态对账清单、摘要对账清单和时间线对账清单，防止周边回写口径分叉
- 准备权限和审计联调用例，覆盖成功、拒绝、越权、重试、失败回查等路径

## 8. 测试、验收与上线准备

### 8.1 测试重点

- 主档测试：创建、编辑、分类、状态推进、时间线记录、权限控制
- 起草测试：空白起草、模板起草、条款引用起草、建稿确认
- 模板 / 条款测试：版本启停、适用范围、多语言展示、来源关系保留
- 主链闭环测试：正文入库、审批发起、审批回写、异常补偿、统一摘要
- 周边挂接测试：签章、履约、变更、终止、归档、搜索、AI 的摘要回写与受控消费
- 多语言测试：主档、模板、条款、台账、详情、搜索和 AI 输入输出口径一致性
- 审计与异常测试：关键动作留痕、回调异常、索引失败、AI 失败、摘要刷新失败、越权拦截

### 8.2 验收重点

- 能证明合同主档是业务真相源，周边模块没有各自长出合同真相源
- 能证明文档中心是文件真相源，合同侧只保存正式引用与摘要
- 能证明流程引擎通过 `contract_id` 绑定合同，不拥有合同主档
- 能证明默认主审批路径仍优先走 `OA`，同时平台流程引擎是一期正式能力
- 能证明模板库 / 条款库是正式能力资源，且条款库已作为后续合同 AI 的重要底座
- 能证明多语言是正式能力，已进入主档、模板、条款、检索和 AI 链路
- 能证明搜索和 AI 只消费与增强，不反向成为合同一级真相源

### 8.3 上线准备

- 配置准备：模板启停状态、条款启用状态、多语言资源、审批承接策略、周边挂接开关
- 数据准备：主数据校验、模板 / 条款初始化数据、联调遗留数据清理、索引初始化
- 运行准备：摘要刷新任务、索引刷新任务、AI 作业、回调补偿、审计落库和告警规则
- 发布准备：灰度顺序、回退条件、回退步骤、上线观察窗口和问题升级路径
- 协同准备：业务方、`OA` 对接方、企业微信对接方、运维和支持人员明确发布窗口与应急路径
