# 合同管理本体专项设计：迁移、压测与切换治理设计

## 1. 文档说明

本文档用于继续下沉 `contract-core` 主线中“合同主链路迁移、负载验证与切换治理”这一能力，重点回答以下问题：

- 历史合同迁移时，哪些对象属于合同主链路正式迁移对象，哪些只属于依赖输入或消费结果
- 历史合同导入、台账重建、摘要回填、索引补建与切换验证，如何按正式批次对象治理，而不是按一次性脚本执行理解
- 性能压测、切换窗口、回滚边界、核查口径与证据留存，如何成为合同域的正式治理约束
- 当迁移、压测、灰度切换与正式切换发生时，合同主档、摘要、投影、搜索、审批桥接和文档绑定各自承诺到哪一层

本文是以下文档的下游专项设计：

- [`contract-core Detailed Design`](../detailed-design.md)

`document-center`、`workflow-engine`、`integration-hub`、`contract-lifecycle` 是本文的强依赖或关键消费方：前者提供文件真相、审批桥接与外围数据承接边界，后者消费迁移后的合同主链路结果并继续回写生命周期摘要。但这些关系都属于跨模块依赖或消费关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不写成接口文档、字段表、脚本参数说明、数据库 `DDL` 或代码实现说明
- 不写成项目排期、上线日历、值班安排、运维操作手册或逐步执行剧本
- 不写文档中心、流程引擎、搜索引擎、签章引擎各自私有迁移细节，只讨论合同域正式承接面

## 2. 设计目标

- 让历史合同迁移、压测、切换不再被理解为临时实施动作，而是成为 `contract-core` 对正式数据真相、读写负载和上线切换负责的治理对象
- 让合同主档、摘要、台账投影、详情聚合、文档绑定、审批摘要、搜索索引之间形成统一迁移口径，而不是各自独立补数据
- 让迁移批次、切换窗口、回滚边界、核查口径和证据留存有稳定对象模型，便于审计、复盘、重放和责任界定
- 让性能验证覆盖正式业务主链路，而不是只跑单点接口压测后就宣称合同域可切换

## 3. 正式治理对象

### 3.1 治理对象分层

合同主链路的迁移、压测、切换要成为正式治理对象，必须先把治理主语固定下来。本文将其收口为五类正式对象：

1. 迁移对象：历史合同及其合同侧正式承接结果。
2. 迁移批次：一组有明确范围、前置条件、核查口径和退出结论的迁移执行单元。
3. 负载验证对象：围绕合同主链路读写、聚合、回写和重建的压测面。
4. 切换窗口对象：从旧承接口径切换到新合同主链路口径的一段受控业务窗口。
5. 治理证据对象：用于证明“迁移结果可信、压测结果可信、切换结果可信”的正式留痕材料。

只有当这五类对象都被正式定义后，迁移、压测、切换才不是脚本集合，而是可审计的治理闭环。

### 3.2 治理目标不是一次性导入

合同域对迁移治理的正式目标不是“把旧数据搬过来”，而是回答以下问题：

- 哪些历史合同已经成为 `cc_contract_master` 可解释的正式合同主档
- 哪些合同已具备可读摘要、可查台账、可看详情、可追溯文档绑定和审批摘要
- 哪些历史对象仍停留在只可追溯、不可切换、不可正式暴露的过渡状态
- 若切换失败，系统会回退到哪一层，哪些结果保留、哪些结果必须废止

## 4. 历史合同迁移对象模型

### 4.1 正式迁移对象范围

合同主链路的正式迁移对象必须只收口到合同域需要承接的正式对象，不把周边模块私有运行态一起吸入。正式迁移对象包括：

- 合同主档：合同编号、名称、分类、归属、主体、金额、期限、主状态、阶段、关键引用键
- 合同主体集合：甲乙方、内部主体、担保方等合同参与方
- 合同摘要：审批、签章、履约、变更、终止、归档、风险、待办、主文档摘要块
- 合同台账投影：列表展示、筛选、排序、统计所需的读模型字段
- 合同文档绑定投影：`document_id + effective_document_version_id + document_role` 的合同侧业务解释
- 合同审批绑定与摘要：当前审批路径、审批引用、终局状态、待办摘要、桥接健康摘要
- 合同状态流转记录：历史迁移过程认定的阶段结论与来源依据

### 4.2 非合同域正式迁移对象

以下内容可以作为依赖输入或核查依据，但不属于合同域正式迁移对象：

- 文档中心完整版本链、对象存储键、预览派生产物
- 流程引擎任务链、节点意见流、`OA` 原始明细
- 履约、变更、终止、归档模块的私有过程表
- 搜索引擎底层分词结果、倒排结构和缓存中间态

这类内容如需迁移，由各自能力域治理；合同域只消费其正式结果面。

### 4.3 历史合同迁移分级

历史合同不应被视为单一大表导入对象，而应按合同主链路完备度分为四级：

1. `MASTER_READY`：主档与主体信息完整，可形成正式 `contract_id`。
2. `LEDGER_READY`：主档之外已具备摘要和台账投影，可进入列表与统计视图。
3. `CHAIN_READY`：已具备文档绑定、审批摘要和关键状态映射，可进入详情主链路。
4. `CUTOVER_READY`：已通过批次核查与负载验证，可进入正式切换口径。

这一分级的作用是把“迁移完成”从单值判断拆成可治理的阶段结论，避免主档刚导入就被误认为可正式切换。

## 5. 迁移批次治理模型

### 5.1 批次必须成为正式对象

迁移不能只以脚本执行批为单位，而要以正式批次对象治理。一个迁移批次至少要固定以下语义：

- 批次标识：唯一 `batch_id`
- 批次范围：合同来源系统、业务域、年份、组织、合同类型或风险等级范围
- 批次前置条件：依赖数据是否就绪、引用系统是否可读、冻结窗口是否满足
- 批次成功标准：主档完整率、摘要完整率、绑定成功率、核查差异阈值、性能阈值。其中特别用于批次放行的性能阈值、切换准入线、回滚准入线，不允许在现场临时拍板，必须继承三层正式来源口径：第一层是需求基线中已承诺的业务容量、时延、稳定性目标；第二层是实施排期基线中已冻结的批次窗口、灰度范围、观察时长与回退时限；第三层是已发布验证结论中由联调、演练、压测专项或上一批次正式复盘确认的验证结论。若三层口径存在差异，以最新已发布且完成审批留痕的正式基线为准，并在批次证据中标明继承版本。
- 批次退出结论：通过、阻塞、回滚、人工收口中

### 5.2 推荐批次切分维度

批次切分应优先按业务风险和切换影响面治理，而不是只按数据量分片。建议优先维度如下：

1. 业务状态分层：草稿 / 审批中 / 已签署生效 / 履约中 / 已归档。
2. 来源系统分层：旧合同台账、旧审批系统导入、纸质备案补录、外围集成同步。
3. 风险等级分层：高金额、高频访问、跨组织、审计敏感合同优先单独治理。
4. 读写活跃度分层：仍在变更中的活跃合同与长期历史归档合同分开处理。

### 5.3 批次状态口径

每个迁移批次应统一使用以下状态解释：

- `PREPARED`：范围已冻结，输入与依赖已校验
- `MIGRATING`：正在导入主档、摘要、投影或绑定结果
- `VERIFYING`：正在做数据核查、链路抽检、性能复核
- `READY_FOR_CUTOVER`：已满足切换前准入
- `CUTOVER_ACTIVE`：正在切换窗口内受控暴露
- `ROLLED_BACK`：已回退到切换前正式口径
- `CLOSED`：治理结论已封板，证据已留存

## 6. 负载验证面设计

### 6.1 压测对象不是单接口，而是主链路能力面

合同主链路的负载验证必须围绕正式业务面，而不是只测试某个查询接口。至少覆盖以下六类验证面：

1. 台账读面：列表查询、筛选、排序、统计、导出入口的稳定性。
2. 详情聚合读面：主档、摘要、文档摘要、审批摘要、生命周期摘要聚合读取。
3. 状态写面：审批回写、文档绑定切换、摘要刷新、状态流转记录写入。
4. 重建写面：台账重建、摘要回填、索引补建、缓存失效后的重新投影。
5. 切换混合面：新旧口径并行期的双读校验、双写比对或受控只读暴露。
6. 恢复面：批次回滚或重放后，主档、摘要、投影重新一致所需的压力承受能力。

### 6.2 负载验证的正式指标口径

每一类验证面都必须至少同时回答四个维度，且所有阈值都应直接引用正式基线，不得由压测现场临时口头调整：

- 吞吐：在既定窗口内可承接多少合同读取、写入或重建任务；正式阈值继承需求基线承诺容量，并由已发布压测结论确认可达。
- 时延：关键查询、关键回写、批量重建达到什么响应时间范围；正式阈值继承需求基线中的时延目标，并以性能专项报告中的结论值作为批次复用口径。
- 稳定性：连续压测期间错误率、超时率、重试率、死锁或补偿比例；正式阈值继承已发布验证与演练结论，不接受现场临时放宽。
- 一致性：高压下主档、摘要、投影、索引之间是否出现可接受阈值外的漂移；正式阈值继承数据核查基线与上一轮正式验证结论。

### 6.3 压测样本必须代表迁移后真实结构

压测不能只使用人工构造的小样本。正式样本必须覆盖：

- 大字段合同、长时间线合同、多附件合同、多主体合同
- 存在审批桥接摘要和文档绑定的完整主链合同
- 已归档历史合同与仍在履约中的活跃合同
- 同组织高并发读热点与跨组织长尾读请求

否则压测只能证明单点程序可运行，不能证明迁移后合同域可切换。

## 7. 切换窗口治理

### 7.1 切换窗口的正式定义

切换窗口不是上线时间点，而是一个有清晰边界的治理对象。它至少包括：

- 切换开始条件：哪些批次达到 `READY_FOR_CUTOVER`。`READY_FOR_CUTOVER` 不是现场口头判断，而是指该批次已同时满足需求基线定义的业务承诺、实施排期基线冻结的切换窗口约束，以及已发布验证结论确认的性能与核查达标结果。
- 切换期间约束：允许哪些读写，禁止哪些高风险变更或补录动作
- 切换观察期：新口径暴露后持续观察的一段受控窗口
- 切换结束条件：核查通过、指标稳定、无未收口阻塞差异

### 7.2 合同域内的切换对象

合同域切换窗口内真正发生切换的是以下解释口径：

- 合同台账默认读取口径从旧台账切换到 `cc_contract_ledger_projection`
- 合同详情默认读取口径从旧聚合口径切换到 `ContractDetailAggregate`
- 合同主状态解释口径切换到 `cc_contract_master + cc_contract_state_transition`
- 合同文档主读版本解释切换到 `cc_contract_document_link`
- 合同审批结果消费口径切换到统一审批摘要与绑定引用

因此，切换窗口治理的是“默认解释权切换”，而不是简单发布脚本完成。

### 7.3 灰度不是父文档关系，而是治理手段

灰度、双读校验、受控组织先行、特定合同类型先切换，都属于切换治理手段。它们可以被使用，但都不改变本文的治理主语：合同域只承认正式批次、正式窗口和正式核查结论，而不承认“某个临时脚本先跑了所以已经切换”。

## 8. 回滚边界设计

### 8.1 回滚对象必须分层

合同主链路切换失败时，回滚边界必须分层定义，不能笼统写成“全部回退”。本文将回滚边界收口为三层：

1. 暴露层回滚：把台账、详情、搜索结果入口退回旧读取口径。
2. 投影层回滚：废止本批次新生成的摘要、台账投影、索引暴露资格。
3. 真相层边界：已落入正式 `cc_contract_master` 的主档事实不做静默删除，只允许按批次状态标记为未切换、未生效或待人工收口。

### 8.2 不可回滚的正式边界

以下内容不应在失败时用“脚本删除”方式伪装回滚成功：

- 已生成并有审计编号的合同主档身份
- 已形成审计留痕的状态流转记录
- 已被外部系统消费的正式绑定引用

这些对象如果需要退出正式暴露，应通过批次失效、引用废止、重新映射或人工收口方式治理，而不是抹去历史。

### 8.3 回滚准入条件

回滚应只在以下情况触发。这里的“允许阈值”“切换准入线”与“短时调参”判定，均必须引用切换前已冻结的正式基线版本；若正式基线未授权更新，则现场不得自行改线：

- 核查差异超过批次允许阈值
- 性能指标未达到切换准入线且短时调参无法消除风险
- 文档绑定、审批摘要、状态映射出现系统性错误，导致合同业务解释不可信
- 观察期内出现跨批次污染，无法保证新旧口径隔离

## 9. 核查口径设计

### 9.1 核查主语必须统一

合同主链路核查的正式主语，不是“表行数对上即可”，而是“同一 `contract_id` 在迁移后是否仍可被正确解释”。因此核查至少包括以下六类口径：

1. 主档核查：合同身份、分类、归属、主体、金额、期限、主状态是否一致。
2. 摘要核查：审批、签章、履约、归档、风险、待办摘要是否一致。
3. 台账核查：列表筛选结果、排序键、统计口径是否一致。
4. 详情核查：文档摘要、审批摘要、生命周期摘要与时间线是否可解释。
5. 引用核查：`document_id`、`effective_document_version_id`、审批引用是否可解引用。
6. 搜索与回放核查：搜索命中后是否能回到同一合同详情，重放后是否仍得到同一业务解释。

### 9.2 抽检与全检边界

不是所有核查都需要全检，但必须明确哪类对象全检、哪类对象抽检：

- 主档唯一键、引用可解引用性、批次数量统计，应全检
- 详情聚合展示、时间线可读性、复杂文档绑定语义，应按风险样本抽检
- 性能类指标应按批次全量采集，不接受只看单次样本

### 9.3 差异处置口径

核查发现差异时，不允许直接把旧系统值覆盖成新系统值或相反。差异必须被归类为：

- 源数据缺失
- 映射规则错误
- 依赖系统引用失效
- 聚合或投影重建错误
- 可接受历史脏数据但需显式豁免

只有显式归类后，批次结论才有审计意义。

## 10. 证据留存与正式收口

### 10.1 证据对象最小集合

若迁移、压测、切换要成为正式治理对象，就必须留下可复核证据。每个批次至少保留以下证据：

- 批次范围冻结记录
- 输入来源与版本快照说明
- 主档 / 摘要 / 投影 / 引用核查结果
- 压测报告摘要与关键指标曲线
- 切换窗口开始、观察、结束结论
- 回滚触发与否的判定记录
- 人工确认、豁免批准与风险接受记录

### 10.2 证据的正式用途

这些证据不是归档附件，而是正式治理用途对象，用于：

- 证明某一批历史合同为何可以进入正式切换
- 解释某次回滚为何触发、回滚到哪一层、哪些事实被保留
- 支撑后续审计、问题复盘、容量评估和再次迁移时的规则复用

同时，证据中必须显式记录本批次性能阈值、切换准入线、回滚准入线分别继承自哪一版需求基线、哪一版实施排期基线、哪一份已发布验证或性能专项结论，确保读者可以直接追溯正式来源，而不是把准入线理解为现场临时决定。

### 10.3 证据留存边界

证据留存应保留结论、口径和引用，不复制周边模块全部原始运行明细。合同域正式保留的是：

- 合同侧核查结果与差异归因
- 批次级性能结果与切换结论
- 对周边正式引用的证据索引

而不是复制全文日志、全量原始审批意见流或文档中心全部对象元数据。

## 11. 本文结论

合同主链路的迁移、压测、切换要成为正式治理对象，核心不在于脚本是否存在，而在于合同域是否建立了稳定的治理主语、批次对象、验证面、切换边界、回滚边界、核查口径和证据闭环。

在这一约束下：

- 历史合同迁移对象被限定为合同域正式承接面，而不是把周边模块私有真相整体吸入
- 迁移批次成为有范围、前置条件、成功标准和退出结论的正式对象，且其中性能阈值、切换准入线、回滚准入线必须继承正式基线版本
- 负载验证围绕合同主链路读写、聚合、重建和恢复面展开，而不是单点接口压测，其达标判定继承需求基线与已发布性能专项结论
- 切换窗口治理的是合同默认解释权从旧口径向新口径的受控切换，其放行判断继承需求基线、实施排期基线与正式验证结论
- 回滚边界、核查口径与证据留存共同保证切换失败时仍可解释、可追责、可再次治理，并能追溯准入线的正式来源

这就是合同主链路迁移、压测、切换从“实施动作”升级为 `contract-core` 正式治理对象的收口方式。
