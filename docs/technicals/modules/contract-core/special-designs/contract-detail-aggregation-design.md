# 合同详情聚合专项设计

## 1. 文档说明

本文档用于收口 `contract-core` 主线中“合同详情聚合”这一能力，重点回答“合同详情聚合如何服务消费面而不变成第二合同真相源”，而不是回答某个详情接口如何拼字段。

本文是以下文档的下游专项设计：

- [`contract-core Detailed Design`](../detailed-design.md)

`document-center`、`workflow-engine`、`contract-lifecycle`、`intelligent-applications`、前端合同工作台是本文的强依赖或关键消费方：前者分别提供文件摘要、审批摘要、生命周期摘要和智能消费上下文，后者消费详情聚合结果；但这些关系都属于跨模块依赖或消费关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不写成对外 `API` 路径、响应字段表、前端组件树或页面交互稿
- 不写成缓存中间件选型、任务脚本、类名、函数签名、数据库 `DDL` 或代码实现说明
- 不写成实施排期、联调计划、负责人拆分、灰度步骤或上线手册
- 不展开文档中心、流程引擎、生命周期模块各自的内部真相模型细节

## 2. 设计目标

- 让合同详情页拥有稳定、可消费、可解释的单合同工作台视图，而不复制合同主档、文件版本链或审批过程链为第二真相源
- 让主档、文档、审批、生命周期、审计摘要围绕同一 `contract_id` 被组织成统一阅读面，但每一块都保持对正式源头的可回指能力
- 让详情聚合在缓存命中、缓存失效、异步回写延迟、下游摘要补偿中都保持“可降级读取、可重放重建、不可越界持有真相”
- 让“详情页所见口径”和“系统正式真相源”明确分离，避免前端工作台展示结果反向固化为业务事实
- 让详情聚合只承接消费所需的受控摘要面，不退化为跨模块过程查询总线

## 3. 真相分层与聚合定位

### 3.1 详情聚合不是合同第二主档

合同详情聚合的正式定位，是单合同消费面的读时装配对象。它回答的是“围绕这份合同，当前工作台需要如何阅读和组织受控事实”，不回答“哪一层数据本身构成正式业务真相”。

因此必须固定以下分层：

1. 合同主档真相：由 `cc_contract_master`、`cc_contract_party`、`cc_contract_summary`、`cc_contract_state_transition` 回答合同是谁、当前阶段是什么、有哪些正式摘要和状态流转。
2. 文件真相：由文档中心与 `cc_contract_document_link` 共同回答文件是谁、当前业务绑定哪个版本、哪些只是文档摘要投影。
3. 审批真相：由流程引擎或 `OA` 桥接治理对象回答审批运行事实；合同域只消费统一审批摘要。
4. 生命周期真相：由签章、履约、变更、终止、归档等模块各自持有过程真相；合同详情只消费当前摘要与关键历史锚点。
5. 详情聚合真相：不存在独立真相。详情聚合只是一层受控阅读面。

### 3.2 详情聚合的唯一职责

详情聚合只做三件事：

- 统一组织：把分散在主档、摘要、文档引用、审批摘要和生命周期摘要中的消费面信息组织成工作台可读结构
- 统一解释：明确每一块信息当前在详情页中以什么口径展示，例如“当前正文摘要”“当前审批摘要”“当前归档摘要”
- 统一回指：每一块展示结果都必须可回指其正式来源对象或来源摘要，而不是停留在详情聚合私有字段中自我闭环

它不承担以下职责：

- 不承接正式编辑真相
- 不承接跨模块过程链全量镜像
- 不承接长期状态存储
- 不承接“页面当前显示内容即正式事实”的倒置语义

## 4. 统一聚合对象模型

### 4.1 聚合对象的块结构

`ContractDetailAggregate` 在语义上应按以下块收口：

- `master_block`：合同主档核心身份、分类、归属、金额、日期、当前主状态与阶段
- `party_block`：主体列表及其展示排序
- `document_block`：当前正文、签章稿、归档稿、补充协议等当前可见文档摘要
- `approval_block`：统一审批摘要、当前审批路径、待办摘要、桥接健康摘要
- `lifecycle_block`：签章、履约、变更、终止、归档等当前阶段摘要
- `audit_block`：面向详情页可读的最近关键审计摘要与恢复 / 异常提示
- `timeline_block`：按业务可读事件组织的时间线片段
- `action_block`：当前允许动作、阻塞原因、下一步建议
- `assistant_block`：提供给 AI、搜索、推荐入口的受控上下文摘要

这些块的作用是让消费面按职责阅读，而不是把所有字段平铺成一个巨型响应对象。

这些块是对父文档 `ContractDetailAggregate` 既有聚合对象的语义分组 / 归并解释，不是第二套并行对象模型。实现层仍应以父文档已有对象口径组织输出；本文使用 block 命名，只是为了把消费面的阅读职责收口得更清楚。

### 4.2 聚合块与父文档对象映射

| 聚合块 | 对应父文档对象 / 摘要 | 映射说明 |
| --- | --- | --- |
| `master_block` | `contract_master` | 直接对应合同主档对象 |
| `party_block` | `party_list` | 直接对应主体列表对象 |
| `document_block` | `document_summary` | 直接对应文档摘要对象 |
| `approval_block` | `approval_summary` | 直接对应统一审批摘要对象 |
| `lifecycle_block` | `signature_summary` + `performance_summary` + `change_summary` + `termination_summary` + `archive_summary` | 对父文档多个生命周期摘要做归并展示，不新增第二套生命周期对象 |
| `audit_block` | `timeline_list` + `action_entry_list` 中与审计、补偿、异常提示直接相关的受控片段 | 属于从既有时间线 / 动作摘要中抽取的可读审计视角，不单独引入新真相对象 |
| `timeline_block` | `timeline_list` | 直接对应时间线对象 |
| `action_block` | `action_entry_list` | 直接对应动作列表对象 |
| `assistant_block` | `ai_panel_summary` | 直接对应 AI 面板摘要对象 |

其中需要特别固定两点：

- `lifecycle_block` 是把父文档中分散的 `signature_summary`、`performance_summary`、`change_summary`、`termination_summary`、`archive_summary` 收口为一个“生命周期阅读块”；它们在来源和回指上仍保持原有摘要边界。
- `audit_block` 是从父文档已有 `timeline_list`、`action_entry_list` 中抽取“详情页可信度判断最需要看到的关键审计 / 恢复 / 异常提示”，不是要求父文档新增独立 `audit_summary` 对象。

### 4.3 每个块只允许承接两类内容

每个聚合块只允许承接两类内容：

- 来源事实的受控摘要
- 来源事实的消费解释元数据

例如：

- `document_block` 可以承接“当前主正文的文档摘要、业务角色、来源事件、默认暴露关系”，但不能复制文档中心完整版本链
- `approval_block` 可以承接“当前审批状态、节点摘要、待办摘要、桥接健康度”，但不能复制任务级意见流全文
- `lifecycle_block` 可以承接“当前签章 / 履约 / 变更 / 终止 / 归档摘要”，但不能吸收各模块过程表

### 4.4 聚合块必须显式携带来源锚点

详情聚合中每个主要块都必须带可解释来源锚点，至少包括以下语义之一：

- `source_type`
- `source_ref_id`
- `source_updated_at`
- `summary_version` 或等价摘要版本标识
- `staleness_hint`

来源锚点的作用不是给页面暴露实现细节，而是保证系统在缓存、重放、审计和异常诊断时，始终知道“这块详情是基于哪一份正式来源拼出来的”。

## 5. 五类聚合边界

### 5.1 主档边界

主档块只消费合同域自身正式真相：

- `cc_contract_master`
- `cc_contract_party`
- `cc_contract_summary` 中与合同本体直接相关的基础摘要

主档块负责回答：

- 这份合同是谁
- 当前在哪个阶段
- 当前归谁负责
- 当前基础展示名词与主状态是什么

主档块不回答：

- 文件版本链细节
- 审批过程细节
- 履约或归档内部步骤细节

### 5.2 文档边界

文档块围绕 `cc_contract_document_link` 组织合同侧对文件真相的业务解释，再消费文档中心提供的受控摘要。

它只负责回答：

- 当前默认阅读的正文是谁
- 当前签章稿、归档稿、补充协议摘要是什么
- 当前这些文档在合同业务上扮演什么角色

它不负责回答：

- 文档中心完整版本链如何展开
- 对象键、预览产物、文本层、冷热迁移位于哪里
- 哪些派生产物仍在后台处理中

因此，详情页所见的“当前正文”是合同业务解释口径，不是文件系统级最新版本口径。

### 5.3 审批边界

审批块只消费合同侧统一审批摘要和审批绑定锚点，不直连流程任务链。

它只负责回答：

- 审批当前走哪条路径
- 当前审批摘要是什么
- 最近结论是什么
- 是否存在桥接补偿或人工介入风险

它不负责回答：

- 当前所有任务节点明细
- 全量审批意见流
- `OA` 原始表单字段与页面状态

### 5.4 生命周期边界

生命周期块只消费签章、履约、变更、终止、归档等模块的当前摘要与关键结果锚点。

它只负责回答：

- 当前签章、履约、变更、终止、归档是否有与合同阅读相关的最新摘要
- 哪些阶段结果会影响合同当前可执行动作
- 哪些异常需要在详情页显式提醒

它不负责承接：

- 履约节点全过程
- 变更审批链全过程
- 档案借阅和归还全过程

### 5.5 审计摘要边界

详情页中的审计块不是审计中心镜像，而是合同工作台可读的关键审计摘要面。

它只应展示：

- 最近关键动作
- 最近恢复或补偿提醒
- 最近权限敏感动作摘要
- 可供人工判断当前页面可信度的受控提示

它不应展示：

- 全量审计事件流
- 原始日志、错误栈、底层 trace 明细

## 6. 详情页所见口径与真相源分离

### 6.1 所见口径是“当前消费解释”，不是“全局唯一事实原文”

详情页总会面临一个问题：页面看到的是“当前最有用的信息”，但系统正式真相可能分布在不同模块、不同摘要刷新节奏和不同恢复边界中。因此，详情聚合必须把“所见口径”定义为一层消费解释，而不是把页面结果误写成新的真相主表。

例如：

- 页面看到“当前审批已通过”，本质上是审批摘要已被合同域承接后的消费解释
- 页面看到“当前正文为签章稿”，本质上是合同域对文档角色的业务解释
- 页面看到“归档处理中”，本质上是生命周期摘要面向消费方的当前结论，而不是归档模块全量过程真相

### 6.2 所见口径必须允许受控陈旧

详情聚合不做强一致锁，它接受受控最终一致。这意味着页面看到的内容在极短时间内允许存在摘要延迟，但必须满足两个约束：

- 不把中间线程态伪装成正式终局事实
- 在任意时刻都能回退到正式源头重新装配

因此，详情页允许出现“摘要更新时间晚于主档更新时间”“某块提示正在刷新中”这类受控陈旧信号，但不允许为了追求页面即时性而让详情聚合自持未确认真相。

### 6.3 页面交互状态不得倒灌为聚合真相

前端工作台可能存在折叠面板、临时筛选、标签切换、卡片展开等交互状态。这些状态只能属于前端会话态，不得回写为 `ContractDetailAggregate` 的正式组成部分。

同理，页面为了渲染方便形成的字段裁剪、分组、懒加载顺序，也只能属于消费层裁剪策略，不能反向定义详情聚合的领域边界。

## 7. 缓存、回放与重建

### 7.1 缓存对象定位

详情聚合缓存只是一层短期读取加速器，缓存内容本质上是“基于某组来源锚点生成的详情阅读结果”。

因此缓存对象必须至少绑定：

- `contract_id`
- `locale_code`
- 关键来源版本戳或摘要版本戳
- 生成时间

缓存对象不应被理解为“详情正式快照主档”。

### 7.2 缓存值应以可替换整块为主

详情聚合缓存不宜把每个微字段拆成大量独立正式缓存真相；更合理的做法是围绕聚合块或整份聚合结果做短期缓存。原因是：

- 详情页核心问题是统一阅读延迟，而不是字段级别强一致
- 块级替换更容易和来源锚点、失效规则、回放重建对齐
- 字段级缓存过细会让缓存本身变成新的耦合源头

### 7.3 回放的对象不是页面结果，而是装配过程

所谓回放，不是把旧的详情页面 JSON 当作正式历史回放材料，而是基于当时可定位的正式来源锚点，重新执行详情装配过程。

因此回放必须遵守：

- 回放输入来自主档、摘要、文档绑定、审批摘要、生命周期摘要、审计摘要的正式记录
- 回放输出是某一时点可解释的详情阅读面
- 回放失败时，应暴露缺失的是哪一个来源块，而不是伪造一个“看起来完整”的详情结果

### 7.4 重建顺序

详情聚合重建顺序应固定为：

1. 校验 `contract_id` 关联的主档和主状态是否可解引用。
2. 重建主档块与主体块。
3. 基于文档绑定重建文档块。
4. 基于审批绑定与审批摘要重建审批块。
5. 基于生命周期各摘要重建生命周期块。
6. 基于关键事件与审计摘要重建时间线块、审计块与动作块。
7. 重新生成整份详情聚合并刷新短期缓存。

该顺序的意义在于保持“先真相、后解释；先主语、后消费块”的稳定装配链。

## 8. 聚合失效设计

### 8.1 失效的正式定义

详情聚合失效，不等于缓存过期本身，而是指“当前详情结果所依赖的来源锚点已变化，原聚合解释不再可靠”。

因此失效来源至少包括：

- 合同主档字段变化
- 合同主体变化
- `cc_contract_summary` 版本变化
- `cc_contract_document_link` 当前有效绑定变化
- 当前审批摘要或审批绑定变化
- 签章、履约、变更、终止、归档任一当前摘要变化
- 影响页面动作可见性的权限或审计提示变化

### 8.2 失效分层

详情聚合失效应分成三层：

- 强失效：合同主档、当前正文绑定、当前审批绑定、主状态等核心锚点变化，必须立即判定旧详情不可继续复用
- 块失效：某一生命周期摘要或审计摘要变化，只需重建对应块并更新整份聚合
- 自然过期：无显式事件，但短 TTL 到点，需要后台或读时重建以控制陈旧窗口

### 8.3 失效优先于 TTL

TTL 只是兜底窗口，不是正确性来源。只要来源锚点变化，详情缓存就应优先失效；不能因为 TTL 未到而继续把旧聚合当作可靠结果。反过来，即便没有事件到达，TTL 到点后也必须允许系统主动重建，以避免长时间悬挂在旧摘要之上。

## 9. 消费方治理边界

### 9.1 前端工作台

前端工作台消费的是详情聚合结果，不拥有详情真相定义权。它可以：

- 对聚合块做展示裁剪
- 按交互需要重排展示顺序
- 为不同角色隐藏或展开部分消费块

但它不可以：

- 以本地状态覆盖正式摘要
- 以页面缓存长期替代服务端聚合缓存
- 把前端临时加工后的字段回写为新的详情正式口径

### 9.2 搜索、消息、AI 等消费方

这些能力可以复用详情聚合的部分块或上下文摘要，但不能把详情聚合当作新的统一事实母本。

具体约束如下：

- 搜索结果卡片优先消费 `ContractSummaryModel`，而不是先依赖详情聚合再裁剪
- AI 助手可消费 `assistant_block` 作为上下文入口，但答案回写仍必须回到正式业务动作
- 消息中心或待办中心只应消费摘要面，不应为了复用而强依赖整份详情聚合

### 9.3 详情聚合不是跨模块查询总线

如果任何消费方开始要求详情聚合提供越来越多周边过程细节，说明边界已经偏移。此时应优先补充该消费方应直连的正式摘要接口，而不是让详情聚合无限长胖成跨模块总线。

## 10. 审计、异常与可信度提示

详情聚合需要显式表达自身可信度，而不是把所有块都渲染成同等新鲜、同等可靠的事实。建议统一使用以下判断语义：

- `READY`：来源锚点齐全，当前块基于最新已承接摘要生成
- `STALE_ALLOWED`：来源块允许短时陈旧，但仍可安全展示
- `REBUILDING`：已检测到来源失效，当前正等待重建
- `SOURCE_MISSING`：某个正式来源暂不可解引用，只能降级展示或提示缺失

该可信度提示服务三个目标：

- 帮助前端工作台做受控降级，而不是误导用户
- 帮助审计与排障快速判断问题是源头缺失还是缓存陈旧
- 避免页面把“暂无最新数据”解释成“没有这类业务事实”

## 11. 本文结论

合同详情聚合的成立前提，不是把所有合同相关信息搬进一个大对象，而是把围绕 `contract_id` 的受控摘要、业务解释和来源锚点组织成单合同消费面。

其稳定边界是：

- 合同主档、文档中心、审批桥接、生命周期模块各自继续持有正式真相
- 详情聚合只承接受控摘要与消费解释，不复制过程真相和版本链真相
- 缓存只负责加速，不负责定真；回放与重建始终基于正式来源重新装配
- 失效由来源锚点变化驱动，`TTL` 只是兜底窗口
- 详情页所见口径必须与正式真相源分离，前端工作台没有定义真相的权力

至此，合同详情聚合既能稳定服务详情页、AI 辅助入口和其他消费面，又不会演化成第二合同真相源。
