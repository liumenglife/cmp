# 合同审批桥接专项设计

## 1. 文档说明

本文档用于收口合同管理本体子模块在审批桥接消费侧的专项设计，重点回答
“合同主档如何消费审批桥接结果，而不把流程引擎或 `OA` 真相吸进合同域”。

本文是以下文档的下游专项设计：

- [`contract-core Detailed Design`](../detailed-design.md)

本文可引用以下强依赖或关键消费方，但它们不是本文父文档关系：

- 合同管理本体子模块 `Architecture Design`：提供合同主档与流程引擎边界前提
- 合同管理本体子模块 `API Design`：提供统一审批摘要与合同侧消费入口口径
- 流程引擎 `OA` 桥接映射与补偿专项设计：提供桥接治理前提与外部摘要承接口径
- 签章、履约、归档、消息、审计：消费合同侧审批摘要、状态结果与异常留痕

本文输出以下内容：

- 合同主档与流程实例 / 审批桥接结果之间的绑定模型
- 审批摘要回写口径、状态推进边界与主审批路径承接关系
- 异常回写、审计留痕、重放与恢复边界
- 合同域如何只消费审批结果摘要，而不复制流程任务链或 `OA` 原生事实

本文不展开以下内容：

- 不写成流程引擎节点 `DSL`、审批任务表或 `OA` 厂商适配说明
- 不写成对外 `API` 文档、回调字段表或接口联调说明
- 不写成实施计划、任务拆分、代码实现说明或运行脚本手册

## 2. 设计目标与边界

本专项设计必须同时满足以下目标：

- 合同主档只持有审批绑定结果、统一审批摘要和状态映射结果
- 流程实例真相、任务真相、`OA` 原生审批明细仍留在流程引擎或 `OA`
- 默认主审批路径优先走 `OA`，但平台承接路径对合同域暴露同一套消费口径
- 审批桥接异常必须可留痕、可补偿、可重放，但不能伪造审批结论
- 合同详情、台账、时间线消费统一审批摘要，不直连流程任务明细

边界约束如下：

- 本文治理的是合同域的消费与承接边界，不是审批流程执行引擎本体
- 合同域不保存流程图、任务链、意见流全文、`OA` 原始表单字段
- 合同域只承接能驱动业务准入、状态推进、工作台展示与审计所需的最小审批结果面

## 3. 合同域消费审批桥接结果的核心原则

### 3.1 消费结果，不消费过程真相

合同域需要知道的是：

- 当前合同是否已进入审批链路
- 本次审批走的是 `OA` 主路径还是 `CMP_WORKFLOW` 承接路径
- 当前审批阶段、最近结论、待办摘要、终局结论与桥接健康度
- 审批结果是否允许合同进入签章、退回修改、终止发起或人工处理

合同域不需要知道的是：

- 节点 `DSL` 如何定义
- 审批任务如何拆分、抢占、转办或会签聚合
- `OA` 原生页面展示了哪些字段和意见流细节

因此，合同域与审批域之间必须通过“稳定绑定键 + 统一审批摘要 + 受控状态映射”交互，而不是通过流程过程表直连。

### 3.2 合同主档是业务真相源，审批桥接是受控输入面

审批桥接结果进入合同域后，只能作为合同主档的受控输入，不能反向成为合同一级真相源。

- 合同主档决定当前业务阶段是否切换
- 审批桥接决定可供消费的审批事实摘要
- 状态映射器决定哪些审批结果可以推动合同主状态变化

这意味着“审批通过”不是直接改写合同全部语义，而是触发合同域按既定映射规则推进到下一业务阶段。

### 3.3 双路径统一消费口径

无论审批实际走 `OA` 还是平台流程引擎，合同域统一只消费以下对象：

- `approval_mode`
- `latest_approval_ref`
- `cc_contract_integration_link` 中的稳定外部 / 平台实例绑定
- `cc_contract_summary.approval_summary_json`
- `cc_contract_state_transition` 中的审批触发记录

这样可以保证合同域看到的是同一种审批消费面，而不是为 `OA` 和平台路径各维护一套字段解释。

## 4. 绑定模型设计

### 4.1 绑定主键与引用锚点

合同侧审批绑定应收口为三层锚点：

- 合同主档锚点：`cc_contract_master.contract_id`
- 主审批引用锚点：`cc_contract_master.latest_approval_ref`
- 外部 / 平台实例锚点：`cc_contract_integration_link(contract_id, target_system, target_resource_id)`

其中：

- `contract_id` 是合同域唯一主语
- `latest_approval_ref` 表示当前生效审批承接对象的稳定引用键
- `cc_contract_integration_link` 负责把合同与 `OA` 或 `CMP_WORKFLOW` 实例做稳定绑定

合同域因此不需要保存流程任务明细表，只需要能稳定找到“这份合同当前绑定的是哪条审批承接记录”。

### 4.2 绑定关系语义

单份合同在同一时刻只允许存在一个当前生效的主审批绑定，但可以保留多次历史审批绑定轨迹。

- 当前生效绑定：服务合同主状态解释、详情摘要和下一步动作判断
- 历史绑定：服务时间线、审计、异常追踪和恢复重放

这意味着：

- 驳回重提、退回修改后再发起，形成新的审批绑定记录
- 旧绑定不删除，只退出当前有效位
- 合同主档始终只指向当前有效审批引用，而不是把历史审批链平铺到主档字段中

### 4.3 合同侧绑定信息的最小集合

合同域最小只承接以下审批绑定信息：

- 审批路径：`approval_mode = OA / CMP_WORKFLOW`
- 当前绑定实例引用：平台实例号或桥接实例引用
- 当前审批阶段摘要：待发起、审批中、通过、驳回、撤销、异常待处理
- 最近审批动作摘要：最近结论、最近处理时间、最近处理人摘要
- 桥接健康摘要：是否补偿中、是否需人工介入、是否存在异常关闭

除此之外的节点明细、意见明细、参与人完整列表，应继续留在流程引擎摘要接口或 `OA` 原生系统中。

## 5. 审批摘要回写设计

### 5.1 回写目标

审批桥接结果进入合同域后，正式业务回写目标只包括三处：

- `cc_contract_master`：更新 `approval_mode`、`latest_approval_ref`、必要阶段字段
- `cc_contract_summary`：更新 `approval_summary_json` 与待办 / 行动摘要
- `cc_contract_state_transition`：记录由审批结果触发的状态推进或拒绝推进结论

其中 `cc_contract_integration_link` 不作为审批结果消费后的业务结果落点，而是承接闭环中的审批绑定锚点：负责把 `latest_approval_ref` 解引用到当前 `OA` 或 `CMP_WORKFLOW` 实例，并为补写、重放、恢复提供稳定实例定位。台账投影与详情聚合继续消费摘要刷新结果，不直接消费桥接回调事件。

### 5.2 摘要口径

合同侧 `approval_summary_json` 应坚持统一摘要口径，至少包含：

- 当前审批路径
- 当前审批状态
- 当前处理节点摘要
- 当前待办人或候选人摘要
- 最近动作结论摘要
- 最终结论摘要
- 桥接治理状态摘要
- 最近成功同步时间

该摘要回答的是“合同现在可如何解释审批状态”，而不是“审批系统内部发生过哪些全部细节”。

### 5.3 回写时序

审批结果回写合同域时应遵守以下顺序：

1. 先确认审批承接事件已被流程引擎或桥接层落库。
2. 再校验或补齐 `cc_contract_integration_link`，并按 `latest_approval_ref` 找到合同当前有效审批绑定。
3. 先更新 `cc_contract_master`，把当前审批消费结果收口到主档。
4. 需要推进主状态时，经状态映射器写入 `cc_contract_state_transition`，记录本次审批触发的状态流转结论。
5. 再刷新 `cc_contract_summary` 中的审批块与行动块。
6. 最后刷新台账投影、详情缓存和时间线摘要。

该顺序确保合同域在已确认承接事实与稳定绑定锚点的前提下，按“主档 -> 状态流转 -> 摘要 -> 投影”收口，不直接吃未落稳的回调线程中间态。

## 6. 状态推进边界设计

### 6.1 审批结果到合同状态的映射边界

审批结果并不等于合同生命周期状态本身，二者之间必须经过合同域映射。

建议边界如下：

- `OA_APPROVING` / `CMP_FLOW_RUNNING`：合同进入 `APPROVING`
- `OA_APPROVED` / `CMP_FLOW_APPROVED`：合同进入 `APPROVED_PENDING_SIGNATURE`
- `OA_REJECTED` / `CMP_FLOW_REJECTED`：合同回到可修改或重新发起审批的业务状态
- `OA_CANCELLED` / `CMP_FLOW_CANCELLED`：合同退出当前审批阶段，但不删除审批历史
- 桥接异常或摘要补偿中：不自动推导为通过 / 驳回，只标记异常治理状态

### 6.2 不允许越界推进的场景

以下场景不得由审批桥接结果直接推动合同状态：

- 仅收到任务级中间动作，但未形成实例级阶段结论
- 仅收到 `OA` 原始事件，平台侧尚未完成归一承接
- 仅完成摘要刷新，但签章、生效等后续前置条件尚未满足
- 仅因补偿任务重跑而重复推进已完成状态

### 6.3 主审批路径与平台承接路径的关系

对合同域而言，`OA` 主审批路径与平台承接路径是“不同来源、同一消费口径”的关系。

- `OA` 是默认主路径，合同域通过桥接摘要消费其结果
- 平台流程引擎是正式承接路径，合同域通过同样的审批摘要消费其结果
- 路径差异只体现在来源字段、桥接健康度和审计证据上，不体现在合同业务字段模型上

因此，合同域既不能把 `OA` 当成黑盒附件，也不能为了统一而复制 `OA` 内部真相。

## 7. 异常回写与审计留痕

### 7.1 异常回写边界

合同域只接收会影响业务判断的异常摘要，不接收审批域全部内部错误栈。

应回写的异常摘要包括：

- 发起后长时间未绑定实例
- 已有终局结论但合同摘要未刷新
- 当前审批绑定失效或引用冲突
- 审批桥接进入人工介入或异常关闭

不应回写的内容包括：

- `OA` 厂商接口字段缺失明细
- 任务执行器并发竞争细节
- 节点级路由求值日志全文

### 7.2 审计留痕对象

合同域应至少对以下审批消费动作保留审计证据：

- 审批绑定建立、切换、失效
- 审批摘要刷新成功或失败
- 审批结果推动合同状态变更的映射决定
- 异常回写、人工确认收口、重放触发

建议留痕字段至少包括：

- `trace_id`
- `contract_id`
- `latest_approval_ref` 或等价审批引用
- `approval_mode`
- `trigger_source`
- `mapping_rule_code`
- 摘要版本或摘要哈希
- 处理结果、处理时间、处理人

### 7.3 时间线留痕原则

合同详情时间线只呈现业务可读事件，例如：

- 已提交审批
- 审批已通过
- 审批已驳回并退回修改
- 审批桥接异常，待人工处理

时间线不直接展开任务级意见流或 `OA` 原始节点日志，避免把流程域真相复制进合同域展示层。

## 8. 重放与恢复边界

### 8.1 可重放对象

合同域允许重放的是“审批结果消费动作”，不是“审批事实本身”。

允许重放的内容：

- 审批摘要重建
- `latest_approval_ref` 补写
- 状态映射重算
- 台账投影、详情缓存、时间线摘要重建

不允许重放的内容：

- 伪造新的审批通过 / 驳回结论
- 代替 `OA` 或流程引擎补写任务动作明细
- 在旧实例仍有效时生成一条并行主审批事实

### 8.2 恢复顺序

合同审批桥接消费面的恢复顺序应固定为：

1. 先校验 `latest_approval_ref` 与 `cc_contract_integration_link` 是否仍能解引用到当前有效审批绑定；缺失时先补齐绑定锚点。
2. 恢复 `cc_contract_master` 中的审批消费字段。
3. 重算合同状态映射与 `cc_contract_state_transition`。
4. 重建 `cc_contract_summary` 中的审批摘要与行动摘要。
5. 重建台账投影、详情聚合缓存与时间线摘要。

这样可以保持“先校验绑定锚点，再按主档 -> 状态流转 -> 摘要 -> 投影恢复”的顺序稳定性。

### 8.3 恢复前提与边界

恢复合同侧审批消费面时，只依赖以下正式来源：

- `cc_contract_master`
- `cc_contract_integration_link`
- `cc_contract_state_transition`
- 流程引擎统一审批摘要接口或桥接正式摘要接口

恢复时不得依赖：

- 缓存中的旧审批卡片
- 前端详情页本地状态
- `OA` 原始页面抓取结果

## 9. 本文结论

合同审批桥接专项设计的核心，不是把审批系统搬进合同域，而是把合同域需要消费的审批结果面稳定收口。

其成立条件是：

- 合同主档只通过稳定引用键绑定当前审批承接对象
- 合同域统一消费审批摘要，而不复制流程任务链和 `OA` 原生真相
- 审批结果推进合同状态时必须经过状态映射器，而不是直接硬覆盖
- 异常回写、审计留痕、重放与恢复都只围绕合同消费面闭环

至此，合同主档既能可靠消费审批桥接结果，又不会把流程引擎或 `OA` 的运行时真相吸进自身边界。
