# 合同管理本体专项设计：合同搜索索引设计

## 1. 文档说明

本文档用于继续下沉 `contract-core` 主线中“合同搜索索引”这一能力，重点回答以下问题：

- 合同搜索如何消费合同主档、合同摘要、台账投影与文档派生结果，而不在搜索侧形成第二合同真相
- 搜索索引对象应围绕哪一个稳定主键组织，哪些字段来自主档，哪些字段来自摘要，哪些字段来自文档派生结果
- 当合同状态、正文绑定、文档文本层、多语言文本或摘要快照变化时，索引应如何保持版本一致性与受控刷新
- 哪些索引内容允许重建，哪些只允许失效后重刷，哪些事实必须回到合同主档或文档中心正式源解释
- 审计、争议调查、历史回放时，搜索链路应能追溯到哪一版合同业务事实与文档派生输入

本文是以下文档的下游专项设计：

- [`contract-core Detailed Design`](../detailed-design.md)

`document-center`、`identity-access`、`intelligent-applications` 以及平台搜索能力是本文的强依赖或关键消费方：前者提供文档文本派生结果与文件真相边界，中间者决定可见性与数据权限，后者消费搜索结果做问答、推荐或分析。但这些关系都属于跨模块依赖或消费关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不写搜索引擎选型、集群拓扑、索引库参数、分词器实现细节或底层检索算法说明
- 不写对外搜索 `API`、请求参数表、响应字段表、排序语法或错误码文档
- 不写实施排期、批量迁移脚本、压测方案、灰度发布步骤或代码实现说明
- 不把本文写成搜索平台通用设计；本文只回答合同域如何向搜索域提供受控输入与一致性边界

## 2. 设计目标与边界

本专项设计必须同时满足以下目标：

- 让搜索索引围绕同一 `contract_id` 组织合同检索对象，而不是围绕索引文档号重新定义合同身份
- 让搜索只消费合同主档、合同摘要、台账投影、多语言文本与文档派生文本，不复制流程、签章、履约或文件版本链真相
- 让主档字段、摘要字段、文档派生字段进入索引时保留清晰来源标签，避免搜索结果倒置成新的业务解释中心
- 让索引刷新、失效、重建、回放都能回指正式源，并按“主档优先、摘要补充、文档派生受控接入”的规则收口
- 让搜索结果服务台账、详情、智能问答和工作台，但最终业务判断仍回到合同主档与各正式摘要接口

边界约束如下：

- 搜索索引是合同域的增强读模型，不是合同业务真相源
- 搜索命中的合同状态、当前正文、审批结果、归档结果都只能解释为“某次索引构建时消费到的正式摘要快照”
- 搜索域不得自行推断合同主状态、不得依据最新文本内容静默覆盖主档分类、不得把文档全文当作合同一级主档
- 搜索域可以保留查询优化字段、召回辅助字段和文本片段，但不能成为审批、签章、变更、归档等业务动作的写入依据

## 3. 合同搜索不形成第二合同真相的核心原则

### 3.1 先有合同真相，再有搜索投影

合同搜索首先是对合同正式业务真相的受控投影，而不是独立知识库。搜索链路中必须坚持以下顺序：

1. 合同主档回答“这份合同是谁、属于谁、当前主状态是什么”。
2. 合同摘要与台账投影回答“当前要给列表与工作台暴露哪些受控摘要”。
3. 文档中心派生文本回答“当前业务有效正文或附件版本里有哪些可检索文本证据”。
4. 搜索索引把前三者拼成检索友好的读模型。

因此，索引文档只负责提升“找得到”，不负责改写“什么是真的”。

### 3.2 搜索消费结果，不消费过程真相

搜索需要消费的是：

- 合同稳定身份与分类字段
- 当前主状态、阶段、归属和列表摘要
- 当前业务有效正文或受控附件的可检索文本
- 多语言标题、别名、主体展示、标签与风险摘要

搜索不需要消费的是：

- 流程实例任务链
- 文档中心完整版本链与对象存储位置
- 履约、变更、终止、归档模块的过程表
- 签章坐标、归档封包成员、`OCR` 明细块的内部结构

因此，搜索域与合同域的关系是“消费受控结果面”，不是“接管原始过程面”。

### 3.3 搜索命中必须回到正式读面

搜索命中后，默认跳转目标只能是合同台账、合同详情或合同摘要视图，不能停留在索引文档自身作为最终解释面。

这意味着：

- 搜索结果卡片展示的状态、金额、相对方、摘要文案，本质上都只是正式源的查询快照
- 一旦用户进入详情页，仍应重新读取合同正式读面，而不是继续依赖搜索结果卡片中的缓存值
- 任何会驱动业务动作的校验，都必须重新读取 `cc_contract_master`、`cc_contract_summary` 或相邻模块正式摘要

## 4. 索引对象模型设计

### 4.1 统一索引主语义

合同搜索索引对象应统一围绕 `contract_id` 建立，不以独立搜索文档号作为业务主语。搜索系统内部可以存在物理索引文档标识，但在合同域语义上，正式主键必须始终回到：

- `contract_id`：合同稳定业务身份
- `locale_code`：索引展示语种或召回语种维度
- `index_scope`：索引暴露范围，如合同主档、正文全文、附件摘要等受控子范围

这意味着同一合同可以存在多语索引投影或多范围索引投影，但它们都属于同一合同的搜索读模型分片，而不是多份合同对象。

### 4.2 最小索引对象语义

合同搜索索引对象在语义上至少应持有以下字段：

- `contract_id`
- `locale_code`
- `index_scope`
- `master_snapshot_version`
- `summary_snapshot_version`
- `document_binding_snapshot_ref`
- `document_text_snapshot_ref`
- `index_schema_version`
- `visibility_scope_ref`
- `indexed_at`
- `exposed_status`

其中：

- `master_snapshot_version` 用于标记本次索引消费的是哪一版合同主档快照语义
- `summary_snapshot_version` 用于标记本次索引消费的是哪一版合同摘要 / 台账快照
- `document_binding_snapshot_ref` 用于标记本次索引采用了哪一组正文 / 附件业务绑定解释
- `document_text_snapshot_ref` 用于标记本次索引采用了哪一版文档派生文本结果
- `visibility_scope_ref` 用于标记本次索引采用了哪一套权限与暴露范围判断结果
- `exposed_status` 只表达该索引投影当前是否允许默认参与检索，不表达合同业务状态本身

### 4.3 索引对象内部块划分

为避免字段混写，索引对象应按四块组织：

- `identity_block`：合同编号、合同名称、分类、归属、相对方等稳定识别字段
- `summary_block`：状态、阶段、风险、待办、归档、签章、审批等受控摘要字段
- `document_block`：正文标题、正文文本片段、关键附件文本摘要、文本来源版本引用
- `governance_block`：索引版本、来源快照、权限范围、刷新时间、审计锚点

这样可以明确区分“哪些字段用于业务识别”“哪些字段用于检索召回”“哪些字段只是治理与回放证据”。

## 5. 三类字段如何进入索引

### 5.1 主档字段进入索引的规则

主档字段只承接稳定身份与业务主分类，不承接过程细节。进入索引的典型字段包括：

- `contract_id`
- `contract_no`
- `contract_name`
- `document_form`
- `business_domain`
- `contract_subcategory`
- `contract_detail_type`
- `owner_org_unit_id`
- `owner_user_id`
- `amount`
- `currency`
- `sign_date`
- `effective_date`
- `expire_date`
- `contract_status`
- `current_stage`

进入规则如下：

- 只消费主档当前正式字段值，不消费主档历史变更链全文
- 可检索值与过滤值都来自主档正式字段，不允许搜索域根据文档全文二次推断新的分类字段
- 主档字段进入索引后只作为检索投影；一旦与详情页展示不一致，应以主档现读结果为准

### 5.2 摘要字段进入索引的规则

摘要字段用于补充列表和结果卡片，但不替代主档业务身份。进入索引的典型字段包括：

- `approval_summary_json` 中的当前审批状态摘要、节点摘要、待办摘要
- `signature_summary_json` 中的签章状态摘要与关键时间点摘要
- `archive_summary_json` 中的归档状态摘要与归档引用摘要
- `risk_summary_json` 中的风险等级、异常计数、关键提示
- `pending_action_json` 中的待办动作标签
- `main_document_summary_json` 中的正文标题、正文类型、当前可读引用摘要
- `counterparty_display`
- `owner_org_unit_name_display`
- `owner_user_name_display`

进入规则如下：

- 摘要字段只能来自 `cc_contract_summary` 或 `cc_contract_ledger_projection` 这类正式消费面
- 搜索可存储受控摘要文本，但不得把这些摘要再扩写成新的业务字段解释
- 摘要字段可用于结果卡片展示、排序、过滤和召回加权，但不能作为合同主状态唯一来源

### 5.3 文档派生字段进入索引的规则

文档派生字段是合同搜索最容易越界的部分，必须明确只消费“当前业务有效绑定 + 文档中心正式派生文本结果”。进入索引的典型字段包括：

- `MAIN_BODY` 对应正文版本的标题、可检索文本层、关键片段摘要
- `SIGNED_COPY`、`CHANGE_BODY`、`ARCHIVE_COPY` 在业务规则允许暴露时的标题与受控文本摘要
- 指定附件角色的标题、文件类型、受控文本摘要
- 多语言正文标题或受控别名
- 文本来源对应的 `document_id`、`document_version_id`、文本结果引用键

进入规则如下：

- 必须先由合同侧确定“当前哪些文档角色、哪些版本具备业务暴露资格”，再由搜索消费其文本结果
- 搜索不得直接以文档中心 `current_version_id` 替代合同业务有效版本判断
- 搜索不得自行读取未被合同绑定解释的临时工作稿、处理中派生稿或未受控附件文本
- 文档派生文本默认只增强召回与片段展示，不反向改写合同标题、类型、状态或当前正文解释

### 5.4 多语言字段进入索引的规则

多语言字段进入索引时，应继续遵守“语义编码不变、展示文本按语种投影”的原则：

- 合同名称、相对方展示、组织展示、摘要标签、多语标题按 `locale_code` 建立投影
- 状态编码、分类编码、风险编码保留稳定值，不因语言不同而复制出多套业务语义
- 同一 `contract_id` 下可存在多语索引文档，但它们共享同一合同真相锚点
- 多语言召回命中后仍需回到正式详情读面重新装配当前语种展示文本

## 6. 刷新顺序与版本一致性

### 6.1 索引一致性必须服从合同主链顺序

合同搜索索引刷新必须遵守父文档已经确定的顺序：先主档，再摘要，再投影，最后搜索。对搜索专项而言，正式刷新顺序应进一步收口为：

1. 合同主档完成正式写入。
2. 合同摘要与台账投影完成刷新。
3. 文档绑定解释更新完成，且当前业务有效版本可解引用。
4. 文档中心对应文本层 / `OCR` / 摘要结果达到可消费状态。
5. 搜索索引消费上述快照并生成新的索引投影。

只要前四步任一尚未稳定，搜索就只能保持旧索引、延迟刷新或显式失效，不能提前推导新结果。

### 6.2 一致性判断锚点

搜索刷新时至少要校验四类锚点是否一致：

- 合同主档版本号或主档快照版本
- 合同摘要版本号或摘要刷新时间
- 文档绑定解释引用，如当前 `MAIN_BODY` / `SIGNED_COPY` 绑定集合
- 文档派生文本引用，如文本层结果哈希、`OCR` 结果引用或文本摘要快照

只有这些锚点共同成立，新的索引投影才可标记为当前可暴露版本。

### 6.3 乱序更新的收口原则

以下乱序场景不得直接覆盖当前索引：

- 新正文绑定尚未完成，但新 `OCR` 文本先回写
- 合同主状态已切换，但摘要刷新尚未完成
- 多语言标题已更新，但正文文本仍对应旧版本
- 历史正文文本重刷完成，但当前业务有效正文未变化

遇到乱序时，搜索域只允许做三件事：

- 保持现有可暴露索引不变
- 标记候选刷新任务待补齐依赖
- 在治理块记录不一致原因，待后续重刷

### 6.4 搜索暴露状态不是合同状态

索引对象的 `exposed_status` 只用于表达：

- `READY`：当前可参与默认检索
- `PENDING`：等待依赖齐备后刷新
- `STALE`：旧索引已退出默认曝光，但仍保留审计 / 回放价值
- `FAILED`：本次刷新失败，待重试或人工处理

该状态绝不能与 `contract_status` 混用。合同是否生效、审批是否通过、是否归档，仍以合同域正式字段为准。

## 7. 重建边界设计

### 7.1 可重建对象

以下对象允许按正式源重新构建：

- 合同搜索索引文档本身
- 多语言索引投影
- 结果卡片片段与高亮缓存
- 索引中的过滤字段、排序字段、召回辅助字段
- 索引治理块中的来源快照摘要

重建前提是：合同主档、合同摘要、文档绑定解释和文档派生文本结果仍可从正式源解引用。

### 7.2 只允许重刷，不负责重造的对象

以下对象不属于合同搜索负责重造的范围：

- 文档中心的正文版本链与文本层结果本体
- 审批、签章、归档、履约等模块的摘要生成逻辑
- 权限系统的数据权限决策真相
- 合同主档与摘要正式字段本身

搜索能做的是等待这些正式输入恢复后再重刷索引，不能为了补齐索引而自行重造业务事实。

### 7.3 历史索引的保留与失效

当合同主状态、当前正文绑定或多语展示文本更新后：

- 旧索引投影可以转为 `STALE`
- 默认检索只暴露当前有效索引投影
- 历史索引只服务审计调查、争议回放、显式版本检索或离线比对

这与“保留历史合同快照”不同。历史索引只是历史查询投影，不是历史合同真相仓库。

### 7.4 全量重建的正式边界

发生全量重建时，搜索链路必须按以下边界执行：

1. 以 `contract_id` 为主键重新枚举合同集合。
2. 重新读取当前主档、当前摘要、当前文档绑定解释与当前可消费文本结果。
3. 重新生成当前默认索引投影。
4. 对无效或无法解引用的旧索引，标记为 `STALE` 或 `FAILED`。

全量重建不是恢复合同历史，也不是重新推导合同状态；它只是重新投影当前正式真相。

## 8. 搜索与合同真相边界

### 8.1 搜索负责“可发现”，合同负责“可解释”

两者边界应固定为：

- 搜索负责让用户按编号、标题、相对方、正文片段、摘要标签、风险标签把合同找出来
- 合同域负责解释这份合同当前是否有效、当前正文是什么、下一步动作是什么

因此，搜索结果可以提示“命中文本来自正文第几段”或“当前风险等级较高”，但不能单独裁定“这份合同已经进入某个业务阶段”。

### 8.2 搜索不能成为业务写入前置真相

任何依赖合同状态、正文版本或审批结论的业务动作，都不能以搜索索引命中结果作为直接前置判断。例如：

- 不能根据搜索结果里的 `contract_status` 决定是否允许发起签章
- 不能根据搜索结果里的正文片段判断某个条款已经正式生效
- 不能根据搜索结果卡片里的归档状态直接决定是否允许借阅

这些动作必须重新读取正式源。

### 8.3 搜索与文档真相的边界

合同搜索消费的是“被合同业务解释后允许暴露的文本结果”，不是“文档中心里一切可见文本”。因此：

- 文档中心负责说明某个文本结果对应哪个 `document_version_id`
- 合同侧负责说明这个 `document_version_id` 当前在业务上是不是正文、签章稿、补充协议或归档稿
- 搜索侧只在两者都成立时，才把文本纳入合同索引

这就是“合同搜索如何消费合同主档和文档派生结果而不形成第二合同真相”的核心收口。

## 9. 审计与回放边界

### 9.1 审计最小单元

合同搜索链路的最小审计单元不是“某次查询命中了什么”，而是“某次索引刷新基于哪一版合同正式事实生成了哪一个可暴露搜索投影”。至少应记录：

- `contract_id`
- `locale_code`
- `index_scope`
- `master_snapshot_version`
- `summary_snapshot_version`
- `document_binding_snapshot_ref`
- `document_text_snapshot_ref`
- `index_schema_version`
- `exposed_status`
- `operator_id` 或 `system_actor`
- `indexed_at`

### 9.2 必须留痕的关键事件

搜索专项至少应覆盖以下审计事件：

- 索引投影首次创建
- 主档或摘要变化触发的增量刷新
- 文档绑定变化触发的文本重刷
- 文本结果失效导致的索引退出默认曝光
- 全量重建、历史回放或人工补刷
- 权限范围变化导致的索引暴露范围调整

### 9.3 回放能力的正式边界

回放时，搜索链路必须能回答：

- 某次搜索命中的合同结果，对应的是哪一版合同主档与摘要快照
- 命中文本来自哪个文档角色、哪个 `document_version_id`、哪一个文本派生结果
- 当时该索引是否为默认暴露版本，还是历史保留版本

但回放边界只到“解释当时索引为何如此”，不负责：

- 回放流程任务运行细节
- 回放签章坐标或档案封包内部结构
- 直接恢复外部搜索集群内部瞬时状态

### 9.4 删除与合规边界

若因权限收紧、合同撤销暴露资格或合规要求需要从默认搜索中移除某些文本，搜索域只应：

- 停止默认暴露对应索引投影
- 保留必要的审计锚点或改为受限回放记录
- 不擅自删除合同主档、文档绑定或摘要正式记录

这样既保证搜索可治理，也不破坏正式真相链。

## 10. 本文结论

合同搜索索引设计的核心，不是把搜索引擎做成合同副本，而是坚持以下四条边界：

1. 索引对象统一围绕 `contract_id` 建立，业务主语始终属于合同主档。
2. 主档字段、摘要字段、文档派生字段进入索引时必须保留来源分层，且先有合同业务解释，后有文本接入。
3. 索引允许重建、失效、回放，但这些动作只重建搜索投影，不重造合同真相、文档真相或周边模块真相。
4. 搜索只负责“可发现”，合同主档与正式摘要接口负责“可解释、可动作、可审计”。

在这一前提下，合同搜索既可以稳定消费合同主档和文档派生结果，支撑多语言检索与结果卡片展示，也不会反向形成第二合同真相源。
