# 文档中心与文档协作子模块专项设计：OCR / 搜索 / 签章 / 归档挂接绑定

## 1. 文档说明

本文档用于继续下沉 `document-center` 主线中“`OCR` / 搜索 / 签章 / 归档挂接绑定”能力，重点回答以下问题：

- 文档中心如何把 `OCR` 结果、搜索索引、签章输入、归档封包统一挂接在同一文件真相源上
- 四类派生结果分别锚定 `DocumentAsset`、`DocumentVersion`、文件主档与版本链的哪一层
- 文档版本变化后，哪些消费面应传播更新、哪些只允许失效等待重建
- 哪些结果可以重建，哪些关系只能失效重挂，哪些边界不能由文档中心替代外部能力域修复
- 整个挂接链路如何形成统一审计留痕，而不把派生结果倒置成新的文件真相源

本文是以下文档的下游专项设计：

- [`document-center Detailed Design`](../detailed-design.md)

`e-signature`、`contract-lifecycle`、`intelligent-applications`、搜索能力与归档能力是本专项设计的强依赖或关键消费方：它们分别消费签章输入引用、归档输入引用、`OCR` / 搜索派生结果或回收写回链路。但这些关系都属于跨模块依赖或消费关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不重写 `OCR` 引擎选型、搜索引擎选型、签章引擎协议或归档系统实现方案
- 不写外部 `API` 路径、请求报文、回调报文与错误码
- 不写坐标计算算法、全文检索分词策略、版式识别模型参数或归档介质格式细节
- 不写实施排期、任务拆分、脚本实现或代码级类图

## 2. 设计目标

- 让 `DocumentVersion` 成为 `OCR`、搜索、签章、归档四类消费面的统一输入锚点
- 保证四类结果都围绕文件真相源派生，不在文档中心之外再长出“正式文件副本主档”
- 让“当前主版本可消费”与“历史版本可追溯”同时成立，而不混淆版本真相与派生结果真相
- 让版本切换、回写结果、失效传播、重建补偿都能以统一绑定模型落地
- 让审计链能解释“谁在什么时候基于哪个版本消费了什么、产出了什么、为何失效或重建”

## 3. 统一挂接原则

### 3.1 文件真相层与派生层分离

文档中心内部必须始终区分两层：

1. 文件真相层：`dc_document_asset` + `dc_document_version`
2. 派生挂接层：围绕指定 `document_version_id` 形成的能力绑定、结果引用、消费快照、回写引用

统一判断规则如下：

- 能被用户当作正式文件继续读取、预览、下载、签署、归档的，必须回到 `DocumentVersion`
- 只用于识别、检索、签署定位、归档打包说明的，属于派生结果或消费快照
- 派生结果允许重建、失效、回放，但无权覆盖 `current_version_id` 指向

### 3.2 绑定锚点统一为版本主键

四类消费面统一以 `document_version_id` 作为一级锚点，不允许直接锚定业务附件表、搜索文档号、签章申请号或归档记录号作为文件真相主键。

同时保留两级引用语义：

- 版本级绑定：回答“这份派生结果对应哪个具体文件内容版本”
- 主档级投影：回答“当前主档对外默认暴露哪一个版本的哪一份派生结果”

因此，主档级永远只做投影和摘要，不持有派生结果真相本体。

### 3.3 `dc_capability_binding` 是统一入口，不是统一明细仓库

`dc_capability_binding` 的职责只到“登记关系、状态、结果引用、校验摘要、最近同步时间”为止。

它承担的是：

- 统一表达版本与能力域之间是否已挂接
- 统一表达当前绑定是否 `READY`、`FAILED`、`STALE`
- 统一表达结果应该去哪个结果对象、索引文档、签章申请或归档封包追溯

状态口径统一如下：

- `PENDING`：等待消费或等待回写
- `READY`：当前绑定可作为正式默认结果或正式历史结果被追溯
- `FAILED`：本次挂接失败，可重试
- `STALE`：旧结果已失去默认曝光资格、输入快照已过期或绑定需重挂；所有此前可能被描述成 `INVALIDATED` 的场景，统一收口到 `STALE`

它不承担的是：

- 存放 `OCR` 全量文本块明细
- 存放搜索引擎倒排结构
- 存放签章业务规则与签署流程实例
- 存放归档主记录、借阅状态与保管业务状态

## 4. 四类派生结果的统一对象模型

### 4.1 `OCR` 结果对象

`OCR` 结果对象是“某个 `DocumentVersion` 的识别派生结果”，不是“新的文档文件”。

其最小语义应包括：

- `source_document_version_id`
- `ocr_result_ref`
- `ocr_schema_version`
- `content_checksum`
- `page_count`
- `text_layer_status`
- `structured_extract_status`
- `producer_job_id`

挂接规则如下：

- 一个版本在同一套识别语义下只允许存在一个当前有效 `OCR` 结果对象
- 识别重跑时，不新建文件主档，只更新结果对象版本或结果引用，并把旧绑定置为 `STALE`
- `OCR` 结果可被搜索、智能应用、辅助比对消费，但这些消费都必须继续引用原 `source_document_version_id`

### 4.2 搜索索引绑定

搜索侧在文档中心内表达的不是搜索文档全文，而是“版本与索引文档之间的绑定关系”。

其最小语义应包括：

- `source_document_version_id`
- `index_document_ref`
- `index_schema_version`
- `index_scope`
- `index_checksum`
- `indexed_at`
- `exposed_by_asset_current_flag`

挂接规则如下：

- 搜索索引主键应优先锚定 `document_version_id`，而不是锚定 `document_asset_id`
- 主档层只额外维护“当前主版本对应哪条可暴露索引绑定”的投影状态
- 历史版本索引可保留作审计或回放，但默认策略是：版本一旦退出 `current_version_id`，其索引绑定立即转为 `STALE` 并退出默认检索曝光，物理索引文档按审计 / 回放需要保留；仅在审计调查、争议回放、合规核查或显式版本检索任务触发时，才允许临时重新暴露或补建历史索引

### 4.3 签章输入引用

签章侧在文档中心内部首先形成的是“输入引用”，而不是结果文件。

签章输入引用的最小语义应包括：

- `source_document_asset_id`
- `source_document_version_id`
- `signature_request_ref`
- `coordinate_protocol_ref`
- `coordinate_snapshot_checksum`
- `input_status`
- `consumed_at`

挂接规则如下：

- 签章申请只能引用某个明确版本作为输入稿，不能引用“当前最新文件”这种漂浮语义
- 坐标协议、签位快照、页码换算结果都属于输入引用快照，不回写覆盖原文件
- 签章完成后，如形成正式签章稿，必须以 `origin_type=SIGNATURE_OUTPUT` 回收为新的 `DocumentVersion`
- 原输入引用继续保留，用于解释“该签章稿由哪个输入版本和哪份坐标快照产生”

### 4.4 归档封包绑定

归档侧在文档中心内部表达的是“归档消费所依据的版本集合与封包引用关系”。

归档封包绑定的最小语义应包括：

- `archive_request_ref`
- `primary_document_asset_id`
- `primary_document_version_id`
- `package_manifest_ref`
- `package_checksum`
- `package_member_snapshot`
- `archive_medium_ref`
- `binding_status`

挂接规则如下：

- 归档封包绑定至少要锚定一个主版本，必要时再补充成员版本快照集合
- 成员集合是归档时刻的消费快照，不会倒推改写各文件主档的版本链结构
- 如果归档封包最终产出实体归档稿或封包文件，该实体文件应回收到文档中心作为 `origin_type=ARCHIVE_OUTPUT` 的 `DocumentVersion` 或独立 `ARCHIVE_COPY` 主档下的新版本
- 如果归档结果只形成外部档案系统介质引用，则文档中心仅保存绑定关系与回收引用，不把外部档案主记录复制成文件真相

## 5. 与文件主档和版本链的挂接关系

### 5.1 版本级挂接是唯一正式绑定关系

四类消费面全部先挂在版本级，再由主档层做当前视图投影：

- `OCR`：版本 -> `OCR` 结果对象
- 搜索：版本 -> 索引绑定
- 签章：版本 -> 签章输入引用 -> 结果版本回收
- 归档：版本 / 版本集合 -> 归档封包绑定 -> 结果版本或外部介质引用

这保证两个关键事实同时成立：

- 历史版本可追溯其当时被哪些能力消费
- 当前主档只暴露当前主版本对应的默认消费结果

### 5.2 主档级只维护当前消费投影

`dc_document_asset` 与聚合视图层只允许维护以下摘要，不允许持有派生结果明细：

- 当前主版本是否已有 `OCR` 结果
- 当前主版本是否已入搜索索引
- 当前主版本是否已声明签章输入可用
- 当前主版本是否已形成最新归档输入或归档封包绑定

因此，文件主档回答的是“当前对外默认消费状态”，不是“派生结果本体存放处”。

### 5.3 输出文件一律回到版本链

四类能力中，只有真正形成“正式文件”的结果才允许进入版本链：

- `OCR` 结果默认不形成新版本
- 搜索索引默认不形成新版本
- 签章完成后的正式签章稿形成新版本
- 归档输出的正式归档稿 / 归档封包文件形成新版本；纯档案介质引用不形成新版本

这条规则是文档中心把多个消费面挂在同一真相源上的核心收口点。

## 6. 更新传播规则

### 6.1 新版本追加

当同一 `DocumentAsset` 追加新版本时：

- 原版本上的 `OCR` 结果、索引绑定、签章输入引用、归档封包绑定仍然保留历史成立性
- 新版本自动进入待挂接状态，生成新的 `dc_capability_binding`
- 主档级“当前可消费投影”切换到新版本后，不再默认暴露旧版本对应的消费结果

### 6.2 当前主版本切换

当 `current_version_id` 切换时：

- `OCR`：旧主版本结果保留但不再作为默认识别结果暴露
- 搜索：默认将旧主版本索引绑定置为 `STALE` 并退出默认检索曝光，新的主版本索引补建后接管默认曝光；只有命中审计调查、争议回放、合规核查或显式版本检索任务时，才重新暴露或补建历史索引
- 签章：如旧主版本已被声明为签章输入，输入引用仍然有效，但不得自动漂移到新主版本
- 归档：已生效归档封包绑定保留其当时消费快照，不因主版本变化被静默改写

### 6.3 派生结果回写

能力回写必须遵循“先校验输入版本仍存在，再更新绑定状态”的顺序：

1. 校验 `source_document_version_id` 未删除且仍属于原主档
2. 校验回写结果摘要与请求快照一致
3. 更新对应 `dc_capability_binding`
4. 触发主档投影和聚合缓存刷新
5. 如形成正式输出文件，再进入版本链追加流程

### 6.4 业务状态变化

合同状态、签署状态、归档状态变化可以影响“是否允许继续消费”与“是否允许默认曝光”，但不能反向改写历史版本绑定事实。

## 7. 重建、失效与恢复边界

### 7.1 可重建对象

以下对象在源版本和任务证据仍可用时允许重建：

- `OCR` 结果对象
- 搜索索引绑定与索引文档
- 主档级消费投影
- 聚合缓存中的挂接摘要

重建原则：重建的是派生结果，不是版本真相。

### 7.2 只允许失效重挂的对象

以下对象不能被文档中心无差别自动重算，只能失效后重新挂接：

- 签章输入引用中的坐标快照
- 已提交给签章域的签章申请输入关系
- 已形成法律 / 业务意义的归档封包绑定

原因在于这些对象除了版本内容，还绑定了消费时刻的签章意图、坐标协议、归档成员集合或业务生效边界。文档中心可以把它们统一标记为 `STALE` 并发起重挂流程，但不能擅自替外部能力域重建业务事实。

### 7.3 失效触发条件

以下事件会触发失效判断：

- 源版本被替换为新的当前主版本
- 预览页码、文本层或渲染校验摘要发生变化，导致旧坐标快照不再可信
- `OCR` / 搜索 schema 版本升级，旧结果不再满足当前消费要求
- 归档成员文件中任一源版本被明确标记为不再可用或已撤销消费资格

### 7.4 恢复边界

文档中心的恢复边界只到以下层面：

- 恢复 `DocumentAsset` / `DocumentVersion` 真相
- 恢复 `dc_capability_binding` 与结果引用关系
- 恢复“哪些结果可重建、哪些引用需重挂”的判断能力
- 恢复统一审计链与任务追溯链

文档中心不承诺直接恢复外部搜索集群内部状态、签章引擎运行态或档案系统业务主记录；它只负责基于文件真相重新发起挂接或标记需人工 / 外部域补偿。

## 8. 审计留痕设计

### 8.1 最小审计单元

四类消费面统一至少记录以下审计键：

- `document_asset_id`
- `document_version_id`
- `capability_type`
- `binding_status`
- `request_ref` / `job_id`
- `result_ref`
- `result_checksum`
- `operator_id` / `system_actor`
- `occurred_at`

### 8.2 关键审计事件

必须覆盖以下事件：

- 挂接创建
- 输入快照冻结
- 派生结果回写成功 / 失败
- 版本切换导致的默认曝光切换
- 结果失效、重建、重挂请求
- 签章输出 / 归档输出回收为新版本

### 8.3 审计解释能力

统一审计链必须能回答以下问题：

- 某份搜索命中结果对应的是哪个正式文件版本
- 某份签章稿是基于哪个输入版本、哪份坐标快照产生的
- 某个归档封包当时包含了哪些文件版本
- 某份 `OCR` 文本为何失效、由哪次重建替换

## 9. 本文结论

文档中心把 `OCR`、搜索、签章、归档挂接在同一文件真相源上的核心方法，不是把这些能力实现收编进一个统一引擎，而是坚持三条边界：

1. 正式文件真相只在 `DocumentAsset` 与 `DocumentVersion`
2. 四类消费面全部先锚定 `document_version_id`
3. 只有真正形成正式文件的输出才允许回到版本链，其他结果都只作为派生对象、绑定关系或消费快照存在

在这一前提下，`OCR` 结果对象、搜索索引绑定、签章输入引用、归档封包绑定都能围绕同一文件真相源建立统一的更新传播、失效重建和审计追溯机制，而不会把搜索索引、识别文本、签章申请或归档记录反向升级成新的文件主档。
