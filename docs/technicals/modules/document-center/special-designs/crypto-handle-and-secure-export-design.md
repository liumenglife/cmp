# 文档中心专项设计：加密句柄与安全导出设计

## 1. 文档说明

本文档用于继续下沉 `document-center` 主线中“加密句柄与安全导出”这一能力，重点回答以下问题：

- 文档中心如何围绕文件真相源建立一层受控访问层，而不是把密文对象或明文导出副本升级为第二文件真相源
- 加密句柄对象、受控读取句柄、导出任务与导出介质如何成为同一套正式访问治理对象
- 访问令牌边界、平台内受控读取、解密下载导出、受控例外、过期撤销与审计如何在文档中心闭环
- 文档中心如何与加密文档能力、授权判断能力、对象生命周期治理能力协同，而不越界写成算法或密钥托管说明

本文是以下文档的下游专项设计：

- [`document-center Detailed Design`](../detailed-design.md)

`encrypted-document`、`identity-access`、对象存储与生命周期治理能力是本文的强依赖或关键消费方：前者提供密文对象与解密能力边界，后两者分别提供授权主体判断与导出介质生命周期承接；但这些关系都属于跨模块依赖或消费关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不重写对外 `API` 路径、字段、错误码、回调报文或下载接口契约
- 不写加密算法、密钥层级、密钥托管、厂商 `SDK`、硬件加密设备接入或明文缓冲区代码实现
- 不写数据库 `DDL`、类名、函数签名、任务主题、部署脚本、实施排期与上线步骤

## 2. 设计目标

- 让文档中心在 `DocumentAsset` / `DocumentVersion` 之上建立正式的受控访问层，统一承接密文读取与明文导出例外
- 让平台内默认读取始终围绕短期受控句柄进行，不把通用下载地址、长期令牌或静态明文副本暴露给业务侧
- 让解密下载导出成为“授权规则命中后的受控例外作业”，而不是把版本对象直接转成平台外长期可读文件
- 让过期、撤销、介质清理、审计追溯都围绕正式访问对象闭环，确保访问事实可重建、可追踪、可失效

## 3. 受控访问层的正式边界

文档中心的文件真相仍然只有两层：`DocumentAsset` 表达“这份文件是谁”，`DocumentVersion` 表达“这份文件当前内容是什么”。加密句柄与安全导出不替代这两层，而是在其上补一层“谁可以在什么边界下接触内容”的正式访问治理层。

这层受控访问层有三个判断原则：

1. 任何平台内读取，都必须先锚定到唯一 `document_version_id`，再由访问层生成短期受控读取句柄；不能绕开版本真相直接按对象路径或密文地址读取。
2. 任何平台外明文导出，都只能由授权命中后的导出作业产生；明文副本只是作业结果介质，不是新版本、不是新文件真相、也不是对外长期稳定地址。
3. 任何访问凭证都只表达一次受控能力放行，不表达所有权转移；授权结束、令牌过期、作业撤销或介质清理后，访问事实仍保留，访问能力必须失效。

因此，文档中心要回答的不是“如何实现解密算法”，而是“如何把加密文档的可访问性收口为文件真相源上的正式受控访问层”。

## 4. 正式治理对象

### 4.1 加密句柄对象

加密句柄对象用于表达“某个 `document_version_id` 当前对应哪一个受控密文载体与解密策略快照”。它不是密文文件本身，也不是密钥实体，而是文档中心在读取与导出前必须先解析的正式引用对象。

其正式职责如下：

- 绑定唯一 `document_version_id`
- 指向当前有效密文对象引用与内容校验摘要
- 固化加密策略快照、访问域摘要、允许消费通道摘要
- 表达该版本是否允许平台内预览、平台内下载流、受控导出三类消费方式
- 保留失效、替换、撤销后的历史引用链路，供审计与恢复使用

加密句柄对象的关键原则是“围绕版本稳定、围绕物理对象可替换”。当底层密文对象迁移、重加密、恢复或隔离时，版本身份不变；访问层重新解析新的有效句柄即可，下游页面和业务模块不应缓存底层对象定位信息。

### 4.2 受控读取句柄

受控读取句柄用于表达一次平台内读取放行。它只服务平台内预览、页流、文本层消费、受控下载流等平台内场景，不服务平台外长期持有。

其正式边界如下：

- 必须绑定 `document_version_id`、访问主体、访问用途、访问通道、过期时间、审计引用
- 必须短时有效，且默认一次会话或一次消费窗口内使用
- 只能换取平台内流式读取能力，不能直接换取脱离平台的明文文件
- 不能被前端、外部系统或其他模块当作长期文档地址缓存

### 4.3 安全导出作业与导出介质

安全导出不是读取句柄的延长线，而是单独的受控例外链路。其正式治理对象包括：

- 导出授权命中结果：表达本次请求命中了哪条管理端授权规则
- 安全导出作业：表达一次从授权通过到明文生成、交付、过期、清理的完整过程
- 导出介质引用：表达该作业产生了哪一个短生命周期明文介质，以及其当前是否仍可交付

导出介质只能挂在导出作业下，不能回写 `DocumentVersion`，也不能生成第二套文档编号。

其中，安全导出作业的正式去重语义固定为“同一 `document_version_id + request_user_id + reason + 授权命中时间窗` 只创建一个活跃作业”。该语义由作业上的 `request_dedup_fingerprint` 承接；交付令牌上的 `download_token_hash` 只表达一次具体发放凭证，不参与作业去重。

## 5. 与加密文档和授权判断的关系

### 5.1 与加密文档能力的关系

`encrypted-document` 决定“文件如何被加密、如何解密、密钥如何管理、哪些策略可被执行”；文档中心在本文中只决定“这些能力如何挂到文件真相源的读取与导出路径上”。

因此：

- 文档中心不保存密钥托管实现细节，只消费加密策略快照与解密能力结果
- 文档中心持有的是可访问性判断与句柄治理，不是算法治理
- 平台内读取和导出前，都必须先由文档中心解析加密句柄对象，再调用下游加密能力完成受控解密

### 5.2 与授权判断的关系

访问判断必须拆成两层，而不能混成一次布尔判断：

1. 基础访问层：用户是否有权看到该合同 / 文档 / 当前版本。
2. 敏感放行层：在已有基础访问权前提下，是否允许获得平台内受控读取句柄，或是否命中“解密下载”例外授权。

`identity-access` 提供部门、人员、范围、组织关系等主体判断能力；文档中心负责把判断结果落到具体版本、具体通道、具体作业上，避免出现“有文档访问权就默认有解密导出权”的越界放大。

## 6. 平台内受控读取链路

平台内受控读取的正式链路如下：

1. 业务页面或内部消费方先解析到唯一 `document_asset_id` 与目标 `document_version_id`。
2. 文档中心校验基础访问权限、业务上下文、版本状态、加密句柄状态。
3. 文档中心生成短期受控读取句柄，并写入访问审计预记录。
4. 消费方在有效窗口内携带该句柄请求平台内预览流、下载流或能力消费输入。
5. 文档中心联动加密能力执行内存流式解密，仅返回本次消费所需的数据流。
6. 句柄过期、消费完成、主体退出上下文或版本状态改变后，读取能力失效。

这里的关键约束有三条：

- 平台内读取默认不落地明文介质，优先以内存流和受控会话完成
- 同一个读取句柄只对声明的访问通道有效，预览句柄不能平移为导出句柄
- 历史版本、隔离版本、撤销版本即使曾经可读，也必须重新过访问层判断，不能复用旧句柄

## 7. 访问令牌边界

访问令牌只是受控读取句柄或导出交付凭证的载体，不是独立业务对象。文档中心必须明确其边界，避免令牌语义膨胀：

- 令牌只绑定单一访问对象，不能一牌多文档复用
- 令牌只绑定单一访问模式，平台内读取与安全导出使用不同令牌语义
- 令牌必须显式携带失效时间、用途、主体绑定与撤销依据
- 令牌不可作为长期书签、接口回填字段或跨业务流程缓存键使用
- 令牌一旦泄露，平台能做的是快速失效与追溯，不能把它视为可无限兜底的安全边界

正式上应区分两类令牌：

1. 受控读取令牌：短时、单用途、偏会话内消费。
2. 导出交付令牌：绑定安全导出作业、短时可交付、过期后必须回收或拒绝。

## 8. 解密下载导出流程

安全导出链路必须独立于平台内读取链路，正式步骤如下：

1. 请求方先完成基础访问校验。
2. 文档中心命中启用中的“解密下载”授权规则，并校验时间窗口、主体归属、范围条件、文档密级与业务状态。
3. 校验通过后创建安全导出作业，冻结授权快照、版本快照、请求原因与审计引用。
4. 作业解析加密句柄对象，调用受控解密能力生成短生命周期明文导出介质。
5. 文档中心只向该作业发放短期导出交付令牌，不暴露底层明文对象路径。
6. 交付完成后，作业进入已发放或已过期状态，等待介质回收与审计闭环。

这个流程中必须保持三个分离：

- 授权规则与导出作业分离：规则表达长期可命中条件，作业表达一次执行事实
- 导出作业与导出介质分离：作业是主对象，介质只是其短期结果
- 平台内读取与平台外导出分离：前者默认不产生可脱离平台使用的明文副本，后者是受控例外

并且，导出作业去重与交付令牌也必须分离：作业侧按照 `request_dedup_fingerprint` 约束同一业务请求的幂等，交付侧按照 `download_token_hash` 管理一次具体下载发放与失效，二者不能混用。

## 9. 受控例外边界

“管理端授权解密下载”是平台默认受控读取模型上的正式例外，但这个例外必须被约束为窄能力，而不是一般性越权通道。

例外边界如下：

- 只放开“解密下载导出”能力，不放开平台内审批、修订、批注、归档、签章等其他高敏动作
- 只对命中的部门、人员与有效时间窗口成立，不继承为永久角色能力
- 只对授权范围内的文档资产或版本成立，不自动扩展到同合同下全部文档
- 只对申请时声明的业务用途成立，必要时应绑定审批引用或业务原因

受控例外的核心不是“让谁能拿到明文”，而是“让例外始终保持例外，不侵蚀默认受控读取主路径”。

## 10. 介质清理与安全擦除语义

本文不展开底层安全擦除算法或设备指令，但必须定义文档中心层面的正式治理语义。对导出介质，文档中心至少要区分四种状态：

1. 待交付：明文已生成，但尚未发放。
2. 可交付：交付令牌生效，介质仍在短窗口内可取用。
3. 待清理：已交付完成、已过期、已撤销或作业失败后的残留介质，等待受控清理。
4. 已清理：介质已完成平台承诺范围内的删除或安全擦除，并保留清理证明。

治理规则如下：

- 明文导出介质必须落在临时受控介质层，不能混入正式对象层和派生对象层
- 介质一旦进入待清理状态，应由生命周期能力接管，不允许继续被页面直接引用
- 清理失败时，应保留失败审计、重试记录和残留介质定位，不得把状态静默改成成功
- 平台承诺清理的是平台可控介质，不承诺回收已被用户带出平台后的外部分发副本

因此，文档中心层面对“安全擦除”的正式承诺是：在平台可控介质范围内完成可审计的失效、删除与清理证明闭环，而不是在本文中规定具体擦除算法。

## 11. 访问审计、过期与撤销

### 11.1 审计范围

访问审计至少覆盖：

- 受控读取句柄创建、消费、过期、拒绝、撤销
- 解密下载授权创建、停用、命中、失效
- 安全导出作业创建、明文生成、交付、过期、取消、失败
- 导出介质清理成功、清理失败、补偿重试

审计必须能回指业务对象、版本对象、访问主体、访问通道、授权快照、作业状态和介质状态，且失败事件与拒绝事件同样要写入正式审计链。

### 11.2 过期规则

受控访问层至少存在三类过期：

1. 读取句柄过期：失去平台内读取能力，但不影响版本真相存在。
2. 导出交付过期：失去下载明文介质能力，并推动介质进入待清理。
3. 授权规则过期：后续请求不再命中，但已发生的历史作业与审计事实保留。

### 11.3 撤销规则

撤销必须支持以下触发源：

- 管理端主动停用授权规则
- 用户、部门、组织关系或数据范围发生变化
- 文档版本被撤销、隔离、替换或进入禁止导出状态
- 安全事件触发批量令牌失效与导出介质回收

撤销后的处理原则如下：

- 未消费的读取句柄立即失效
- 未交付的导出作业应取消并清理介质
- 已交付但仍在平台可控窗口内的介质应尽快失效并转入清理
- 已脱离平台的外部明文副本只保留审计与追责事实，不虚假承诺可回收

## 12. 恢复与异常边界

受控访问层的恢复目标不是“恢复某个旧令牌仍然可用”，而是“恢复访问事实、访问边界和平台内可控状态”。因此：

- 句柄和令牌属于可重建短态，丢失后按审计和版本真相重新生成，不作为长期恢复对象
- 授权规则、导出作业、介质清理状态属于正式记录，必须可恢复、可追溯
- 导出介质如果已清理，不因恢复流程重新默认生成明文，必须重新发起受控导出作业
- 异常场景下宁可收紧为拒绝访问，也不能因恢复不完整而放宽明文访问边界

## 13. 本文结论

文档中心中的加密句柄与安全导出设计，本质上是在 `DocumentAsset` / `DocumentVersion` 这一文件真相源之上，再建立一层正式、可审计、可过期、可撤销的受控访问层。加密句柄对象负责把版本真相稳定映射到受控密文载体，受控读取句柄负责平台内短期放行，安全导出作业负责承接授权命中的明文例外，导出介质只作为短生命周期结果存在。

因此，文档中心回答的是“如何把加密句柄与安全导出做成文件真相源上的受控访问层”，而不是“如何实现算法、密钥或厂商能力”。只要访问对象、令牌边界、导出作业、介质清理、审计、过期与撤销都围绕版本真相闭环，平台就能在保持默认受控使用的同时，为解密下载例外提供可治理、可追溯的正式承接。
