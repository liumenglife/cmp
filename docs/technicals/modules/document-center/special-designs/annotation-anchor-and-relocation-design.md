# 文档中心专项设计：批注锚点与跨版本重定位设计

## 1. 文档说明

本文档用于继续下沉 `document-center` 主线中“批注锚点与跨版本重定位”这一能力，重点回答以下问题：

- 批注锚点如何稳定挂载在文件真相源上，使批注不因预览渲染、版本推进或格式差异而漂移或丢失
- 锚点序列化载荷如何自包含足够信息，支持跨版本重定位、失效检测和审计回溯
- 不同文件格式和锚点类型下，定位策略如何统一建模而不陷入具体坐标算法
- 版本升级后锚点失效的判定、重定位尝试、失败留痕和人工介入边界如何收口
- 锚点稳定性如何通过审计事件闭环，保证批注协作在文件演进过程中可追踪、可恢复、可解释

本文是以下文档的下游专项设计：

- [`document-center Detailed Design`](../detailed-design.md)

`preview-rendering-and-text-layer-design` 是本文的强依赖方：它定义了页坐标系、文本层快照和校正对象，本文在此基础上定义锚点如何定位和重定位。`object-storage-and-lifecycle-design` 是锚点对象存储层和保留窗口的消费方。`contract-core`、`e-signature`、`encrypted-document` 是批注锚点的关键消费方：它们分别引用批注视图、签章前批注摘要或受控阅读批注上下文，但这些关系属于跨模块协作关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不重写批注的 `API` 路径、字段、错误码、回调协议或鉴权流程
- 不写具体坐标转换算法、前端渲染逻辑、文本选区交互实现或浏览器兼容策略
- 不写数据库 `DDL`、具体类名、函数签名、批处理任务脚本或部署拓扑细节
- 不写实施排期、压测方案、迁移步骤或上线计划

## 2. 设计目标

- 让批注锚点成为 `document-center` 文件真相源上的正式附着对象，而不是页面临时坐标或前端私有状态
- 让锚点序列化载荷自包含定位信息与校验信息，支持跨版本重定位、失效检测和审计回溯
- 让文本类文档与版面类文档使用统一的锚点对象模型，只在定位策略层区分锚定方式
- 让版本推进时锚点的存活、重定位和失效判定有明确规则，不静默删除也不伪造定位成功
- 让锚点稳定性审计与恢复边界清晰，保证批注协作在文件演进过程中可追踪、可恢复、可解释

## 3. 锚点对象模型

### 3.1 锚点与批注的关系

批注（`Annotation`）是协作层业务对象；锚点（`Anchor`）是批注在文件真相源上的正式定位对象。两者关系如下：

- 一个批注必须拥有一个创建时锚点，记录批注最初锚定在哪个版本的哪个位置
- 锚点不独立于批注存在，但锚点的定位语义与批注的业务语义（评论、风险、待办等）解耦
- 持久化边界统一为“`dc_annotation` 承接批注业务语义，`dc_annotation_anchor` 承接锚点正式对象”；父文档中不再允许把锚点载荷内嵌回 `dc_annotation`
- 批注可以被解决、驳回或重新打开，但这些状态变更不改变锚点的原始定位留痕
- 锚点重定位时，原锚点记录保留为历史留痕，新锚点通过重定位链路与原批注关联

### 3.2 锚点类型体系

锚点类型定义了“锚点以什么粒度定位到文件内容”。类型体系统一如下：

1. `TEXT_RANGE`：文本范围锚点，锚定到一个或多个连续文本片段。适用于合同正文、条款、附录等文本可提取的文档。
2. `PAGE_RECT`：页面矩形锚点，锚定到某一页上的一个矩形区域。适用于版面类文档、扫描件、签章区域等无法稳定提取文本的场景。
3. `PAGE_POINT`：页面点锚点，锚定到某一页上的一个位置点。适用于单点批注、标记或轻量评论。
4. `WHOLE_PAGE`：整页锚点，锚定到某一页的完整内容。适用于页面级批注、页眉页脚审查或整页风险标记。

类型选择规则如下：

- 文本类文档优先使用 `TEXT_RANGE`，因为文本锚点在版本演进中具有最强的内容级重定位能力
- 版面类文档或扫描件优先使用 `PAGE_RECT` 或 `PAGE_POINT`，因为文本层可能不稳定或不可用
- 整页批注或页面级审查场景使用 `WHOLE_PAGE`
- 锚点类型在创建时确定，后续不因重定位而改变类型

### 3.3 锚点核心字段

一个锚点对象至少包含以下核心字段：

- `anchor_id`：锚点唯一标识
- `annotation_id`：所属批注标识
- `document_version_id`：锚点所锚定的文档版本，是锚点的版本级根绑定
- `preview_generation_id`：锚点创建时引用的预览代次正式对象
- `page_slice_id`：页级锚点引用的页级切片正式对象
- `text_layer_snapshot_id`：文本范围锚点引用的文本层快照正式对象
- `preview_generation_no`：锚点创建时对应的预览代次，用于坐标系回溯
- `anchor_type`：锚点类型，取值为上述四种之一
- `anchor_payload`：锚点序列化载荷，承载定位信息、校验信息和格式版本
- `anchor_hash`：锚点内容哈希，用于检测载荷篡改与去重
- `anchor_status`：锚点状态，表达当前锚点是否有效、是否已重定位、是否已失效
- `degraded_reason_code`：降级定位原因；未降级时为空
- `degraded_granularity`：降级后的定位粒度，如 `PAGE`、`REGION`、`POINT`
- `reanchored_from_anchor_id`：重定位来源锚点标识，形成重定位链路
- `created_at`、`created_by`：创建留痕

父子对象关系补充如下：

- `dc_annotation` 保存 `annotation_id`、业务状态、正文内容以及 `current_anchor_id`
- `dc_annotation_anchor` 保存锚点载荷、哈希、状态、降级语义、重定位链路和所锚定的 `document_version_id`、`preview_generation_id`、`page_slice_id`、`text_layer_snapshot_id`
- 锚点重定位通过新增 `dc_annotation_anchor` 记录实现，不允许覆盖式更新原锚点载荷来伪装“原地成功”

锚点引用对象规则如下：

- 所有锚点必须引用 `dc_preview_generation.preview_generation_id`，用于固定坐标系代次
- `PAGE_RECT`、`PAGE_POINT`、`WHOLE_PAGE` 必须引用 `dc_preview_page_slice.page_slice_id`，用于固定页号、页尺寸、旋转方向和页校验摘要
- `TEXT_RANGE` 必须引用 `dc_text_layer_snapshot.text_layer_snapshot_id`，并同时保留其上游 `page_slice_id`，用于固定文本层校验摘要与页级兜底定位
- 若文本层校正影响当前展示结果，锚点载荷可记录当前有效 `text_layer_correction_id` 作为解释依据，但锚点真相仍以创建时的 `text_layer_snapshot_id` 和载荷哈希为准

### 3.4 锚点状态机

锚点状态表达锚点在其生命周期中的正式状态，统一如下：

- `ACTIVE`：锚点有效，定位信息可信，可正常展示和消费
- `RELOCATED`：锚点已被重定位到新位置，当前记录为历史留痕，新锚点接管定位
- `STALE`：锚点因版本切换、预览代次更替或文本层变化而进入待验证状态，尚未完成重定位判定
- `RELOCATION_FAILED`：锚点尝试重定位但失败，保留失效证据，等待人工介入或用户决策
- `EXPIRED`：锚点因文件版本被永久替代、超出保留窗口或业务闭环而不再需要定位

状态流转规则如下：

- `ACTIVE -> STALE`：版本切换、预览代次更替或文本层校验摘要变化时触发
- `STALE -> ACTIVE`：重定位成功且新位置可信时，原锚点标记为 `RELOCATED`，新锚点状态为 `ACTIVE`
- `STALE -> RELOCATION_FAILED`：重定位尝试失败，保留失败证据
- `RELOCATED`：终态之一，原锚点不再参与定位，但保留为历史链路
- `RELOCATION_FAILED -> ACTIVE`：人工介入修复后，锚点恢复有效
- `ACTIVE / RELOCATED / RELOCATION_FAILED -> EXPIRED`：超出保留窗口或业务闭环后清理

### 3.5 锚点与预览代次的关系

锚点创建时锚定到特定 `document_version_id`、`preview_generation_id` 和对应页级 / 文本层正式对象。关系如下：

- `document_version_id` 是锚点的版本级根绑定，决定锚点属于哪个正式内容版本
- `preview_generation_id` 是锚点的预览代次正式对象绑定，决定锚点的坐标信息基于哪一次预览渲染；`preview_generation_no` 仅作为审计回放摘要
- `page_slice_id` 是页级坐标系绑定，决定 `PAGE_RECT`、`PAGE_POINT`、`WHOLE_PAGE` 的页尺寸、旋转方向和页校验摘要
- `text_layer_snapshot_id` 是文本层绑定，决定 `TEXT_RANGE` 的文本偏移、词元跨度和文本层校验摘要
- 同一 `document_version_id` 下可能存在多个预览代次，锚点只绑定创建时的代次
- 当预览代次切换时，锚点不一定失效，但需要基于新代次的坐标系进行重定位验证

## 4. 锚点序列化协议

### 4.1 序列化原则

锚点序列化载荷不是坐标数据的原始转储，而是承载“定位信息 + 校验信息 + 格式版本”的自包含结构。序列化必须满足：

- 自包含：载荷本身包含重定位所需的全部必要信息，不依赖运行时缓存或前端私有状态
- 可版本化：载荷格式带有版本号，支持后续格式演进而不破坏历史锚点的可读性
- 可校验：载荷包含校验摘要，用于检测内容篡改、预览失配和重定位可行性
- 可审计：载荷可追溯到创建时的版本、代次和操作者

### 4.2 通用载荷结构

无论锚点类型如何，序列化载荷都包含以下通用字段：

- `format_version`：载荷格式版本号，当前为 `1`
- `anchor_type`：锚点类型，与核心字段一致
- `document_version_id`：创建时锚定的版本
- `preview_generation_id`：创建时引用的 `dc_preview_generation` 主键
- `page_slice_id`：创建时引用的 `dc_preview_page_slice` 主键，文本类锚点也必须携带
- `text_layer_snapshot_id`：`TEXT_RANGE` 锚点引用的 `dc_text_layer_snapshot` 主键；非文本类锚点为空
- `preview_generation_no`：创建时的预览代次
- `coordinate_space_version`：坐标系版本，来自预览代次的坐标空间标识
- `content_checksum`：定位目标内容的校验摘要，用于重定位时的内容匹配
- `type_specific_payload`：类型特定载荷，由锚点类型决定其内部结构

### 4.3 `TEXT_RANGE` 载荷结构

`TEXT_RANGE` 锚点的类型特定载荷至少包含：

- `page_no`：文本起始所在页号
- `text_layer_snapshot_id`：锚点创建时的文本层快照主键
- `page_slice_id`：文本起始页的页级切片主键
- `text_layer_checksum`：创建时文本层快照的校验摘要
- `start_offset`、`end_offset`：文本在文本层中的起止偏移
- `text_content_hash`：锚定文本片段的内容哈希
- `context_prefix`、`context_suffix`：锚定文本前后的上下文片段，用于模糊匹配
- `token_span_list`：可选，锚定文本的词元级跨度清单，用于精确定位

### 4.4 `PAGE_RECT` 载荷结构

`PAGE_RECT` 锚点的类型特定载荷至少包含：

- `page_no`：锚定所在页号
- `page_slice_id`：锚点创建时的页级切片主键
- `page_checksum`：页级切片校验摘要
- `rect_x`、`rect_y`、`rect_width`、`rect_height`：矩形区域坐标，基于预览代次的页坐标系
- `rect_normalized_x`、`rect_normalized_y`、`rect_normalized_width`、`rect_normalized_height`：归一化坐标，用于跨分辨率适配
- `visual_hash`：矩形区域内的视觉内容哈希，用于重定位时的视觉匹配

### 4.5 `PAGE_POINT` 载荷结构

`PAGE_POINT` 锚点的类型特定载荷至少包含：

- `page_no`：锚定所在页号
- `page_slice_id`：锚点创建时的页级切片主键
- `page_checksum`：页级切片校验摘要
- `point_x`、`point_y`：点坐标，基于预览代次的页坐标系
- `point_normalized_x`、`point_normalized_y`：归一化坐标
- `context_visual_hash`：点周围的视觉内容哈希

### 4.6 `WHOLE_PAGE` 载荷结构

`WHOLE_PAGE` 锚点的类型特定载荷至少包含：

- `page_no`：锚定页号
- `page_slice_id`：锚点创建时的页级切片主键
- `page_checksum`：页级切片校验摘要
- `page_content_hash`：整页内容哈希

### 4.7 锚点哈希计算

`anchor_hash` 基于序列化载荷计算，计算范围包括：

- `format_version`
- `anchor_type`
- `document_version_id`
- `type_specific_payload` 中的定位核心字段

不包括：

- `anchor_status`、`reanchored_from_anchor_id` 等状态与链路字段
- `created_at`、`updated_at` 等时间戳字段

当载荷中任何定位核心字段发生变化时，`anchor_hash` 必须随之变化，用于检测内容篡改与去重。

## 5. 锚点定位策略

### 5.1 定位策略原则

锚点定位策略回答“锚点如何在预览视图上找到其锚定的内容位置”。定位策略必须满足：

- 内容优先：优先基于内容匹配而非布局位置，因为布局可能随预览渲染变化
- 冗余定位：载荷中包含多种定位线索，当主定位方式失败时可尝试备选方式
- 校验先行：定位前先校验坐标系版本和内容校验摘要，判断是否需要重定位
- 降级可控：精确定位失败时可降级到粗粒度定位，但必须标记降级状态

### 5.2 `TEXT_RANGE` 定位策略

`TEXT_RANGE` 锚点的定位优先级如下：

1. 精确偏移定位：使用 `start_offset` / `end_offset` 在当前文本层快照中定位，前提是 `text_layer_checksum` 匹配
2. 词元级定位：使用 `token_span_list` 在当前文本层中进行词元匹配，允许微小偏移
3. 内容哈希定位：使用 `text_content_hash` 在当前文本层中搜索匹配的文本片段
4. 上下文模糊匹配：使用 `context_prefix` / `context_suffix` 在当前文本层中进行上下文搜索，允许文本内容有小幅变化
5. 页码兜底：若上述方式均失败，定位到 `page_no` 并标记为降级定位

### 5.3 `PAGE_RECT` 定位策略

`PAGE_RECT` 锚点的定位优先级如下：

1. 归一化坐标定位：使用 `rect_normalized_*` 在当前页坐标系中定位，前提是 `page_checksum` 匹配
2. 绝对坐标适配：使用 `rect_*` 结合当前页宽高进行缩放适配
3. 视觉哈希匹配：使用 `visual_hash` 在当前页中搜索视觉内容匹配的区域
4. 页码兜底：若上述方式均失败，定位到 `page_no` 并标记为降级定位

### 5.4 `PAGE_POINT` 定位策略

`PAGE_POINT` 锚点的定位优先级如下：

1. 归一化坐标定位：使用 `point_normalized_*` 在当前页坐标系中定位
2. 绝对坐标适配：使用 `point_*` 结合当前页宽高进行缩放适配
3. 上下文视觉匹配：使用 `context_visual_hash` 在当前页中搜索匹配区域
4. 页码兜底：定位到 `page_no` 并标记为降级定位

### 5.5 `WHOLE_PAGE` 定位策略

`WHOLE_PAGE` 锚点的定位优先级如下：

1. 页码直接定位：使用 `page_no` 定位到指定页
2. 内容哈希验证：定位后使用 `page_content_hash` 验证页面内容是否变化
3. 若页码越界，标记为失效

### 5.6 降级定位的约束

降级定位是锚点精确定位失败后的受控降级，不是默认行为。约束如下：

- 降级定位必须在锚点状态中标记降级原因和降级粒度
- 降级定位的正式承接字段固定为 `degraded_reason_code` 与 `degraded_granularity`，不能只写在审计事件、页面提示或任务日志里
- 降级定位不自动升级为精确定位，除非重新生成锚点载荷
- 降级定位的锚点仍可展示和消费，但消费方应感知其定位精度降低
- 降级定位的锚点在下一次预览重建或文本层更新时应优先触发重定位验证

## 6. 版本演进与锚点重定位

### 6.1 版本切换时的锚点失效判定

当 `document_asset_id` 的 `current_version_id` 发生切换时，旧版本上的锚点需要进行失效判定。判定规则如下：

1. 若旧版本仍处于可访问状态（未被永久删除或归档清理），其锚点不自动失效，但标记为 `STALE` 进入待验证
2. 若旧版本已被永久替代且超出保留窗口，其锚点可标记为 `EXPIRED`
3. 若旧版本处于归档区但仍有恢复可能，其锚点保持 `STALE`，等待版本恢复后再验证

### 6.2 预览代次切换时的锚点失效判定

同一 `document_version_id` 下预览代次切换时，锚点需要进行坐标系验证。判定规则如下：

1. 若新代次的 `coordinate_space_version` 与锚点载荷一致，锚点保持 `ACTIVE`
2. 若 `coordinate_space_version` 变化但 `page_checksum` 未变，锚点标记为 `STALE` 并触发坐标系适配验证
3. 若 `page_checksum` 变化，锚点标记为 `STALE` 并触发完整重定位流程

### 6.3 文本层变化时的锚点失效判定

文本层快照变化时，`TEXT_RANGE` 锚点需要进行内容验证。判定规则如下：

1. 若 `text_layer_checksum` 未变，锚点保持 `ACTIVE`
2. 若 `text_layer_checksum` 变化但 `text_content_hash` 仍可匹配，锚点标记为 `STALE` 并触发偏移量重算
3. 若 `text_content_hash` 无法匹配，锚点标记为 `STALE` 并触发上下文模糊匹配

### 6.4 跨版本重定位流程

当锚点需要跨版本重定位时，执行以下流程：

1. 获取目标版本的预览代次和文本层快照
2. 基于锚点载荷中的定位信息，在目标版本上执行定位策略
3. 若定位成功，创建新锚点记录，状态为 `ACTIVE`，原锚点标记为 `RELOCATED`
4. 若定位失败，原锚点标记为 `RELOCATION_FAILED`，保留失败证据

新锚点与原锚点的关系：

- 新锚点的 `document_version_id` 指向目标版本
- 新锚点的 `reanchored_from_anchor_id` 指向原锚点
- 新锚点的 `anchor_payload` 基于目标版本的坐标系和文本层重新生成
- 原锚点的 `anchor_status` 更新为 `RELOCATED`，不再参与定位

### 6.5 重定位优先级

当锚点需要跨版本重定位时，优先级如下：

1. 文本内容重定位：优先尝试基于文本内容匹配在新版本上定位，因为文本内容是最稳定的定位依据
2. 页码 + 规则重定位：若文本匹配失败，尝试基于页码变化规则（如页码偏移、章节映射）进行定位
3. 视觉相似度重定位：若规则重定位失败，尝试基于视觉相似度在新版本上搜索匹配区域
4. 人工标注重定位：若自动重定位均失败，标记为 `RELOCATION_FAILED`，由用户手动标注新位置

### 6.6 重定位失败的处理

重定位失败时，必须遵循以下原则：

- 不静默删除原批注和原锚点记录
- 原锚点标记为 `RELOCATION_FAILED`，保留失败证据（失败原因、尝试的定位方式、目标版本信息）
- 批注状态不因锚点失效而自动变更，仍保留其业务状态
- 重定位失败的批注在协作视图中标记为“锚点待确认”，由用户决定是否手动重新标注或忽略

### 6.7 批量重定位

当版本切换或预览重建导致大量锚点需要重定位时，应采用批量重定位策略：

- 批量重定位作为异步任务执行，不阻塞版本切换主事务
- 批量重定位按锚点类型分组执行，`TEXT_RANGE` 优先于 `PAGE_RECT` / `PAGE_POINT`
- 批量重定位的结果（成功数、失败数、降级数）必须写入审计事件
- 批量重定位失败不影响版本切换的成立性，只影响批注的可用性

## 7. 锚点稳定性审计

### 7.1 审计原则

锚点稳定性审计不是记录“谁创建了批注”，而是追踪锚点在整个生命周期中经历了哪些正式状态变迁和重定位动作。审计必须满足：

- 可追溯：审计事件可回溯到具体文档版本、预览代次和操作者
- 可解释：审计事件记录状态变迁的原因、触发条件和定位依据
- 可恢复：审计事件包含足够信息支持锚点状态的人工恢复

### 7.2 必须审计的关键事件

以下事件必须写入审计记录：

- 锚点创建：记录创建时的版本、代次、锚点类型和操作者
- 锚点重定位成功：记录原锚点、新锚点、目标版本、重定位方式和定位精度
- 锚点重定位失败：记录原锚点、目标版本、失败原因和尝试的定位方式
- 锚点状态变更：记录状态从何值变更为何值、触发原因和触发主体
- 锚点降级定位：记录降级原因、降级粒度和影响范围；其中正式对象字段以 `degraded_reason_code` 与 `degraded_granularity` 为准，审计只负责补充事件上下文
- 锚点过期清理：记录清理原因、保留窗口和清理结果

### 7.3 审计记录核心字段

锚点审计记录至少包含以下字段：

- `annotation_id`、`anchor_id`：关联的批注和锚点
- `document_asset_id`、`document_version_id`：关联的文档对象和版本
- `preview_generation_no`：关联的预览代次
- `event_type`：事件类型，如 `CREATED`、`RELOCATED`、`RELOCATION_FAILED`、`STALE`、`EXPIRED`
- `event_reason`：事件原因
- `source_anchor_id`：重定位来源锚点（如适用）
- `target_document_version_id`：重定位目标版本（如适用）
- `relocation_method`：重定位方式（如适用）
- `relocation_accuracy`：重定位精度（如适用）
- `operator_id`、`job_id`：操作主体

### 7.4 审计查询入口

锚点审计查询必须支持以下入口：

- 按 `annotation_id` 查询某批注的全部锚点状态变迁
- 按 `document_version_id` 查询某版本上所有锚点的状态摘要
- 按 `document_asset_id` 查询某文档对象全部版本的锚点稳定性概况
- 按 `event_type` 和时间范围查询特定类型的锚点事件

## 8. 恢复边界

### 8.1 可自动恢复的场景

以下场景允许系统自动恢复或自动处理：

- 预览代次重建后坐标系未变，锚点从 `STALE` 自动恢复为 `ACTIVE`
- 文本层更新后内容未变，`TEXT_RANGE` 锚点自动更新偏移量并恢复为 `ACTIVE`
- 版本切换后新版本与旧版本内容高度相似，锚点自动重定位成功
- 批量重定位任务失败后自动重试，在重试窗口内成功

### 8.2 需人工介入的场景

以下场景超出自动恢复边界，需要人工介入：

- 锚点重定位失败且自动重试耗尽，需人工标注新位置或确认批注是否仍有效
- 文本层与页坐标持续失配，导致锚点定位结果不可信，需人工核查预览渲染质量
- 版本内容大幅重构，导致大量锚点无法自动重定位，需人工批量处理
- 锚点载荷格式版本升级后历史锚点无法解析，需人工迁移或补建

### 8.3 不承诺自动恢复的边界

以下场景超出本文自动恢复承诺：

- 源文件损坏导致所有预览代次和文本层均无法重建，锚点定位基础丧失
- 锚点载荷因存储损坏而丢失，且无审计记录可供重建
- 已脱离平台的批注导出副本，平台不承诺其锚点在外部环境中的有效性

### 8.4 恢复优先级

当锚点需要恢复时，优先级如下：

1. 基于审计记录重建锚点状态和重定位链路
2. 基于当前有效预览代次和文本层快照重新执行定位策略
3. 若自动定位失败，提供人工标注入口，由用户手动确认新位置
4. 若人工标注也无法完成，标记批注为“锚点失效”，保留批注业务内容但不再参与定位消费

## 9. 与其他专项设计的边界

### 9.1 与预览渲染与文本层设计的边界

- 预览渲染与文本层设计定义页坐标系、文本层快照和校正对象的正式来源
- 本文定义锚点如何基于这些坐标系和文本层进行定位和重定位
- 当预览代次或文本层变化时，由本文定义锚点的失效判定和重定位规则

### 9.2 与对象存储与生命周期设计的边界

- 对象存储与生命周期设计定义派生对象的存储层、迁移和清理规则
- 锚点对象作为批注的定位附着对象，其存储和保留窗口由对象生命周期规则承接
- 锚点载荷中引用的预览代次和文本层快照的存储生命周期，由对象存储专项设计承接

### 9.3 与 `OCR` / 搜索 / 签章 / 归档挂接设计的边界

- `OCR` 结果可作为文本层抽取的辅助来源，间接影响 `TEXT_RANGE` 锚点的定位基础
- 搜索索引可消费锚点定位的批注上下文，但不接管锚点定位职责
- 签章坐标与锚点坐标共享页坐标系，但签章坐标有独立的精度和法律语义要求
- 归档封包中的批注快照应包含锚点序列化载荷，但归档后的锚点不再参与重定位

## 10. 本文结论

批注锚点是 `document-center` 文件真相源上的正式附着对象，承载批注在文件内容上的定位语义。锚点序列化载荷必须自包含定位信息与校验信息，支持跨版本重定位、失效检测和审计回溯。锚点定位策略基于内容优先、冗余定位、校验先行和降级可控的原则，在不同锚点类型和文件格式下统一建模。版本推进时锚点的存活、重定位和失效判定有明确规则，不静默删除也不伪造定位成功。锚点稳定性审计与恢复边界清晰，保证批注协作在文件演进过程中可追踪、可恢复、可解释。

因此，批注锚点不是“页面上的坐标点”，也不是“前端选区的序列化”，而是文档中心在版本演进过程中维持协作语义稳定性的正式治理对象。只有这样，批注才能在文件版本推进、预览重建、文本层更新和格式差异的背景下，持续稳定地挂载在文件真相源上。
