## 2. 计划边界与原则

本计划只覆盖文档中心与文档协作子模块的实施推进顺序，不替代需求、架构、
接口或详细设计文档。

实施过程中遵循以下原则：

- 文档中心是文件对象真相源，正式文件写入、版本演进、预览入口和能力挂接都先
  围绕文档中心成立。
- 文档协作建立在文档中心之上，批注、修订、协同处理只围绕既有文件对象和版本链
  展开，不另建第二套文件真相源。
- 默认不可脱离平台使用，但管理端可按部门、人员授权解密下载；解密下载后的导出
  明文文件可脱离 `CMP` 使用，且必须纳入统一权限控制与审计留痕。
- 预览、`OCR`、搜索、签章、加密、归档均按“挂接能力”推进，先建立统一输入源和
  调用挂点，再进入能力联调，不倒置成各能力先行定义自己的文件体系。
- 模块实施优先保证主链路闭环，再逐步补齐派生能力和联调能力，避免一开始并行铺开
  过多外围能力导致真相源与版本链反复返工。
- 使用总平台既定正式文档链路，不在本计划中引入新的平行规格体系。

## 3. 前置条件

### 3.1 文档中心相关环境与存储准备

- 文档中心所依赖的文件存储、对象寻址规则、受控访问方式和备份恢复策略已在环境侧
  可用，至少满足正式文件入库、版本追加、预览产物存放和归档产物回收。
- 文档中心运行所需的异步任务执行环境、任务重试能力、日志采集和审计落库链路已具备。
- 文档中心与合同主档、权限底座、组织架构、审计中心之间的基础集成接口可用，能够
  支撑文件归属、权限判断和关键动作留痕。
- 开发、联调、预发环境中的文件存储策略保持一致的访问语义，避免开发阶段形成一套
  临时文件路径约定、上线前再整体迁移。

### 3.2 预览、OCR、搜索、签章、加密挂接前置条件

- 预览能力所需的文档转换运行环境、任务编排入口和预览访问域已准备完成，至少可在
  联调环境生成并读取受控预览产物。
- `OCR` 挂接前，需先确认文档版本可稳定作为识别输入，且识别结果以派生结果回写，
  不覆盖文件真相。
- 搜索挂接前，需先确认索引构建与刷新链路可消费文档版本事件，且索引失败不会阻断
  文档主链路。
- 签章挂接前，需先确认签章模块可接受文档中心指定版本作为签章输入稿，并能把签章
  结果稿回收到文档中心版本链。
- 加密挂接前，需先确认写入路径自动加密、读取路径平台内受控解密、授权解密下载作业
  三条链路的执行边界已经明确并具备实现条件。

### 3.3 管理端授权解密下载前置条件

- 管理端已具备按部门、人员配置授权范围的权限模型，不以临时白名单代替正式授权能力。
- 审计中心已具备记录授权发放、授权变更、解密下载申请、导出结果与失败原因的能力。
- 文档访问权限、组织架构归属和管理端授权规则之间的优先级已明确，避免出现“有下载
  权限但无文档访问上下文”或“有文档查看权限但无明文导出权限”的口径冲突。
- 解密下载导出物的临时存放、有效期和回收策略已在环境侧准备完成。

### 3.4 外部条件前置说明

- 企业微信相关联调需要测试账号、测试应用或等价联调凭证。
- 自研加密软件、自研电子签章属于平台内子模块实施范围，不依赖外部测试账号才能启动。
- 其余平台内挂接能力均以平台环境、平台任务链路和平台权限审计能力就绪为主，不额外
  设定外部账号作为前置条件。

## 4. 总体阶段划分

本模块按五个阶段推进：

1. 文件真相源与基础写入能力
2. 版本链与统一预览能力
3. 文档协作与修订能力
4. 加密、`OCR`、搜索、签章、归档挂接能力
5. 联调、验收与上线准备

阶段之间以前一阶段稳定产物为下一阶段输入，不建议跳阶段并行铺开。

## 5. 分阶段实施计划

### 5.1 阶段一：文件真相源与基础写入能力

#### 目标

- 建立文档中心作为文件对象真相源的最小可运行闭环。
- 打通正式文件写入、业务绑定、主版本建立和基础读取能力。
- 让合同正文、附件等正式文件不再以临时附件路径先落别处、再补录到文档中心。

#### 核心产出

- 文档主档与版本主链路可用，支持正式写入首版文件。
- 与合同主档的业务绑定能力可用，能稳定表达“哪个合同拥有哪份文档对象”。
- 基础读取与受控下载能力可用，能在权限校验后读取当前正式版本。
- 文档写入、读取、版本创建等关键动作纳入统一审计。
- 基础异常处理与补偿规则成型，能处理重复上传、入库失败、存储失败等主链问题。

#### 前置依赖

- 文件存储和异步任务基础环境已准备完成。
- 合同主档、权限底座、组织架构、审计中心已有可调用基础接口。
- 文档中心内部详细设计已能支撑文件对象和首版版本的正式实现。

#### 完成标志

- 新建合同正文和附件可以直接进入文档中心并形成正式文件对象。
- 合同详情、基础业务查询可稳定读取当前主版本引用。
- 关键写入失败不会留下“合同已绑定但文件不存在”的不一致状态。
- 审计可追踪谁在何时对哪份文档完成写入、替换或读取。

### 5.2 阶段二：版本链与统一预览能力

#### 目标

- 在真相源之上建立稳定版本链，明确主版本切换与版本追溯能力。
- 建立统一预览入口，使预览成为版本视图，而不是第二份文件真相。

#### 核心产出

- 版本追加、版本切换、版本追溯、版本关系校验能力可用。
- 预览产物生成、预览状态跟踪和预览访问控制链路可用。
- 版本切换后，预览、绑定视图和后续能力消费入口能跟随主版本变化刷新。
- 可支撑后续批注锚点、修订比对和能力挂接所需的稳定版本标识。

#### 前置依赖

- 阶段一的文件真相源与基础写入能力已稳定可用。
- 文档转换运行环境和预览任务链路已准备完成。
- 版本切换对合同详情、只读聚合视图和审计链路的影响已明确。

#### 完成标志

- 同一文档对象可以稳定追加多个版本，且任何时刻只有一个正式主版本。
- 业务侧可按版本查看历史稿、当前稿和切换记录。
- 预览失败不会破坏文件真相；预览成功后可被后续协作与能力挂接复用。
- 版本切换的权限、审计和刷新动作具备明确闭环。

### 5.3 阶段三：文档协作与修订能力

#### 目标

- 在既有版本链上建立批注、修订、协作处理与采纳机制。
- 让文档协作成为文档中心上的上层能力，而不是独立文件系统。

#### 核心产出

- 面向版本视图的批注、评论、处理状态能力可用。
- 修订记录、候选版本、采纳 / 驳回动作和版本推进链路可用。
- 文档协作工作区、摘要视图和绑定视图可支撑合同详情、审批摘要和归档准备页。
- 锚点失效、重定位和跨版本协作延续规则可支撑主流程使用。

#### 前置依赖

- 阶段二的版本链和统一预览能力已稳定可用。
- 预览视图可提供批注锚点附着基础。
- 权限模型能够区分查看、评论、修订、采纳等动作权限。

#### 完成标志

- 用户可以围绕同一文档对象发起批注与修订，而不生成第二份正式文件真相。
- 修订采纳后可推动主版本切换，并可回写协作摘要状态。
- 协作视图可被合同详情、审批协同等上层页面稳定复用。
- 批注和修订全程留痕，可审计、可追溯、可恢复。

### 5.4 阶段四：加密、OCR、搜索、签章、归档挂接能力

#### 目标

- 在文档中心主链稳定后，把关键挂接能力统一接入同一文件对象与版本链。
- 形成“文档中心先成立，挂接能力围绕其消费和回写”的正式运行方式。

#### 核心产出

- 加密挂接：写入自动加密、平台内受控读取、管理端授权解密下载链路可用。
- `OCR` 挂接：可基于受控版本发起识别并回写派生结果。
- 搜索挂接：可基于文档版本刷新索引并提供检索输入。
- 签章挂接：可把指定版本声明为签章输入稿，并将签章结果稿回收到版本链。
- 归档挂接：可基于正式版本形成归档输入并把归档稿或归档封包引用回收至文档中心。

#### 前置依赖

- 阶段一至阶段三已形成稳定文件真相、版本链和协作基础。
- 管理端授权模型与审计能力已满足解密下载要求。
- 平台任务中心可以承接 `OCR`、索引刷新、预览、解密下载、签章回收等异步任务。
- 签章、归档等相关子模块已提供最小可联调入口。

#### 完成标志

- 默认文档读取与下载仍受平台控制，只有管理端按部门、人员授权后才可触发解密下载。
- 导出的明文文件可脱离 `CMP` 使用，但授权、生成、交付、过期和失败都可审计。
- `OCR`、搜索、签章、归档均以文档版本为正式输入，不再接受私有附件表作为输入源。
- 任一挂接能力失败都不会破坏文档主链与版本真相。

### 5.5 阶段五：联调、验收与上线准备

#### 目标

- 以完整业务主链验证文档中心与文档协作的可交付性。
- 收口跨模块联调、测试验收、数据治理和上线切换准备。

#### 核心产出

- 跨模块联调清单与执行记录完成，包括合同主档、审批协同、签章、归档、搜索、
  企业微信相关触达场景。
- 测试清单、验收用例、上线检查清单和回退预案完成。
- 环境配置、容量评估、日志观测、备份恢复和审计抽查方案完成。
- 存量附件治理、历史稿回收策略和上线切换规则形成可执行方案。

#### 前置依赖

- 前四个阶段的核心能力已经完成并通过模块级验证。
- 需要外部条件的联调资源已经就绪，尤其是企业微信测试账号与对应联调环境。
- 业务验收口径、审计检查项和安全检查项已统一。

#### 完成标志

- 主链场景能够覆盖“上传 / 起草 -> 版本演进 -> 协作修订 -> 预览 / 检索 / 签章 /
  归档 -> 审计追踪”。
- 高敏场景能够覆盖“管理端授权 -> 按部门、人员命中授权 -> 解密下载 -> 导出明文
  文件 -> 审计回查”。
- 上线准备清单明确，具备环境校验、配置校验、数据校验、容量校验和回退方案。
- 模块进入上线窗口时，不再存在“正式文件真相仍散落在历史附件路径”的结构性问题。

## 6. 模块推进顺序

本模块按“文件真相源 -> 版本链 -> 协作 -> 挂接能力 -> 联调验收”的顺序推进，原因如下：

1. 先文件真相源，是因为文档中心必须先成为正式写入入口。
   如果文件仍先散落在业务模块或临时附件路径，后续版本链、批注、签章、搜索、归档
   都会建立在不稳定输入源上，最终导致多套文件真相并存。

2. 再版本链，是因为协作、预览、签章、归档都依赖稳定版本标识。
   没有版本链，只能围绕“当前文件”做一次性处理，无法支撑修订、追溯、回收和主版本
   切换，也无法形成统一预览视图。

3. 再协作，是因为文档协作不是独立文件系统，而是围绕既有版本链做批注与修订。
   如果在真相源和版本链未稳定前先做协作，后续每次版本调整都会推倒协作锚点和修订关系。

4. 最后挂接能力，是因为 `OCR`、搜索、签章、加密、归档都应消费同一文件对象与版本。
   只有文档中心主链稳定后，这些能力才能真正复用统一输入源，并把结果回收到正确位置。

5. 联调与验收放在最后收口，是因为外部协作、跨模块场景和上线准备都建立在前述能力
   已具备稳定边界的前提上。过早进入大联调，只会把设计问题误暴露成联调问题。

## 7. 外部联调准备

本模块只对真正依赖外部条件的能力做联调准备说明。

### 7.1 需要外部条件的联调项

- 企业微信相关触达、登录态联动或消息联动场景，需要测试账号、测试应用配置和联调环境。
- 若上线前需要通过企业微信触发文档处理通知、审批提醒或协作消息，应提前准备联调名单、
  回调地址和验证路径。

### 7.2 不作为外部账号依赖的联调项

- 自研加密软件属于平台内挂接模块，不以外部测试账号作为前置条件。
- 自研电子签章属于平台内子模块能力，不以外部测试账号作为前置条件。
- `OCR`、搜索、预览、归档等能力的联调重点在平台内任务链路、权限链路和结果回收，
  不额外引入“外部测试账号”作为实施启动条件。

### 7.3 联调前应先准备的内容

- 明确每个联调对象只消费哪类文档版本、回写什么结果、失败如何回查。
- 准备跨模块联调数据集，至少覆盖正文、附件、修订稿、签章稿、归档稿等典型对象。
- 准备权限与审计联调用例，确保高敏场景不是只测业务成功路径，也覆盖拒绝、越权、过期、
  重试和失败回查路径。

## 8. 测试、验收与上线准备

### 8.1 测试准备

- 模块测试按阶段推进，先测文档主链，再测版本链，再测协作，再测挂接能力，最后测跨模块链路。
- 测试数据需覆盖正文、附件、补充协议、签章结果稿、归档稿等不同文档角色。
- 必测场景包括：重复上传、版本切换、预览失败重试、锚点失效重定位、修订采纳、
  加密写入失败、解密下载越权、索引失败不阻断主链、签章结果回收失败补偿。
- 高敏能力必须单列测试项，尤其是管理端授权解密下载、部门 / 人员命中授权、审计回查、
  导出文件过期与清理。

### 8.2 验收准备

- 验收按“文件真相是否统一、版本链是否可追溯、协作是否建立在其上、挂接能力是否围绕
  同一输入源运行”四条主线收口。
- 验收需覆盖默认平台内受控使用路径，以及管理端授权解密下载这一受控例外路径。
- 验收需确认导出的明文文件可脱离 `CMP` 使用，但平台内默认路径仍不放开脱离平台的常规明文流转。
- 验收需抽查审计日志，确认写入、读取、版本切换、修订采纳、授权发放、解密下载均可回查。

### 8.3 上线准备

- 明确上线前存量附件治理策略：哪些历史文件需要迁入文档中心，哪些可保留只读引用，
  哪些必须完成版本归并后才能开放主链使用。
- 明确上线切换窗口、数据校验方法、预览与索引初始化方式、任务积压清理规则。
- 明确文件存储、预览产物、导出明文文件和审计日志的容量监控与备份恢复方案。
- 明确回退原则：如挂接能力异常，优先保证文档中心主链可用，不允许因搜索、`OCR` 或
  其他派生能力异常破坏正式文件真相。
