# 印章资源与证书介质专项设计

## 1. 文档定位

本文档是 `CMP` 电子签章子模块的印章资源与证书介质治理设计，用于明确印章资源、印章样式、授权矩阵与证书介质在平台内的治理对象、归属关系、生命周期边界与跨模块协作约束。

本文档不承担以下内容：

- 不输出 API 路径、请求样例、错误码全集
- 不输出表级 DDL、索引细节、字段长度与枚举值全集
- 不输出签章坐标算法、渲染实现、底层加密生产代码
- 不输出实现排期、联调顺序、负责人拆分
- 不输出运维手册、部署清单、监控配置

## 2. 印章资源治理

### 2.1 治理对象

印章资源治理以"印章资源聚合"为最小治理单元，其内部由以下对象构成：

- **印章资源主体**：表达一个组织实体拥有的印章本体，包含印章名称、印章类型、所属组织、法律效力范围、生效与失效时间、资源状态。
- **印章资源版本**：表达印章资源在特定时间窗口内的有效快照，每次变更印章样式、授权矩阵或证书绑定关系时，产生新版本而不是覆写旧版本。
- **印章样式引用**：表达印章在签署页面、签署结果稿、验签证明中的视觉呈现方式，不直接持有图片二进制，只持有文档中心中的样式文件引用。
- **授权矩阵快照**：表达哪些组织、角色、人员在什么条件下可以使用该印章，每次印章使用准入时消费授权矩阵快照，而不是实时查询最新授权关系。
- **证书绑定关系**：表达该印章资源当前关联哪一张签名证书、哪一个证书链、哪一个时间戳证书，作为签章执行时的证书真相源。

### 2.2 稳定锚点

印章资源的稳定锚点设计如下：

- **印章资源 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **印章资源编号**：面向用户与管理的稳定业务标识，允许按组织规则自定义，但在平台内作为辅助索引存在，不替代印章资源 ID 作为真相键。
- **当前有效版本 ID**：每一枚印章资源在任意时刻有且只有一个"当前有效版本"，所有签章准入、会话建立、结果登记都消费该版本的快照，而不是实时计算最新版本。
- **印章样式文档引用**：印章样式文件统一托管在文档中心，印章资源只保存 `document_asset_id` / `document_version_id`，不保存样式文件副本。

### 2.3 版本归属

印章资源版本归属遵循以下原则：

- 每一版印章资源必须归属到某一枚印章资源主体，不可独立存在。
- 版本号由平台统一生成，反映版本先后顺序，但不表达"哪个版本更好"或"哪个版本是业务主版本"——业务主版本由"当前有效版本"字段显式指向。
- 版本一旦生成，其内部的印章样式引用、授权矩阵快照、证书绑定关系即冻结，不因后续组织变更、人员调整、证书更新而自动漂移。
- 版本变更必须记录变更类型：新建、样式更新、授权更新、证书更新、失效归档。

### 2.4 发布/回滚单元

印章资源的发布与回滚以"印章资源版本"为最小单元：

- **发布单元**：将某一版本设为"当前有效版本"，同时记录发布时间、发布人、发布原因。发布操作后，新发起的签章申请将消费新版本快照。
- **回滚单元**：将"当前有效版本"指向先前的某一历史版本，同时记录回滚时间、回滚人、回滚原因。回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在执行中的签章会话不受影响：已建立会话的签章申请继续消费会话建立时绑定的版本快照，不因发布/回滚操作而中途切换。
- 发布/回滚操作本身纳入印章资源审计事件，但不触发签章申请、会话、结果的级联变更。

### 2.5 审计最小单元

印章资源审计以"印章资源版本"为最小单元，必须记录以下事件类型：

- 印章资源创建、印章资源失效、印章资源归档
- 印章资源版本发布、版本回滚、版本冻结
- 印章样式变更、授权矩阵变更、证书绑定变更
- 印章使用准入判定（含拒绝记录）、印章使用人工申诉

审计事件必须携带：印章资源 ID、印章资源版本 ID、操作类型、操作人、操作时间、操作原因、关联的合同 ID（若适用）。

## 3. 印章样式与授权矩阵治理边界

### 3.1 印章样式治理边界

印章样式治理遵循以下边界：

- 印章样式文件统一进入文档中心治理，电子签章模块只持有样式文件引用，不保存样式文件副本。
- 印章样式的视觉呈现规则（如尺寸、位置、透明度、旋转角度）属于签章引擎执行参数，不在印章资源层固化，由签章会话根据实际场景动态计算或从签章方案配置中读取。
- 印章样式与印章资源版本是一对一绑定关系：每个印章资源版本最多引用一个有效样式文件版本；样式文件更新时，必须产生新的印章资源版本，而不是覆写旧版本的样式引用。
- 印章样式的有效性校验（如图片格式、尺寸范围、透明度通道）在样式文件进入文档中心时由文档中心负责，电子签章模块在绑定样式引用前只做引用可达性检查，不做样式内容级校验。

### 3.2 授权矩阵治理边界

授权矩阵治理遵循以下边界：

- 授权矩阵表达"谁可以在什么条件下使用这枚印章"，其输入来源是组织与权限底座，但快照固化在印章资源版本内部。
- 授权矩阵维度包括：组织范围、角色范围、人员范围、签署场景限制、生效时间窗口、签署金额上限（若适用）。
- 授权矩阵只决定"是否允许使用印章"，不决定"签署顺序、签署顺位、参与方展开规则"——后者由签章申请时的签署编排策略独立控制。
- 授权矩阵变更不实时生效于正在执行的签章会话：已建立的会话继续消费会话建立时的授权快照，新发起的申请才消费最新发布的版本。
- 授权矩阵的原始规则由组织与权限底座维护，印章资源只消费其快照，不反向写入组织与权限底座。

### 3.3 印章样式与授权矩阵的关系

印章样式与授权矩阵在印章资源版本内是平行关系，而非依赖关系：

- 一枚印章资源版本同时包含一个样式引用和一个授权矩阵快照。
- 允许只更新样式而不改变授权矩阵（产生新版本，授权矩阵复用上一版本的快照）。
- 允许只更新授权矩阵而不改变样式（产生新版本，样式引用复用上一版本的引用）。
- 样式与授权矩阵之间没有继承、覆盖或优先级关系。

## 4. 证书介质治理

### 4.1 治理对象

证书介质治理以"证书链聚合"为顶层治理对象，其内部由以下对象构成：

- **根证书**：平台信任的顶级证书颁发机构证书，由平台统一预置或由管理员在受控流程下导入，作为所有下级证书的有效性锚点。
- **签名证书**：与印章资源绑定的实体证书，用于为签章动作生成数字签名，包含证书主体、序列号、有效期、公钥信息、关联印章资源 ID。
- **时间戳证书**：用于为签章结果提供可信时间戳的证书，可由平台统一配置，也可按签章场景指定特定时间戳服务。
- **证书绑定关系**：表达印章资源与签名证书的绑定、签名证书与时间戳证书的信任路径、证书链验证顺序。

### 4.2 证书介质生命周期

#### 4.2.1 根证书生命周期

- **预置/导入**：根证书由平台统一预置或通过受控管理界面导入，导入时记录证书指纹、颁发者、有效期、导入人与导入时间。
- **生效**：根证书导入后默认为生效状态，作为证书链验证的信任锚点。
- **失效**：根证书到达有效期或被发现存在安全风险时，标记为失效，失效后不再作为新签章的信任锚点，但已完成的签章结果不因根证书失效而变为无效。
- **归档**：失效根证书经过存续期后归档，归档后不再参与任何证书链验证，只保留审计可达性。

#### 4.2.2 签名证书生命周期

- **申请/导入**：签名证书可通过平台向 CA 申请，或由管理员在受控流程下导入已有证书，导入时记录证书与印章资源的绑定关系。
- **绑定**：签名证书必须绑定到某一枚印章资源，一个印章资源在任意时刻最多绑定一张有效签名证书。
- **生效**：签名证书绑定后，经有效性校验通过（有效期、根证书信任链、密钥对可用性）进入生效状态。
- **续期**：签名证书到期前，可通过证书续期流程更新证书，续期后产生新的印章资源版本，旧版本在存续期内仍然有效。
- **吊销**：签名证书因密钥泄露或安全事件需要吊销时，标记吊销状态，吊销后该证书不再用于新签章，但已完成的签章结果不因证书吊销而自动变为无效——需要结合验签结论与业务规则判断历史签章的有效性。
- **解绑**：签名证书与印章资源解绑时，必须先将印章资源切换到新证书或标记为"证书待更新"状态，不允许印章资源在无有效签名证书的情况下继续承接新签章申请。

#### 4.2.3 时间戳证书生命周期

- **配置**：时间戳证书由平台统一配置，作为签章结果时间戳服务的信任锚点。
- **生效与失效**：时间戳证书按有效期管理，失效后自动切换到备用时间戳证书或时间戳服务。
- **场景覆盖**：不同签章场景可配置不同时间戳证书或服务端点，由签章方案在会话建立时确定，而不是在印章资源层固化。

### 4.3 证书介质与印章资源的边界

- 印章资源持有"当前绑定的签名证书 ID"与"证书绑定历史"，但不持有证书私钥或证书文件副本——私钥由签章引擎或密钥管理服务在运行时提供。
- 证书介质层负责证书链验证、有效期检查、吊销状态检查，但签章结果的业务有效性由电子签章模块综合签章状态、验签结果、回调结论共同判定。
- 证书更新（签名证书续期、根证书轮换）必须产生新的印章资源版本，确保正在执行的签章会话不因证书变更而中断。

## 5. 跨模块边界说明

### 5.1 与 `contract-core` 的边界

- **合同主档是业务真相源**：印章资源不持有合同业务状态，只通过 `contract_id` 挂接。
- **印章使用准入消费合同摘要**：签章申请受理时读取合同主档摘要（合同状态、审批结论、签署要求），但不将印章资源状态反向写入合同主档。
- **签章结果回写合同摘要**：签章完成后，只将稳定摘要（`signature_status`、签署时间、验签结论）回写合同主档，不将印章资源细节、证书链细节、样式信息写入合同主档。
- **印章资源不直接消费合同模板**：合同模板中的签章占位信息属于合同起草与审批阶段的业务规则，印章资源层不参与模板解析。

### 5.2 与 `document-center` 的边界

- **文档中心是文件真相源**：印章样式文件、签章结果稿、验签产物均由文档中心统一治理。
- **印章资源只持有样式文件引用**：印章样式文件进入文档中心后，印章资源只保存 `document_asset_id` / `document_version_id`，不保存样式文件副本，不负责样式文件的版本链管理。
- **签章结果稿回写文档中心**：签章执行完成后，结果稿由 `writeback-coordinator` 调用文档中心建立正式版本，印章资源不参与结果稿的生成与存储。
- **印章样式校验分工**：样式文件的格式校验、病毒扫描、内容安全检测由文档中心负责，印章资源层只做引用可达性检查。

### 5.3 与 `integration-hub` 的边界

- **集成边界负责外部签章服务适配**：若平台选择对接外部签章服务（如第三方 CA、时间戳服务、电子签章平台），适配逻辑由 `integration-hub` 统一承接。
- **印章资源不感知外部系统差异**：印章资源只持有标准化证书绑定关系与签章方案引用，外部系统的参数映射、协议转换、签名验签细节由 `integration-hub` 在适配层处理。
- **回调事件统一进入集成交换中心**：外部签章服务的回调、时间戳回执、验签回执统一由 `integration-hub` 接收、归一化后进入电子签章模块的回调事件处理链路。
- **证书导入/更新可经由集成边界**：根证书、时间戳证书的配置与更新可通过 `integration-hub` 管理的外部信任源同步，但最终收口到印章资源与证书介质治理层。

## 6. 最小可执行基线

### 6.1 配置结构

印章资源与证书介质配置以 `seal_resource_config` 为发布单元，最小结构如下：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `seal_resource_id` | string | 是 | 无 | 印章资源稳定主键 |
| `seal_version_id` | string | 是 | 无 | 本次发布的印章资源版本 |
| `owner_org_id` | string | 是 | 无 | 印章归属组织 |
| `style_document_asset_id` | string | 是 | 无 | 印章样式文件资产引用 |
| `style_document_version_id` | string | 是 | 无 | 印章样式文件版本引用 |
| `certificate_id` | string | 是 | 无 | 当前绑定签名证书 |
| `certificate_chain_id` | string | 是 | 无 | 证书链引用 |
| `timestamp_certificate_id` | string | 否 | 平台默认时间戳证书 | 时间戳证书引用 |
| `authorization_snapshot_ref` | string | 是 | 无 | 授权矩阵快照引用 |
| `effective_at` | datetime | 是 | 无 | 版本生效时间 |
| `expires_at` | datetime | 否 | 空 | 版本失效时间 |
| `config_status` | enum | 是 | `DRAFT` | `DRAFT` / `PUBLISHED` / `FROZEN` / `REVOKED` / `ARCHIVED` |

### 6.2 核心状态机

| 对象 | 状态 | 允许转换 | 触发条件 |
| --- | --- | --- | --- |
| 印章资源版本 | `DRAFT` | `PUBLISHED` / `ARCHIVED` | 管理员完成样式、授权、证书校验后发布；草稿废弃时归档 |
| 印章资源版本 | `PUBLISHED` | `FROZEN` / `REVOKED` | 被签章会话消费后冻结；证书吊销或授权失效时撤销 |
| 印章资源版本 | `FROZEN` | `ARCHIVED` | 存续期结束且无活动会话引用 |
| 证书绑定 | `PENDING_VERIFY` | `ACTIVE` / `INVALID` | 证书链、有效期、吊销状态校验完成 |
| 证书绑定 | `ACTIVE` | `REVOKED` / `EXPIRED` | CA 吊销、管理员撤销或有效期到期 |

### 6.3 关键算法伪代码

```text
resolve_seal_snapshot(request):
  seal = load_seal_resource(request.seal_resource_id)
  version = load_current_version(seal.current_version_id)
  assert version.config_status in [PUBLISHED, FROZEN]
  assert request.started_at between version.effective_at and version.expires_at_or_infinity
  assert can_use_seal(request.actor, version.authorization_snapshot_ref, request.contract_snapshot)
  cert = load_certificate(version.certificate_id)
  assert cert.status == ACTIVE and cert.valid_from <= now <= cert.valid_to
  assert verify_certificate_chain(cert, version.certificate_chain_id)
  return freeze_snapshot(version, cert, request.signature_request_id)
```

### 6.5 异常分流表

| 异常 | 分流结果 | 处理动作 |
| --- | --- | --- |
| 印章资源不存在或未发布 | 阻断签章准入 | 申请进入 `ADMISSION_REJECTED`，记录审计 |
| 授权矩阵未命中 | 阻断签章准入 | 返回权限拒绝原因，允许人工申诉 |
| 样式文件引用不可达 | 转人工或补偿 | 创建文档引用修复任务，不启动会话 |
| 证书过期或吊销 | 阻断新签章 | 标记证书绑定异常，要求更换证书版本 |
| 时间戳证书不可用 | 降级或阻断 | 若策略允许备用服务则切换，否则进入人工确认 |

### 6.6 与详细设计字段映射

| 专项字段 | 详细设计字段 / 对象 | 映射说明 |
| --- | --- | --- |
| `seal_resource_id` | `es_signature_request.seal_scheme_id` | 申请受理时保存所选印章方案或印章资源引用 |
| `seal_version_id` | `es_signature_session.runtime_checkpoint` | 会话建立时固化印章版本快照摘要 |
| `style_document_asset_id` / `style_document_version_id` | `es_request_document_binding.document_asset_id` / `document_version_id` | 样式文件仍由文档中心治理，签章域保存引用 |
| `certificate_id` / `certificate_chain_id` | `es_signature_result.verification_summary_json` | 结果登记时保存证书与验签摘要，不保存私钥 |
| `authorization_snapshot_ref` | `es_signature_audit_event.before_snapshot_ref` | 准入审计记录授权快照引用 |

## 7. 治理原则总结

- 印章资源以"版本快照"为核心治理模式，确保正在执行的签章链路不因资源变更而中断。
- 证书介质以"信任链"为核心治理模式，确保签章结果的可验证性与长期有效性。
- 印章样式、授权矩阵、证书绑定三者平行治理，变更互不影响，但共同构成印章资源版本的完整快照。
- 跨模块协作以"引用而非持有、消费快照而非实时查询"为原则，确保各模块边界清晰、责任明确、审计可追溯。
