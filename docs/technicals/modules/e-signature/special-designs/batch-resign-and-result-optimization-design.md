# 批量签章与结果优化专项设计

## 1. 文档定位

本文档是 `CMP` 电子签章子模块的批量签章与结果优化治理设计，用于明确批量补签策略、结果稿生成性能优化、历史数据迁移策略在平台内的治理对象、归属关系、责任边界与跨模块协作约束。

本文档不承担以下内容：

- 不输出 API 路径、请求样例、错误码全集
- 不输出表级 DDL、索引细节、字段长度与枚举值全集
- 不输出具体批处理、性能调优、并发控制的生产代码
- 不输出实现排期、联调顺序、负责人拆分
- 不输出运维手册、部署清单、监控配置

## 2. 批量补签策略治理

### 2.1 治理对象

批量补签策略治理以"补签策略配置"为最小治理单元，其内部由以下对象构成：

- **补签策略主体**：表达一种补签场景的类型、适用条件、生效与失效时间、策略状态，包含策略名称、补签类型（主动补签、被动补签、批量补签、异常补签）、关联合同类型、关联业务流程。
- **补签范围规则**：表达在该策略下如何确定补签范围，包括范围来源（合同主档查询、审批流程追溯、签章结果分析、人工指定）、范围过滤条件（合同状态、签署状态、时间窗口、失败原因分类）、范围去重规则（同一合同不重复补签、同一参与方不重复补签）。
- **补签优先级规则**：表达补签任务的执行优先级，包括优先级维度（合同重要性、签署截止时间、业务紧急程度、失败时长）、优先级计算方式（固定优先级、动态评分、人工指定）、优先级调整规则（超时自动升级、人工干预调整）。
- **补签执行策略**：表达补签任务如何执行，包括执行方式（单独完成、批量完成、并行完成）、执行约束（最大批量大小、最大并行数、执行超时时间）、执行回退规则（部分失败处理、整体失败处理、人工介入条件）。

### 2.2 稳定锚点

批量补签策略的稳定锚点设计如下：

- **补签策略 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **补签策略编码**：面向用户与管理的稳定业务标识，允许按组织规则自定义，但在平台内作为辅助索引存在，不替代补签策略 ID 作为真相键。
- **当前有效版本 ID**：每一份补签策略在任意时刻有且只有一个"当前有效版本"，所有新发起的补签任务消费该版本的快照，而不是实时计算最新版本。
- **补签类型枚举**：作为平台内稳定分类，包括主动补签、被动补签、批量补签、异常补签等，不因业务发展而随意扩展枚举值。

### 2.3 版本归属

补签策略版本归属遵循以下原则：

- 每一版补签策略必须归属到某一补签策略主体，不可独立存在。
- 版本号由平台统一生成，反映版本先后顺序，但不表达"哪个版本更好"或"哪个版本是业务主版本"——业务主版本由"当前有效版本"字段显式指向。
- 版本一旦生成，其内部的补签范围规则、补签优先级规则、补签执行策略即冻结，不因后续合同变更、组织调整、业务流程变更而自动漂移。
- 版本变更必须记录变更类型：新建、范围规则更新、优先级规则更新、执行策略更新、失效归档。

### 2.4 最小治理单元

批量补签的最小治理单元是"补签策略版本"：

- **发布单元**：将某一版本设为"当前有效版本"，同时记录发布时间、发布人、发布原因。发布操作后，新发起的补签任务将消费新版本快照。
- **回滚单元**：将"当前有效版本"指向先前的某一历史版本，同时记录回滚时间、回滚人、回滚原因。回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在执行中的补签任务不受影响：已建立的补签任务继续消费任务建立时绑定的补签策略快照，不因发布/回滚操作而中途切换。
- 发布/回滚操作本身纳入补签策略审计事件，但不触发补签任务、签章申请、会话的级联变更。

### 2.5 审计最小单元

批量补签策略审计以"补签策略版本"为最小单元，必须记录以下事件类型：

- 补签策略创建、补签策略失效、补签策略归档
- 补签策略版本发布、版本回滚、版本冻结
- 补签范围规则变更、补签优先级规则变更、补签执行策略变更
- 补签任务创建、补签任务执行、补签任务完成、补签任务失败
- 补签策略使用准入判定（含拒绝记录）、补签策略使用人工申诉

审计事件必须携带：补签策略 ID、补签策略版本 ID、操作类型、操作人、操作时间、操作原因、关联的合同 ID（若适用）、关联的补签任务 ID（若适用）。

## 3. 结果稿生成性能优化治理

### 3.1 治理边界

结果稿生成性能优化治理以"结果稿生成策略"为治理对象，明确以下边界：

- **优化范围边界**：只治理结果稿生成的策略选择、资源分配、任务拆分、并行控制，不治理签章引擎内部的渲染算法、加密算法、签名算法。
- **与签章引擎适配层边界**：结果稿生成策略只消费签章引擎适配层（见 `signature-engine-adapter-design.md`）提供的"引擎能力声明"，根据能力声明选择生成策略（如某引擎支持批量签，则优先使用批量生成策略；如不支持，则使用单文件串行生成策略）。
- **与文档中心边界**：结果稿生成完成后，由签章模块负责将结果稿注册到文档中心（见 `document-binding-registry` 组件），不将生成过程中的临时文件、中间产物传递到文档中心。
- **与合同主档边界**：结果稿生成完成后，由 `writeback-coordinator` 组件负责将结果稿引用回写到合同主档，结果稿生成策略本身不直接操作合同主档。

### 3.2 与签章引擎适配层的接口约束

结果稿生成策略与签章引擎适配层的接口约束如下：

- **能力查询约束**：结果稿生成策略在任务建立时，通过签章引擎适配层查询目标引擎的"引擎能力声明"，包括是否支持批量签、是否支持异步生成、是否支持结果回调、最大批量大小、最大并发数。
- **参数传递约束**：结果稿生成策略将"渲染上下文"集合（包含印章引用、坐标信息、证书绑定、渲染参数）传递给签章引擎适配层，由适配层负责参数映射（见 `signature-engine-adapter-design.md` 第 3 节），结果稿生成策略不感知具体引擎的参数格式。
- **结果回收约束**：结果稿生成策略从签章引擎适配层回收生成结果（包含结果稿文档引用、验签证明引用、生成状态），适配层负责将引擎原始响应归一化为平台统一的结果格式，结果稿生成策略不解析引擎原始响应。
- **异常处理约束**：结果稿生成策略消费签章引擎适配层归一化的错误分类（如引擎不可达、参数非法、证书无效、签名失败、验签失败），根据错误分类决定重试、降级、人工介入策略，不根据引擎原始错误码做分支判断。

### 3.3 与文档中心的接口约束

结果稿生成策略与文档中心的接口约束如下：

- **版本链归属约束**：结果稿必须注册到文档中心，成为合同主档文档版本链上的一个正式版本，结果稿生成策略不直接操作文档中心的版本链，而是通过 `document-binding-registry` 组件完成注册。
- **文件引用约束**：结果稿生成策略只持有文档中心返回的 `document_asset_id` / `document_version_id`，不持有文件二进制、存储路径、访问 URL，所有后续引用都通过文档中心接口查询。
- **结果稿元数据约束**：结果稿生成策略负责生成结果稿的元数据（包含签章参与方列表、签章时间、签章结果、验签状态），元数据随结果稿一起注册到文档中心，不单独存储在签章模块外部。
- **生成过程临时文件约束**：结果稿生成过程中的临时文件、中间产物由签章模块内部管理，生成完成后及时清理，不传递到文档中心，不成为文档中心的正式版本。

### 3.4 性能优化策略治理

结果稿生成性能优化策略以"生成策略配置"为最小治理单元：

- **策略类型**：包括单文件串行生成、单文件并行生成、批量打包生成、异步回调生成，策略类型由"引擎能力声明"和"业务场景要求"共同决定。
- **资源分配规则**：表达生成任务可使用的资源上限，包括最大并发数、单个任务超时时间、批量任务最大批量大小、队列优先级。
- **任务拆分规则**：表达如何将大批量生成任务拆分为多个子任务，包括按合同拆分、按参与方拆分、按文件页拆分、按批次拆分。
- **容错降级规则**：表达生成失败时的降级策略，包括重试次数、重试间隔、降级方案（如从批量生成降级为单文件生成）、人工介入条件。

## 4. 历史数据迁移策略治理

### 4.1 治理对象

历史数据迁移策略治理以"迁移方案"为最小治理单元，其内部由以下对象构成：

- **迁移方案主体**：表达一次历史数据迁移的类型、作用范围、生效与失效时间、方案状态，包含方案名称、迁移类型（签章结果迁移、文档引用迁移、审计事件迁移）、关联合同类型、关联时间范围。
- **数据范围规则**：表达在该方案下如何确定迁移数据范围，包括范围来源（合同主档查询、现有签章记录查询、文档中心查询）、范围过滤条件（合同状态、签署状态、时间窗口、数据完整性检查）、范围去重规则（已迁移数据不重复迁移、重复数据合并规则）。
- **数据转换规则**：表达历史数据如何转换为平台标准格式，包括字段映射（历史字段到当前字段的映射关系）、默认值填充（历史数据缺失字段的默认值）、数据校验规则（迁移前校验、迁移后校验、不一致数据处理）。
- **迁移执行策略**：表达迁移任务如何执行，包括执行方式（全量迁移、增量迁移、分批迁移）、执行时序（与合同主档迁移的时序、与文档中心迁移的时序）、执行回退规则（迁移失败回退、数据不一致回退、人工介入条件）。

### 4.2 责任边界

历史数据迁移策略的责任边界划分如下：

- **签章模块责任**：负责迁移签章模块内部数据（签章申请、会话、参与方、结果、审计事件），负责将历史签章结果与文档中心建立正确的引用关系，负责迁移后的数据一致性校验。
- **contract-core 责任**：负责迁移合同主档中与签章相关的基础数据（合同签署状态、签署时间、签署参与方基本信息），不负责迁移签章模块的专业状态（会话状态、结果详情、验签证明）。
- **document-center 责任**：负责迁移历史签章结果稿文件、验签证明文件的存储与版本链建立，不负责迁移签章模块的引用关系表（引用关系由签章模块在迁移过程中建立）。
- **迁移协调责任**：跨模块迁移由各自模块分别执行，签章模块负责协调整体迁移时序（先迁移合同主档基础数据，再迁移文档中心文件，最后迁移签章模块内部数据并建立引用关系），不代替其他模块执行迁移。

### 4.3 与 contract-core 的数据边界

历史数据迁移与 contract-core 的数据边界如下：

- **合同主档数据边界**：合同主档是业务真相源，迁移过程中只从合同主档读取历史签署状态（如已签署、未签署、签署中），不将签章模块的内部状态（如会话状态、参与方详情）写回合同主档。
- **合同主档摘要边界**：迁移完成后，签章模块通过 `writeback-coordinator` 向合同主档回写稳定的签章摘要（包含签章状态、最后签章时间、签章参与方数量），不回写签章过程全量状态树。
- **合同主档迁移时序边界**：历史数据迁移时，contract-core 先完成合同主档基础数据迁移，签章模块再基于迁移后的合同主档建立签章数据与合同主档的关联关系，不假设历史合同 ID 在迁移前后保持不变（通过合同编号等业务标识建立映射关系）。
- **合同主档回写幂等边界**：迁移过程中向合同主档回写摘要时，必须遵循幂等约束（同一合同只回写一次迁移后的摘要，不因多次迁移尝试而重复回写）。

### 4.4 与 document-center 的数据边界

历史数据迁移与 document-center 的数据边界如下：

- **文档中心文件边界**：文档中心是文件真相源，迁移过程中只将历史签章结果稿文件注册到文档中心，不将签章模块的内部引用关系（如哪个结果对应哪个合同）存储在文档中心。
- **文档版本链边界**：历史签章结果稿注册到文档中心后，必须成为对应合同文档版本链上的正式版本，签章模块只持有 `document_asset_id` / `document_version_id`，不持有文件二进制或存储路径。
- **文档中心迁移时序边界**：历史数据迁移时，document-center 先完成历史文件迁移并生成新的版本 ID，签章模块再基于迁移后的版本 ID 建立签章结果与文档中心的引用关系，不假设历史文件 ID 在迁移前后保持不变（通过文件哈希、合同编号、签署时间等组合标识建立映射关系）。
- **文档引用一致性边界**：迁移完成后，签章模块内的结果稿引用必须与文档中心的实际版本链一致，不一致时以文档中心已建立的正式版本链为准，签章模块修正引用关系，不强迫文档中心修改版本链。

### 4.5 审计最小单元

历史数据迁移审计以"迁移方案"为最小单元，必须记录以下事件类型：

- 迁移方案创建、迁移方案失效、迁移方案归档
- 迁移任务启动、迁移任务完成、迁移任务失败、迁移任务回退
- 数据范围确认、数据转换执行、数据校验执行
- 与 contract-core 的数据对齐、与 document-center 的数据对齐
- 迁移不一致处理、迁移人工介入、迁移后数据修复

审计事件必须携带：迁移方案 ID、迁移任务 ID、操作类型、操作人、操作时间、操作原因、关联的合同 ID（若适用）、关联的文档 ID（若适用）、数据不一致详情（若适用）。

## 5. 最小可执行基线

### 5.1 配置结构

批量补签与结果优化配置以 `batch_signature_policy_config` 为发布单元，最小结构如下：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `resign_policy_id` | string | 是 | 无 | 补签策略主键 |
| `resign_policy_version_id` | string | 是 | 无 | 补签策略版本 |
| `scope_filter` | object | 是 | 无 | 合同状态、签章状态、失败原因、时间窗口过滤条件 |
| `dedupe_key_rule` | string | 是 | `contract_id + signer_id + resign_reason` | 补签任务去重键 |
| `priority_rule` | object | 是 | 固定中优先级 | 优先级计算规则 |
| `max_batch_size` | integer | 是 | `100` | 单批最大数量 |
| `max_parallel_count` | integer | 是 | `5` | 最大并发数 |
| `fallback_strategy` | enum | 是 | `SPLIT_TO_SINGLE` | 批量失败后的降级策略 |
| `migration_mapping_version` | string | 否 | 空 | 历史迁移字段映射版本 |

### 5.2 核心状态机

| 对象 | 状态 | 允许转换 | 触发条件 |
| --- | --- | --- | --- |
| 批量补签任务 | `PENDING` | `SCANNING` / `CANCELLED` | 任务创建或撤销 |
| 批量补签任务 | `SCANNING` | `READY` / `FAILED` | 范围扫描完成或过滤条件异常 |
| 批量补签任务 | `READY` | `RUNNING` / `CANCELLED` | 达到执行窗口或人工撤销 |
| 批量补签任务 | `RUNNING` | `PARTIAL_FAILED` / `COMPLETED` / `FAILED` | 子任务执行完成 |
| 结果优化任务 | `PENDING` | `GENERATING` / `FAILED` | 引擎能力匹配或上下文异常 |
| 结果优化任务 | `GENERATING` | `WRITEBACK_PENDING` / `FAILED` | 结果稿生成完成或失败 |

### 5.3 关键算法伪代码

```text
build_batch_resign_task(policy):
  candidates = query_contracts(policy.scope_filter)
  normalized = filter_by_signature_status(candidates, policy.allowed_signature_status)
  unique_items = deduplicate(normalized, policy.dedupe_key_rule)
  scored_items = sort_by_priority(unique_items, policy.priority_rule)
  batches = chunk(scored_items, policy.max_batch_size)
  for batch in batches:
    submit_batch(batch, max_parallel_count=policy.max_parallel_count)
  return task_summary(batches)
```

### 5.5 异常分流表

| 异常 | 分流结果 | 处理动作 |
| --- | --- | --- |
| 范围过滤条件为空 | 阻断任务创建 | 要求补充时间窗口或合同范围 |
| 同一合同存在活动签章申请 | 跳过并记录 | 不重复发起补签，进入冲突清单 |
| 批量生成失败 | 降级执行 | 按 `fallback_strategy` 拆成单文件任务 |
| 部分子任务失败 | 部分完成 | 成功项继续回写，失败项进入人工或重试 |
| 历史文件与文档中心无法匹配 | 迁移人工处理 | 生成迁移不一致清单 |

## 6. 与 implementation-batch-plan.md 的批次规划承接关系

根据 [`implementation-batch-plan.md`](../../implementation-batch-plan.md) 的批次规划，`e-signature` 模块被安排在**第三批：依赖主链路真相源的业务能力**中实施。

本专项设计与批次规划的承接关系如下：

- **第三批实施主体**：`e-signature`、`encrypted-document`、`contract-lifecycle` 在第三批并行展开，但必须建立在合同状态、审批状态和文档版本真相已稳定的前提上。
- **本专项的批次归属**：批量签章与结果优化专项设计属于 `e-signature` 模块的治理能力建设，与第三批实施主体保持一致，不参与第一批（底座主线）和第二批（合同核心主链路）的实施。
- **与第二批的关键依赖**：
  - 依赖 `contract-core` 在第二批完成一轮主链路收口，至少明确合同状态流转与挂载点边界。
  - 依赖 `document-center` 在第二批完成一轮主链路收口，至少明确文档版本链、文件引用边界。
  - 依赖 `workflow-engine` 在第二批完成一轮主链路收口，至少明确审批结果回写边界。
- **本专项的启动条件**：
  - 第二批主链路收口完成，合同主档、文档主档、审批回写链路已稳定。
  - `contract-core`、`document-center`、`workflow-engine` 的 API 已冻结，数据边界已明确。
  - 签章引擎适配层（见 `signature-engine-adapter-design.md`）已完成基础能力建设，引擎能力声明已可查询。
- **本专项的交付节奏**：
  - 先完成批量补签策略治理（作为签章模块内部治理能力）。
  - 再完成结果稿生成性能优化治理（作为与签章引擎适配层、文档中心的协作约束）。
  - 最后完成历史数据迁移策略治理（作为与 contract-core、document-center 的数据边界约束）。
  - 三项治理均不涉及具体实现代码，只明确治理边界、责任划分、协作约束，为后续实现阶段提供稳定上游约束。

## 7. 跨专项设计引用关系

本专项设计在 `e-signature` 模块专项设计体系中的位置和引用关系如下：

- **印章资源与证书介质专项设计**（[seal-resource-and-certificate-design.md](seal-resource-and-certificate-design.md)）：本专项的批量补签策略、结果稿生成策略均消费印章资源版本快照和证书绑定关系，不实时查询印章资源最新状态。
- **签章坐标与渲染专项设计**（[signature-coordinate-and-rendering-design.md](signature-coordinate-and-rendering-design.md)）：本专项的结果稿生成策略消费坐标方案版本快照，不实时查询坐标方案最新状态；骑缝签等特殊渲染策略由该专项定义，本专项只负责生成任务的编排。
- **签署参与方编排专项设计**（[signing-party-orchestration-design.md](signing-party-orchestration-design.md)）：本专项的批量补签策略消费签署场景策略版本快照，根据场景策略确定补签参与方展开规则和顺序控制规则。
- **签章引擎适配专项设计**（[signature-engine-adapter-design.md](signature-engine-adapter-design.md)）：本专项的结果稿生成策略通过签章引擎适配层查询引擎能力声明、传递渲染上下文、回收生成结果，不直接操作具体签章引擎。

以上引用关系表明，本专项设计是 `e-signature` 模块专项设计体系的收口环节，其治理边界建立在前四个专项设计的基础之上，共同构成完整的电子签章子模块治理能力。
