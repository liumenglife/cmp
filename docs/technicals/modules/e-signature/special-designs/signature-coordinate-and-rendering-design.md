# 签章坐标与渲染专项设计

## 1. 文档定位

本文档是 `CMP` 电子签章子模块的签章坐标、盖章渲染、文件合成与验签算法治理设计，用于明确签章坐标体系、渲染执行边界、文件合成责任、验签算法策略与证明文件生成在平台内的治理对象、归属关系、责任边界与跨模块协作约束。

本文档不承担以下内容：

- 不输出 API 路径、请求样例、错误码全集
- 不输出表级 DDL、索引细节、字段长度与枚举值全集
- 不输出具体坐标计算、渲染实现、加密算法的生产代码
- 不输出实现排期、联调顺序、负责人拆分
- 不输出运维手册、部署清单、监控配置

## 2. 签章坐标治理

### 2.1 治理对象

签章坐标治理以"签章坐标方案"为最小治理单元，其内部由以下对象构成：

- **坐标方案主体**：表达一份合同模板或一份待签文件在签署时的签章位置规划，包含方案名称、适用场景、关联印章资源、生效与失效时间、方案状态。
- **坐标锚点定义**：表达签章在文档中的定位依据，分为绝对坐标锚点（固定页码+固定坐标）和相对坐标锚点（基于关键字、基于表单字段、基于文档结构元素）。
- **坐标版本快照**：表达坐标方案在特定时间窗口内的有效快照，每次变更锚点定义、适用条件或关联印章时，产生新版本而不是覆写旧版本。
- **坐标绑定关系**：表达某一签章申请最终消费的是哪个坐标方案版本，由签章申请准入时快照固化，不因后续坐标方案变更而自动漂移。

### 2.2 稳定锚点

签章坐标的稳定锚点设计如下：

- **坐标方案 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **坐标方案编号**：面向用户与管理的稳定业务标识，允许按组织规则自定义，但在平台内作为辅助索引存在，不替代坐标方案 ID 作为真相键。
- **当前有效版本 ID**：每一份坐标方案在任意时刻有且只有一个"当前有效版本"，所有新发起的签章申请消费该版本的快照，而不是实时计算最新版本。
- **文档页码+坐标对**：表达签章在文档中的具体落点，属于坐标版本快照的内部数据，不单独作为跨模块关联键。

### 2.3 版本归属

坐标方案版本归属遵循以下原则：

- 每一版坐标方案必须归属到某一坐标方案主体，不可独立存在。
- 版本号由平台统一生成，反映版本先后顺序，但不表达"哪个版本更好"或"哪个版本是业务主版本"——业务主版本由"当前有效版本"字段显式指向。
- 版本一旦生成，其内部的锚点定义、适用条件、关联印章列表即冻结，不因后续模板调整、印章变更而自动漂移。
- 版本变更必须记录变更类型：新建、锚点更新、适用条件更新、关联印章更新、失效归档。

### 2.4 最小治理单元

签章坐标的最小治理单元是"坐标方案版本"：

- **发布单元**：将某一版本设为"当前有效版本"，同时记录发布时间、发布人、发布原因。发布操作后，新发起的签章申请将消费新版本快照。
- **回滚单元**：将"当前有效版本"指向先前的某一历史版本，同时记录回滚时间、回滚人、回滚原因。回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在执行中的签章会话不受影响：已建立会话的签章申请继续消费会话建立时绑定的坐标快照，不因发布/回滚操作而中途切换。
- 发布/回滚操作本身纳入坐标方案审计事件，但不触发签章申请、会话、结果的级联变更。

### 2.5 审计最小单元

签章坐标审计以"坐标方案版本"为最小单元，必须记录以下事件类型：

- 坐标方案创建、坐标方案失效、坐标方案归档
- 坐标方案版本发布、版本回滚、版本冻结
- 锚点定义变更、适用条件变更、关联印章变更
- 坐标使用准入判定（含拒绝记录）、坐标使用人工申诉

审计事件必须携带：坐标方案 ID、坐标方案版本 ID、操作类型、操作人、操作时间、操作原因、关联的合同 ID（若适用）。

## 3. 盖章渲染治理边界

### 3.1 渲染治理边界

盖章渲染治理遵循以下边界：

- 渲染执行由 `signature-engine-adapter` 负责，电子签章模块只持有渲染参数引用与渲染结果引用，不保存渲染中间状态或渲染缓存。
- 渲染参数包括：印章样式引用（来自印章资源版本）、签章坐标（来自坐标方案版本）、渲染尺寸、透明度、旋转角度等视觉呈现规则。
- 渲染结果以文件形式进入文档中心治理，电子签章模块只保存 `document_asset_id` / `document_version_id` 引用，不保存渲染结果文件副本。
- 渲染失败的分类与重试策略由电子签章模块定义，但具体渲染执行与错误诊断由 `signature-engine-adapter` 负责。

### 3.2 与印章样式的接口约束

盖章渲染与印章样式的接口约束如下：

- 印章样式文件统一托管在文档中心，渲染时由 `signature-engine-adapter` 通过文档中心引用获取样式文件内容。
- 电子签章模块在建立签章会话时，将印章资源版本中的样式引用、坐标方案版本中的坐标信息、签署配置中的渲染参数打包为"渲染上下文"，传递给 `signature-engine-adapter`。
- 印章样式的视觉呈现规则（如尺寸、位置、透明度、旋转角度）在渲染上下文中显式传递，不在印章资源层固化，由签章方案配置或签署场景动态确定。
- 渲染上下文一旦传递给 `signature-engine-adapter`，即视为冻结快照，不因后续印章样式更新、坐标方案变更而自动漂移。

### 3.3 与文档中心的接口约束

盖章渲染与文档中心的接口约束如下：

- 渲染输入稿由文档中心提供引用，渲染结果稿由 `signature-engine-adapter` 生成后通过文档中心接口建立正式版本。
- 电子签章模块不直接操作文件二进制，渲染输入通过文档中心引用传递，渲染输出通过文档中心版本链回收。
- 渲染结果的版本关系由文档中心治理：结果稿是哪个输入稿的签章版本、哪个样式文件的渲染产物，由文档中心的版本链和元数据表达。
- 渲染失败后的重试不直接操作文档中心版本链，由电子签章模块通过重新发起渲染请求（携带相同渲染上下文）实现，文档中心按需创建新版本。

## 4. 文件合成与验签算法治理

### 4.1 文件合成治理边界

文件合成治理遵循以下边界：

- 文件合成由 `signature-engine-adapter` 负责，电子签章模块只定义合成策略（如是否合成附件、是否保留原文批注、是否扁平化表单字段）并消费合成结果引用。
- 合成策略在签章申请准入时快照固化，不因后续策略变更而自动漂移。
- 合成结果以文件形式进入文档中心治理，电子签章模块只保存合成结果引用，不保存合成中间文件。
- 合成失败的分类与重试策略由电子签章模块定义，但具体合成执行与错误诊断由 `signature-engine-adapter` 负责。

### 4.2 验签算法治理对象

验签算法治理以"验签策略"为顶层治理对象，其内部由以下对象构成：

- **验签策略主体**：表达在什么条件下使用哪种验签算法、哪种证书链验证规则、哪种时间戳验证规则，包含策略名称、适用场景、生效与失效时间、策略状态。
- **算法选择规则**：表达根据签章类型、证书类型、时间戳类型选择具体验签算法的决策逻辑，如 RSA-SHA256、SM2-SM3 等。
- **证书链验证规则**：表达如何验证签名证书的有效性，包括根证书信任检查、证书有效期检查、证书吊销状态检查（CRL/OCSP）、证书路径构建规则。
- **时间戳验证规则**：表达如何验证时间戳的有效性，包括时间戳证书信任检查、时间戳有效期检查、时间戳与签名时间的逻辑关系检查。

### 4.3 验签算法责任边界

验签算法的责任边界如下：

- **电子签章模块责任**：定义验签策略、记录验签结论（`PASSED`/`FAILED`/`WARNING`）、保存验签摘要与失败原因、回写验签状态到合同主档与签章摘要。
- **`signature-engine-adapter` 责任**：执行具体验签计算、执行证书链验证、执行时间戳验证、返回原始验签结果与详细验证报告。
- **印章资源层责任**：提供签名证书绑定关系、证书链信任锚点（根证书），作为验签算法执行时的输入，但不参与验签计算过程。
- **文档中心责任**：提供待验签文件内容、提供签名元数据（如签名时间、签名者信息），但不参与验签算法执行。

### 4.4 验签结果消费约束

验签结果的消费遵循以下约束：

- 验签结果以摘要形式保存在 `es_signature_result` 中，包括 `verification_status`、`verification_summary_json`。
- 验签详细报告（如证书链验证每一步的结果、时间戳验证细节）由 `signature-engine-adapter` 返回后，电子签章模块只保存摘要，不保存完整报告原文。
- 验签失败时，电子签章模块负责记录失败原因分类（证书过期、证书吊销、时间戳无效、签名被篡改等），但具体的失败诊断由 `signature-engine-adapter` 负责。
- 合同主档、归档、搜索、AI 只消费验签摘要（`verification_status`），不消费验签详细报告或原始验签数据。

## 5. 证明文件治理

### 5.1 治理对象

证明文件治理以"证明文件生成策略"为顶层治理对象，其内部由以下对象构成：

- **证明文件模板**：表达验签证明、签署证明、归档证明的文档结构与内容要素，模板文件统一托管在文档中心，电子签章模块只保存模板引用。
- **证明生成规则**：表达在什么条件下生成哪种证明文件、包含哪些签章结论、包含哪些验签结论、包含哪些合同摘要信息。
- **证明消费渠道**：表达证明文件的使用场景，如供用户下载、供归档系统消费、供第三方验证平台查询。

### 5.2 消费约束

证明文件的消费遵循以下约束：

- 证明文件生成由 `signature-engine-adapter` 或专用证明生成服务负责，电子签章模块只持有证明文件引用与生成策略。
- 证明文件以文件形式进入文档中心治理，电子签章模块只保存 `document_asset_id` / `document_version_id` 引用，不保存证明文件副本。
- 证明文件的生成时机由电子签章模块控制：通常在签章完成且验签通过后生成，或在用户/系统主动请求时生成。
- 证明文件的消费方（如归档系统、第三方验证平台）只通过文档中心引用获取证明文件，电子签章模块不提供直接的文件流式传输，除非用户下载场景。

## 6. 跨模块边界说明

### 6.1 与 `seal-resource-and-certificate-design.md` 的边界

- **印章资源层**负责印章样式引用、证书绑定关系、授权矩阵快照，本文档负责消费这些快照进行渲染与验签。
- **印章资源版本**是渲染上下文与验签策略的输入来源之一，但渲染执行与验签执行不在印章资源层。
- **印章样式的视觉呈现规则**不在印章资源层固化，由本文档定义的渲染参数动态确定，印章资源只提供样式文件引用。
- 两份文档共同构成签章执行的完整输入：印章资源层提供"用什么签"，本文档提供"签在哪里、怎么签、签完怎么验"。

### 6.2 与 `document-center` 的边界

- **文档中心是文件真相源**：渲染输入稿、渲染结果稿、验签证明文件均由文档中心统一治理。
- **电子签章模块只持有文件引用**：签章坐标与渲染模块不保存文件二进制，所有文件操作通过文档中心接口完成。
- **渲染与验签不直接操作文档中心版本链**：渲染上下文通过文档引用传递，渲染结果由 `signature-engine-adapter` 调用文档中心接口建立版本。
- **文档中心负责文件级校验**：文件格式校验、病毒扫描、内容安全检测由文档中心负责，签章坐标与渲染模块只做引用可达性检查。

### 6.3 与 `signature-engine-adapter` 的边界

- **`signature-engine-adapter` 是签章执行层**：负责具体渲染、合成、验签、证明生成的计算工作。
- **电子签章模块是签章治理层**：负责定义渲染参数、坐标方案、验签策略、证明生成规则，并消费执行结果。
- **接口契约**：电子签章模块通过标准化接口（渲染上下文、验签请求、证明生成请求）调用 `signature-engine-adapter`，`signature-engine-adapter` 返回标准化结果（渲染结果引用、验签结论、证明文件引用）。
- **错误处理分工**：`signature-engine-adapter` 负责诊断执行错误的具体内容（如渲染引擎异常、证书验证失败详情），电子签章模块负责根据错误类型执行重试策略或进入人工介入流程。

## 7. 最小可执行基线

### 7.1 配置结构

坐标与渲染配置以 `rendering_context_config` 为会话冻结单元，最小结构如下：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `coordinate_scheme_id` | string | 是 | 无 | 坐标方案主键 |
| `coordinate_version_id` | string | 是 | 无 | 坐标方案版本 |
| `anchor_type` | enum | 是 | `ABSOLUTE` | `ABSOLUTE` / `KEYWORD` / `FORM_FIELD` / `STRUCTURE_NODE` |
| `page_no` | integer | 条件必填 | 无 | 绝对坐标时必填 |
| `x` / `y` | decimal | 条件必填 | 无 | 页面坐标，左下角为原点 |
| `keyword` | string | 条件必填 | 无 | 关键字锚点文本 |
| `offset_x` / `offset_y` | decimal | 否 | `0` | 相对锚点偏移 |
| `seal_width` / `seal_height` | decimal | 是 | 按印章样式默认值 | 渲染尺寸 |
| `opacity` | decimal | 否 | `1.0` | 透明度 |
| `rotation` | decimal | 否 | `0` | 旋转角度 |
| `verification_policy_id` | string | 是 | 平台默认验签策略 | 验签策略引用 |

### 7.2 核心状态机

| 对象 | 状态 | 允许转换 | 触发条件 |
| --- | --- | --- | --- |
| 坐标方案版本 | `DRAFT` | `PUBLISHED` / `ARCHIVED` | 坐标校验通过后发布；废弃时归档 |
| 坐标方案版本 | `PUBLISHED` | `FROZEN` / `REVOKED` | 被会话消费后冻结；模板变更导致不可用时撤销 |
| 渲染任务 | `PENDING` | `RENDERING` / `FAILED` | 渲染上下文生成完成或输入引用异常 |
| 渲染任务 | `RENDERING` | `RENDERED` / `FAILED` | 引擎返回结果或异常 |
| 验签任务 | `PENDING` | `PASSED` / `FAILED` / `WARNING` | 引擎返回验签结论 |

### 7.3 关键算法伪代码

```text
resolve_render_position(document, anchor):
  if anchor.anchor_type == ABSOLUTE:
    assert page_exists(document, anchor.page_no)
    return point(anchor.page_no, anchor.x, anchor.y)
  if anchor.anchor_type == KEYWORD:
    hits = find_keyword(document.text_layer, anchor.keyword)
    assert size(hits) == 1
    return point(hits[0].page_no, hits[0].x + anchor.offset_x, hits[0].y + anchor.offset_y)
  if anchor.anchor_type == FORM_FIELD:
    field = find_form_field(document, anchor.field_name)
    assert field exists
    return point(field.page_no, field.x + anchor.offset_x, field.y + anchor.offset_y)
  fail("UNSUPPORTED_ANCHOR_TYPE")
```

### 7.4 输入输出样例

输入：

```json
{
  "document_version_id": "doc_ver_contract_001",
  "coordinate_version_id": "coord_ver_001",
  "anchor_type": "KEYWORD",
  "keyword": "甲方盖章",
  "offset_x": 12,
  "offset_y": -8,
  "seal_width": 120,
  "seal_height": 120
}
```

输出：

```json
{
  "page_no": 8,
  "x": 420,
  "y": 96,
  "seal_width": 120,
  "seal_height": 120,
  "rendering_status": "PENDING"
}
```

### 7.5 异常分流表

| 异常 | 分流结果 | 处理动作 |
| --- | --- | --- |
| 绝对页码不存在 | 阻断渲染 | 坐标方案版本进入修正流程 |
| 关键字未命中 | 转人工 | 生成坐标人工确认任务 |
| 关键字多处命中 | 转人工 | 要求选择命中位置或改用绝对坐标 |
| 文档版本不可读 | 补偿重试 | 触发文档中心引用修复或解密访问检查 |
| 验签失败 | 结果异常 | `es_signature_result.verification_status=FAILED` |

### 7.6 与详细设计字段映射

| 专项字段 | 详细设计字段 / 对象 | 映射说明 |
| --- | --- | --- |
| `coordinate_scheme_id` / `coordinate_version_id` | `es_signature_request.seal_scheme_id`、`es_signature_session.runtime_checkpoint` | 申请保存方案引用，会话固化版本快照 |
| `document_version_id` | `es_request_document_binding.document_version_id` | 待签输入稿来自文档绑定表 |
| `rendering_status` | `es_signature_result.result_status` | 渲染完成后推进结果状态 |
| `verification_policy_id` | `es_signature_result.verification_summary_json` | 验签策略与结论进入结果摘要 |
| `verification_status` | `es_signature_result.verification_status`、`es_signature_summary.verification_status` | 结果表持有专业结论，摘要表输出稳定读模型 |

## 8. 治理原则总结

- 签章坐标以"版本快照"为核心治理模式，确保正在执行的签章链路不因坐标方案变更而中断。
- 盖章渲染以"渲染上下文快照"为核心治理模式，确保渲染执行与印章样式、坐标定义的版本解耦。
- 验签算法以"验签策略"为核心治理模式，确保验签结论的可解释性与长期有效性。
- 证明文件以"生成策略"为核心治理模式，确保证明内容的可审计性与消费渠道的清晰性。
- 跨模块协作以"引用而非持有、消费快照而非实时查询、执行与治理分离"为原则，确保各模块边界清晰、责任明确、审计可追溯。
