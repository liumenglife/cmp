# 签章引擎适配层专项设计

## 1. 文档定位

本文档是 `CMP` 电子签章子模块的签章引擎适配层治理设计，用于明确签章引擎适配层的治理对象、参数映射边界、多引擎并存规则、与周边模块的协作约束。

本文档不承担以下内容：

- 不输出 API 路径、请求样例、错误码全集
- 不输出表级 DDL、索引细节、字段长度与枚举值全集
- 不输出具体签章引擎接口、SDK 调用、协议转换的生产代码
- 不输出实现排期、联调顺序、负责人拆分
- 不输出运维手册、部署清单、监控配置

## 2. 签章引擎适配层治理

### 2.1 治理对象

签章引擎适配层治理以"引擎适配配置"为最小治理单元，其内部由以下对象构成：

- **适配配置主体**：表达一种签章引擎的类型、适配协议版本、生效与失效时间、配置状态，包含引擎类型标识、引擎厂商、适配层版本、协议类型（如 PKCS#7、CAdES、XAdES、PAdES 等）。
- **参数映射规则**：表达电子签章模块内部参数（印章引用、坐标信息、证书绑定、渲染参数）与具体签章引擎接口参数之间的映射关系，包括字段映射、类型转换、默认值填充、条件裁剪规则。
- **引擎端点配置**：表达签章引擎的服务地址、认证方式、超时参数、重试策略、熔断规则，不包含引擎内部的业务逻辑或实现细节。
- **引擎能力声明**：表达该签章引擎支持的能力集合，如支持的签名算法（RSA-SHA256、SM2-SM3 等）、支持的时间戳类型、支持的文档格式、是否支持骑缝签、是否支持批量签、是否支持验签证明生成。

### 2.2 稳定锚点

签章引擎适配层的稳定锚点设计如下：

- **适配配置 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **适配配置编码**：面向用户与管理的稳定业务标识，允许按组织规则自定义，但在平台内作为辅助索引存在，不替代适配配置 ID 作为真相键。
- **当前生效配置 ID**：每一类签章引擎在任意时刻有且只有一个"当前生效配置"，所有新发起的签章会话消费该配置的快照，而不是实时查询最新配置。
- **引擎类型枚举**：作为平台内稳定分类，包括内置引擎、第三方 CA 引擎、本地签章服务引擎、云端签章服务引擎等，不因厂商变更而随意扩展枚举值。

### 2.3 版本归属

适配配置版本归属遵循以下原则：

- 每一版适配配置必须归属到某一适配配置主体，不可独立存在。
- 版本号由平台统一生成，反映版本先后顺序，但不表达"哪个版本更好"或"哪个版本是业务主版本"——业务主版本由"当前生效配置"字段显式指向。
- 版本一旦生成，其内部的参数映射规则、引擎端点配置、引擎能力声明即冻结，不因后续引擎升级、协议变更、厂商切换而自动漂移。
- 版本变更必须记录变更类型：新建、参数映射更新、端点配置更新、能力声明更新、失效归档。

### 2.4 最小治理单元

签章引擎适配层的最小治理单元是"适配配置版本"：

- **发布单元**：将某一版本设为"当前生效配置"，同时记录发布时间、发布人、发布原因。发布操作后，新发起的签章申请将消费新版本快照。
- **回滚单元**：将"当前生效配置"指向先前的某一历史版本，同时记录回滚时间、回滚人、回滚原因。回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在执行中的签章会话不受影响：已建立会话的签章申请继续消费会话建立时绑定的适配配置快照，不因发布/回滚操作而中途切换。
- 发布/回滚操作本身纳入适配层审计事件，但不触发签章申请、会话、结果的级联变更。

### 2.5 审计最小单元

签章引擎适配层的审计以"适配配置版本"为最小单元，必须记录以下事件类型：

- 适配配置创建、适配配置失效、适配配置归档
- 适配配置版本发布、版本回滚、版本冻结
- 参数映射规则变更、引擎端点配置变更、能力声明变更
- 引擎适配异常记录（含引擎不可达、参数映射失败、响应解析失败）、引擎适配人工介入

审计事件必须携带：适配配置 ID、适配配置版本 ID、操作类型、操作人、操作时间、操作原因、关联的签章请求 ID（若适用）。

## 3. 参数映射治理边界

### 3.1 参数映射的治理范围

参数映射治理遵循以下边界：

- 参数映射只定义"电子签章模块内部参数"到"具体签章引擎接口参数"的转换规则，不持有引擎接口的完整定义或实现代码。
- 参数映射分为必填字段映射、可选字段映射、条件字段映射：必填字段映射失败时阻断签章执行；可选字段映射失败时降级执行；条件字段映射根据运行时条件动态决定是否映射。
- 参数映射规则支持默认值填充：当电子签章模块未提供某参数时，适配层按映射规则填充默认值，而不是依赖引擎侧默认值。
- 参数映射规则支持类型转换：如日期格式转换、编码转换、枚举值映射，转换规则在适配配置中显式定义，不在代码中硬编码。

### 3.2 与具体引擎的解耦要求

签章引擎适配层与具体引擎的解耦遵循以下要求：

- **接口协议解耦**：适配层消费标准化"渲染上下文"和"签章请求上下文"，而不是消费具体引擎的专有接口对象。电子签章模块不感知具体引擎的接口细节。
- **参数映射解耦**：参数映射规则以配置化方式定义，而不是以代码分支方式硬编码。新增引擎类型时，只需新增适配配置，不修改电子签章模块核心代码。
- **错误码解耦**：适配层负责将引擎返回的错误码、错误信息归一化为平台统一的错误分类（如引擎不可达、参数非法、证书无效、签名失败、验签失败），电子签章模块只消费归一化错误，不消费引擎原始错误。
- **能力差异解耦**：不同引擎的能力差异通过"引擎能力声明"表达，电子签章模块根据能力声明调整签章策略（如某引擎不支持骑缝签，则编排层不分配骑缝签任务），不通过代码分支判断引擎类型。

### 3.3 参数映射的版本快照

参数映射以"适配配置版本"为单位冻结：

- 签章会话建立时，将当前生效的适配配置版本快照固化到会话上下文中，包括参数映射规则、引擎端点配置、引擎能力声明。
- 会话执行过程中，即使适配配置发布新版本，正在执行的会话仍消费会话建立时的快照，不因配置变更而中途切换。
- 参数映射快照与渲染上下文、印章资源版本快照、坐标方案版本快照共同构成签章执行的完整输入，四者平行治理，变更互不影响。

## 4. 多引擎并存治理规则

### 4.1 多引擎并存场景

平台可能同时对接多种签章引擎，多引擎并存遵循以下规则：

- **按签章场景分配引擎**：不同签章场景（如合同签署、审批单据签署、电子公文签署）可配置使用不同的签章引擎，场景与引擎的映射关系在签章方案中定义，不在适配层中硬编码。
- **按印章资源分配引擎**：不同印章资源（如合同专用章、财务专用章、法人章）可配置使用不同的签章引擎，印章与引擎的映射关系在印章资源中定义，适配层根据印章资源绑定的引擎类型选择适配配置。
- **按组织归属分配引擎**：不同组织可使用不同的签章引擎（如总部使用自建签章服务，分公司使用第三方 CA 签章服务），组织与引擎的映射关系在组织配置中定义，适配层根据签署参与方的组织归属选择适配配置。

### 4.2 引擎切换与回滚单元

引擎切换与回滚以"适配配置版本"为最小单元：

- **切换单元**：将某一印章资源或签章场景关联到的引擎适配配置切换为新版本，切换后新发起的签章申请消费新版本适配配置。
- **回滚单元**：将引擎适配配置指针指回先前的历史版本，回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在执行中的签章会话不受影响：已建立会话的签章申请继续消费会话建立时的适配配置快照，不因切换/回滚操作而中途变更引擎。
- 切换/回滚操作纳入适配层审计事件，但不触发签章申请、会话、结果的级联变更。

### 4.3 引擎并存时的冲突处理

多引擎并存时的冲突处理遵循以下规则：

- **同一签章任务不允许跨引擎**：一个签章会话中的所有签署动作必须使用同一签章引擎，不允许部分参与方用引擎 A、部分参与方用引擎 B。
- **引擎能力不匹配时阻断**：签章会话建立时，适配层检查所选引擎的能力声明是否满足签章场景的需求（如需要骑缝签但引擎不支持），能力不匹配时阻断会话建立，而不是降级执行。
- **引擎不可达时熔断**：适配层检测到引擎连续不可达时，触发熔断机制，暂停向该引擎发起新请求，但不影响正在执行的会话。熔断恢复后，新请求恢复正常路由。

## 5. 跨模块边界说明

### 5.1 与 `seal-resource-and-certificate-design.md` 的边界

- **印章资源层**负责印章样式引用、证书绑定关系、授权矩阵快照，本文档负责将这些快照映射为具体签章引擎接口所需的参数。
- **印章资源版本**是适配层参数映射的输入来源之一，但参数映射和协议转换不在印章资源层。
- **证书绑定关系**通过印章资源版本传递到适配层，适配层根据引擎类型选择对应的证书格式和签名算法，但证书的私钥由签章引擎或密钥管理服务在运行时提供，适配层不持有私钥。
- 两份文档共同构成签章执行的完整输入：印章资源层提供"用什么签"（印章、证书），本文档提供"怎么调用签章引擎"（参数映射、协议转换、端点配置）。

### 5.2 与 `signing-party-orchestration-design.md` 的边界

- **签署编排层**负责确定参与方、顺序、完成判定，本文档负责根据编排结果选择对应的签章引擎和参数映射规则。
- **签署场景策略**可定义该场景下使用的签章引擎类型或适配配置，适配层根据场景策略选择具体的适配配置版本，但引擎选择和参数映射不在编排层。
- **参与方与印章的匹配关系**由编排层确定，适配层消费该匹配结果（哪个参与方使用哪枚印章、哪个印章关联哪个引擎），但不参与匹配逻辑。
- 两份文档共同构成签署执行的完整链路：编排层决定"谁、在什么顺序、用什么印章签"，本文档决定"调用哪个引擎、用什么参数签"。

### 5.3 与 `signature-coordinate-and-rendering-design.md` 的边界

- **坐标与渲染层**负责定义签章坐标、渲染参数、验签策略，本文档负责将这些参数映射为签章引擎接口所需的格式。
- **渲染上下文**是适配层参数映射的核心输入之一，包含印章样式引用、坐标信息、渲染参数，适配层将其转换为引擎接口参数，但渲染执行不在适配层。
- **验签策略**定义验签算法和证书链验证规则，适配层负责调用引擎的验签接口并归一化返回结果，但验签算法选择不在适配层。
- 两份文档共同构成签章执行的完整技术链路：坐标与渲染层定义"签在哪里、怎么签、签完怎么验"，本文档定义"如何调用引擎完成签和验"。

### 5.4 与 `integration-hub` 的边界

- **`integration-hub` 是外部系统适配层**：负责对接外部签章服务、第三方 CA、时间戳服务等外部系统，处理外部系统的协议适配、认证适配、报文转换。
- **签章引擎适配层是内部签章执行适配**：负责将电子签章模块的内部参数映射为签章引擎接口参数，不管是内置引擎还是通过 `integration-hub` 对接的外部引擎，都通过统一的适配层接口调用。
- **调用链路**：电子签章模块 → 签章引擎适配层 → （内置引擎 或 `integration-hub` 对接的外部引擎）。适配层屏蔽引擎类型差异，电子签章模块不感知引擎是内置还是外部。
- **错误处理分工**：适配层负责参数映射、协议转换、错误归一化；`integration-hub` 负责外部系统的连通性、认证、外部协议处理；电子签章模块负责根据归一化错误执行重试策略或进入人工介入流程。

## 6. 最小可执行基线

### 6.1 配置结构

签章引擎适配配置以 `engine_adapter_config` 为发布单元，最小结构如下：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `adapter_config_id` | string | 是 | 无 | 适配配置主键 |
| `adapter_config_version_id` | string | 是 | 无 | 适配配置版本 |
| `engine_type` | enum | 是 | 无 | `BUILT_IN` / `THIRD_PARTY_CA` / `LOCAL_SERVICE` / `CLOUD_SERVICE` |
| `protocol_type` | enum | 是 | 无 | `PKCS7` / `CADES` / `XADES` / `PADES` |
| `endpoint_ref` | string | 是 | 无 | 引擎端点配置引用 |
| `auth_profile_ref` | string | 是 | 无 | 调用认证配置引用 |
| `capability_profile_ref` | string | 是 | 无 | 引擎能力声明引用 |
| `parameter_mapping_version` | string | 是 | 无 | 参数映射规则版本 |
| `timeout_ms` | integer | 是 | `30000` | 单次调用超时 |
| `retry_policy_id` | string | 是 | 平台默认重试策略 | 重试策略引用 |
| `config_status` | enum | 是 | `DRAFT` | `DRAFT` / `PUBLISHED` / `FROZEN` / `REVOKED` / `ARCHIVED` |

### 6.2 内部参数到引擎参数映射表

| 内部参数 | 来源 | 引擎参数 | 转换规则 | 必填级别 | 失败处理 |
| --- | --- | --- | --- | --- | --- |
| `signature_request_id` | `es_signature_request` | `bizNo` | 原值传递 | 必填 | 阻断调用 |
| `signature_session_id` | `es_signature_session` | `sessionNo` | 原值传递 | 必填 | 阻断调用 |
| `document_version_id` | `es_request_document_binding` | `inputDocumentRef` | 转为文档中心可访问句柄 | 必填 | 阻断调用 |
| `seal_resource_id` | 印章资源快照 | `sealId` | 按映射字典转引擎印章编号 | 必填 | 阻断调用 |
| `certificate_id` | 证书绑定快照 | `certRef` | 转为引擎证书引用 | 必填 | 阻断调用 |
| `page_no` / `x` / `y` | 坐标快照 | `position` | 统一坐标系转引擎坐标系 | 必填 | 阻断调用 |
| `seal_width` / `seal_height` | 渲染上下文 | `sealSize` | 单位转换为引擎要求单位 | 可选 | 使用引擎能力默认值 |
| `opacity` / `rotation` | 渲染上下文 | `renderOptions` | 缺省填充 `1.0` / `0` | 可选 | 默认值填充 |
| `verification_policy_id` | 验签策略 | `verifyPolicy` | 映射为引擎验签策略编码 | 条件必填 | 降级为平台默认验签策略 |
| `callback_event_id` | 回调事件 | `callbackTraceNo` | 原值传递 | 可选 | 仅记录审计 |

### 6.3 核心状态机

| 对象 | 状态 | 允许转换 | 触发条件 |
| --- | --- | --- | --- |
| 适配配置版本 | `DRAFT` | `PUBLISHED` / `ARCHIVED` | 映射规则与端点校验通过；废弃时归档 |
| 适配配置版本 | `PUBLISHED` | `FROZEN` / `REVOKED` | 被会话消费后冻结；端点或能力不可用时撤销 |
| 引擎调用 | `PENDING` | `MAPPING` / `FAILED` | 调用任务创建或缺少必填上下文 |
| 引擎调用 | `MAPPING` | `CALLING` / `FAILED` | 参数映射成功或失败 |
| 引擎调用 | `CALLING` | `SUCCEEDED` / `FAILED` / `TIMEOUT` | 引擎返回、异常或超时 |

### 6.4 适配伪代码

```text
adapt_and_call_engine(context, adapter_config):
  assert adapter_config.config_status in [PUBLISHED, FROZEN]
  assert engine_capability_match(adapter_config.capability_profile_ref, context.required_capability)
  payload = {}
  for rule in load_mapping_rules(adapter_config.parameter_mapping_version):
    value = read_path(context, rule.internal_path)
    if value is null and rule.required:
      fail("PARAMETER_MAPPING_REQUIRED_MISSING", rule.internal_path)
    if value is null:
      value = rule.default_value
    payload[rule.engine_path] = convert(value, rule.convert_type, rule.convert_args)
  response = call_engine(adapter_config.endpoint_ref, adapter_config.auth_profile_ref, payload)
  return normalize_engine_response(response)
```

### 6.6 异常分流表

| 异常 | 分流结果 | 处理动作 |
| --- | --- | --- |
| 必填参数缺失 | 阻断调用 | 记录参数映射失败审计，任务进入失败 |
| 能力声明不匹配 | 阻断会话建立 | 返回引擎能力不支持，要求切换引擎或场景策略 |
| 引擎不可达 | 重试 / 熔断 | 按 `retry_policy_id` 重试，连续失败后熔断 |
| 引擎返回未知错误码 | 归一化为 `ENGINE_UNKNOWN_ERROR` | 保留原始错误引用，进入人工介入 |
| 响应解析失败 | 补偿重试 | 原始响应入交换日志，避免直接推进结果 |

## 7. 治理原则总结

- 签章引擎适配层以"版本快照"为核心治理模式，确保正在执行的签章链路不因适配配置变更而中断。
- 参数映射以"配置化规则"为核心治理模式，确保新增引擎类型时只需新增配置，不修改核心代码。
- 多引擎并存以"场景/印章/组织维度的引擎分配规则"为核心治理模式，确保不同场景、不同印章、不同组织可以使用不同的签章引擎。
- 适配层与印章资源层、编排层、渲染层、integration-hub 平行治理，变更互不影响，但共同构成签章执行的完整上下文。
- 跨模块协作以"引用而非持有、消费快照而非实时查询、参数映射配置化而非代码硬编码"为原则，确保各模块边界清晰、责任明确、审计可追溯。
