# 签章运行时参数专项设计

## 1. 文档定位

本文档是 `CMP` 电子签章子模块的签章运行时参数治理设计，用于明确审计事件枚举、任务重试参数、锁 TTL、摘要重建批处理窗口在平台内的治理对象、归属关系、责任边界与跨模块协作约束。

本文档不承担以下内容：

- 不输出 API 路径、请求样例、错误码全集
- 不输出表级 DDL、索引细节、字段长度与枚举值全集
- 不输出具体参数、配置、批处理的生产代码
- 不输出实现排期、联调顺序、负责人拆分
- 不输出运维手册、部署清单、监控配置

## 2. 审计事件枚举治理

### 2.1 治理对象

审计事件枚举治理以"审计事件类型目录"为顶层治理对象，其内部由以下对象构成：

- **事件类型主体**：表达一种签章域事件的类型标识、业务分类、事件级别、是否可补偿、是否可重放，包含事件类型编码、事件名称、事件描述、适用模块、适用组件。
- **事件负载schema**：表达该事件类型标准负载结构，包括必填字段（事件时间、事件源、关联业务ID、操作人）、可选字段（失败原因、补偿结论、人工介入标记）、扩展字段（预留业务上下文）。
- **事件消费规则**：表达哪些下游组件必须消费该事件，哪些组件可选消费，包括合同主档消费规则、文档中心消费规则、搜索投影消费规则、告警指标消费规则、审计归档消费规则。

### 2.2 稳定锚点

审计事件枚举的稳定锚点设计如下：

- **事件类型编码**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **事件类型名称**：面向用户与管理的稳定业务标识，允许按组织规则自定义，但在平台内作为辅助索引存在，不替代事件类型编码作为真相键。
- **事件类型版本 ID**：每一类事件类型在任意时刻有且只有一个"当前有效版本"，所有新产生的审计事件消费该版本的 schema 快照，而不是实时计算最新版本。
- **事件级别枚举**：作为平台内稳定分类，包括 `INFO`、`WARN`、`ERROR`、`CRITICAL` 等，不因业务发展而随意扩展枚举值。
- **事件分类枚举**：作为平台内稳定分类，包括签章申请类、会话控制类、签署动作类、验签结果类、回写类、补偿恢复类、人工介入类，不因组件调整而随意扩展枚举值。

### 2.3 版本归属

审计事件枚举版本归属遵循以下原则：

- 每一版事件类型必须归属到某一事件类型主体，不可独立存在。
- 版本号由平台统一生成，反映版本先后顺序，但不表达"哪个版本更好"或"哪个版本是业务主版本"——业务主版本由"当前有效版本"字段显式指向。
- 版本一旦生成，其内部的事件负载 schema、事件消费规则即冻结，不因后续组件升级、消费逻辑变更而自动漂移。
- 版本变更必须记录变更类型：新建、负载 schema 更新、消费规则更新、事件级别调整、失效归档。

### 2.4 最小治理单元

审计事件枚举的最小治理单元是"事件类型版本"：

- **发布单元**：将某一版本设为"当前有效版本"，同时记录发布时间、发布人、发布原因。发布操作后，新产生的审计事件将消费新版本 schema 快照。
- **回滚单元**：将"当前有效版本"指向先前的某一历史版本，同时记录回滚时间、回滚人、回滚原因。回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在执行中的签章会话不受影响：已建立会话产生的审计事件继续消费会话建立时绑定的事件类型 schema 快照，不因发布/回滚操作而中途切换。

## 3. 任务重试参数治理

### 3.1 治理对象

任务重试参数治理以"重试策略配置"为顶层治理对象，其内部由以下对象构成：

- **重试策略主体**：表达一种任务类型的重试行为，包含策略名称、适用任务类型（签章引擎回调、文档中心回写、合同主档回写、摘要重建、补偿执行）、最大重试次数、初始退避时间、退避倍率、最大退避时间。
- **重试条件规则**：表达在什么错误条件下触发重试，包括可重试错误码集合、不可重试错误码集合、超时判定阈值、死锁判定阈值。
- **重试熔断规则**：表达在什么条件下停止重试并转入人工介入，包括连续失败次数阈值、单位时间内失败率阈值、重试资源消耗阈值。

### 3.2 稳定锚点

任务重试参数的稳定锚点设计如下：

- **重试策略 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **重试策略编码**：面向用户与管理的稳定业务标识，允许按组织规则自定义，但在平台内作为辅助索引存在，不替代重试策略 ID 作为真相键。
- **当前生效策略 ID**：每一类任务在任意时刻有且只有一个"当前生效策略"，所有新入队的重试任务消费该策略的快照，而不是实时查询最新策略。
- **任务类型枚举**：作为平台内稳定分类，包括引擎回调任务、文档回写任务、合同回写任务、摘要重建任务、补偿任务，不因新增任务类型而随意扩展枚举值。

### 3.3 版本归属

任务重试参数版本归属遵循以下原则：

- 每一版重试策略必须归属到某一重试策略主体，不可独立存在。
- 版本号由平台统一生成，反映版本先后顺序，但不表达"哪个版本更好"或"哪个版本是业务主版本"——业务主版本由"当前生效策略"字段显式指向。
- 版本一旦生成，其内部的重试条件规则、重试熔断规则即冻结，不因后续引擎升级、错误码调整而自动漂移。
- 版本变更必须记录变更类型：新建、重试次数调整、退避参数更新、条件规则更新、熔断规则更新、失效归档。

### 3.4 最小治理单元

任务重试参数的最小治理单元是"重试策略版本"：

- **发布单元**：将某一版本设为"当前生效策略"，同时记录发布时间、发布人、发布原因。发布操作后，新入队的重试任务将消费新版本快照。
- **回滚单元**：将"当前生效策略"指向先前的某一历史版本，同时记录回滚时间、回滚人、回滚原因。回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在执行中的重试任务不受影响：已进入重试队列的任务继续消费任务入队时绑定的重试策略快照，不因发布/回滚操作而中途切换。

### 3.5 责任划分

任务重试参数的责任划分如下：

- **电子签章模块**：负责定义签章域内的重试策略、维护重试策略版本、记录重试执行审计。
- **签章引擎适配层**（见 [signature-engine-adapter-design.md](signature-engine-adapter-design.md)）：负责消费重试策略，执行具体的回调重试逻辑，但不拥有重试策略的定义权。
- **文档中心**：负责定义文档回写相关的错误码语义，但不拥有回写重试策略的定义权。
- **合同主档**：负责定义合同回写相关的错误码语义，但不拥有回写重试策略的定义权。

## 4. 锁 TTL 治理

### 4.1 治理对象

锁 TTL 治理以"分布式锁配置"为顶层治理对象，其内部由以下对象构成：

- **锁配置主体**：表达一种锁资源的 TTL 行为，包含锁名称、锁用途、默认 TTL、最大 TTL、续期步长、续期上限次数、等待队列超时时间。
- **锁作用域规则**：表达该锁在哪些场景下生效，包括签章申请准入锁、会话状态推进锁、签章结果登记锁、文档绑定锁、回写执行锁。
- **锁竞争处理规则**：表达锁获取失败时的处理策略，包括快速失败、进入等待队列、降级为只读模式、触发人工介入。

### 4.2 稳定锚点

锁 TTL 的稳定锚点设计如下：

- **锁配置 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **锁名称**：面向运维与调试的稳定标识，允许按组织规则自定义，但在平台内作为辅助索引存在，不替代锁配置 ID 作为真相键。
- **当前生效配置 ID**：每一类锁在任意时刻有且只有一个"当前生效配置"，所有新申请的锁资源消费该配置的快照。
- **锁作用域枚举**：作为平台内稳定分类，包括申请准入锁、会话推进锁、结果登记锁、文档绑定锁、回写执行锁，不因新增锁场景而随意扩展枚举值。

### 4.3 版本归属

锁 TTL 版本归属遵循以下原则：

- 每一版锁配置必须归属到某一锁配置主体，不可独立存在。
- 版本号由平台统一生成，反映版本先后顺序，但不表达"哪个版本更好"或"哪个版本是业务主版本"——业务主版本由"当前生效配置"字段显式指向。
- 版本一旦生成，其内部的锁作用域规则、锁竞争处理规则即冻结，不因后续性能调优、业务调整而自动漂移。
- 版本变更必须记录变更类型：新建、TTL 调整、续期参数更新、作用域更新、竞争规则更新、失效归档。

### 4.4 最小治理单元

锁 TTL 的最小治理单元是"锁配置版本"：

- **发布单元**：将某一版本设为"当前生效配置"，同时记录发布时间、发布人、发布原因。发布操作后，新申请的锁资源将消费新版本快照。
- **回滚单元**：将"当前生效配置"指向先前的某一历史版本，同时记录回滚时间、回滚人、回滚原因。回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在持有中的锁不受影响：已获取锁资源的执行流程继续按获取时的 TTL 规则执行，不因发布/回滚操作而中途变更。

### 4.5 责任划分

锁 TTL 的责任划分如下：

- **电子签章模块**：负责定义签章域内的锁配置、维护锁配置版本、记录锁竞争审计。
- **`signature-control-plane`（见 detailed-design.md 第 3.8 节）**：负责消费锁配置，执行具体的锁申请、续期、释放逻辑，但不拥有锁配置的定义权。
- **基础设施层**：负责提供分布式锁能力，但不拥有业务锁配置的定义权。

## 5. 摘要重建批处理窗口治理

### 5.1 治理对象

摘要重建批处理窗口治理以"批处理窗口配置"为顶层治理对象，其内部由以下对象构成：

- **窗口配置主体**：表达一种摘要重建批处理的窗口行为，包含窗口名称、适用摘要类型（签章摘要、搜索投影、归档消费投影、AI 摘要投影）、窗口大小、窗口滑动间隔、最大积压阈值、并行度上限。
- **窗口触发规则**：表达在什么条件下触发批处理窗口，包括定时触发（cron 表达式）、积压触发（待处理数量阈值）、事件驱动触发（特定审计事件到达）。
- **窗口异常处理规则**：表达窗口执行过程中的异常处理策略，包括单条失败不影响同窗口其他条目、窗口整体失败重试策略、窗口长时间未完成告警阈值。

### 5.2 稳定锚点

摘要重建批处理窗口的稳定锚点设计如下：

- **窗口配置 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。

- 版本一旦生成，其内部的窗口触发规则、窗口异常处理规则即冻结，不因后续性能调优、摘要结构变更而自动漂移。
- 版本变更必须记录变更类型：新建、窗口大小调整、触发规则更新、异常规则更新、失效归档。

### 5.4 最小治理单元

摘要重建批处理窗口的最小治理单元是"窗口配置版本"：

- **发布单元**：将某一版本设为"当前生效配置"，同时记录发布时间、发布人、发布原因。发布操作后，新入窗口的重建任务将消费新版本快照。

## 6. 最小可执行基线

### 6.1 配置结构

运行时参数以 `signature_runtime_parameter_set` 为发布单元，最小结构如下：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `parameter_set_id` | string | 是 | 无 | 参数集主键 |
| `parameter_set_version_id` | string | 是 | 无 | 参数集版本 |
| `event_schema_version` | string | 是 | 无 | 审计事件负载版本 |
| `retry_policy_id` | string | 是 | 平台默认重试策略 | 重试策略引用 |
| `lock_config_id` | string | 是 | 平台默认锁配置 | 锁配置引用 |
| `summary_rebuild_window_id` | string | 是 | 平台默认窗口 | 摘要重建窗口引用 |
| `publish_status` | enum | 是 | `DRAFT` | `DRAFT` / `PUBLISHED` / `FROZEN` / `REVOKED` / `ARCHIVED` |
| `effective_at` | datetime | 是 | 无 | 生效时间 |
| `expires_at` | datetime | 否 | 空 | 失效时间 |

重试参数基线如下：

| 任务类型 | 最大重试次数 | 初始退避 | 退避倍率 | 最大退避 | 转人工条件 |
| --- | --- | --- | --- | --- | --- |
| `ES_CALLBACK_RETRY` | 5 | 30 秒 | 2 | 10 分钟 | 连续失败 5 次或签名校验持续失败 |
| `ES_DOCUMENT_WRITEBACK` | 8 | 1 分钟 | 2 | 30 分钟 | 文档中心版本冲突或连续失败 8 次 |
| `ES_CONTRACT_WRITEBACK` | 8 | 1 分钟 | 2 | 30 分钟 | 合同状态冲突或连续失败 8 次 |
| `ES_SUMMARY_REBUILD` | 3 | 5 分钟 | 2 | 30 分钟 | 摘要与结果长期不一致 |

### 6.2 核心状态机

| 对象 | 状态 | 允许转换 | 触发条件 |
| --- | --- | --- | --- |
| 参数集版本 | `DRAFT` | `PUBLISHED` / `ARCHIVED` | 校验通过后发布；废弃时归档 |
| 参数集版本 | `PUBLISHED` | `FROZEN` / `REVOKED` | 被运行任务消费后冻结；参数错误时撤销 |
| 重试任务 | `READY` | `RUNNING` / `DEAD_LETTER` | 到达下次触发时间或超过重试上限 |
| 重试任务 | `RUNNING` | `SUCCEEDED` / `READY` / `FAILED` | 执行成功、可重试失败或不可重试失败 |
| 摘要重建窗口 | `PENDING` | `RUNNING` / `SKIPPED` | 到达窗口或无待处理项 |
| 摘要重建窗口 | `RUNNING` | `COMPLETED` / `FAILED` | 批处理完成或异常 |

### 6.3 关键算法伪代码

```text
schedule_retry(job, policy):
  if job.error_code in policy.non_retryable_errors:
    return mark_failed(job, reason="NON_RETRYABLE")
  if job.retry_count >= policy.max_retry_count:
    return move_to_manual_pool(job)
  delay = min(policy.initial_backoff * pow(policy.backoff_multiplier, job.retry_count), policy.max_backoff)
  job.next_trigger_at = now + delay
  job.retry_count += 1
  return save(job)
```

### 6.5 异常分流表

| 异常 | 分流结果 | 处理动作 |
| --- | --- | --- |
| 事件负载缺少必填字段 | 阻断发布或记录失败 | schema 校验失败，不进入消费链 |
| 可重试错误 | 延迟重试 | 按重试参数计算 `next_trigger_at` |
| 不可重试错误 | 失败或人工 | 标记失败并进入人工池 |
| 锁续期超过上限 | 释放并补偿 | 任务回到可重试队列，记录锁异常审计 |
| 摘要窗口超时 | 告警并重试 | 生成窗口失败事件，下一窗口补扫 |

## 9. 跨模块协作约束

签章运行时参数的跨模块协作遵循以下约束：

- **审计事件枚举**：合同主档、文档中心、工作流引擎等下游模块，只消费事件类型目录中已定义的事件类型和负载 schema，不得自行定义签章域事件类型。
- **任务重试参数**：签章引擎、文档中心、合同主档等外部依赖，只提供错误码语义，不得介入签章模块内部重试策略的定义。
- **锁 TTL**：基础设施层只提供分布式锁能力，不得介入业务锁配置的定义和版本管理。
- **摘要重建批处理窗口**：搜索服务、归档服务、AI 服务等下游消费者，只消费重建后的摘要结果，不得介入批处理窗口配置的定义。

## 10. 治理闭环

签章运行时参数的治理闭环如下：

1. **定义**：电子签章模块定义各类运行时参数的治理对象、稳定锚点、版本归属。
2. **发布**：通过版本发布机制，将运行时参数配置推送到各消费方。
3. **消费**：各消费方（签章引擎适配层、`signature-control-plane`、下游服务）消费运行时参数快照。
4. **审计**：所有运行时参数变更、消费、异常均产生标准化审计事件。
5. **优化**：基于审计事件和指标数据，触发运行时参数的版本更新，回到步骤 2。
