# 签署参与方编排专项设计

## 1. 文档定位

本文档是 `CMP` 电子签章子模块的签署参与方编排策略治理设计，用于明确不同签署场景下的参与方编排策略、治理对象、归属关系、生命周期边界与跨模块协作约束。

本文档不承担以下内容：

- 不输出 API 路径、请求样例、错误码全集
- 不输出表级 DDL、索引细节、字段长度与枚举值全集
- 不输出具体编排算法、顺序计算、并行控制的生产代码
- 不输出实现排期、联调顺序、负责人拆分
- 不输出运维手册、部署清单、监控配置

## 2. 签署场景治理

### 2.1 治理对象

签署场景治理以"签署场景策略"为顶层治理对象，其内部由以下对象构成：

- **场景策略主体**：表达一种签署场景的类型、适用条件、生效与失效时间、策略状态，包含场景名称、场景类型、关联合同类型、关联业务流程。
- **参与方展开规则**：表达在该场景下如何确定签署参与方集合，包括参与方来源（合同主体、审批流程节点、人工指定、组织规则解析）、参与方类型（`USER`、`ORG_UNIT`、`EXTERNAL_PARTY`、`SYSTEM_PROXY`）、参与方角色（`APPLICANT`、`PRIMARY_SIGNER`、`COUNTERSIGNER`、`WITNESS`）。
- **顺序控制规则**：表达在该场景下签署顺序如何确定，包括顺序模式（顺序签、并行签、混合签）、顺序来源（合同指定、审批流程节点顺序、人工指定）、顺序约束（前序签署完成才能进入后续、并行组内部无顺序要求）。
- **完成判定规则**：表达在该场景下何时判定签署完成，包括完成条件（全部参与方签署完成、关键参与方签署完成、满足最低签署人数）、完成动作（自动闭环、等待人工确认、触发后续流程）。

### 2.2 稳定锚点

签署场景的稳定锚点设计如下：

- **场景策略 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **场景策略编码**：面向用户与管理的稳定业务标识，允许按组织规则自定义，但在平台内作为辅助索引存在，不替代场景策略 ID 作为真相键。
- **当前有效版本 ID**：每一份场景策略在任意时刻有且只有一个"当前有效版本"，所有新发起的签章申请消费该版本的快照，而不是实时计算最新版本。
- **场景类型枚举**：作为平台内稳定分类，包括单人签、多人顺序签、多人并行签、骑缝签、补签、重签等，不因业务发展而随意扩展枚举值。

### 2.3 版本归属

场景策略版本归属遵循以下原则：

- 每一版场景策略必须归属到某一场景策略主体，不可独立存在。
- 版本号由平台统一生成，反映版本先后顺序，但不表达"哪个版本更好"或"哪个版本是业务主版本"——业务主版本由"当前有效版本"字段显式指向。
- 版本一旦生成，其内部的参与方展开规则、顺序控制规则、完成判定规则即冻结，不因后续合同变更、组织调整、流程变更而自动漂移。
- 版本变更必须记录变更类型：新建、参与方规则更新、顺序规则更新、完成规则更新、失效归档。

### 2.4 最小治理单元

签署场景的最小治理单元是"场景策略版本"：

- **发布单元**：将某一版本设为"当前有效版本"，同时记录发布时间、发布人、发布原因。发布操作后，新发起的签章申请将消费新版本快照。
- **回滚单元**：将"当前有效版本"指向先前的某一历史版本，同时记录回滚时间、回滚人、回滚原因。回滚不改变历史版本的冻结内容，只改变指针指向。
- 正在执行中的签章会话不受影响：已建立会话的签章申请继续消费会话建立时绑定的场景策略快照，不因发布/回滚操作而中途切换。
- 发布/回滚操作本身纳入场景策略审计事件，但不触发签章申请、会话、结果的级联变更。

### 2.5 审计最小单元

签署场景审计以"场景策略版本"为最小单元，必须记录以下事件类型：

- 场景策略创建、场景策略失效、场景策略归档
- 场景策略版本发布、版本回滚、版本冻结
- 参与方展开规则变更、顺序控制规则变更、完成判定规则变更
- 场景策略使用准入判定（含拒绝记录）、场景策略使用人工申诉

审计事件必须携带：场景策略 ID、场景策略版本 ID、操作类型、操作人、操作时间、操作原因、关联的合同 ID（若适用）。

## 3. 不同签署场景的治理边界

### 3.1 单人签署场景

- **治理对象**：单一签署参与方，通常为合同发起人或指定签署人。
- **参与方展开**：从合同主体或审批流程中确定唯一签署人，生成一条 `es_signer_assignment` 记录。
- **顺序控制**：无顺序要求，单一参与方签署完成即判定场景完成。
- **完成判定**：该参与方签署动作完成，视为整个签署场景完成。
- **边界说明**：不涉及多人协调、顺序推进、并行组管理，是其他复杂场景的基础单元。

### 3.2 多人顺序签署场景

- **治理对象**：多个签署参与方，按预定顺序依次签署。
- **参与方展开**：根据合同要求或审批流程节点顺序，展开有序参与方列表，为每个参与方生成 `es_signer_assignment` 记录并分配 `sign_sequence_no`。
- **顺序控制**：严格按 `sign_sequence_no` 顺序推进，前一参与方完成签署后，才激活下一参与方。会话状态推进时，必须检查前序参与方是否全部完成。
- **完成判定**：所有顺序参与方均完成签署，视为场景完成。若中途有参与方拒绝签署或失败，场景进入异常状态。
- **边界说明**：顺序信息来源于合同主档的签署顺序要求或审批流程节点的顺序定义，签章模块只消费快照，不直接修改合同主档或流程实例的顺序定义。

### 3.3 多人并行签署场景

- **治理对象**：多个签署参与方，在同一轮次内并行签署，无先后顺序约束。
- **参与方展开**：展开并行参与方集合，为所有参与方生成 `es_signer_assignment` 记录，可分配相同 `sign_sequence_no` 或标记为同一并行组。
- **顺序控制**：无顺序约束，所有参与方可同时收到签署任务。会话状态推进时，只检查是否所有并行参与方均完成。
- **完成判定**：并行组内所有参与方均完成签署，视为场景完成。
- **边界说明**：适用于合同各方无先后顺序要求的场景，参与方之间的并行关系由场景策略定义，不因个别参与方延迟而阻塞其他参与方。

### 3.4 骑缝签署场景

- **治理对象**：骑缝章签署，通常用于多页合同文档的骑缝盖章。
- **参与方展开**：确定执行骑缝签署的操作人（通常为指定印章持有人或系统代理），生成对应的 `es_signer_assignment` 记录。
- **顺序控制**：骑缝签署通常为单一动作，或作为其他签署动作的附属动作。
- **完成判定**：骑缝章渲染完成并验签通过，视为骑缝签署完成。
- **边界说明**：骑缝签署的核心是渲染策略（见 `signature-coordinate-and-rendering-design.md`），参与方编排只负责确定"谁执行骑缝签署"和"骑缝签署与其他签署动作的先后顺序"。骑缝签署不单独构成完整的签署场景，通常作为其他场景的附属环节。

### 3.5 补签场景

- **治理对象**：针对已完成的签署场景中遗漏的参与方，或合同变更后需要追加的签署参与方。
- **参与方展开**：只展开需要补签的参与方，生成新的 `es_signer_assignment` 记录，不重复展开已完成签署的参与方。
- **顺序控制**：补签参与方的顺序根据业务规则确定，可以是追加到原有顺序之后，也可以独立成轮次。
- **完成判定**：补签参与方完成签署后，更新签章摘要，不自动触发原签署场景的状态回退。
- **边界说明**：补签场景必须关联原签署场景的 `signature_request_id` 或 `signature_result_id`，作为审计追溯依据。补签不影响已完成签署的法律效力，只作为补充登记。

### 3.6 重签场景

- **治理对象**：因验签失败、合同变更、法律要求等原因，需要重新执行的签署场景。
- **参与方展开**：根据重签原因重新展开参与方集合，可能是原参与方全集，也可能是部分参与方。
- **顺序控制**：按重签场景的策略重新确定顺序，不受原签署场景顺序的约束。
- **完成判定**：重签场景的所有参与方完成签署，且验签通过后，生成新的签章结果，更新签章摘要。
- **边界说明**：重签场景必须关联原签署场景的 `signature_request_id` 和 `signature_result_id`，作为审计追溯和结果对比依据。重签完成后，原签署结果标记为"被重签替代"，但不物理删除，以保留完整审计链。

## 4. 参与方编排策略治理

### 4.1 治理对象

参与方编排策略治理以"编排策略版本"为最小治理单元，其内部由以下对象构成：

- **编排策略主体**：表达一套参与方编排规则的稳定身份，包含策略名称、策略类型、适用场景、生效与失效时间、策略状态。
- **参与方来源规则**：表达参与方从哪里获取，包括合同主体映射、审批流程节点映射、组织规则解析结果、人工指定列表。
- **角色分配规则**：表达每个参与方被分配什么角色（`APPLICANT`、`PRIMARY_SIGNER`、`COUNTERSIGNER`、`WITNESS`）。
- **顺序编排规则**：表达参与方之间的顺序关系，包括顺序模式、顺序优先级、并行组划分、依赖关系。
- **快照固化规则**：表达在签章申请准入时，哪些动态信息需要固化为快照（如组织名称、用户姓名、角色关系），以避免后续变更影响已发起的签署。

### 4.2 稳定锚点

参与方编排的稳定锚点设计如下：

- **编排策略 ID**：全局唯一，一经分配不再变更，是跨表、跨模块、跨审计事件的核心关联键。
- **编排策略编码**：面向用户与管理的稳定业务标识，作为辅助索引存在。
- **当前有效版本 ID**：每一套编排策略在任意时刻有且只有一个"当前有效版本"，所有新发起的签章申请消费该版本的快照。
- **参与方快照 ID**：每次编排执行后生成的参与方快照，作为 `es_signer_assignment` 的数据来源，不因后续编排策略变更而自动漂移。

### 4.3 版本归属

编排策略版本归属遵循以下原则：

- 每一版编排策略必须归属到某一编排策略主体，不可独立存在。
- 版本一旦生成，其内部的参与方来源规则、角色分配规则、顺序编排规则即冻结。
- 版本变更必须记录变更类型：新建、参与方来源更新、角色分配更新、顺序编排更新、失效归档。

### 4.4 最小治理单元

参与方编排的最小治理单元是"编排策略版本"：

- **发布单元**：将某一版本设为"当前有效版本"，新发起的签章申请消费新版本快照。
- **回滚单元**：将"当前有效版本"指向先前的某一历史版本。
- 正在执行中的签章会话不受影响：已建立会话的签章申请继续消费会话建立时绑定的编排快照。
- 发布/回滚操作纳入编排策略审计事件，不触发签章申请、会话、结果的级联变更。

## 5. 跨模块边界说明

### 5.1 与 `contract-core` 的边界

- **合同主档是业务真相源**：签署参与方编排不持有合同业务状态，只通过 `contract_id` 挂接。
- **参与方来源消费合同摘要**：签章申请受理时读取合同主档摘要（合同主体、签署顺序要求、合同类型），作为参与方展开的输入之一，但不将编排结果反向写入合同主档。
- **签署顺序要求来自合同主档**：合同主档中定义的签署顺序要求（如有）作为编排策略的输入，但编排策略可以基于业务规则覆盖或补充合同主档的顺序定义。
- **签章结果回写合同摘要**：签署完成后，只将稳定摘要（`signature_status`、签署时间、参与方摘要）回写合同主档，不将参与方临时态、顺序计算过程、编排细节写入合同主档。
- **合同主体不参与编排执行**：合同主体信息作为参与方来源之一，但编排执行不直接修改合同主体，只生成签署任务责任人快照。

### 5.2 与 `workflow-engine` 的边界

- **审批流程节点作为参与方来源之一**：若签章场景关联审批流程，流程引擎的审批节点、节点参与人可作为签署参与方的来源，但签章模块只消费审批摘要快照，不直接查询流程实例运行时状态。
- **参与方映射由签章模块负责**：流程引擎的 `Org Binding Resolver` 输出审批节点的参与人集合，签章模块的编排策略负责将这些参与人映射为签署参与方，并分配签署角色和顺序。
- **签章模块不拥有流程实例**：流程引擎负责流程实例的生命周期管理，签章模块只通过 `contract_id` 和审批摘要引用挂接，不反向控制流程实例的状态推进。
- **签章完成可触发流程后置节点**：签署完成后，签章模块可通过合同主档或集成边界触发流程引擎的后置节点（如"已签署"节点），但不直接操作流程实例。

### 5.3 与 `identity-access` 的边界

- **用户与组织身份是参与方真相源**：签署参与方的身份信息（用户 ID、组织 ID、组织名称、用户姓名）来自 `identity-access` 主线，签章模块只消费身份快照，不直接修改身份主档。
- **组织规则解析结果作为参与方来源之一**：`identity-access` 的 `org-rule-resolver` 负责将组织规则（如"部门负责人"、"发起人上级"）解析为具体参与人集合，签章模块的编排策略消费该解析结果。
- **权限判定由 `identity-access` 负责**：哪些用户可以使用哪枚印章、哪些用户可以发起签章申请，由 `identity-access` 的权限中心统一判定，签章模块只消费权限判定快照，不直接执行权限检查逻辑。
- **签署参与方快照固化**：为避免后续身份或组织变更影响已发起的签署，签章申请准入时将参与方的身份信息固化为快照（保存在 `es_signer_assignment` 中），后续身份或组织变更不影响已固化的快照。

## 6. 签署场景与印章资源、渲染策略的关联关系

### 6.1 与印章资源的关联关系

- **签署场景决定印章使用范围**：不同签署场景可能要求使用不同的印章资源（如合同章、财务章、法人章），场景策略中可以定义"该场景下允许使用的印章类型或印章资源列表"。
- **参与方角色关联印章资源**：不同的参与方角色（如 `PRIMARY_SIGNER`、`COUNTERSIGNER`）可能关联不同的印章资源，编排策略负责将参与方角色与印章资源匹配。
- **印章资源快照在编排时固化**：签章申请准入时，根据场景策略和编排策略，确定每个参与方使用的印章资源，并将印章资源版本快照（含授权矩阵快照）固化到签章申请中，不因后续印章资源变更而自动漂移。
- **本文档不展开印章资源细节**：印章资源的治理细节见 `seal-resource-and-certificate-design.md`，本文档只描述签署场景与印章资源的关联关系和快照固化边界。

### 6.2 与渲染策略的关联关系

- **签署场景决定渲染要求**：不同签署场景可能有不同的渲染要求（如骑缝签需要跨页渲染、普通签需要在固定位置渲染），场景策略中可以定义"该场景下的渲染策略类型"。
- **参与方与坐标方案关联**：编排策略确定参与方后，每个参与方的签章位置由坐标方案（见 `signature-coordinate-and-rendering-design.md`）确定，编排策略负责将参与方与坐标方案中的锚点定义匹配。
- **渲染上下文在会话建立时固化**：签章会话建立时，根据场景策略、编排策略和坐标方案，生成渲染上下文（含印章样式引用、坐标锚点、渲染参数），固化到签章会话中，不因后续坐标方案变更而自动漂移。
- **本文档不展开渲染实现细节**：渲染策略的治理细节见 `signature-coordinate-and-rendering-design.md`，本文档只描述签署场景与渲染策略的关联关系和上下文固化边界。

## 7. 最小可执行基线

### 7.1 配置结构

参与方编排配置以 `signer_orchestration_config` 为发布单元，最小结构如下：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `scenario_policy_id` | string | 是 | 无 | 签署场景策略主键 |
| `scenario_policy_version_id` | string | 是 | 无 | 场景策略版本 |
| `orchestration_policy_id` | string | 是 | 无 | 编排策略主键 |
| `participant_source_list` | array | 是 | 无 | 合同主体、审批节点、组织规则、人工指定等来源 |
| `order_mode` | enum | 是 | `SEQUENTIAL` | `SINGLE` / `SEQUENTIAL` / `PARALLEL` / `MIXED` |
| `completion_rule` | enum | 是 | `ALL_REQUIRED` | `ALL_REQUIRED` / `KEY_SIGNER_REQUIRED` / `MIN_COUNT_REQUIRED` |
| `min_completed_count` | integer | 条件必填 | 无 | 最低签署人数规则使用 |
| `parallel_group_rule` | object | 否 | 空 | 并行组划分规则 |
| `snapshot_required` | boolean | 是 | `true` | 是否固化参与方身份快照 |

### 7.2 核心状态机

| 对象 | 状态 | 允许转换 | 触发条件 |
| --- | --- | --- | --- |
| 编排策略版本 | `DRAFT` | `PUBLISHED` / `ARCHIVED` | 规则校验通过后发布；废弃时归档 |
| 编排策略版本 | `PUBLISHED` | `FROZEN` / `REVOKED` | 被申请消费后冻结；规则不再适用时撤销 |
| 参与方任务 | `WAITING` | `READY` / `CANCELLED` | 前序任务完成或申请撤销 |
| 参与方任务 | `READY` | `SIGNED` / `REJECTED` / `FAILED` | 签署动作完成、拒签或失败 |
| 会话 | `OPEN` | `PARTIALLY_SIGNED` / `COMPLETED` / `FAILED` | 参与方状态汇总推进 |

### 7.3 关键算法伪代码

```text
build_signer_assignments(request, policy):
  candidates = []
  for source in policy.participant_source_list:
    candidates += resolve_participants(source, request.contract_snapshot, request.approval_summary)
  signer_list = deduplicate_by_identity_and_role(candidates)
  assert signer_list not empty
  for signer in signer_list:
    signer.snapshot = freeze_identity_snapshot(signer)
  ordered = apply_order_mode(signer_list, policy.order_mode, policy.parallel_group_rule)
  assert validate_completion_rule(ordered, policy.completion_rule, policy.min_completed_count)
  return create_es_signer_assignment_rows(request, ordered)
```

### 7.5 异常分流表

| 异常 | 分流结果 | 处理动作 |
| --- | --- | --- |
| 参与方来源为空 | 阻断会话建立 | 申请停留在 `ADMITTED`，生成编排修复任务 |
| 组织规则解析多义 | 转人工确认 | 固化候选清单，等待管理员选择 |
| 顺序规则成环 | 阻断发布或建立 | 编排策略版本不得发布，已发布版本进入撤销流程 |
| 关键参与方拒签 | 会话失败或转人工 | 按场景策略进入 `FAILED` 或人工确认 |
| 并行组部分超时 | 部分失败处理 | 未完成任务转人工或按完成规则判断是否闭环 |

## 8. 治理原则总结

- 签署场景以"版本快照"为核心治理模式，确保正在执行的签章链路不因场景策略变更而中断。
- 参与方编排以"编排快照"为核心治理模式，确保已固化的参与方列表、角色分配、顺序关系不因后续规则变更而自动漂移。
- 签署场景、编排策略、印章资源、渲染策略四者平行治理，变更互不影响，但共同构成签署执行的完整上下文。
- 跨模块协作以"引用而非持有、消费快照而非实时查询、执行与治理分离"为原则，确保各模块边界清晰、责任明确、审计可追溯。
- 不同签署场景（单人签、多人顺序签、多人并行签、骑缝签、补签、重签）有各自独立的治理边界，不因场景复杂度的差异而混淆治理层次。
