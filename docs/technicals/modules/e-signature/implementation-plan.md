## 2. 计划边界与原则

本计划只回答“先做什么、后做什么、依赖什么、完成到什么程度才进入下一阶段”。
以下内容不在本文展开：

- 不复述需求范围、一期功能清单与业务背景
- 不重写技术选型理由、系统拓扑与模块分层
- 不下沉物理表设计、接口字段、错误码、状态枚举与回调报文
- 不展开签章坐标、证书介质、验签算法、底层文件处理细节
- 不写成人天粒度排期表、负责人排班表或逐日项目管理表

本计划执行时遵循以下原则：

- 电子签章是平台内正式子模块，按平台正式能力建设，不按外部接入壳推进
- 合同主档是业务真相源，电子签章围绕 `contract_id` 挂接，不拥有合同主档
- 文档中心是文件真相源，电子签章只消费和回写文件引用，不自建第二套文件真相
- 签章动作建立在审批通过后的正式合同主链上，准入判断以平台承接后的审批事实为准
- 签章结果必须回写合同主档摘要与时间线，同时回收到文档中心版本链
- 不把电子签章内部过程状态整体塞进合同主档接口，合同侧只接收稳定业务摘要
- 阶段推进遵循“先准入、再会话、再结果回写、再纸质备案与周边挂接、最后联调验收上线”
- 电子签章不依赖外部测试账号作为成立前提，外部条件只用于真实需要的联调环节

## 3. 前置条件

### 3.1 合同主档、文档中心、流程引擎前置条件

- 合同主档需已具备稳定 `contract_id`、合同当前业务状态、主文档引用、
  合同时间线写入口与签章摘要挂接位置
- 合同主档需能表达“允许发起签章”的正式业务前置，不要求暴露电子签章内部过程态
- 文档中心需已具备正式待签输入稿引用能力，能稳定提供
  `document_asset_id`、`document_version_id`、文档归属与版本有效性判断
- 文档中心需已具备结果稿和验签产物的正式回收入口，保证结果稿进入版本链后再作为正式文件引用
- 流程引擎或审批摘要承接层需已形成统一审批通过事实，至少可稳定提供审批结果、完成时间、
  审批承载方式与摘要引用
- `OA` 为主审批路径时，平台侧审批承接链路需先收口为平台可消费的审批摘要，
  电子签章不直接依赖外部审批原始过程态

### 3.2 签章输入稿、签章结果稿、验签产物回写前置条件

- 签章输入稿必须来自审批通过后的正式合同主链，并在发起时冻结到明确文档版本
- 签章结果稿只能以文档中心正式回写成功后的版本作为稳定引用，不能先写合同摘要再补文件真相
- 验签产物按文档中心受控引用治理，不作为签章模块私有文件沉淀
- 输入稿、结果稿、验签产物之间需有稳定绑定关系，保证后续对账、恢复与审计可追踪

### 3.3 审计、任务、通知、权限前置条件

- 平台审计链路需已能记录签章申请、准入拒绝、会话推进、结果回写、人工介入等关键动作
- 统一任务中心需可承接会话建立、验签整理、文档回写、合同回写、摘要重建与补偿任务
- 通知能力需具备最小可用出口，用于签署推进、异常告警与人工介入提示
- 权限体系需已具备签章发起、印章使用、结果查看、纸质备案确认、管理操作等授权边界
- 组织与身份数据需可用于签署参与方快照固化，但电子签章不反向维护组织主数据

## 4. 总体阶段划分

本模块按五个阶段推进：

1. 签章申请与准入基础能力
2. 签章会话与参与方控制能力
3. 签章结果、文档回写与合同摘要回写能力
4. 纸质备案与周边挂接能力
5. 联调、验收与上线准备

阶段之间的总体依赖关系如下：

- 阶段 1 为签章正式入口，没有准入能力就无法保证后续链路建立在正式合同主链上
- 阶段 2 建立签署运行时控制，没有会话和参与方控制就无法形成稳定结果
- 阶段 3 建立结果闭环，没有结果回写闭环就不能把签章从模块内事件变成平台正式事实
- 阶段 4 在主链闭环已成立后，再补纸质备案与归档、搜索、AI 等周边挂接，避免外围消费倒逼主链返工
- 阶段 5 用于把前述能力从“可开发”推进到“可联调、可验收、可上线”

## 5. 分阶段实施计划

### 5.1 阶段一：签章申请与准入基础能力

**目标**

- 建立电子签章的正式业务入口
- 固化“只有审批通过后的正式合同主链才能发起签章”这一本模块准入前提
- 固化待签输入稿必须来自文档中心正式版本的入口约束

**核心产出**

- 签章申请受理链路与申请编号机制
- 准入校验规则，包括合同状态、审批摘要、输入稿版本、权限与幂等控制
- 合同主档、审批摘要、文档中心输入稿的申请快照绑定机制
- 申请失败、准入拒绝、重复申请冲突的稳定处理口径
- 面向审计和任务中心的最小接入点

**前置依赖**

- 合同主档可提供稳定合同摘要与主文档引用
- 平台已可消费审批通过事实或等价审批摘要
- 文档中心已可提供正式输入稿引用与版本有效性判断
- 权限与审计底座可提供最小可用能力

**完成标志**

- 平台可针对一份审批通过的正式合同受理签章申请
- 审批未通过、审批撤回、合同状态不合法、输入稿版本失效等场景能稳定拒绝准入
- 同一合同同一输入稿的重复发起具备稳定幂等或稳定冲突结果
- 申请受理与拒绝结果可被审计和追踪

### 5.2 阶段二：签章会话与参与方控制能力

**目标**

- 在准入成立后，建立可推进、可超时、可恢复的签章会话机制
- 固化参与方快照、签署顺序和会话推进规则
- 让签章运行时具备独立治理能力，但不把运行时状态外泄为合同主档真相

**核心产出**

- 会话创建、续轮、关闭、超时检查与异常终止机制
- 参与方展开、签署顺序控制、当前待签人聚合与会话完成判断
- 会话推进与回调归一的状态控制边界
- 会话相关任务、锁、重试与人工介入入口
- 面向通知能力的签署推进、超时和异常提醒挂点

**前置依赖**

- 阶段一的申请受理与准入快照已稳定
- 组织与身份数据可用于参与方快照固化
- 统一任务中心可承接会话建立和超时处理
- 通知能力可承接最小提示动作

**完成标志**

- 通过准入的申请可稳定进入会话，并能跟踪当前进度与参与方摘要
- 回调乱序、重复推进、会话超时等场景有稳定控制结果
- 会话内部状态留在签章模块治理，不要求合同主档感知全过程
- 失败会话可进入恢复或人工介入，而不是形成黑盒中断

### 5.3 阶段三：签章结果、文档回写与合同摘要回写能力

**目标**

- 让签章完成从“模块内完成”升级为“平台正式闭环完成”
- 先把签章结果稿和验签产物回收到文档中心，再把稳定摘要回写合同主档与时间线
- 建立失败补偿和部分成功可恢复机制

**核心产出**

- 签章结果登记、验签结果登记与结果摘要读模型
- 文档中心结果稿回写链路与验签产物引用回收链路
- 合同主档摘要与时间线回写链路
- 文档回写优先、合同回写随后、摘要重建收尾的顺序控制
- 文档回写失败、合同回写失败、验签失败的补偿与恢复路径

**前置依赖**

- 阶段二已可形成稳定会话结果
- 文档中心已准备好结果稿正式建链与产物回收入口
- 合同主档已准备好签章摘要与时间线写入口
- 统一任务中心、审计中心与告警能力可承接回写补偿

**完成标志**

- 签章完成后可在文档中心看到正式结果稿引用和验签产物引用
- 合同主档可看到稳定 `signature_status`、`signed_at`、验签结论摘要与时间线事件
- 不把 `request_status`、`session_status`、参与方临时态整体写回合同主档
- 文档回写失败或合同回写失败时，系统能保持部分成功状态并持续补偿，不把合同误判为已闭环完成

### 5.4 阶段四：纸质备案与周边挂接能力

**目标**

- 在电子签章主链闭环稳定后，再补纸质备案这一辅助形态
- 建立归档、搜索、AI 等周边模块对稳定签章摘要的消费路径
- 保证周边只消费稳定摘要和文档中心正式引用，不反向依赖签章内部过程态

**核心产出**

- 纸质备案登记、核验、确认、作废与互斥规则
- 纸质扫描件与文档中心正式版本的绑定机制
- 面向归档、搜索、AI 的稳定摘要投影与刷新机制
- 纸质备案与电子签正式结果的优先级与替代规则
- 周边挂接时的审计、任务与异常恢复口径

**前置依赖**

- 阶段三的结果回写闭环已成立
- 文档中心可提供 `PAPER_SCAN` 等价正式引用
- 归档、搜索、AI 已有可消费签章摘要的稳定入口
- 审计与任务中心可承接纸质备案重核与摘要刷新

**完成标志**

- 纸质备案可作为辅助路径登记线下签署事实，但不创建电子签章会话
- 归档、搜索、AI 只消费稳定签章摘要和文档中心结果引用，不直接读取签章会话过程表
- 电子签章与纸质备案并存、替代、作废场景有稳定判断，不冲突合同主档和文档中心真相

### 5.5 阶段五：联调、验收与上线准备

**目标**

- 把前四阶段能力从模块内可运行推进到跨模块可联调、可验收、可上线
- 收口准入、会话、回写、纸质备案和补偿恢复的全链路验证结论
- 准备上线切换、回滚、人工介入和运行监控方案

**核心产出**

- 合同主档、文档中心、流程摘要、审计、任务、通知等联调用例与对账口径
- 关键主链路、异常链路、补偿链路、恢复链路的验收场景清单
- 上线前数据检查项、配置检查项、权限检查项、告警检查项
- 上线观察指标、人工介入值班口径与问题分流原则
- 首批上线范围和灰度策略的执行准备

**前置依赖**

- 前四阶段核心能力已具备可验证实现
- 联调对象已提供稳定测试环境或等价联调入口
- 上线环境已具备最小部署、日志、任务执行与审计查询能力

**完成标志**

- 端到端主链路可跑通：审批通过 -> 发起签章 -> 会话推进 -> 结果回写文档中心 -> 合同摘要回写
- 关键异常链路可验证：准入拒绝、会话超时、验签失败、文档回写失败、合同回写失败、纸质备案冲突
- 上线检查项、回滚口径、人工介入口径和监控项已准备完成
- 本模块可以按平台正式子模块身份进入首批上线范围，而不是以“后续补齐”方式挂空壳上线

## 6. 模块推进顺序

本模块按“先准入、再会话、再结果回写、再纸质备案与周边挂接”的顺序推进，原因如下：

- 先做准入，是为了先锁死本模块成立边界。没有准入能力，就无法保证所有签章动作都建立在审批通过后的正式合同主链上，也无法保证输入稿来自文档中心正式版本。
- 再做会话，是因为签章不是一次性写结果动作，而是需要参与方控制、顺序控制、超时控制和异常控制的运行时过程。没有会话层，后续结果只能依赖临时流程拼接，难以审计和恢复。
- 再做结果回写，是因为只有在运行时结果稳定后，才适合建立文档中心回写和合同主档回写的正式闭环。若过早做回写，容易把未稳定的过程态误写入合同主档或文档中心。
- 最后再做纸质备案与周边挂接，是为了让归档、搜索、AI 等外围消费建立在稳定摘要之上，避免外围模块提前绑定签章内部过程态，也避免纸质辅助路径反向污染电子签章主链设计。

对应的实施判断是：

- 准入解决“能不能开始”
- 会话解决“过程怎么推进”
- 回写解决“结果怎么成为平台正式事实”
- 纸质备案与周边挂接解决“主链稳定后如何扩展与消费”

## 7. 外部联调准备

本模块只对真正需要外部条件的部分做联调准备，不把电子签章整体写成外部测试账号依赖系统。

需要外部条件的部分主要包括：

- `OA` 主审批路径场景下，平台审批承接链路需能稳定提供审批通过事实或审批摘要，
  供签章准入消费
- 如签署通知需要走企业级外发通道，需准备对应通道配置与可验证出口，但这属于通知链路联调，
  不是电子签章成立前提
- 如存在企业侧真实证照、印章授权或法务确认口径，需要在联调阶段确认其配置与操作边界，
  但不改变“电子签章是平台内正式子模块”的定位

不作为本模块成立前提的事项包括：

- 不要求外部测试账号先行到位后，电子签章子模块才可开发或自测
- 不把合同主档、文档中心、流程引擎等平台内模块写成“外部依赖系统”
- 不把签章输入稿、结果稿、验签产物的内部回写链路写成外部接口前置

联调准备重点应放在：

- 审批摘要口径对齐
- 合同主档摘要回写口径对齐
- 文档中心结果稿建链与引用对齐
- 审计、任务、通知、人工介入的跨模块责任边界对齐

## 8. 测试、验收与上线准备

### 8.1 测试重点

- 准入测试：合同状态、审批摘要、输入稿版本、权限、幂等冲突
- 会话测试：参与方顺序、会话超时、重复回调、乱序回调、异常关闭
- 结果测试：验签通过、验签失败、结果登记、摘要生成
- 回写测试：文档中心回写成功、合同主档回写成功、先文档后合同的顺序控制
- 补偿测试：文档回写失败、合同回写失败、摘要重建失败、人工介入转单
- 纸质备案测试：登记、核验、确认、作废、与电子签结果冲突处理

### 8.2 验收重点

- 电子签章以平台正式子模块方式运行，不依赖外部测试账号前置
- 签章动作只建立在审批通过后的正式合同主链上
- 合同主档仍是业务真相源，文档中心仍是文件真相源
- 签章结果已同时回写合同主档摘要与时间线，并回收到文档中心版本链
- 合同主档未被塞入电子签章内部全过程状态
- 关键异常、补偿、恢复与审计链路可验证

### 8.3 上线准备

- 核对签章相关配置、权限、印章可用性、任务执行器、通知出口、审计查询入口
- 核对合同主档与文档中心的联调版本，避免上线时出现回写契约漂移
- 准备首批合同样本、首批签署组织范围与灰度启用策略
- 准备上线观察指标，至少覆盖申请成功率、会话超时率、验签失败率、
  文档回写失败率、合同回写失败率、人工介入量
- 准备回滚和人工接管口径，确保在回写链路异常时可暂停新申请、保留已产生结果并继续补偿
