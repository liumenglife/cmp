# 流程引擎专项设计：并行与会签聚合设计

## 1. 文档说明

本文档用于继续下沉 `workflow-engine` 主线中"并行分叉收敛与会签聚合治理"这一能力，重点回答以下问题：

- 并行分叉与收敛如何在运行时成为正式受控语义，而不是散落在配置字段中的自由分支
- 会签如何在不另起独立实例的前提下，形成可治理的参与人票数、聚合结果与收敛判定
- 百分比阈值、全票通过、多数票、一票否决等收敛规则如何具备精确参数与取整边界
- 超时降级、投票冲突回收、异常中断如何在并行与会签语义下被正式收口
- 节点 DSL 如何约束并行与会签的声明边界，避免运行时猜测
- 审计留痕如何覆盖聚合判定全过程，保证事后可回放、可追责

## 2. 设计目标

- 让并行分叉收敛与会签聚合成为流程引擎内部正式治理对象，而不是配置端可随意拼装的运行时黑盒
- 让收敛规则具备精确的算法参数、边界条件和取整约定，避免不同节点自行解释"通过"
- 让超时、异常、投票冲突等边缘场景在聚合层被统一收口，而不是由各分支任务各自为政
- 让节点 DSL 中的并行与会签声明具备发布级约束力，不允许运行时遭遇未声明的收敛语义
- 让聚合判定全过程可审计、可回放、可争议仲裁，避免事后无法还原"为什么在这一刻收敛了"

## 3. 正式治理对象

并行与会签聚合的正式治理对象，不是"分支任务都完成了就往下走"这一运行时直觉，而是以下四类对象的组合：

1. `Concurrency Declaration`：节点 DSL 中对并行分叉、并行收敛、会签参与模式的正式声明，是版本发布时冻结的静态治理入口。
2. `Aggregation Rule`：收敛判定所依据的规则集，包括收敛模式、阈值参数、取整策略、冲突处理策略和超时降级策略。
3. `Aggregation State`：运行时为每个并行组或会签节点维护的聚合计数、分支状态、收敛结果和判定证据快照。
4. `Aggregation Decision`：一次收敛判定的正式结论，包括判定时间、依据的规则版本、输入事实摘要、判定结果和触发原因（正常收敛、超时降级、管理员干预）。

只有当一个并行组或会签节点同时具备"声明 + 规则 + 状态 + 决策"四层治理对象时，其收敛行为才算正式成立。由此，聚合的治理重心是"版本化规则 + 运行时证据"，而不是"所有子任务都完成了就往前走"。

在父文档口径中，这一治理关系已经正式收口为两层承接：

- 静态声明与规则冻结在 `wf_node_definition.participant_mode`、`join_group_key`、`route_config`、`time_limit_config`。
- 运行时 `Aggregation State` / `Aggregation Decision` 统一落到 `wf_node_aggregation`：当前聚合状态由 `aggregation_status + rule_snapshot + participant_snapshot + counter_snapshot + last_fact_action_id` 承接；每次正式判定则通过 `latest_decision_action_id + latest_decision_snapshot_ref` 引用不可变证据快照，供恢复、审计与回放使用。

## 4. 并行节点语义

### 4.1 并行分叉

并行分叉是流程从单一执行路径分裂为多条可并发执行分支的语义动作。其治理边界如下：

- 分叉必须由 `PARALLEL_GATEWAY` 类型节点显式声明，不允许审批节点隐式生成多分支
- 分叉时 `Runtime Orchestrator` 为每个分支生成独立任务集，各分支任务的 `task_id` 不同但共享同一 `process_id` 和 `join_group_key`
- 分叉节点本身不承载审批参与人，不产生审批任务
- 分叉后各分支独立推进，互不阻塞，直到遭遇同一收敛组的收敛节点

### 4.2 并行收敛

并行收敛是多条分支汇合为单一执行路径的语义动作。其治理边界如下：

- 收敛必须由与分叉节点配对的收敛节点声明，通过 `join_group_key` 关联
- 收敛节点在同一版本内必须且只能关联一个分叉节点，不允许一对多或多对多
- 收敛判定必须基于 `Aggregation Rule` 执行，不允许运行时自行选择收敛策略
- 未命中收敛条件前，实例的 `current_node_id` 停留在收敛组语义节点，不推进到后续节点
- 收敛判定完成后，所有未完成分支任务统一关闭为 `CANCELED`

### 4.3 拓扑约束

- 并行分叉与收敛必须成对出现，不允许只存在分叉而无收敛
- 同一版本内，分叉节点的出边数必须与收敛节点的入边数一致，且通过 `join_group_key` 一一配对
- 并行分支内部不允许再嵌套同一 `join_group_key` 的并行分叉，避免递归收敛歧义
- 分叉节点与收敛节点之间允许存在不同长度的分支路径，但每个分支都必须最终可达收敛节点

## 5. 会签节点语义

### 5.1 会签定位

会签是审批节点的一种参与模式，由 `participant_mode=COUNTERSIGN` 声明，不另起独立流程实例。其核心语义是：多个参与人各自独立审批，节点层根据聚合规则判定是否收敛。

### 5.2 任务生成

- 每个会签参与人生成独立 `wf_approval_task`，任务类型为 `COUNTERSIGN`
- 所有会签子任务共享同一 `process_id`、`node_id` 和 `node_key`
- 会签子任务之间互不依赖，各参与人可独立审批
- `COUNTERSIGN_ADD` 只允许在节点仍处于会签打开态时追加新任务，不允许在收敛后补加

### 5.3 聚合计数

节点层在运行时维护以下聚合计数：

- `total_expected`：预期总参与人数（含动态追加）
- `submitted_count`：已提交审批意见的人数
- `approve_count`：通过票数
- `reject_count`：驳回票数
- `pending_count`：未完成人数
- `timeout_count`：超时未处理人数
- `transferred_count`：转办后由他人代处理的人数（原参与人退出计数）

聚合计数必须在事务内与任务状态同步更新，不允许任务已关闭但计数未刷新。

### 5.4 会签追加

- `COUNTERSIGN_ADD` 动作只允许在会签节点处于打开态时执行
- 追加后 `total_expected` 递增，新任务按当前绑定规则生成
- 追加不影响已完成参与人的投票结果
- 追加的审计记录必须标注追加来源（管理员、规则自动追加、转办后补人）

## 6. 聚合规则

### 6.1 收敛模式

并行收敛与会签收敛共享以下收敛模式枚举：

- `ALL`：全部分支或全部参与人完成才收敛
- `ANY`：任一分支或任一参与人命中指定结果即收敛
- `PERCENTAGE`：按比例阈值收敛
- `MAJORITY`：按多数票收敛（超过半数）
- `UNANIMOUS`：全票通过才收敛（仅适用于会签）

### 6.2 百分比阈值与取整规则

`PERCENTAGE` 模式下的参数与取整约定如下：

- `threshold_percent`：收敛所需通过百分比，取值范围 `(0, 100]`
- `rounding_mode`：百分比计算后的取整方式，固定为以下三种之一：
  - `CEILING`：向上取整。例如 7 人的 60% 计算为 4.2，取整为 5，即至少需要 5 人通过
  - `FLOOR`：向下取整。例如 7 人的 60% 计算为 4.2，取整为 4，即至少需要 4 人通过
  - `ROUND_HALF_UP`：四舍五入。例如 7 人的 60% 计算为 4.2，取整为 4
- `basis`：百分比计算基数，固定为以下两种之一：
  - `TOTAL_EXPECTED`：以预期总人数为基数，含动态追加
  - `SUBMITTED`：以已提交人数为基数，用于"已提交中通过比例"语义
- 最小通过人数公式：当 `basis=TOTAL_EXPECTED` 时，`min_pass_count = rounding(threshold_percent / 100 * total_expected)`。当结果为 0 且 `threshold_percent > 0` 时，`min_pass_count` 强制为 1，避免"阈值大于零但无需任何人通过"的语义矛盾。

`basis=SUBMITTED` 的附加约束如下：

- 只允许用于 `participant_mode=COUNTERSIGN` 的节点，不用于 `PARALLEL_GATEWAY` 收敛节点。
- 若节点允许 `COUNTERSIGN_ADD`，则必须同时声明“首次提交后禁止动态追加”；一旦 `submitted_count > 0`，不得再追加参与人。
- 发布校验时，若同一节点同时声明 `basis=SUBMITTED` 与“提交后仍允许动态追加”，必须直接阻断发布。

取整模式与计算基数必须在节点 DSL 中显式声明，不允许运行时使用默认值。

### 6.3 全票通过

`UNANIMOUS` 模式下的规则如下：

- 所有预期参与人（含动态追加）均以通过结果提交时才收敛
- 任意一人驳回即触发收敛，结论为驳回
- 超时未处理的参与人不视为通过，不计入全票基数

### 6.4 多数票

`MAJORITY` 模式下的规则如下：

- 通过人数超过预期总人数的半数时收敛，结论为通过
- 驳回人数超过预期总人数的半数时收敛，结论为驳回
- 计算公式：`pass_threshold = floor(total_expected / 2) + 1`，`reject_threshold = floor(total_expected / 2) + 1`
- 当总人数为偶数时，恰好半数不构成多数，需超过半数

### 6.5 规则优先级

当节点同时配置多种收敛条件时，按以下优先级判定：

1. 驳回优先：任意驳回结果满足一票否决条件时，立即以驳回收敛（若启用一票否决）
2. 通过收敛：通过票数满足阈值条件时，以通过收敛
3. 超时降级：超时触发后按降级策略判定（见第 7 节）

一票否决是 `UNANIMOUS` 模式的默认行为。在 `PERCENTAGE` 和 `MAJORITY` 模式下，一票否决需显式启用 `veto_enabled=true`。

## 7. 超时降级

### 7.1 超时触发

- 会签或并行节点的超时由 `wf_task_timer` 驱动，不在请求线程内轮询
- 超时以节点级 `time_limit_config` 中的 `timeout_at` 或 `timeout_duration` 为依据
- 超时触发后，未完成的分支或参与人任务标记为 `TIMEOUT`，不自动关闭为 `CANCELED`

### 7.2 降级策略

超时触发后可执行的降级策略，与父文档 `time_limit_config.timeout_strategy` 使用同一套正式枚举：

- `FORCE_PASS`：超时参与人的票数按通过计入，用于"不反对即同意"语义；仅允许用于并行收敛或会签聚合节点
- `FORCE_REJECT`：超时参与人的票数按驳回计入，用于"不表态即反对"语义
- `EXCLUDE_FROM_BASE`：超时参与人不计入总基数，百分比和多数票基于剩余有效人数重算；仅允许用于并行收敛或会签聚合节点
- `ESCALATE`：超时不自动判定，升级到管理员或上级审批人人工裁定

### 7.3 降级后的重算

- `FORCE_PASS` 和 `FORCE_REJECT` 下，超时票数直接加入对应计数，触发重新收敛判定
- `EXCLUDE_FROM_BASE` 下，`total_expected` 递减，阈值基于新基数重算，已提交票数不变
- `ESCALATE` 下，聚合状态转入 `AWAITING_MANUAL_DECISION`，不自动收敛

### 7.4 降级策略声明

降级策略必须在节点 DSL 中显式声明，不允许运行时使用隐式默认值。声明至少包括：

- `timeout_strategy`：降级策略枚举
- `timeout_escalation_target`：当策略为 `ESCALATE` 时的升级目标
- `timeout_force_result`：当策略为 `FORCE_PASS` 或 `FORCE_REJECT` 时的结果映射

## 8. 投票冲突回收

### 8.1 冲突场景

投票冲突是指在会签收敛判定中，通过与驳回同时满足不同收敛条件，或因动态追加、超时降级导致收敛结论不稳定的场景。典型冲突包括：

- 通过票数已达到 `PERCENTAGE` 阈值，但同时存在未处理参与人且未启用超时
- 驳回票数触发一票否决，但通过票数也满足多数票阈值
- 超时降级后，强制票数导致收敛结论与之前部分参与人的实际投票结果矛盾
- 动态追加参与人后，此前已达标的阈值被重新稀释

### 8.2 冲突消解原则

- 驳回优先：当一票否决启用时，驳回结果的优先级高于任何通过收敛条件
- 决策冻结：一旦聚合判定形成正式 `Aggregation Decision`，后续动态追加或超时降级不得回溯修改已冻结的决策，只允许基于新事实形成新决策
- 证据保留：冲突消解过程必须写入 `Aggregation Decision` 的判定依据字段，标注冲突来源、消解规则和最终裁定理由
- 管理员兜底：当自动冲突消解无法形成唯一结论时，聚合状态转入 `AWAITING_MANUAL_DECISION`

### 8.3 回收机制

回收是指因冲突或异常导致已做出的收敛判定需要被撤销或修正的场景。回收边界如下：

- 只允许在实例仍处于运行态时回收，已进入终态的实例不允许回收
- 回收必须由管理员显式触发或由系统补偿逻辑触发，不允许静默回收
- 回收后，聚合状态回退到回收前的最后一个稳定快照，重新执行收敛判定
- 回收动作必须写入审计记录，标注回收原因、回收前结论和回收后结论

## 9. 与节点 DSL 的约束关系

### 9.1 DSL 声明边界

并行与会签在节点 DSL 中的声明必须具备以下字段：

- `participant_mode`：`SINGLE`、`PARALLEL`、`COUNTERSIGN`，决定节点是否进入并发语义
- `join_group_key`：并行收敛组键，用于分叉与收敛节点配对
- `convergence_mode`：收敛模式枚举（`ALL`、`ANY`、`PERCENTAGE`、`MAJORITY`、`UNANIMOUS`）
- `convergence_params`：收敛参数对象，包括 `threshold_percent`、`rounding_mode`、`basis`、`veto_enabled`
- `timeout_strategy`：超时降级策略枚举
- `timeout_params`：超时降级参数对象

### 9.2 发布级约束

- `participant_mode=COUNTERSIGN` 时，`convergence_mode` 和 `convergence_params` 为必填，不允许省略
- `participant_mode=PARALLEL` 只表示单个审批节点内部的并发参与人处理，不表达流程拓扑分叉；凡是分叉 / 收敛结构，必须使用 `node_type=PARALLEL_GATEWAY`
- `node_type=PARALLEL_GATEWAY` 的分叉节点与收敛节点必须显式声明 `join_group_key` 并在同版本内完成配对
- `convergence_mode=PERCENTAGE` 时，`threshold_percent`、`rounding_mode`、`basis` 三者均为必填
- `basis=SUBMITTED` 时，节点必须声明“首次提交后禁止动态追加”；否则阻断发布
- `timeout_strategy=ESCALATE` 时，`timeout_escalation_target` 为必填
- `timeout_strategy=FORCE_PASS` 或 `EXCLUDE_FROM_BASE` 时，节点必须属于并行收敛或会签聚合语义；否则阻断发布
- 上述约束在发布校验阶段由校验器强制执行，缺失或矛盾时阻断发布

### 9.3 运行时不可变性

- 收敛模式和参数一经发布冻结，运行时不得修改
- 运行时只能基于冻结的规则执行聚合判定，不得自行选择更宽松或更严格的收敛策略
- 若运行时发现规则无法执行（如基数为零导致百分比无意义），应进入 `FAILED` 状态并触发人工介入，不得静默降级为 `ALL` 或 `ANY`

## 10. 审计留痕

### 10.1 审计事件覆盖

围绕并行与会签聚合，审计留痕至少覆盖以下事件：

1. 分叉事件：何时进入并行分叉、生成了哪些分支任务、各分支的初始参与人
2. 投票事件：每个参与人的投票时间、投票结果、是否超时、是否转办后由他人代投
3. 追加事件：何时追加了会签参与人、追加来源、追加后总基数变化
4. 收敛判定事件：何时触发收敛判定、判定依据的规则版本、输入计数快照、判定结论
5. 超时降级事件：何时触发超时、执行了何种降级策略、降级后计数变化
6. 冲突消解事件：是否发生冲突、冲突类型、消解规则、最终裁定
7. 回收事件：何时触发回收、回收原因、回收前后的结论差异

### 10.2 审计原则

- 审计记录引用 `process_id`、`node_id`、`join_group_key`、收敛规则版本
- 每次收敛判定必须记录完整的 `Aggregation Decision`，包括输入事实、规则参数、判定结论和触发原因
- 超时降级和冲突消解必须记录消解路径，不允许只记录最终结论
- 审计保存判定证据和规则快照，不保存与聚合无关的审批意见正文和附件内容
- 如后续发生实例争议，应能回答"在什么时刻、基于什么规则、依据什么事实、做出了什么收敛判定"

这正是并行与会签成为正式治理对象的关键证据链。

## 11. 与其他专项设计的边界

- 与 `workflow-dsl-and-validator-design` 的边界：本文只定义并行与会签的聚合算法参数、取整规则和收敛语义；节点 DSL 的正式治理对象、分层校验器和发布校验证据链由前者下沉。
- 与 `org-rule-evaluator-design` 的边界：本文只定义会签参与人如何被消费为聚合输入；组织规则求值、候选人生成和选人算法由后者下沉。
- 与 `oa-bridge-mapping-and-compensation-design` 的边界：本文只定义平台引擎路径下的会签聚合语义；`OA` 主路径下的外部会签回调映射和补偿由后者下沉。
- 与 `task-executor-and-notification-runtime-design` 的边界：本文只定义聚合判定如何驱动任务状态变更；任务执行器的并发参数、扫描周期和通知触发时机由后者下沉。

## 12. 本文结论

并行与会签聚合之所以必须成为正式治理对象，是因为流程引擎真正需要治理的不是"所有分支都完成了就往下走"，而是"在什么规则下、基于什么事实、以什么结论收敛，以及谁可以事后追责这一判定"。

因此，本设计将并行与会签聚合收口为"声明 + 规则 + 状态 + 决策"四层治理对象，并明确了收敛模式、百分比取整、全票与多数票语义、超时降级、投票冲突回收、DSL 约束和审计留痕边界。

在这一前提下，流程引擎的并行与会签不再是配置端可随意拼装的运行时黑盒，而是具备版本化规则、精确参数和完整证据链的正式平台治理能力。
