# 流程引擎专项设计：流程 DSL 与校验器设计

## 1. 文档说明

本文档用于继续下沉 `workflow-engine` 主线中“流程定义语言与校验治理”这一能力，重点回答以下问题：

- 流程定义语言如何从画布保存载荷提升为平台内正式可治理对象
- 节点、边、条件、组织绑定、版本约束如何形成统一对象模型，而不是散落在配置页面字段中
- 校验器如何分层判断草稿可编辑、发布可运行、运行可审计三类不同语义
- 组织绑定校验与组织主数据解析之间如何划清接口边界，避免流程引擎复制组织规则真相
- 兼容升级、错误分类、审计留痕如何保证流程定义长期可发布、可回放、可追责

本文是以下文档的下游专项设计：

- [`workflow-engine Detailed Design`](../detailed-design.md)

`identity-access`、`contract-core`、`OA` 桥接能力、流程配置端是本文的强依赖或关键消费方：它们分别提供组织主数据规则、消费已发布流程版本、消费发布校验结果或承接配置录入，但这些关系属于跨模块协作关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不写前端画布保存协议、节点坐标、连线样式、版本比较页面或交互细节
- 不写对外 `API` 路径、请求字段、响应样例、错误码枚举全集
- 不写解析器类图、编译器代码结构、表达式执行器实现细节或数据库 `DDL`
- 不写实施排期、迁移项目计划、灰度步骤或上线手册

## 2. 设计目标

- 让 `Workflow DSL` 成为流程引擎内部正式受控语言，而不是配置端私有 JSON
- 让发布前校验成为流程版本生成的硬门禁，而不是保存成功后的人工约定
- 让节点、边、条件、组织绑定和版本元数据共享统一治理口径，避免运行时再做二次猜测
- 让草稿态与发布态具有不同约束强度，既允许配置迭代，又保证发布版本可执行、可审计
- 让流程升级和历史版本并存时保持稳定引用，不把新规则强行覆盖到旧实例上

## 3. 正式治理对象

流程定义语言的正式治理对象，不是“画布 JSON 文件”，而是以下四类对象的组合：

1. `Definition Envelope`：定义流程主身份、业务归属、适用组织域、当前 DSL 架构版本、治理元数据。
2. `Draft DSL Snapshot`：定义草稿态下的节点、边、条件、绑定引用与编辑态元数据，用于持续编辑和预校验。
3. `Published Runtime Snapshot`：由发布动作冻结出的可运行版本快照，是实例创建、运行时回放和审计追责的唯一依据。
4. `Validation Report`：记录本次校验的规则集版本、校验范围、结果级别、证据摘要和阻断原因，是发布治理对象的一部分，而不是临时日志。

只有当同一版本同时具备“可追溯的运行时快照 + 发布时校验报告 + 审计事件”时，该流程版本才算正式成立。由此，`Workflow DSL` 的治理重心是“版本化语言资产”，而不是“页面上能画出来的图”。

在父文档口径中，这一治理关系通过 `wf_process_version` 收口：版本主表至少稳定持有 `dsl_schema_version`、`validation_ruleset_version`、发布校验结果摘要、正式 `Validation Report` 引用和发布审计事件引用；整份报告正文以不可变证据对象持久化。这样发布校验结果不会退化成“发布前跑过一次检查”，而会成为版本治理、审计追溯和历史争议回放的正式组成部分。

## 4. DSL 对象模型

### 4.1 根对象

根对象 `WorkflowDefinitionDsl` 至少承载以下逻辑域：

- `definition_identity`：`definition_id`、`process_code`、`business_type`、适用承接模式
- `dsl_meta`：`dsl_schema_version`、草稿序列号、来源说明、校验规则集版本
- `graph`：节点集合、边集合、入口约束、结束约束
- `binding_policy`：节点绑定是否强制、绑定来源域、绑定缺失策略
- `publish_policy`：发布门禁、兼容级别声明、版本备注、冻结策略
- `audit_meta`：创建人、最近编辑人、编辑来源、变更摘要

根对象不承载页面坐标、缩放、颜色、前端临时选中态等表现信息。若配置端需要保留这些信息，应由独立画布协议承接，不能混入正式 DSL 真相。

### 4.2 节点对象

节点对象 `NodeSpec` 是流程 DSL 的核心治理单元，至少包含：

- 稳定节点键 `node_key`
- 节点类型 `node_type`
- 节点名称与业务说明
- 参与人策略与组织绑定引用
- 动作约束，如可审批、可驳回、可转办、可抄送
- 时限、提醒、升级等治理性约束摘要
- 与外部桥接或运行时能力相关的扩展声明

节点对象的关键约束如下：

- 一个版本内 `node_key` 必须稳定且唯一
- 节点类型决定其允许出现的属性族，禁止把所有节点做成自由字典
- 审批节点必须具备组织绑定语义；开始、结束、网关节点不得伪装成审批节点携带审批人配置
- 节点允许保留扩展槽，但扩展槽只能挂载已登记扩展域，不能成为绕过校验的任意字段口袋

### 4.3 边对象

边对象 `EdgeSpec` 用于表达节点之间的正式流转关系，至少包含：

- 稳定边键 `edge_key`
- `from_node_key` 与 `to_node_key`
- 路由类型，如默认流转、条件流转、并行分叉、收敛回边禁用声明
- 顺序优先级与互斥关系
- 条件引用或命中策略

边的核心治理目标不是描述画线方式，而是限定“何种节点之间允许形成何种流转关系”。因此，边对象必须受拓扑规则与节点类型规则双重约束。

### 4.4 条件对象

条件对象 `ConditionSpec` 只定义正式可治理条件，而不在本文展开表达式语法细节。其治理边界至少包括：

- 条件标识、所属边或所属节点
- 条件输入变量引用集合
- 比较与组合运算类别
- 空值、缺值、未知值处理策略
- 条件说明与命中优先级

条件对象必须只引用平台已登记的业务上下文字段、流程上下文字段或系统计算字段，不允许直接读取未登记对象、任意脚本结果或前端临时变量。

### 4.5 组织绑定对象

组织绑定对象 `BindingSpec` 只表达“流程节点需要什么绑定声明”，不表达“最终解析出谁”。它至少包括：

- `binding_type`：`USER`、`ORG_UNIT`、`ORG_RULE`
- 绑定目标引用
- 是否强制命中
- 组合关系与顺序
- 解析时需要传递的受控上下文摘要

这一定义很关键：`BindingSpec` 是 DSL 内的治理声明，不是运行时参与人结果。运行时参与人快照属于实例和任务真相，不能回写污染 DSL。

## 5. 节点、边、条件与版本约束

### 5.1 拓扑约束

- 每个流程版本必须且只能有一个正式开始节点
- 结束节点允许多个，但每个结束节点都必须可从开始节点可达
- 禁止无出口审批节点、不可达节点、孤立子图和未声明的回环
- 并行分叉与并行收敛必须成组出现，不能只存在一端
- 同一节点的多条条件边必须存在可判定的优先级或互斥声明，避免运行时结果不稳定

### 5.2 节点语义约束

- `START` 节点不得承载审批参与人和审批动作能力
- `APPROVAL` 节点必须绑定组织声明，且必须声明允许动作集合
- 网关节点必须声明分叉或收敛语义，不能同时兼任人工审批节点
- `END` 节点不得再有出边

### 5.3 条件语义约束

- 条件引用字段必须来自已登记变量域
- 条件运算符必须与字段类型兼容
- 同一条边的条件不能自相矛盾
- 条件链如存在兜底分支，兜底分支必须显式可识别，避免“未命中即随机落边”

### 5.4 版本约束

DSL 需要同时维护两类版本：

1. 流程业务版本：对应 `version_no`，用于业务发布、实例引用和历史追溯。
2. DSL 架构版本：对应 `dsl_schema_version`，用于解释该版本快照按哪一套语言规则生成。

二者不能互相替代。流程版本升级不必然意味着 DSL 架构升级；DSL 架构升级也不应强迫所有历史流程版本重写。

## 6. 校验器分层设计

校验器不是单个布尔判断，而是围绕正式治理对象的分层判定系统。建议收口为五层：

1. 结构校验：检查对象缺字段、类型错误、非法枚举、重复键。
2. 拓扑校验：检查开始结束、可达性、并行成对、条件边顺序、非法回路。
3. 语义校验：检查节点类型与属性族是否匹配、条件字段是否合法、动作是否冲突。
4. 绑定校验：检查组织绑定声明是否完整、引用目标是否存在、规则参数是否满足最小约束。
5. 发布校验：检查当前草稿是否具备发布前提，包括是否存在阻断错误、是否完成版本备注、是否满足兼容声明和审计要求。

分层的目的，是把“能保存”和“能发布”拆开。草稿阶段可允许部分高阶校验仅给出警告，但发布阶段必须执行完整阻断判断。

## 7. 草稿态与发布态差异

### 7.1 草稿态

草稿态的核心目标是允许持续编辑，因此遵循“可编辑优先、风险显性化”原则：

- 允许存在未完成节点命名、未闭合边、待补全绑定等中间状态
- 允许保存结构完整但语义未收口的条件草稿
- 校验结果可包含 `INFO`、`WARNING`、`ERROR`
- `ERROR` 阻断发布，但不必阻断草稿保存，除非已破坏最基本结构可读性

### 7.2 发布态

发布态的核心目标是形成正式运行快照，因此遵循“可执行优先、可追责优先”原则：

- 禁止任何阻断级错误进入发布版本
- 必须冻结节点键、边键、条件引用、绑定声明和 DSL 架构版本
- 必须生成正式 `Validation Report` 与版本摘要
- 必须留下发布操作者、发布时间、发布原因和变更摘要

其中“版本摘要”不只是描述版本备注，还应包含本次发布使用的规则集版本、阻断项是否清零、告警数量摘要和报告指纹，并由 `wf_process_version` 稳定引用，供实例启动校验、审计检索和版本回放复用。

### 7.3 状态差异的治理含义

草稿与发布不是同一份 JSON 的两个标签，而是两种不同的治理义务。草稿面向编辑协作，发布面向运行时与审计。因此，发布动作必须生成不可被编辑态覆盖的独立快照，避免“后来又改了草稿，历史实例解释被改写”。

## 8. 发布前校验与组织绑定接口边界

发布前校验应把流程引擎与组织底座的关系限定为“声明校验 + 可解析性确认”，而不是“把组织结果预先固化进版本”。边界如下：

- 流程引擎负责检查 `BindingSpec` 是否完整、引用标识是否合法、规则参数是否满足必填约束
- 组织底座负责确认被引用人员、部门、组织规则标识在当前组织域内存在且可用
- 对于 `ORG_RULE`，流程引擎只验证规则类型、参数结构和调用契约，不在 DSL 校验阶段重演完整选人结果
- 发布前允许做“样本解析”或“最小可解析性探测”，但探测结果只能作为校验证据，不能写回版本快照替代正式绑定声明

由此可避免两个问题：一是流程引擎复制组织真相导致规则漂移，二是把发布时的人员名单错误冻结为 DSL 内容。

## 9. 兼容升级策略

兼容升级关注的是“新 DSL 规则如何与历史版本共存”，而不是“如何把所有历史流程立即迁移”。建议收口为三类：

1. 向后兼容升级：旧版本可按原 `dsl_schema_version` 继续解释运行，新版本可使用新增能力。
2. 受控变更升级：允许新版本调整节点、边、条件或绑定模型，但必须生成新的业务版本，不覆盖旧发布版本。
3. 破坏性升级：当 DSL 架构升级导致旧解释器不再适用时，必须通过显式迁移动作生成新快照，并保留迁移前后映射审计。

兼容判断至少应覆盖以下场景：

- 节点键是否复用或被删除
- 条件字段是否更换语义
- 组织绑定类型是否发生不兼容变化
- 并行或会签节点是否改变收敛语义
- 外部桥接字段是否需要新增强制声明

历史实例始终绑定创建时的发布版本快照。无论后续草稿如何变动，都不得让运行中或已归档实例被新的 DSL 解释器重新解释为另一套流程。

## 10. 错误分类

校验器输出的错误必须具备正式分类，避免全部落为“配置错误”。建议至少分为六类：

1. `STRUCTURE_ERROR`：字段缺失、类型错误、非法枚举、重复键。
2. `TOPOLOGY_ERROR`：不可达、孤立、死路、非法回路、并行结构失衡。
3. `SEMANTIC_ERROR`：节点类型与属性冲突、条件字段或运算不合法、动作集合冲突。
4. `BINDING_ERROR`：组织绑定缺失、引用对象不存在、规则参数不完整、组织域不匹配。
5. `COMPATIBILITY_ERROR`：版本升级声明与实际变更不一致、历史兼容策略缺失。
6. `GOVERNANCE_ERROR`：缺少发布备注、审计字段不完整、规则集版本缺失、冻结前提未满足。

每条错误至少应带：错误分类、严重级别、对象定位、规则标识、面向人的解释、是否阻断发布。这样配置端、审计中心和后续专项治理才能基于同一份校验结果协作。

## 11. 审计留痕

围绕 DSL 与校验器，审计留痕至少覆盖三类事件：

1. 编辑事件：谁在何时修改了哪些节点、边、条件或绑定声明。
2. 校验事件：使用了哪一版校验规则、发现了哪些阻断项、结果是否通过。
3. 发布事件：谁基于哪份草稿发布了哪个版本、生成了什么运行时快照、兼容声明是什么。

审计留痕应坚持以下原则：

- 审计记录引用 `definition_id`、`version_id`、`dsl_schema_version`、校验规则集版本
- 审计保存摘要、指纹和差异说明，不保存与运行无关的前端表现噪声
- 发布时的 `Validation Report` 哈希和值班追踪号应可回查
- 如后续发生实例争议，应能回答“当时发布出去的正式流程到底长什么样、为什么被允许发布”

这正是 DSL 成为正式治理对象的关键证据链。

## 12. 与其他专项设计的边界

- 与 `workflow-canvas-protocol-design` 的边界：本文定义正式 DSL 治理对象；画布协议只负责编辑表现与页面协作，不拥有正式运行真相。
- 与 `org-rule-evaluator-design` 的边界：本文只定义 `ORG_RULE` 在 DSL 中如何被声明和校验；规则求值语言、求值上下文和运行时选人算法由后者下沉。
- 与 `parallel-and-countersign-aggregation-design` 的边界：本文只定义并行与会签节点的 DSL 约束和版本约束；具体聚合算法参数由后者下沉。
- 与 `oa-bridge-mapping-and-compensation-design` 的边界：本文只保留外部桥接声明在 DSL 中的治理位置，不展开 `OA` 字段映射与补偿细节。

## 13. 本文结论

流程定义语言与校验器之所以必须成为正式治理对象，是因为流程引擎真正需要治理的不是“画布长什么样”，而是“什么样的流程版本可以被平台正式信任并投入运行”。

因此，本设计将 `Workflow DSL` 收口为“版本化语言快照 + 分层校验报告 + 发布冻结规则 + 审计证据链”的组合治理对象，并明确了节点、边、条件、组织绑定、版本约束、草稿与发布差异、兼容升级和错误分类边界。

在这一前提下，后续专项设计才有稳定前提继续下沉画布协议、组织规则求值器、并行聚合算法和 `OA` 桥接细节。
