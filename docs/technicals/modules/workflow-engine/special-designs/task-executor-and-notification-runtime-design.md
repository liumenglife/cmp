# 流程引擎专项设计：任务执行器与通知运行时设计

## 1. 文档说明

本文档用于继续下沉 `workflow-engine` 主线中"任务执行器运行时治理"与"通知运行时协作治理"两个能力，重点回答以下问题：

- 任务执行器如何从"异步任务跑批脚本"提升为流程引擎内部正式受控运行时对象
- 扫描周期、并发参数、告警阈值如何在流程引擎内部被统一治理，而不是散落在部署配置或外部调度系统中
- 通知事件、变量契约、渠道声明、消息合并声明如何在流程引擎侧形成统一输出契约，而不是由通知中心猜测业务语义
- 任务执行器与通知运行时的正式治理对象分别由哪个控制面和正式对象承接，而不是停留在口头治理

本文是以下文档的下游专项设计：

- 不写成对外 API 路径、请求字段、响应样例、错误码枚举全集

## 2. 设计目标与边界

本专项设计必须同时满足以下目标：

- 任务执行器正式治理对象统一由流程引擎 `Runtime Control Plane` 承接，不是外部调度器的被动回调终端
- 扫描周期、并发上限、重试策略、告警阈值必须在流程引擎内部有统一治理口径，可审计、可追溯、可恢复
- 通知运行时的事件语义与策略对象统一由流程引擎 `Runtime Control Plane` 承接，经 `Workflow Collaboration Gateway` 输出到通知中心，不是流程代码中散落的消息发送调用
- 通知模板变量契约、渠道优先级声明、消息合并策略声明必须由流程引擎侧定义输出语义；通知中心只承接模板资产、渠道路由与分发执行
- 审批事实不因通知失败而回滚，但通知状态必须作为运行时可观测输入纳入补偿与告警链路

边界约束如下：

- 本文治理对象是流程引擎内部的执行器运行时与通知输出契约，不是外部任务调度平台或消息通道本体
- 本文只定义流程引擎侧"输出什么、如何治理、如何观测"，不定义通知中心内部路由引擎或任务中心内部调度实现
- 本文只回答流程引擎域内的治理问题，不展开通用异步任务平台的设计

## 3. 任务执行器成为正式治理对象的原则

### 3.1 治理对象判断

当流程引擎产生异步任务（如超时扫描、催办分发、补偿重试）后，不能只保留"已提交到任务中心"的结果，而必须把以下内容纳入正式治理：

- 该异步任务属于哪个流程实例、哪个任务、哪条计时器记录
- 该异步任务当前处于什么执行阶段、已重试几次、下次何时触发
- 该异步任务的并发控制参数是什么、当前并发水位如何
- 该异步任务失败后进入什么恢复路径、是否需要人工介入

因此，任务执行器的正式治理对象不是"一个异步 job"，而是由流程引擎 `Runtime Control Plane` 统一承接的四类正式对象：

- 任务类型注册表：定义异步任务类型、业务归属、触发模式与失败归类
- 治理参数集：定义扫描周期、并发上限、批量大小、锁租约、重试与告警阈值
- 运行时任务记录：即异步任务表 / `jobs` 资源中的实例化记录，承接 `PENDING` 到终态的执行状态
- 治理审计变更：记录注册表与治理参数的版本变更、发布人、发布时间与变更原因

### 3.2 治理主语

任务中心负责统一调度承载、队列管理与执行器生命周期管理。但以下治理责任必须留在流程引擎 `Runtime Control Plane`：

- 异步任务类型的业务语义注册与参数定义
- 扫描周期、并发上限、重试策略、告警阈值的业务级配置
- 任务执行结果与流程实例、审批任务、计时器记录之间的状态回写契约
- 执行异常的业务分类与人工介入判定

换言之，任务中心是执行器的运行底座，只按流程引擎下发的注册表与参数引用执行；流程引擎 `Runtime Control Plane` 才是执行器治理主语与正式对象持有者。

## 4. 异步任务类型注册与分类

### 4.1 类型注册表

流程引擎域内的异步任务类型必须登记在 `Runtime Control Plane` 持有的任务类型注册表中，每种类型至少声明以下属性：

- 类型标识：如 `TASK_TIMEOUT_SCAN`、`TASK_REMIND_DISPATCH`、`NOTIFICATION_RETRY`、`CONTRACT_WRITEBACK`、`OA_DISPATCH`、`OA_CALLBACK_REPLAY`、`SUMMARY_REFRESH`
- 业务归属：绑定到流程实例、审批任务还是桥接记录
- 触发模式：定时扫描型、事件触发型还是补偿重试型
- 默认扫描周期：定时扫描型任务的基础扫描间隔
- 默认并发上限：该类型任务允许的最大并行执行数
- 重试策略族：最大重试次数、退避算法、重试间隔上界
- 告警阈值：连续失败次数或积压量达到多少时触发告警
- 失败归类：可自动恢复、需人工介入、需降级处理

### 4.2 类型分层

建议按执行语义将异步任务类型分为三层：

**扫描层**：由定时调度驱动，按周期扫描待处理记录。

- `TASK_TIMEOUT_SCAN`：扫描 `wf_task_timer` 中到期的超时计时器
- `TASK_REMIND_DISPATCH`：扫描 `wf_task_timer` 中到期的催办计时器

**事件层**：由流程运行事件驱动，异步执行副作用。

- `NOTIFICATION_RETRY`：通知发送失败后的补偿重试
- `CONTRACT_WRITEBACK`：审批结果回写合同主档
- `SUMMARY_REFRESH`：审批摘要刷新

**桥接层**：由 `OA` 桥接链路驱动，承担外部系统交互。

- `OA_DISPATCH`：向 `OA` 发起审批实例
- `OA_CALLBACK_REPLAY`：`OA` 回调失败后的补偿重放

分层的治理意义在于：扫描层关注周期与并发，事件层关注幂等与补偿，桥接层关注外部可用性与人工介入边界。三者共享统一注册表，但执行参数和告警策略允许差异化配置。

## 5. 扫描周期与并发模型

### 5.1 扫描周期治理

扫描周期不是部署时一次性配置的基础设施参数，而是 `Runtime Control Plane` 治理参数集的一部分。治理原则如下：

- 每种扫描型异步任务类型必须声明默认扫描周期
- 扫描周期允许按环境（开发、测试、生产）差异化配置，但配置来源必须统一收敛到流程引擎治理配置域
- 扫描周期变更必须可审计，不允许通过运维脚本静默修改
- 扫描周期过长会导致超时处理延迟，过短会导致数据库扫描压力，必须在治理配置中声明合理区间

建议扫描周期区间：

- `TASK_TIMEOUT_SCAN`：默认 `30s`，允许区间 `10s ~ 120s`
- `TASK_REMIND_DISPATCH`：默认 `60s`，允许区间 `30s ~ 300s`

### 5.2 并发模型

任务执行器的并发模型必须同时满足"不重复处理同一条记录"和"不因单条阻塞导致整体扫描停滞"两个目标。建议采用以下分层并发控制：

**调度层并发**：由任务中心控制，决定同时拉起多少个执行器实例。该层并发参数由任务中心平台治理，流程引擎只声明每种异步任务类型的期望并发上限。

**业务层并发**：由流程引擎控制，决定同一条业务记录（如同一个 `task_id` 或 `timer_id`）是否允许被并发处理。该层并发通过分布式锁或数据库乐观锁实现。

- 锁粒度：`workflow:async:{task_type}:{owner_id}`，如 `workflow:async:TASK_TIMEOUT_SCAN:{timer_id}`
- 锁持有时间：不超过单次扫描周期的一半，避免锁堆积
- 锁失效恢复：锁丢失后依靠数据库状态重放恢复，不依赖锁状态作为业务真相

**执行层并发**：单个执行器实例内部的并发控制。对于扫描型任务，建议采用"批量拉取 + 逐条处理"模式，批量大小由治理配置控制，单条处理失败不阻塞批次内其他记录。

### 5.3 并发参数治理

以下参数必须在流程引擎 `Runtime Control Plane` 的治理参数集中统一管理：

- `max_concurrent_instances`：该类型异步任务允许的最大并行执行器实例数
- `scan_batch_size`：单次扫描拉取的最大记录数
- `lock_ttl_ms`：业务层锁的存活时间
- `backoff_base_ms`：重试退避基准间隔
- `backoff_max_ms`：重试退避最大间隔

这些参数不允许散落在环境变量、配置文件或代码常量中，必须收敛到流程引擎 `Runtime Control Plane` 持有的正式治理对象，支持运行时查询、变更审计与版本回溯；任务中心只消费参数快照或参数引用，不拥有这些参数的主数据。

## 6. 扫描状态机

### 6.1 异步任务执行状态

每条异步任务记录在执行器内部经历以下状态：

- `PENDING`：已注册，等待调度或扫描触发
- `CLAIMED`：已被执行器实例领取，正在处理中
- `SUCCEEDED`：执行成功，副作用已完成
- `FAILED`：执行失败，等待重试或进入人工介入
- `EXHAUSTED`：重试次数耗尽，进入终态，需人工处理
- `CANCELED`：因业务原因取消（如实例已终止，无需继续扫描）

状态转换规则：

- `PENDING -> CLAIMED`：扫描周期触发，执行器领取记录
- `CLAIMED -> SUCCEEDED`：执行成功
- `CLAIMED -> FAILED`：执行失败且未达重试上限
- `CLAIMED -> PENDING`：领取租约超时或执行器心跳丢失，被回收后重新入队
- `FAILED -> PENDING`：重试间隔到达，重新进入待处理队列
- `FAILED -> EXHAUSTED`：重试次数耗尽
- `PENDING -> CANCELED`：业务条件不再满足（如任务已被人工处理）
- `CLAIMED -> CANCELED`：执行过程中发现业务条件已变化

### 6.2 与 `wf_task_timer` 的关系

`wf_task_timer` 是流程引擎内部计时器的真相表。异步任务执行器扫描 `wf_task_timer` 时，必须遵循以下原则：

- `wf_task_timer.timer_status` 为计时器生命周期真相，异步任务执行器不得绕过计时器状态直接操作任务
- 超时触发后，执行器负责"触发超时动作并写入 `wf_approval_action`"，同时更新 `wf_task_timer` 的 `last_triggered_at`、`trigger_count` 和 `timer_status`
- 催办触发后，执行器负责"触发催办动作并写入 `wf_approval_action`"，同时更新 `wf_task_timer` 触发计数，但催办频控判断由 `Task Manager` 负责
- 执行器不得自行修改 `wf_approval_task.task_status`，任务状态推进必须通过 `Task Manager` 的正式接口

### 6.3 `CLAIMED` 僵死回收路径

`CLAIMED` 不是不可回收的中间态。流程引擎 `Runtime Control Plane` 必须为运行时任务记录保存领取租约边界，并提供正式回收路径：

- 领取成功时写入 `claimed_at`、`claim_expires_at`、`claimed_by_executor`
- 执行器续租失败、进程崩溃或超过 `claim_expires_at` 未完成时，该记录进入“僵死 `CLAIMED`”候选
- 回收扫描器只允许回收满足“租约过期 + 未写入终态 + `row_version` 未变化”的记录，回收结果统一转回 `PENDING`
- 每次回收都必须写入治理审计变更与运行日志，区分“超时回收”“人工重置”“重复领取冲突”三类原因
- 达到连续回收阈值后不再无限重入，而是转入 `FAILED` 或 `EXHAUSTED`，触发人工介入

### 6.4 与异步任务表的关系

流程引擎域内的异步任务记录（对应 `api-design.md` 中的 `jobs` 资源）承载异步任务的注册、调度与执行状态。其与扫描状态机的关系如下：

- 异步任务记录的 `job_status` 映射扫描状态机中的 `PENDING`、`CLAIMED`、`SUCCEEDED`、`FAILED`、`EXHAUSTED`、`CANCELED`
- 异步任务记录持有 `owner_type` 和 `owner_id`，指向流程引擎内部的计时器、任务、实例或桥接记录
- 异步任务记录的 `result_code` 和 `result_message` 承载执行结果摘要，用于告警和人工排查
- 异步任务记录不替代 `wf_approval_action` 的事实表地位，只承载"执行调度视角"的状态

## 7. 告警阈值与恢复边界

### 7.1 告警阈值治理

告警阈值不是运维监控系统的私有配置，而是流程引擎执行器治理的一部分。每种异步任务类型必须声明以下告警维度：

- 连续失败次数阈值：同一条记录连续失败达到该值时触发告警
- 积压量阈值：待处理队列中 `PENDING` 状态记录数达到该值时触发告警
- 执行延迟阈值：单条记录从 `PENDING` 到 `CLAIMED` 的等待时间超过该值时触发告警
- 扫描停滞阈值：扫描周期内未成功处理任何记录的连续次数达到该值时触发告警

建议默认阈值：

- `TASK_TIMEOUT_SCAN`：连续失败 `3` 次、积压 `100` 条、延迟 `120s`、停滞 `5` 轮
- `TASK_REMIND_DISPATCH`：连续失败 `3` 次、积压 `200` 条、延迟 `180s`、停滞 `5` 轮
- `NOTIFICATION_RETRY`：连续失败 `5` 次、积压 `500` 条、延迟 `300s`、停滞 `10` 轮
- `CONTRACT_WRITEBACK`：连续失败 `3` 次、积压 `50` 条、延迟 `60s`、停滞 `3` 轮

### 7.2 恢复边界

- 执行器重启后，由扫描周期自动恢复 `PENDING` 和 `FAILED` 状态的记录处理
- 锁丢失后，依靠数据库状态和 `row_version` 重新领取，不依赖内存或分布式锁状态作为业务真相
- 达到 `EXHAUSTED` 状态的记录必须进入人工待处理队列，不允许被扫描周期自动重新领取
- 人工介入只允许重置执行状态、补录参数或触发重放，不允许直接改写执行结果事实

## 8. 通知运行时成为正式治理对象的原则

### 8.1 治理对象判断

当流程引擎运行过程中需要发送通知（如待审批提醒、催办通知、超时告警、审批结果通知）时，不能只在代码中调用通知中心发送接口，而必须把以下内容纳入正式治理：

- 本次通知的触发源是什么（任务生成、催办触发、超时触发、状态变更）
- 本次通知使用哪一版模板、填充了什么变量、选择了什么渠道
- 本次通知的优先级是什么、是否允许与其他通知合并
- 本次通知发送结果如何、失败后进入什么补偿路径

因此，通知运行时的正式治理对象不是"一次消息发送"，而是由流程引擎 `Runtime Control Plane` 持有、并经 `Workflow Collaboration Gateway` 对外输出的五类正式对象：

- 通知事件注册表：定义事件类型、触发源、接收人解析规则与默认模板引用
- 模板变量契约：定义变量清单、必填约束、变量版本与数据来源
- 渠道 / 合并策略参数：定义优先级声明、降级顺序、合并键与合并窗口
- 运行时通知命令：每次触发时由协作网关发往通知中心的标准事件载荷
- 治理审计变更：记录事件注册、变量契约、策略参数的版本变更与发布审计

### 8.2 治理主语

通知中心负责模板资产管理、模板渲染、渠道路由、消息分发与回执管理。但以下治理责任必须留在流程引擎 `Runtime Control Plane`：

- 通知事件类型的业务语义定义与触发条件
- 模板变量的数据来源与填充契约版本
- 渠道优先级的业务级声明
- 消息合并策略的业务规则
- 通知失败后的业务补偿判定

换言之，通知中心治理模板资产、渠道适配器与发送执行，流程引擎治理通知事件语义、变量契约和策略声明；两者不再双重主语。

## 9. 通知事件类型注册

### 9.1 类型注册表

流程引擎域内的通知事件类型必须登记在 `Runtime Control Plane` 持有的通知事件注册表中，每种类型至少声明以下属性：

- 事件标识：如 `TASK_CREATED`、`TASK_REMIND`、`TASK_TIMEOUT`、`PROCESS_COMPLETED`、`PROCESS_REJECTED`、`PROCESS_WITHDRAWN`、`PROCESS_TERMINATED`、`ESCALATION_TRIGGERED`
- 触发源：绑定到任务生成、催办触发、超时触发还是实例状态变更
- 默认模板标识：引用通知中心的模板编码或模板资产版本
- 默认渠道优先级：该事件类型的首选渠道和降级渠道
- 默认合并策略：是否允许合并、合并窗口、合并键
- 通知对象解析规则：如何从流程上下文中提取接收人

### 9.2 事件类型与流程生命周期的对应

- 实例创建时触发 `TASK_CREATED`（对应新任务分发）
- 催办计时器触发时触发 `TASK_REMIND`
- 超时计时器触发时触发 `TASK_TIMEOUT`
- 升级策略触发时触发 `ESCALATION_TRIGGERED`
- 实例进入终态时触发 `PROCESS_COMPLETED`、`PROCESS_REJECTED`、`PROCESS_WITHDRAWN`、`PROCESS_TERMINATED`

每种事件类型的通知对象集合由流程引擎从任务快照、实例上下文和组织解析结果中提取，通知中心只接收已解析的接收人列表和变量字典。

## 10. 通知模板变量契约

### 10.1 变量来源

通知模板变量由流程引擎 `Runtime Control Plane` 统一定义和填充，通知中心只消费已填充的变量字典。变量来源包括：

- 流程实例上下文：`process_id`、`definition_name`、`contract_id`、`business_title`、`starter_name`、`started_at`
- 任务上下文：`task_id`、`node_name`、`assignee_name`、`due_at`、`task_type`
- 动作上下文：`action_type`、`operator_name`、`comment`、`acted_at`
- 业务上下文：`contract_name`、`contract_amount`、`contract_party`（从合同主档关联获取）
- 审批摘要：`current_node_name`、`current_approver_list`、`summary_status`

### 10.2 变量分类

建议将模板变量分为三类：

- 必填变量：模板渲染时必须存在，缺失时通知发送失败并记录补偿
- 可选变量：模板渲染时允许为空，缺失时使用默认值或留白
- 计算变量：由流程引擎在触发时实时计算，如"距超时还有 X 小时"、"已催办 N 次"

### 10.3 变量治理约束

- 变量名必须在通知事件类型注册表中声明，不允许在代码中临时拼装未注册变量
- 变量值必须在通知事件触发时一次性填充完成，不允许通知中心回调流程引擎补填
- 变量中的敏感字段（如审批意见全文）在通知上下文中应做脱敏或摘要处理
- 变量快照必须与通知发送记录一起持久化，便于失败重试和审计追溯

### 10.4 模板变量契约变更闭环

模板变量契约变更不能只改模板正文或流程代码，必须走完整闭环：

- 变量增删改时，先更新通知事件注册表中的变量契约版本、变量清单与来源声明
- `Workflow Collaboration Gateway` 发出的运行时通知命令必须显式携带 `event_code + contract_version + template_code`
- 通知中心发布新模板版本前，必须基于该契约版本完成变量完整性校验；未通过校验的模板不得切为默认模板
- 契约版本升级后，旧失败重试记录仍按原 `contract_version` 重放，不允许用新变量字典覆盖旧事件快照
- 契约变更、模板切换、默认模板生效都必须写入治理审计变更，形成“事件注册 -> 模板发布 -> 运行时发送”的可追溯闭环

## 11. 渠道优先级

### 11.1 优先级声明

每种通知事件类型必须在 `Runtime Control Plane` 的渠道 / 合并策略参数中声明渠道优先级。渠道优先级不是通知中心的内部路由规则，而是流程引擎对"该事件类型应该以什么顺序尝试触达接收人"的业务声明。

建议渠道族：

- 站内消息：平台内部消息中心
- 邮件：企业邮件系统
- 即时通讯：企业微信、钉钉等
- 短信：短信网关

建议优先级声明格式：

- 首选渠道：`IM`（即时通讯）
- 降级渠道：`EMAIL`（即时通讯不可用或超时时降级）
- 兜底渠道：`IN_APP`（站内消息始终写入，作为最低保障）

### 11.2 优先级治理约束

- 渠道优先级由流程引擎按事件类型声明，通知中心按声明执行路由，不得自行覆盖业务优先级
- 渠道降级触发条件（如发送超时、通道不可用、接收人未绑定该渠道）由通知中心判定，但降级结果必须回传流程引擎
- 渠道优先级变更必须纳入治理配置审计，不允许静默修改

## 12. 消息合并策略

### 12.1 合并动机

在高频操作场景下（如批量审批、连续催办、多任务同时超时），若每条事件独立发送通知，会导致接收人被消息淹没。消息合并策略的目标是在"及时性"和"噪声控制"之间取得平衡。

### 12.2 合并键与合并窗口

每种通知事件类型必须声明：

- 是否允许合并：部分高优先级事件（如超时告警）不允许合并，必须即时发送
- 合并键：决定哪些通知可以被合并。建议合并键为 `contract_id + assignee_user_id + event_type`，即同一合同、同一接收人、同一事件类型的通知可合并
- 合并窗口：在该时间窗口内到达的同合并键通知被合并为一条。建议窗口：
  - `TASK_REMIND`：`10min`
  - `TASK_TIMEOUT`：`5min`
  - `TASK_CREATED`：`15min`
  - 状态变更类：`30min`
- 合并摘要格式：合并后通知的标题和正文如何表达"您有 N 条待办任务需要处理"

### 12.3 合并治理约束

- 合并策略由流程引擎按事件类型声明，通知中心按声明执行合并，不得自行决定合并规则
- 合并窗口内若有更高优先级事件到达，应打断合并立即发送
- 合并后的通知记录必须保留被合并事件的引用列表，便于审计追溯
- 合并策略变更必须纳入治理配置审计

## 13. 与任务中心的协作边界

### 13.1 职责划分

| 能力 | 流程引擎职责 | 任务中心职责 |
| --- | --- | --- |
| 异步任务类型注册 | 定义业务语义、参数族、重试策略族 | 按注册表承载调度 |
| 扫描周期 | 声明业务合理区间 | 按配置执行调度 |
| 并发控制 | 声明业务层锁策略 | 控制调度层并发 |
| 执行状态追踪 | 消费执行结果并回写流程状态 | 维护执行器运行时状态 |
| 告警阈值 | 声明业务级告警条件 | 触发告警并通知流程引擎 |
| 失败恢复 | 判定业务恢复路径 | 执行重试调度 |

### 13.2 交互契约

流程引擎与任务中心之间的交互必须满足以下契约：

- 流程引擎向任务中心提交异步任务时，必须携带任务类型标识、业务归属键、治理参数引用
- 任务中心执行完成后，必须向流程引擎回调执行结果，流程引擎据此更新内部状态
- 任务中心不得自行修改流程引擎内部的计时器状态、任务状态或实例状态
- 流程引擎不得绕过任务中心自行管理异步任务调度和执行

### 13.3 与通知中心的协作边界

流程引擎与通知中心之间的交互必须满足以下契约：

- 流程引擎经 `Workflow Collaboration Gateway` 向通知中心提交通知事件时，必须携带事件类型标识、契约版本、已填充的变量字典、渠道优先级声明、合并策略声明
- 通知中心发送完成后，必须向流程引擎回传发送结果（成功、失败、降级）
- 通知中心不得自行决定通知触发时机，触发条件由流程引擎控制
- 流程引擎不得绕过通知中心自行对接外部消息通道

## 14. 运行控制面落地对象与映射

流程引擎 `Runtime Control Plane` 的治理对象以本模块表为主、平台中心为执行承载。治理参数不允许散落为代码常量，任务中心、通知中心只消费已发布的注册表、参数快照和通知命令。

| 治理对象 | 流程引擎落地表 | 平台中心映射 | 必填状态与控制字段 |
| --- | --- | --- | --- |
| 任务类型注册表 | `wf_async_task_type_registry` | 任务中心按 `task_type_code` 注册执行器能力 | `registry_status`、`owner_object_type`、`trigger_mode`、`default_param_set_id` |
| 治理参数集 | `wf_governance_param_set` | 配置中心可保存发布快照，任务中心按 `param_set_id + param_set_version` 消费 | `param_status`、`scan_interval_ms`、`max_concurrent_instances`、`scan_batch_size`、`lock_ttl_ms`、`max_retry_count`、`backoff_base_ms`、`backoff_max_ms` |
| 异步任务记录 | `wf_async_task_record` | 映射 `platform_job` / `jobs` | `job_status`、`claimed_at`、`claimed_by_executor`、`claim_expires_at`、`retry_count`、`next_trigger_at`、`owner_object_type`、`owner_object_id`、`param_snapshot_ref` |
| 通知事件注册表 | `wf_notification_event_registry` | 通知中心按 `event_code + contract_version` 接收业务事件 | `event_status`、`trigger_source_type`、`variable_contract_id`、`notification_policy_id` |
| 通知变量契约 | `wf_notification_variable_contract` | 通知中心模板发布前按契约校验变量完整性 | `contract_status`、`contract_version`、`variable_schema_snapshot`、`required_variable_list`、`data_source_snapshot` |
| 通知策略 | `wf_notification_policy` | 通知中心按策略执行渠道路由、降级与合并 | `policy_status`、`policy_version`、`channel_priority_snapshot`、`merge_enabled`、`merge_key_expression`、`merge_window_ms`、`retry_task_type_id` |
| 治理审计变更 | `wf_governance_audit_change` | 同步审计中心高等级配置审计事件 | `governance_object_type`、`governance_object_id`、`before_version`、`after_version`、`change_action`、`change_reason`、`approval_ref` |

字段映射约束如下：

- `wf_async_task_record.platform_job_id` 与任务中心 `platform_job.job_id` 一一对应；任务中心的执行状态变化必须回写 `job_status`、`claimed_at`、`claim_expires_at`、`retry_count`、`next_trigger_at`
- `wf_async_task_record.owner_object_type + owner_object_id` 固定表达归属对象，允许指向 `process_id`、`task_id`、`timer_id`、`bridge_record_id` 或通知事件实例，不允许只保存自然语言说明
- `wf_async_task_record.param_snapshot_ref` 必须引用已发布的 `wf_governance_param_set` 版本，重试和回放按创建时快照执行，不被最新参数静默覆盖
- `wf_notification_event_registry.variable_contract_id` 与 `notification_policy_id` 必须指向当前发布版本；运行时通知命令必须携带 `event_code + contract_version + policy_version + variables_snapshot`
- `wf_governance_audit_change` 是注册表、参数集、变量契约和通知策略变更的统一审计入口；任何发布、停用、归档动作必须先产生该记录，再同步审计中心

## 15. 本文结论

本设计将任务执行器与通知运行时收口为流程引擎内部两个正式治理对象：

- 任务执行器治理域：由流程引擎 `Runtime Control Plane` 持有任务类型注册表、治理参数集、运行时任务记录与治理审计变更，任务中心只按引用执行
- 通知运行时治理域：由流程引擎 `Runtime Control Plane` 持有通知事件注册表、模板变量契约、渠道 / 合并策略参数与治理审计变更，经 `Workflow Collaboration Gateway` 发出运行时通知命令；通知中心只治理模板资产、渠道路由与发送执行

在此前提下：

- 任务执行器的扫描周期、并发参数、告警阈值在流程引擎内部有统一治理口径
- 通知模板变量契约、渠道优先级、消息合并策略由流程引擎定义输出语义，由通知中心承接模板与分发执行
- 审批事实不因通知失败而回滚，但通知状态纳入运行时补偿与告警链路
- 任务中心和通知中心分别承担执行底座，不再与流程引擎形成双重治理主语

本文到此为止，不继续展开任务中心或通知中心自身的平台级实现、对外 API 契约或实施计划。
