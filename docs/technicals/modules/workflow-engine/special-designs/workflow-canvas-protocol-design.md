# 流程引擎专项设计：画布协议设计

## 1. 文档说明

本文档用于继续下沉 `workflow-engine` 主线中“流程配置画布协议”这一能力，重点回答以下问题：

- 画布保存协议如何服务流程配置编辑，但不成为正式运行真相
- 节点属性、连线关系、布局元数据、草稿编辑态如何形成稳定协议，而不是散落在页面状态中
- 版本比较视图如何基于协议稳定生成，而不是依赖前端临时计算
- 多人编辑、校验、发布与协作时，哪些信息属于画布层，哪些信息必须留在正式 `DSL` 治理层
- 画布协议与正式 `Workflow DSL` 之间如何划清边界，避免“为了页面好画”而污染可发布流程版本

本文是以下文档的下游专项设计：

- [`workflow-engine Detailed Design`](../detailed-design.md)

`workflow-dsl-and-validator-design`、流程配置端、发布校验链路、版本比较视图是本文的强依赖或关键消费方：前者定义正式 `DSL` 真相与校验治理，后者消费本文定义的编辑态对象与比较素材；这些都属于协作或消费关系，不属于本文档的父文档关系。

本文不展开以下内容：

- 不写对外 `API` 路径、请求字段、响应样例、错误码枚举
- 不写前端组件结构、拖拽实现、渲染引擎选型、状态管理库接入方式
- 不写正式 `DSL` 的全部语法、校验器算法细节、发布编译器实现或数据库 `DDL`
- 不写实施排期、任务拆分、灰度计划、迁移脚本或联调手册

## 2. 设计目标

- 让画布协议成为“编辑与协作协议”，而不是“运行时协议”
- 让草稿保存能够保留页面编辑必要信息，但不把坐标、选中态、比较视图噪声写进正式 `DSL`
- 让节点与连线在编辑态具备稳定标识，支持差异比较、草稿恢复、协作冲突判断与发布前映射
- 让配置端可以保存未完成草稿，但这些中间态不能被误判为可运行版本
- 让发布动作始终以正式 `DSL` 快照为唯一真相，画布协议只提供上游输入与辅助证据

## 3. 画布协议的治理定位

画布协议不是“页面存档格式”，而是流程配置端与流程引擎定义治理层之间的受控编辑边界。它的职责只包括三类：

1. 承接草稿编辑所需的结构化图信息，如节点、边、分组、布局、局部编辑状态。
2. 承接版本比较、变更高亮、草稿恢复、协作冲突检测所需的可比较元数据。
3. 为草稿期持续治理中的正式 `Draft DSL Snapshot` 提供受控输入，但不直接替代发布冻结后的 `Published Runtime Snapshot`。

因此，画布协议的设计原则必须是“面向编辑、受控映射、禁止越权”。

- 面向编辑：允许表达未闭合连线、待补全节点、布局调整、比较标记等编辑期信息。
- 受控映射：只有被 `DSL` 规范承认的语义字段，才能进入正式定义语言。
- 禁止越权：画布层不得偷偷持有运行时参与人、任务结果、审计结论或发布冻结后的正式版本事实。

## 4. 协议分层

建议将画布保存对象收口为四层，而不是一个混杂大 `JSON`：

1. `Canvas Envelope`
   - 标识当前协议版本、所属 `definition_id`、草稿序列号、来源端、最近保存摘要。
   - 只承接编辑会话所需治理元数据，不承接运行时真相。
2. `Graph Payload`
   - 承接节点、边、分组、泳道或其他图结构对象。
   - 是画布协议中最接近正式 `DSL` 的部分，但仍保留编辑态容错。
3. `Layout Payload`
   - 承接坐标、尺寸、折叠态、局部展开态、视口锚点、分组框位置等纯表现元数据。
   - 明确禁止进入正式 `DSL` 真相。
4. `Draft Collaboration Payload`
   - 承接草稿编辑态、比较锚点、冲突检测指纹、局部锁提示、未保存变更摘要等协作信息。
   - 这些对象只服务编辑协作，不参与发布运行。

这四层中，只有 `Graph Payload` 的受控子集可以映射进正式 `DSL`。其余三层都必须被明确视为“编辑层附属信息”。

## 5. 节点属性模型

### 5.1 节点模型定位

画布节点对象 `CanvasNode` 的目标，不是直接等价正式 `NodeSpec`，而是提供“编辑上可操作、映射上可审计”的中间对象。它至少应包含以下字段族：

- 稳定标识：`canvas_node_id`
- 语义映射锚点：`node_key`
- 节点类型：如 `START`、`APPROVAL`、`PARALLEL_GATEWAY`、`END`
- 标题与说明：服务配置阅读与比较展示
- 语义属性区：审批动作约束、组织绑定声明引用、时限声明、条件入口声明等
- 展现属性区：图标、颜色标记、展示标签引用等最小展示语义
- 编辑状态区：是否新建、是否待补全、是否局部报错、是否被比较视图标红

其中必须坚持两个稳定键并存：

- `canvas_node_id` 用于本次画布对象身份、拖拽排序、局部替换和协作冲突定位。
- `node_key` 用于正式语义身份、版本比较归一和发布时映射到正式 `DSL`。

如果只保留一个键，会产生两个问题：一是布局改动和语义改动无法区分，二是比较视图无法判断“节点是重命名、挪位置，还是被删后重建”。

### 5.2 节点属性分区原则

节点属性必须在协议层明确分区，避免前端任意加字段：

1. `semantic_props`
   - 只允许承载未来可能进入正式 `DSL` 的受控语义，如节点类型、绑定声明、动作限制、时限策略、并行或会签语义声明。
2. `presentation_props`
   - 只承载最小展示语义或展示引用，如图标、颜色标记、标签样式引用、注释显隐开关。
   - 坐标、尺寸、折叠态、注释锚点等布局元数据统一落在 `Layout Payload`，不得在节点对象内重复承载。
3. `draft_props`
   - 只承载编辑期状态，如待补全标记、草稿错误提示、比较视图高亮原因、局部脏标。

任何字段进入 `semantic_props`，都必须满足“未来可被正式 `DSL` 识别”；否则只能留在 `presentation_props` 或 `draft_props`。这样可以从协议结构上防止画布污染正式语言。

### 5.3 节点绑定边界

审批节点在画布协议中可以承接绑定声明的编辑输入，但不能承接运行时选人结果。

- 可以出现：`binding_type`、绑定目标引用、规则参数草稿、兜底策略声明。
- 不可以出现：发布时解析出的具体审批人名单、运行时参与人快照、任务执行结果。

这条边界必须与 `workflow-dsl-and-validator-design` 保持一致，即画布层编辑的是“声明”，不是“结果”。

## 6. 连线模型

### 6.1 连线对象定位

连线对象 `CanvasEdge` 负责表达编辑态流转关系。它既要支持画线操作，又要为正式 `EdgeSpec` 提供稳定输入，因此必须同时保留：

- 画布对象身份 `canvas_edge_id`
- 正式语义锚点 `edge_key`
- `from_node_key` 与 `to_node_key`
- 连线类型，如默认流转、条件流转、并行分叉、收敛关联
- 顺序与优先级信息
- 条件引用或条件草稿占位
- 最小展示语义，如线型样式引用、标签样式引用

### 6.2 连线语义与表现分离

连线协议要解决的核心不是“线怎么画”，而是“线的语义如何稳定保存而不被样式噪声淹没”。因此应明确区分：

- 语义层：源节点、目标节点、优先级、条件关联、并行/收敛语义。
- 表现层：线型样式引用、标签样式引用等最小展示语义；折线路径点、标签坐标、曲率等布局数据统一进入 `Layout Payload`。

版本比较与发布映射默认基于语义层；表现层只在“布局变化比较”场景下单独使用，不能与流程语义差异混为一谈。

### 6.3 未完成连线的协议容忍

草稿编辑中允许存在以下中间态：

- 起点已确定但终点未落定
- 条件分支已创建但条件体未补全
- 并行分叉已画出但收敛端未完成

这些状态可以在画布协议中保存，但必须被明确标记为 `draft_incomplete` 或等效语义，且默认不可进入正式 `DSL`。这正是画布协议服务编辑、但不污染正式真相的关键差异之一。

## 7. 布局元数据

布局元数据是画布协议的必要组成，但它的治理目标是“可恢复编辑体验”，不是“定义流程含义”。`Layout Payload` 是布局元数据的唯一正式承载层，节点、连线、分组对象不得重复写入同类布局字段。建议至少包括：

- 节点坐标与尺寸
- 视口位置、缩放比、默认聚焦节点
- 分组框、泳道、辅助线、对齐锚点
- 折叠 / 展开态、注释锚点、辅助标签位置
- 自动排布来源标记与最后一次手工调整摘要

布局元数据必须遵守三条约束：

1. 任何布局信息都不得进入正式 `DSL` 根对象。
2. 布局变化默认不构成“流程语义变更”。
3. 版本比较必须能区分“只改布局”和“改了流程语义”。

如果没有这三条约束，后续发布审计、版本差异和争议回放都会被大量无意义坐标噪声污染。

## 8. 草稿编辑态

### 8.1 草稿态目标

草稿态服务的是“持续编辑”，不是“即时可运行”。因此画布协议必须原生支持中间态，而不是强迫所有保存都达到正式发布标准。

### 8.2 草稿编辑态对象

建议在 `Draft Collaboration Payload` 中至少收口以下信息：

- `draft_revision`：草稿递增序列
- `base_version_id` 或 `base_revision_ref`：本次编辑基线
- `dirty_scope`：哪些节点、连线、布局区块已变更
- `draft_validation_summary`：当前草稿的提示、警告、错误摘要
- `incomplete_object_list`：未闭合节点、未完成连线、待补全绑定等对象定位
- `editor_context`：最近编辑人、编辑入口、最近保存时间

这些信息的作用是支撑恢复、预校验和协作提示，不是替代正式 `Validation Report`。正式发布校验结果仍然归 `DSL` 与校验器治理层持有。

### 8.3 草稿保存与发布冻结的关系

草稿保存只保证“当前编辑状态被忠实保存”，不保证“当前版本可以发布”。

发布时必须发生一次明确的边界切换：

1. 从画布协议读取可映射的语义子集。
2. 基于画布协议的可映射语义子集，刷新草稿期持续存在的正式 `Draft DSL Snapshot`；它不是“发布时才临时生成”的对象，而是草稿治理层的正式语义快照。
3. 进入校验器分层校验。
4. 只有通过发布门禁后，才基于当次通过校验的 `Draft DSL Snapshot` 冻结为正式运行版本 `Published Runtime Snapshot`。

因此，画布协议从来不是发布态真相本身，而只是草稿期 `Draft DSL Snapshot` 的编辑与表现层输入之一；发布冻结针对的是 `Published Runtime Snapshot`，不是把画布载荷直接升级为正式版本。

## 9. 版本比较视图

### 9.1 比较视图的治理目标

版本比较视图不应只做“页面截图对比”，而应建立在协议级稳定标识上，回答以下三类差异：

- 语义差异：节点类型、绑定声明、条件、连线关系、时限策略是否变更
- 结构差异：新增节点、删除边、并行结构重组、条件分支重排
- 表现差异：位置移动、尺寸变化、标签调整、折叠态变化

### 9.2 比较锚点

比较视图的主锚点应是 `node_key` 与 `edge_key`，而不是页面位置。页面位置只能作为辅助视图定位信息。

当比较系统发现以下情形时，应优先按语义身份解释：

- 同一 `node_key` 坐标变化：视为布局变化
- 同一 `node_key` 标题或绑定变化：视为语义变化
- `canvas_node_id` 变化但 `node_key` 未变：视为对象重建或局部替换，不必直接判定语义删除
- `node_key` 消失：视为正式语义删除候选

### 9.3 比较结果输出边界

比较视图可以消费画布协议中的布局与草稿信息，但比较结果本身不应反向写入正式 `DSL`。它只是变更说明与评审辅助材料，不是版本真相。

## 10. 编辑协作边界

画布协议需要支持编辑协作，但这里的“协作”应限定在草稿治理层，而不是引入运行时多主真相。

建议明确以下边界：

- 协作对象是“草稿版本”而不是“已发布版本”
- 冲突判断优先基于 `draft_revision`、对象指纹和局部变更范围
- 协作提示可以包含“某节点最近被谁编辑”“当前草稿落后基线多少版本”
- 不在画布协议层定义实时操作变换算法或前端同步框架细节
- 已发布版本一经形成，只能作为比较基线和重新编辑来源，不能被协作会话直接原地修改

这意味着本文只收口“协作边界与治理对象”，不下沉到实时协同算法实现。

## 11. 与正式 DSL 的边界

### 11.1 画布层进入 DSL 的内容

只有以下受控语义可从画布协议进入正式 `DSL`：

- 节点正式身份与节点类型
- 连线正式身份与拓扑关系
- 条件声明引用
- 组织绑定声明
- 时限、动作限制、并行/会签等正式治理语义

### 11.2 绝不进入 DSL 的内容

- 节点坐标、尺寸、折叠态、颜色
- 连线路径点、标签坐标、曲率等布局数据
- 画布缩放比、视口位置、辅助线、选中态
- 比较视图高亮结果、审阅批注定位、局部脏标
- 协作提示、临时锁、最近编辑面板信息
- 运行时选人结果、任务状态、审批动作结果

### 11.3 边界转换原则

画布协议到正式 `DSL` 的转换，不是“整个对象原样透传”，而是一次正式收敛动作：

- 先过滤表现层与协作层字段，其中所有布局元数据统一从 `Layout Payload` 剥离
- 再把语义字段映射为正式 `DSL` 对象
- 再由校验器判断哪些内容只是可保存、哪些内容可发布

这套边界的核心价值，是让流程配置端可以保留丰富编辑体验，同时保证平台真正信任的流程版本只来自正式 `DSL` 快照，而不是来自某次页面保存载荷。

## 12. 与其他专项设计的边界

- 与 `workflow-dsl-and-validator-design` 的边界：本文定义画布编辑协议、草稿对象与比较锚点；后者定义正式 `DSL` 真相、校验分层与发布冻结规则。
- 与 `org-rule-evaluator-design` 的边界：本文只承接组织规则在画布中的声明编辑形态，不定义规则求值语言和运行时选人算法。
- 与 `parallel-and-countersign-aggregation-design` 的边界：本文只承接并行与会签在画布中的表达槽位，不定义收敛算法与判定细则。
- 与 `oa-bridge-mapping-and-compensation-design` 的边界：本文只保留桥接配置的编辑承载位置，不定义 `OA` 映射模板、签名与补偿策略。

## 13. 结论

本设计将 `workflow-engine` 的画布协议收口为“编辑与协作协议”，而不是“正式运行协议”。

在这一前提下：

- 画布协议负责保存节点、连线、布局、草稿协作与比较所需信息
- 正式 `DSL` 负责承接可发布、可运行、可审计的流程真相
- 版本比较基于稳定语义锚点工作，而不是基于页面截图或坐标猜测
- 草稿编辑态可以保留中间状态，但这些状态不会直接进入正式发布版本

因此，画布协议服务的是“流程配置编辑的连续性与可协作性”，而正式 `DSL` 服务的是“平台可发布、可运行、可追责的真相治理”。二者必须协同，但不能混同。
