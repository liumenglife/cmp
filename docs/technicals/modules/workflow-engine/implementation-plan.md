## 2. 计划边界与原则

- 默认主审批路径仍优先走 `OA`，但平台流程引擎是一期正式能力，
  本计划按“可独立运行、可正式承接”组织推进。
- 流程引擎不拥有合同主档，只通过 `contract_id` 绑定业务对象，
  任何阶段都不得生成第二合同真相源。
- 每个审批节点必须绑定组织架构中的部门、人员或组织规则，
  节点可执行性优先于画布完成度。
- 推进顺序遵循“先定义与版本，再运行时主链，再组织绑定，再 `OA` 桥接”，
  先固化可运行骨架，再补齐外部承接与切换能力。
- 能复用总平台既有正式文档链路的内容不在本计划重复展开，
  仅将其转译为实施前置条件、阶段依赖与交付判断。
- 外部联调只写确实依赖外部环境的事项，尤其是 `OA` 主路径联调；
  平台内部模块协作按平台内前置条件管理，不扩写成外围系统计划。

## 3. 前置条件

### 3.1 `OA` 桥接前置条件

- 已明确一期内哪些流程默认走 `OA`，哪些场景触发平台承接，
  且该判断口径已与业务方和平台侧统一。
- `OA` 侧已提供可用于联调的流程编码、调用方式、回调方式、
  鉴权方式、测试环境连通条件和问题响应窗口。
- `OA` 侧已确认平台需要负责的平台侧内容，至少包括：
  发起参数承接、状态回传处理、摘要同步、失败补偿与联调回放。
- `OA` 联调前，平台侧流程摘要、合同状态回写和审计留痕能力必须先具备，
  避免外部回调到达后无法形成稳定闭环。

### 3.2 组织架构、权限、任务中心、审计中心、通知中心前置条件

- 组织架构主数据已可稳定提供部门、人员和组织规则读取能力，
  并具备基础在岗、启停、归属校验能力。
- 权限体系已具备管理端配置权限、审批执行权限、查看权限和
  数据范围控制能力，至少可支撑流程配置和任务处理分权。
- 任务中心已具备统一任务承载、待办查询、异步调度和恢复能力，
  使催办、超时扫描和异常重试不需要流程引擎自建第二套任务体系。
- 审计中心已具备关键事件落库与查询能力，确保流程定义发布、
  审批动作、路径切换、回调异常、状态回写异常都能留痕。
- 通知中心已具备统一消息投递与回执承接能力，确保待办、催办、
  超时和异常消息不由流程引擎直接耦合具体通道。

### 3.3 平台路径与 `OA` 路径切换前置条件

- 已形成清晰的承接策略判断口径，能够区分“默认走 `OA`”与
  “由平台流程引擎正式承接”的触发条件。
- 合同主链路已接受统一审批入口与统一审批摘要口径，
  不要求上层业务根据 `OA` 或平台路径分别接两套业务逻辑。
- 平台侧已具备审批结果统一回写合同状态与时间线的能力，
  路径切换不改变合同主链路的真相源边界。
- 联调前已准备好路径切换验证样例，覆盖至少一个 `OA` 主路径样例和
  一个平台承接样例，避免切换逻辑只停留在文档层。

## 4. 总体阶段划分

本模块按五个阶段推进：

1. 流程定义与版本基础能力
2. 平台运行时主链与任务推进能力
3. 组织架构绑定与任务控制能力
4. `OA` 桥接与双路径承接能力
5. 联调、验收与上线准备

阶段之间采用“骨架先闭环、再接组织、再接外部、最后总体验证”的顺序。
前一阶段的完成标志，是后一阶段进入实现与联调的前置门槛。

## 5. 分阶段实施计划

### 5.1 阶段一：流程定义与版本基础能力

**目标**

- 先建立流程引擎可配置、可发布、可追溯的定义层骨架。
- 先锁定“什么流程可以运行”，再进入“流程如何推进”。

**核心产出**

- 流程定义、草稿、版本、发布与停用的基础能力
- 发布前完整性校验能力，至少覆盖定义可运行性和组织绑定必填性校验
- 面向管理端的定义维护基础闭环
- 可被运行时稳定引用的版本化流程快照

**前置依赖**

- 子模块 [`Architecture Design`](./architecture-design.md) 与
  [`Detailed Design`](./detailed-design.md) 已明确定义、版本、节点、绑定的边界
- 管理端基础权限可用，能够区分配置操作与普通业务操作

**完成标志**

- 能创建流程定义、保存草稿、发布版本、停用版本
- 未绑定组织架构的节点不能通过发布校验
- 运行时能够按版本号或版本引用稳定加载流程定义
- 定义变更和版本发布已进入审计链路

### 5.2 阶段二：平台运行时主链与任务推进能力

**目标**

- 先形成平台引擎可独立执行的审批主链，确保在不依赖 `OA` 的情况下，
  平台已经具备正式审批承接骨架。
- 打通实例启动、任务生成、审批动作、状态推进和合同回写闭环。

**核心产出**

- 流程实例创建、加载与推进能力
- 审批任务生成、处理、关闭与历史动作记录能力
- 串行、并行、会签、转办等一期基础运行方式
- 统一审批摘要与合同状态/时间线回写闭环
- 与任务中心协作的催办、超时、异常恢复基础挂接点

**前置依赖**

- 阶段一定义与版本能力完成，运行时只消费已发布版本
- 合同主档侧已提供基于 `contract_id` 的业务绑定与状态回写入口
- 任务中心已具备基础任务承载与查询能力
- 审计中心已可承接审批动作和异常事件

**完成标志**

- 能基于已发布流程版本发起平台审批实例
- 审批动作能稳定推进实例状态并生成统一审批摘要
- 审批完成、驳回、终止等关键结果可回写合同主链路
- 催办、超时和异常恢复已有任务中心承接入口，不存在模块内第二套任务机制

### 5.3 阶段三：组织架构绑定与任务控制能力

**目标**

- 在运行时主链已成立后，补齐“节点一定能落到真实组织对象”的执行约束。
- 让流程不只是能跑，而是能按组织架构稳定选人、控权和控任务。

**核心产出**

- 节点绑定部门、人员、组织规则的解析能力
- 基于组织状态、权限和可执行性的候选人求值能力
- 待办归属、转办控制、可执行动作控制与超时控制的组织化约束
- 组织变更场景下的运行时处理原则和异常收敛规则

**前置依赖**

- 阶段二运行时主链已可稳定创建实例和任务
- 组织架构主数据、权限体系、任务中心已提供稳定查询和校验能力
- 审计中心与通知中心已可承接组织解析失败、转办、催办、超时等事件

**完成标志**

- 每个审批节点都能解析到合法执行对象或给出明确失败结果
- 转办、候选人控制、催办、超时已受组织与权限约束
- 组织解析失败不会静默丢失，能进入审计、通知和人工处理链路
- 平台路径下的审批执行，不再依赖手工指定审批人才可运行

### 5.4 阶段四：`OA` 桥接与双路径承接能力

**目标**

- 在平台本地审批骨架与组织绑定都成立后，再接入 `OA` 主路径，
  避免桥接先行导致平台内部能力空心化。
- 打通 `OA` 主审批路径与平台承接路径的统一入口、统一摘要和统一回写。

**核心产出**

- 平台向 `OA` 发起审批的桥接能力
- `OA` 回调接收、状态同步、异常补偿与幂等处理能力
- 平台路径与 `OA` 路径的承接策略判断能力
- 面向合同侧的统一审批摘要和统一业务回写口径

**前置依赖**

- 阶段二、阶段三已完成，平台本地流程可独立运行
- `OA` 联调环境、测试流程、调用方式、回调方式、鉴权方式已落实
- 合同主链路已接受统一摘要和统一回写入口，不要求按路径分别适配

**完成标志**

- 默认主审批路径可稳定走 `OA`
- `OA` 无法满足时，平台流程引擎可按既定策略正式承接
- 两条路径下，合同侧读取到的是统一审批摘要而非两套口径
- `OA` 发起失败、回调异常、状态同步失败都进入补偿与审计链路

### 5.5 阶段五：联调、验收与上线准备

**目标**

- 在模块能力完成后，收口跨模块联调、验收样例、发布检查与上线回退准备。
- 把“可开发完成”收口为“可交付上线”。

**核心产出**

- 面向合同主链、组织权限、任务中心、审计中心、通知中心和 `OA` 的联调清单
- 覆盖双路径的验收样例与测试结论
- 上线前配置检查、数据检查、审计检查、告警检查和回退预案
- 模块运行手册、异常处理入口和问题定位入口

**前置依赖**

- 前四个阶段的功能闭环均已达成
- `OA` 侧和平台侧联调窗口可用，关键问题已完成一轮收口
- 测试环境与生产环境部署形态保持一致或足够接近

**完成标志**

- 核心主链、双路径切换、组织绑定、催办超时和异常补偿均完成联调验证
- 验收口径已能证明“默认 `OA` 主路径 + 平台正式承接能力”同时成立
- 上线检查项、回退步骤和问题处置入口已明确
- 模块具备进入正式发布窗口的条件

## 6. 模块推进顺序

本模块按“先定义/版本，再运行时，再组织绑定，再 `OA` 桥接”的顺序推进，
原因如下：

1. 先定义与版本，是为了先固化流程可运行对象。
   没有稳定的定义与版本，后续运行时、联调和回归都没有统一基准。
2. 再做运行时主链，是为了先证明平台流程引擎本身具备正式执行能力。
   如果先做 `OA` 桥接，容易把模块做成外部系统适配层，而不是正式审批能力。
3. 再补组织绑定，是因为节点绑定组织架构是可执行性的硬约束，
   但它必须建立在实例、任务和动作主链已经成立的基础上。
4. 最后接 `OA` 桥接，是因为 `OA` 联调依赖外部条件最多，
   应建立在平台内部骨架、摘要口径、合同回写和异常链路已稳定之后。

这一顺序保证平台流程引擎先成为“能独立成立的正式能力”，
再成为“可与 `OA` 协同的正式能力”。

## 7. 外部联调准备

本模块真正依赖外部条件的联调重点是 `OA` 主路径联调。
其准备项应集中收口，不向无关模块扩散。

### 7.1 `OA` 联调准备

- 确认 `OA` 测试环境、网络连通、调用白名单和回调可达性
- 确认 `OA` 流程模板、流程编码和字段映射口径已冻结到可联调版本
- 确认 `OA` 发起、撤回、审批完成、驳回等关键事件的回调样例可获得
- 准备重复回调、乱序回调、失败重发、人工补发等异常联调样例
- 准备至少一条默认走 `OA` 的真实样例合同流程用于端到端验证

### 7.2 双路径联调准备

- 准备一组 `OA` 主路径样例和一组平台承接样例，
  用于验证统一入口、统一摘要、统一回写是否成立
- 准备路径切换判定样例，验证何时仍走 `OA`、何时切到平台引擎
- 准备合同状态回写与审批摘要对账清单，避免两条路径状态口径分叉

## 8. 测试、验收与上线准备

### 8.1 测试重点

- 定义与版本测试：定义发布、停用、版本切换、非法发布拦截
- 运行时主链测试：实例启动、任务生成、审批动作、并行/会签、转办、终止
- 组织绑定测试：部门绑定、人员绑定、组织规则选人、权限与在岗约束
- 双路径测试：`OA` 主路径、平台承接路径、统一摘要、统一回写
- 异常链路测试：组织解析失败、`OA` 发起失败、回调异常、状态回写失败、
  催办超时失败、异常补偿重试
- 审计与通知测试：关键事件留痕完整性、待办和异常消息触达完整性

### 8.2 验收重点

- 能证明默认主审批路径仍优先走 `OA`
- 能证明平台流程引擎是一期正式能力，且不是仅保留接口壳子
- 能证明每个节点都必须绑定组织架构中的部门、人员或组织规则
- 能证明流程引擎只通过 `contract_id` 绑定合同，不拥有合同主档
- 能证明模块没有生成第二合同真相源，合同状态仍以平台主档为准
- 能证明 `OA` 路径和平台路径在合同侧呈现统一审批摘要和统一结果回写

### 8.3 上线准备

- 配置准备：生产可用流程定义版本、启停状态、承接策略、通知策略
- 数据准备：组织主数据、权限数据、历史定义版本和联调遗留数据清理
- 运行准备：任务扫描、异常补偿、审计落库、告警规则、问题排查入口
- 发布准备：灰度顺序、回退条件、回退步骤、上线观察窗口
- 协同准备：业务方、`OA` 对接方、运维和支持人员明确发布窗口与应急路径
