# CMP 实现阶段统一执行规格

## 1. 规格定位

本文是合同管理平台进入实现阶段时的 Superpowers 执行合同。所有实现任务先以 `docs/technicals/` 的正式技术文档为设计来源，再以本文定义的启动门禁、SubAgent 执行协议、验收闭环与质量门禁落地执行。

本文不替代正式技术文档，不复制技术设计正文；它只把已收口的正式设计转化为可执行、可验证、可审查的实现阶段基准。

## 2. 正式依据来源

### 2.1 项目级依据

- 行为与文档治理依据：[`PRINCIPLE.md`](../../../PRINCIPLE.md)
- 当前项目真相：[`docs/planning/current.md`](../../planning/current.md)
- 决策真相：[`docs/planning/decisions.md`](../../planning/decisions.md)
- 实现批次依据：[`docs/technicals/implementation-batch-plan.md`](../../technicals/implementation-batch-plan.md)

### 2.2 实现设计依据

- 总平台正式实施入口：[`docs/technicals/implementation-plan.md`](../../technicals/implementation-plan.md)
- 第一批底座主线正式计划：[`docs/superpowers/plans/2026-04-26-batch-1-foundations-implementation-plan.md`](../plans/2026-04-26-batch-1-foundations-implementation-plan.md)
- 第一批三条主线的 `Architecture Design`、`API Design`、`Detailed Design`、`Implementation Plan` 及其 `special-designs/` 目录。

### 2.3 第一批主线基准

- `identity-access`：统一身份、组织、角色、菜单权限、功能权限、数据权限、组织规则解析、授权判定、身份审计与解密下载授权边界。
- `agent-os`：围绕 `QueryEngine` 的薄 `Harness Kernel`，以及任务、运行、结果、Prompt 快照、检查点、审计、工具、上下文、记忆、确认、委派与验证治理边界。
- `integration-hub`：统一外部接入、入站、出站、回调、绑定、集成任务、审计、补偿、恢复、对账与原始报文治理边界。

## 3. 启动门禁

任何实现批次正式开工前，必须同时满足以下条件：

- 批次范围与顺序已回到 [`implementation-batch-plan.md`](../../technicals/implementation-batch-plan.md) 的正式分组，不按临时偏好改排期。
- 对应主线已具备正式 `Architecture Design`、`API Design`、`Detailed Design`、`Implementation Plan` 与专项设计输入。
- 当前批次目标、非目标、外部依赖、数据样本、接口冻结点、异常路径和验收口径已经写清楚。
- 开发、测试、联调环境符合 `Docker Compose / 企业内网` 基线，`MySQL` 作为正式真相源，缓存、队列、对象引用、日志与审计能力有可执行入口。
- 统一身份、组织、权限、任务、审计与集成边界不得被业务模块各自复制实现。
- 每个实现任务都有独立验证入口，不能只靠实现者自述完成。

## 4. SubAgent 执行协议

### 4.1 角色边界

- 主 Agent 维护目标、排期、批次真相与验收口径。
- 实现 SubAgent 只执行被派发的局部任务，不改写全局 planning 真相。
- 质量审查 SubAgent 独立复核实现结果、测试证据、文档一致性与风险项。
- 同一个 SubAgent 不同时承担实现与最终质量审查职责。

### 4.2 执行输入

每个 SubAgent 的任务输入必须包含：

- 任务目标与不做内容。
- 必读正式文档链接。
- 允许创建或修改的文件范围。
- 期望完成标志。
- 需要运行的验证命令或验证方式。
- 质量审查要求与交付回执格式。

### 4.3 执行过程

- 先读指定设计依据，再实施；不得凭记忆或经验补写未确认边界。
- 先写可验证任务，再实现最小正确闭环；非必要不扩大抽象层、配置层或兼容层。
- 发现设计冲突时停止扩大实现，返回冲突证据、影响范围与推荐收口方式。
- 任何数据库真相、权限真相、审计真相、外部交换真相只能有一个正式归属。
- 外部系统本体改造不得写入平台内部实现责任。

### 4.4 交付回执

SubAgent 完成后必须回传：

- 已完成文件与关键能力摘要。
- 已运行验证、结果与失败修复记录。
- 对照输入文档的覆盖说明。
- 未覆盖风险、外部依赖与需要主 Agent 决策的事项。

## 5. 验收闭环

每个任务、每条主线、每个批次均按以下闭环验收：

1. 需求与设计对齐：能回溯到正式需求、正式技术文档和批次计划。
2. 实现闭环：代码、配置、数据迁移、运行脚本、测试样本和审计事件形成可运行链路。
3. 异常闭环：失败、超时、重复、权限拒绝、幂等冲突、补偿和人工介入均有可验证处理。
4. 审计闭环：关键动作有 `trace_id`、主体、对象、结果、失败原因和恢复动作留痕。
5. 独立验证：质量审查或验证角色基于证据包给出结论，不接受实现者自证。
6. 回归闭环：失败用例进入回归基线，修复后重新验证。

## 6. 质量门禁

### 6.1 通用门禁

- 运行项目已配置的测试、类型检查、构建或 lint；若缺失，交付回执必须明确说明。
- 新增接口、状态、表结构、事件、配置项必须与正式技术文档命名保持一致。
- 实现不得引入第二套用户、组织、权限、任务、审计、合同、文档、流程或外部交换真相。
- 文档、注释和报告正文默认使用简体中文；代码标识、协议字段、路径和第三方库名保持英文。

### 6.2 第一批门禁

- `identity-access` 先证明统一主体、组织、角色、权限、数据权限与审计链路成立。
- `agent-os` 先证明最小 `QueryEngine / Harness Kernel` 可执行闭环成立，再扩展工具、上下文、记忆、确认、委派和自演进。
- `integration-hub` 先证明统一接入、适配、入站、出站、回调、绑定、任务与审计治理成立，再接单个外部系统细节。
- 三条主线必须共享统一身份、组织、权限、任务、审计和集成边界，不得各自形成重复底座。

## 7. 禁止偏离项

- 禁止绕过 [`implementation-batch-plan.md`](../../technicals/implementation-batch-plan.md) 的第一批范围，提前把第二批业务主链路作为实现前提。
- 禁止把 `agent-os` 漂移为面向开发者写代码的 Coding Agent OS；它必须服务合同、审批、文档、签署、履约、风控、运维审计等企业业务场景。
- 禁止业务模块直连模型、外部系统、权限存储、组织树或审计表。
- 禁止在 `integration-hub` 中维护外部身份协议治理主对象；外部身份进入平台后的主体、绑定预检查、冲突冻结、人工处置与会话准入归 `identity-access`。
- 禁止把 `OA`、企业微信、`CRM`、`SF`、`SRM`、`SAP` 本体改造写成 `CMP` 内部实现责任。
- 禁止只实现成功路径，不实现幂等、失败、补偿、审计、权限拒绝和人工介入路径。
- 禁止在 Superpowers 文档中大段复制正式技术文档正文；Superpowers 文档只保留执行合同、任务拆解、验证与审查基准。

## 8. 完成判定

本规格被正确执行的标志是：每个实现任务都能从正式技术文档进入、从任务清单落地、从测试和审计证据验收、从独立质量审查放行，并且批次边界始终回到统一身份、组织、权限、任务、审计与集成底座这一第一批目标。
