# 最小可运行工程骨架设计

## 背景与目标

本文定义 `CMP` 首个功能点“最小可运行工程骨架”的设计。该功能点位于第一批底座主线实现之前，目标不是实现 `identity-access`、`agent-os`、`integration-hub` 的业务能力，而是先建立后续实现可依赖的工程入口、运行入口、测试入口、质量验证入口和最小健康检查链路。

本设计继承以下已确认口径：后端采用 `Java 21 + Spring Boot + Maven`，前端采用 `React SPA + Tailwind CSS + shadcn/ui` 方向与 `pnpm`，部署与本地编排按 `Docker Compose / 企业内网` 基线收口，整体架构保持模块化单体 / 平台化单体，不在首个功能点中引入微服务拆分。

首个功能点的目标是：

- 建立后端工程、前端工程与本地编排的最小目录和启动方式。
- 提供后端健康检查接口、前端可访问页面和本地反向代理或跨域联调入口。
- 提供单元测试、构建检查、格式 / lint 检查和一键质量验证入口。
- 提供本地 `Docker Compose` 启动链路，至少能编排后端、前端、`MySQL`、`Redis` 及必要的本地网络。
- 为后续第一批底座能力实现提供清晰的模块挂载位置，但不提前实现业务对象。

## 非目标

本功能点明确不做以下内容：

- 不实现身份、组织、角色、菜单权限、功能权限、数据权限、授权判定、审计业务对象。
- 不实现 `AgentTask`、`AgentRun`、`QueryEngine`、工具调用、模型调用或 `Harness Kernel` 运行闭环。
- 不实现入站、出站、回调、绑定、补偿、对账、原始报文治理等 `integration-hub` 业务对象。
- 不实现合同、文档、流程、签章、履约、归档、智能应用等业务模块功能。
- 不设计正式业务表结构和业务迁移脚本；首个功能点只允许存在运行健康检查所需的非业务配置。
- 不引入 `MQ` 优先架构、`gRPC` 主通信协议或微服务拆分。
- 不把示例页面、示例接口包装成业务功能；示例只用于证明工程链路可运行。

## 架构设计

最小骨架按“一个前端应用 + 一个后端应用 + 一组本地基础设施 + 一组验证脚本”组织。

```text
CMP workspace
├── backend/                  Java 21 + Spring Boot + Maven 后端工程
├── frontend/                 React SPA + Tailwind CSS + shadcn/ui 方向前端工程
├── docker-compose.yml        本地编排入口
├── scripts/                  本地验证与质量入口
└── docs/                     已有正式文档与 Superpowers 执行基准
```

后端工程提供最小 `Spring Boot` 应用，暴露 `/actuator/health` 或等价健康检查端点，并预留后续 `foundations`、`modules`、`shared`、`infrastructure` 包边界。首个功能点只要求应用可启动、配置可加载、测试可运行、健康检查可返回稳定结果。

前端工程提供 `React SPA` 最小页面，使用 `pnpm` 作为包管理入口，接入 `Tailwind CSS` 基础配置，并按 `shadcn/ui` 方向保留组件组织位置。首个页面只展示平台骨架、后端健康状态或静态运行状态，不承载业务菜单和权限逻辑。

本地编排通过 `Docker Compose` 统一启动本地依赖，最小包含：

- `backend`：后端应用容器或本地运行映射入口。
- `frontend`：前端开发服务或静态资源服务入口。
- `mysql`：本地开发数据库，作为后续正式业务真相源的环境依赖入口。
- `redis`：本地缓存 / 协调增强层环境依赖入口，不作为业务真相源。

质量验证通过仓库级脚本收口，避免每个开发者或 Agent 自行拼命令。最小入口包括后端测试、前端测试、前端 lint、构建验证和健康检查验证。

## 组件边界

后端边界：

- `backend` 只负责启动 `Spring Boot` 应用、加载环境配置、暴露健康检查、提供后续模块挂载包结构。
- `backend` 可以包含通用异常响应、基础配置绑定和健康检查配置，但不得包含业务域对象。
- `backend` 不直接实现 `identity-access`、`agent-os`、`integration-hub` 的领域模型，只为后续实现提供包和测试入口。

前端边界：

- `frontend` 只负责启动 `React SPA`、加载样式体系、展示最小运行页、验证与后端健康接口的联通。
- `frontend` 可以保留 `components`、`lib`、`routes` 等基础目录，但不得实现业务页面、菜单权限或组织权限逻辑。
- `shadcn/ui` 在本功能点中作为组件方向和结构约束，不要求导入完整业务组件库。

本地编排边界：

- `docker-compose.yml` 只承接本地开发和验证依赖，不表达生产容量规划。
- `MySQL` 与 `Redis` 在本功能点中只证明环境可启动和后续可接入，不承载业务数据设计。
- 本地网络、端口、环境变量命名应保持清晰稳定，为后续批次复用。

验证脚本边界：

- 脚本入口负责串联已有工具命令，不隐藏失败，不吞掉错误输出。
- 任一验证失败都应使脚本以非零状态退出，供 Agent、CI 或人工终端直接判断。

## 数据与状态

首个功能点不引入正式业务数据和业务状态。

允许存在的状态包括：

- 后端应用启动状态。
- 前端应用构建和运行状态。
- `MySQL`、`Redis` 容器健康状态。
- 健康检查端点返回的运行状态。
- 测试、lint、构建、健康检查命令产生的验证结果。

禁止在本功能点中新增以下状态：

- 用户、组织、角色、权限、审计、任务、集成、合同、文档、流程等业务状态。
- 与第一批底座主线等价的临时 mock 真相源。
- 只能在进程内存中保存、后续又被业务依赖的状态。

后续业务状态必须在对应主线正式设计和实现任务中建立，不能通过本骨架中的示例数据延伸出来。

## 错误处理

最小骨架只处理工程级错误，不处理业务级错误。

后端错误处理要求：

- 应用启动失败时，日志需要能定位配置缺失、端口占用、数据库连接失败等工程原因。
- 健康检查失败时，应返回明确的非健康状态，并让本地验证脚本失败。
- 非业务接口异常可以返回统一基础错误结构，但不预设业务错误码体系。

前端错误处理要求：

- 后端健康检查不可达时，页面显示“后端不可用”或等价工程状态，不伪装为业务错误。
- 前端构建、lint、测试失败必须阻断质量验证入口。

本地编排错误处理要求：

- 容器启动失败、端口冲突、健康检查失败应能通过 `docker compose ps`、服务日志和验证脚本定位。
- 不通过重试掩盖确定性配置错误；本功能点只保留最小必要的健康等待机制。

## 测试策略

测试入口必须覆盖“能启动、能构建、能测试、能检查健康链路”。

后端测试：

- 使用 Maven 标准测试入口运行后端单元测试。
- 至少覆盖 `Spring Boot` 应用上下文加载和健康检查端点可用性。
- 如果接入 `Actuator`，需要验证健康端点返回预期状态。

前端测试：

- 使用 `pnpm` 统一执行前端测试和构建。
- 至少覆盖最小页面渲染或基础组件渲染。
- 构建产物应能证明 `React SPA` 与 `Tailwind CSS` 配置可用。

本地编排测试：

- 使用 `docker compose up` 启动本地依赖和应用入口。
- 使用脚本或命令访问后端健康检查端点。
- 如前端页面读取后端健康状态，应验证前后端联通路径可用。

建议的仓库级验证入口：

```bash
scripts/verify-backend.sh
scripts/verify-frontend.sh
scripts/verify-local-stack.sh
scripts/verify-all.sh
```

具体脚本命名可在实现阶段按仓库既有约定收口，但必须形成一个总入口，供后续“编码、质量审查、修复、再次质量审查直到通过”的循环复用。

## 质量审查策略

本功能点完成后，必须进入独立质量审查。质量审查不接受“文件已创建”作为完成证据，只接受可运行、可验证、可复现的证据。

审查范围包括：

- 技术栈是否严格符合 `Java 21 + Spring Boot + Maven` 与 `React SPA + Tailwind CSS + shadcn/ui + pnpm`。
- 是否只建立工程骨架，没有提前实现身份、组织、权限、任务、审计、集成或合同业务对象。
- 本地 `Docker Compose` 是否能启动最小依赖，且端口、网络、环境变量清晰。
- 后端健康检查、前端页面、前后端联通、构建、测试、lint 是否均有可执行入口。
- 验证脚本是否失败即退出，是否适合后续 Agent 和 CI 复用。
- 文档、命名、目录结构是否与正式架构的 `Foundations`、`Modules`、平台共享能力、基础设施层边界一致。

推进方式固定为：

1. 实现 SubAgent 按本文和执行规格完成最小骨架编码。
2. 实现 SubAgent 运行全部验证入口并提交证据。
3. 独立质量审查 SubAgent 对代码、配置、脚本、测试证据和范围边界进行审查。
4. 若存在阻断问题，主 Agent 派发修复任务，不进入下一功能点。
5. 修复后再次由独立质量审查 SubAgent 复审。
6. 只有审查通过后，才允许进入第一批底座能力的后续实现任务。

## 完成判定

本功能点完成必须同时满足：

- 后端工程可用 `Maven` 构建和测试，应用可启动，健康检查端点可访问。
- 前端工程可用 `pnpm` 安装、测试、lint 和构建，最小页面可运行。
- 本地 `Docker Compose` 可启动后端、前端、`MySQL`、`Redis` 或等价最小本地链路。
- 仓库级质量验证入口可一次性执行后端、前端和本地健康检查。
- 没有新增身份、组织、权限、任务、审计、集成、合同、文档、流程等业务对象。
- 后续第一批底座能力能够基于该骨架继续编码，而不需要重建工程结构。
- 独立质量审查结论为通过，或所有阻断问题已修复并复审通过。

## 后续实施计划入口

本设计是进入第一批底座实现前的工程入口。后续实施顺序应为：

1. 先按本文实现“最小可运行工程骨架”。
2. 通过工程骨架质量审查后，再回到 [`CMP 实现阶段统一执行规格`](./2026-04-26-cmp-implementation-execution-spec.md)。
3. 继续执行 [`Batch 1 Foundations Implementation Plan`](../plans/2026-04-26-batch-1-foundations-implementation-plan.md) 中的第一批底座任务。
4. 每个后续任务均按“编码、质量审查、修复、再次质量审查直到通过”的循环推进，不跳过独立审查，不把阻断问题留给下一批。
