### 决策 12：平台审批节点必须绑定组织架构主数据

- 决策：合同管理平台的审批流配置必须与组织架构主数据直接关联；组织架构主数据至少包括部门、人员，且可视化审批流中的每一个节点都必须支持绑定部门、人员或基于组织架构规则选人，不允许仅配置抽象节点类型。
- 原因：用户已明确审批流是基于组织架构运行，若节点不能落到部门、人员或组织架构规则，就无法形成可执行、可承接的真实审批路径。
- 影响：后续需求基线、流程方案、数据模型、组织同步设计、审批配置界面与验收口径，均必须体现“节点和组织架构直接绑定”的约束，不能把审批节点设计成脱离组织主数据的纯类型占位。

### 决策 13：数据库选型按 MySQL 收口

- 决策：当前项目的正式数据库口径按 `MySQL` 收口；`PostgreSQL`、`Pg` 等表述不再作为当前项目最终交付的候选数据库写入 planning 真相。但架构设计不得强绑定某一个具体数据库，必须保留一层数据库抽象层。
- 原因：正式招标文件 `4.1` 仅给出一般性数据库兼容条款，而 `4.2` 更具体地明确写明“选用MySQL数据库”；当前项目应按“具体技术条款优先于一般兼容条款”的判断原则执行。
- 影响：后续需求基线、技术方案、实施计划、接口与数据设计、验收口径及相关说明文档中，数据库选型均应统一表述为 `MySQL`，不得再以 `PostgreSQL`/`Pg` 作为最终交付候选项并列保留；同时仓储层、SQL 访问层或数据访问接口必须保留数据库抽象边界，避免业务层与某一数据库实现细节直接耦合。

### 决策 14：后端技术栈继续沿用当前既定口径

- 决策：后端技术栈不做本轮调整，继续按当前项目已经确认的后端技术口径推进。

- 影响：后续实施计划、技术草稿与任务拆解应默认继承现有后端方案，不再把后端框架或基础设施作为待定项反复展开。

### 决策 15：前端技术栈统一为 React SPA + Tailwind CSS + shadcn/ui

- 决策：前端正式技术栈统一为 `React (SPA) + Tailwind CSS + shadcn/ui`。

- 影响：后续前端架构、页面实现、组件规范、联调方式和实施计划，均应以 `React (SPA)` 作为应用形态，以 `Tailwind CSS` 作为样式体系，以 `shadcn/ui` 作为基础组件方案。

### 决策 16：AI 能力必须保留模型抽象层

- 决策：项目中的 AI 能力不能绑定到某一个具体模型或模型供应商，必须保留一层 AI/模型抽象层。
- 原因：用户已明确 AI 能力需要具备模型可替换性，避免当前阶段的具体模型选择反向固化整体架构。
- 影响：后续 AI 接入设计、提示词编排、任务路由、配置管理与调用接口应基于统一抽象边界展开，业务侧不得直接依赖单一模型 SDK、模型名称或厂商特性作为唯一实现前提。

### 决策 17：部署环境按 Docker Compose / 企业内网收口

- 决策：项目部署环境按 `Docker Compose / 企业内网` 收口。

- 影响：后续部署方案、环境依赖、网络假设、集成方式、运维脚本与交付说明，均应以企业内网部署和 `Docker Compose` 编排为基线，不应默认依赖公网 SaaS、托管数据库或外部云原生平台能力。

### 决策 18：测试账号与加密软件平台语义按当前澄清收口

- 决策：只有企业微信对接需要外部测试账号；加密软件、电子签章属于平台内自研子模块，不作为外部测试账号依赖项。加密软件的正式平台语义统一为：文件进入文档中心后自动加密，默认仅可在平台内按权限解密使用；管理端授权模块必须支持按部门、人员授予“解密下载”权限，获授权对象执行解密下载后导出的明文文件可脱离 `CMP` 使用，且解密下载行为必须纳入权限控制与审计留痕。
- 原因：用户补充澄清了管理端授权解密下载这一例外机制，需要在保留“加密软件仍是文档中心上的自研加密子模块”这一主语义前提下，消除“平台外绝对无法解密”的错误绝对化表述。
- 影响：后续范围说明、测试准备、架构设计、实施计划与需求文档中，不得再把加密软件主语义写成“外部对接对象”或“外部依赖系统”，也不得再使用“平台外绝对不可解密/不可使用”这类绝对化表述；必须统一写明“默认不可脱离平台使用，管理端授权解密下载是受控例外，且授权对象至少覆盖部门、人员”。

### 决策 24：`agent-os` 正式设计按 QueryEngine 驱动的薄 Harness 内核重构

- 决策：`agent-os` 正式设计不再停留在 Claude Code / Harness Engineering 最佳实践的术语补充层面，而是按 `QueryEngine` 驱动的薄 Harness 内核执行结构性设计手术。主文档先建立 `Harness Kernel`、`Tool & Execution Plane`、`Context Plane`、`Governance Plane`、`Evolution Plane` 的分层主线，再由专项设计分别承接 Prompt / Context / Memory / Tool / Provider / Human Gate / Verification / Auto Dream / Persona / Delegation / Ops 等外围能力。
- 原因：此前评估显示 `agent-os` 已具备企业级审计、权限、配额、运维、人工确认和多 Agent 委派等优势，但主线仍容易退化为能力清单式聚合；若不把 `QueryEngine` 作为运行时心脏，后续实现阶段会出现 Prompt、工具、上下文、记忆、安全、验证、自演进各自为政的问题。
- 影响：本次手术按 `docs/superpowers/plans/2026-04-26-agent-os-harness-surgery.md` 执行；目标设计文档由专职 SubAgent 修改，独立 QA SubAgent 复审。完成前，`agent-os` 的实现准备应先以该手术收口结果为依据，避免直接按旧能力清单口径进入实现。所有相关设计必须保持企业业务系统定位，面向合同管理、审批、文档、签署、履约、风控、运维审计等业务场景吸收 Harness Engineering，不得漂移为面向开发者写代码的 Coding Agent OS。

### 决策 26：实现阶段以 Superpowers 执行规格与五批计划作为开发执行基准

- 决策：在 `docs/technicals/` 正式技术文档与 `docs/technicals/implementation-batch-plan.md` 已完成的前提下，新增 1 份 Superpowers 执行规格与 5 份批次执行计划，作为进入实现阶段时的开发执行基准。统一执行规格为 `docs/superpowers/specs/102-cmp-implementation-execution-spec.md`；五批计划分别对应底座主线、合同核心主链路、依赖主链路真相源的业务能力、智能与增强能力、综合联调验收与上线准备。
- 原因：`docs/technicals/` 负责正式技术设计与排期矩阵，但实现阶段仍需要面向 Agent 执行的任务拆分、SubAgent 协作协议、启动门禁、验证闭环与质量审查要求；直接让实现 Agent 只读技术文档会造成执行边界不稳定。
- 影响：后续开发执行时，应先读取 Superpowers 执行规格，再按批次读取对应计划；第一批正式入口为 `docs/superpowers/plans/102-01-batch-1-foundations-implementation-plan.md`。Superpowers 文档不得替代 `docs/technicals/` 正式技术基线，只承接“如何执行、如何派发、如何验证”。

### 决策 27：编码阶段固定采用测试驱动开发与独立质量审查闭环

- 决策：自首个可编码功能点起，编码阶段固定采用测试驱动开发范式；每完成一个功能点立即进入独立质量审查，若审查发现问题则先修复并再次审查，直到明确通过后才进入下一个功能点。
- 原因：用户已明确要求编码、质量审查、修复、再次质量审查的循环必须持续到全绿，且质量审查结论必须完全基于事实表达。
- 影响：主 Agent 只负责协调和回写真相文档；实现与质量审查必须由相互独立的子代理执行。质量审查有问题必须明确列出问题点，没有问题必须明确报告通过，不得使用含糊或防御性口吻替代事实结论。
